8.1 Mathematical Principles of Autoregressive Generation

After training in Chapter 7, we obtain a model capable of predicting the next token. However, the model itself is merely a "scorer": given a prefix, it outputs a probability distribution. How to transform this probability distribution into an actual text sequence is the core problem of text generation.

Autoregressive Decomposition

The mathematical foundation of generation remains autoregressive decomposition — the joint probability of a sequence is decomposed into the product of conditional probabilities:

Equation 8.1 Autoregressive Generation

At each step, the model, based on the existing prefix , outputs a probability distribution over the vocabulary . Then, is obtained by sampling from this distribution or taking the maximum. This process proceeds token by token until an end token is generated or the maximum length is reached.

Two Phases: Prefilling and Decoding

The generation process is divided into two phases with completely different computational characteristics:

  • Prefilling: The user's input prompt is fed into the model all at once. This is a compute-intensive phase — a large number of matrix multiplications are executed in parallel. The bottleneck is GPU compute power (FLOPS). Time to First Token (TTFT) is dominated by this phase.
  • Decoding: Tokens are generated one by one. Each step inputs only one token, using the KV Cache for incremental computation. This is typically a bandwidth-intensive phase — with small batches or low concurrency, GPU compute utilization can be significantly lower than during prefill. Subsequent throughput (TPS/TPOT) is dominated by this phase.
🔑
Important Concept: TTFT (Time to First Token) is primarily affected by prefill, queuing, network, and routing; the user perceives it as "the time to wait for the first character to appear." TPS/TPOT is primarily affected by decoding, batching, KV cache, and bandwidth; the user perceives it as "typing speed." Numerical values vary greatly across different models, context lengths, concurrency levels, and service providers. A reproducible generation benchmark should report these metrics under a fixed model, prompt set, context length, concurrency level, and serving configuration.

8.2 Why Greedy Decoding Fails — Repetition Traps and Exposure Bias

Greedy Decoding selects the token with the highest probability at each step: . It is the simplest strategy, but it has two fundamental flaws.

Flaw 1: Repetition Loop

Once the model enters a high-probability local pattern, later argmax steps may keep selecting similar or identical tokens. This is especially noticeable near common phrases, punctuation, or function words in the training data. The cause is usually a combination of factors: an overly sharp conditional distribution, repetitive patterns already present in the context, no random exploration, and weak stopping constraints. These factors can trap greedy decoding in a repetition loop; it should not be reduced to a single Key/Query self-reinforcement story.

Flaw 2: Exposure Bias

During training, the model sees the true ground truth token at each step. However, during inference, the model sees the self-generated token. The distributions in these two scenarios differ — the conditional distribution during training is , while during inference it is . An error at one step gets amplified in subsequent steps; this is error accumulation.

Entropy Control Perspective of the Temperature Parameter

A common remedy is to introduce randomness: sample from the probability distribution instead of always taking the maximum. The temperature parameter controls the degree of this randomness:

Equation 8.2 Temperature Sampling
  • : Degrades to argmax (completely deterministic)
  • : Original distribution
  • : Degrades to a uniform distribution (completely random)

"Thermodynamic Analogy" of Temperature: In the Boltzmann distribution, temperature controls the system's degree of "exploration" — high temperature makes the system explore more states (more diverse vocabulary), while low temperature makes the system exploit known high-probability patterns (more deterministic output).

Exercise 1: Greedy Decoding

Task: Implement the generate_greedy(model, input_ids, max_new_tokens) function.

Requirements:

  • Input input_ids: (1, seq_len), generate up to max_new_tokens new tokens one by one

# Possible output: "Deep learning is deep learning is deep learning is..."
def generate_greedy(model, input_ids, max_new_tokens=100): """Greedy decoding: select the token with the highest probability at each step""" model.eval() device = next(model.parameters()).device input_ids = input_ids.to(device) for _ in range(max_new_tokens): with torch.no_grad(): logits = model(input_ids) # Take the logits at the last position next_logits = logits[:, -1, :] # (1, vocab_size) next_token = next_logits.argmax(dim=-1, keepdim=True) # (1, 1) input_ids = torch.cat([input_ids, next_token], dim=-1) if next_token.item() == model.tokenizer.eos_token_id: break return input_ids

💡 Observation: Even on small models, greedy decoding is extremely prone to falling into repetition. Note the -1 in logits[:, -1, :] — we are taking the logits at the "last position", which is the model's prediction for the next token.

Temperature sampling controls the "sharpness" of the distribution by applying a division to the logits. The core change is: argmax → softmax + multinomial.

Exercise 2: Temperature Sampling

Requirements:

  • Use torch.multinomial to sample from the distribution
  • Handle the special case of : degrade to argmax (avoid division by zero)
  • Running multiple times with the same prompt should yield different results (due to random sampling)

prompt = tokenizer.encode("The future of artificial intelligence", return_tensors='pt')

def generate_temperature(model, input_ids, max_new_tokens=100, temperature=1.0): """Temperature sampling: use temperature to control the sharpness of the distribution""" model.eval() device = next(model.parameters()).device input_ids = input_ids.to(device) for _ in range(max_new_tokens): with torch.no_grad(): logits = model(input_ids) next_logits = logits[:, -1, :] # (1, vocab_size) # T=0 degrades to greedy if temperature == 0: next_token = next_logits.argmax(dim=-1, keepdim=True) else: scaled = next_logits / temperature probs = torch.softmax(scaled, dim=-1) next_token = torch.multinomial(probs, num_samples=1) input_ids = torch.cat([input_ids, next_token], dim=-1) if next_token.item() == model.tokenizer.eos_token_id: break return input_ids

💡 Handling T=0: When T→0, can cause numerical overflow. In practice, we never divide by true 0 — use argmax directly when T=0, and only perform sampling when T>0.

8.5 From Temperature to Top-K / Top-P: Truncating the Low-Value Tail

Temperature sampling helps, but a large vocabulary still leaves a long softmax tail of extremely low-probability tokens. Even with , the model can still sample low-quality or off-topic tokens. Top-K and Top-P reduce this risk by truncating the low-value tail.

The Logic of Top-K

Keep only the K tokens with the highest probability, re-normalize them, and sample from this set. K is a fixed value (GPT-2 default K=40).

Inherent Problem with Top-K

K is fixed. In some contexts, there might only be 3 reasonable choices (K=40 introduces too much noise), while in other contexts, there might be 100 reasonable continuations (K=40 truncates too many possibilities). For example:

  • "The capital of the United States is ____" — The distribution is extremely sharp; the probability of "Washington" might be over 90%. With K=40, you still sample from 40 tokens, with a small chance of selecting "New York" (clearly wrong)
  • "Write her a poem about ____" — The distribution is very flat; K=40 might be too restrictive for creativity

The Adaptive Solution of Top-P (Nucleus)

Top-P (Holtzman et al., 2020) does not fix K. Instead, it selects the smallest set of tokens whose cumulative probability reaches P:

  1. Sort tokens by probability from highest to lowest
  2. Start accumulating from the highest probability until the cumulative probability (typical value 0.9 or 0.95)
  3. Sample only from this "nucleus"

When the distribution is sharp, the nucleus is small (the "Washington" example might only have 1-3 tokens); when the distribution is flat, the nucleus automatically expands. It does not require manually fixing K, and it is one of the common sampling strategies in open-ended generation; practical deployment still involves tuning parameters in conjunction with temperature, repetition penalty, stop conditions, and task constraints.

Worked Example: Candidate Sets for Top-K vs. Top-P on the Same Distribution

Assume that at a certain generation step, there are only 6 candidate tokens, with logits [4, 3, 2, 1, 0, -1]. With temperature T=1, the probabilities after softmax are approximately:

tokenABCDEF
logit43210-1
probability0.6340.2330.0860.0320.0120.004
cumulative0.6340.8670.9530.9850.9961.000

top-k=3 would always retain A,B,C, then re-normalize them to approximately [0.718, 0.264, 0.097]. top-p=0.9 would start accumulating from the highest probability until the cumulative probability exceeds 0.9; here A+B=0.867 is not enough, adding C brings it to 0.953, so the candidate set is also A,B,C. If changed to top-p=0.8, the candidate set would only need A,B.

The key difference is not this singleton, but the behavior when the distribution shape changes. If the distribution is very sharp, e.g., [0.92, 0.04, 0.02, 0.01, 0.01], top-p=0.9 only retains the highest probability token; if the distribution is flat, e.g., [0.22, 0.20, 0.18, 0.16, 0.14, 0.10], top-p=0.9 needs to retain the first 5 tokens. A fixed top-k does not automatically adjust the candidate set size based on contextual uncertainty.

8.5A Sampling Distribution Ledger: From raw logits to final token

Real generation systems do not directly apply softmax to raw logits. A generation step typically first applies multiple transformations: repetition penalty, valid token constraints, temperature, top-k/top-p truncation, and only then renormalizes and samples. Writing the order incorrectly changes the final distribution, and also renders logprob, entropy, and tuning conclusions meaningless.

1. Recommended Processing Order

A reproducible generation policy should use a clearly declared processing order. Real systems can have more processors, but each step should be explicitly written into the generation policy:

raw logits -> repetition penalty / frequency penalty -> token constraints or grammar mask -> temperature scaling -> top-k or top-p filtering -> softmax over remaining tokens -> sample or argmax

The repetition penalty should be applied before truncation; otherwise, tokens that have already appeared might enter the top-k/top-p candidate set first, and the penalty won't take effect in time. The constraint mask should be applied before normalization, setting the logits of illegal tokens to -inf; the subsequent softmax only distributes probability among valid candidates. Temperature changes the sharpness of the distribution, while top-k/top-p changes the candidate set; they are not the same type of operation.

2. Worked Example: Processing trajectory of logits for the same step

Assume the current vocab has only 5 tokens, and the raw logits are:

token: A B C D E raw logits: 4.0 3.0 2.0 1.0 -1.0

A has appeared in history, use repetition penalty=2.0; the grammar currently only allows A/B/C; temperature=1.0; top-p=0.8. The processing trajectory can be written as:

StageABCDEDescription
raw logits4.03.02.01.0-1.0Model raw output
repetition penalty2.03.02.01.0-1.0A has positive logit, divide by penalty
constraint mask2.03.02.0-inf-infD/E currently illegal
top-p=0.82.03.02.0-inf-infB has highest probability, adding A or C reaches threshold

The final probability is not the softmax of the raw logits, but the softmax of the processed logits over the retained candidates. If top-p retains B, A in case of tied probabilities, the candidate probabilities are approximately B=0.731, A=0.269; if the implementation retains B, C, the probabilities are isomorphic but the candidate tokens differ. Therefore, stable tie-breaking, token id order, and reporting the candidate set are all important.

3. logprob, entropy, and candidate set must be bound to the same distribution

Generation APIs often return logprobs, a list of top tokens, or entropy. It must be stated which distribution these quantities come from:

  • raw model logprob: Unconstrained and untruncated, suitable for comparing the model's own preference for tokens.
  • processed logprob: Probability after penalty, mask, temperature, and truncation, suitable for explaining the actual sampling in this instance.
  • entropy: If calculated after truncation, it only represents the uncertainty within the candidate set and cannot be directly compared with the raw full-vocab entropy.
  • selected token: The token id, text snippet, processed probability, and whether it was affected by constraints should all be reported.

If a system claims "the model is very confident in this answer," but cites the probability after top-p truncation and JSON grammar mask, this statement is not rigorous. A high probability in a constrained distribution only indicates "it is the most likely within the valid set," which is not equivalent to the model being highly confident over the open vocabulary.

4. What to look at when tuning parameters

When tuning temperature, top-p, and repetition penalty, do not judge from a few generated texts. At a minimum, record the following together:

kept_token_count / vocab_size selected_token_probability entropy_after_filtering repetition rate or distinct-n stop reason: eos / stop string / max_new_tokens / constraint failure generated tokens and latency

These quantities help distinguish several types of problems: boring output might be due to too low temperature or too small a candidate set; off-topic output might be due to too high temperature or too wide top-p; repetition might require repetition penalty, stop conditions, or training data processing; format errors are better suited for constrained decoding rather than continuing to lower the temperature.

🔑
Section Takeaway: Generation is not as simple as "sample after softmax." A rigorous generation experiment must report the transformation chain from raw logits to the processed distribution; otherwise, the experiment cannot be reproduced, and it is impossible to determine whether quality changes come from the model, the sampling strategy, or the constraint system.

8.6 Programming Exercise 3-4: Implementing Top-K and Top-P Sampling

Exercise 3: Top-K Sampling

Task: Implement generate_topk(model, input_ids, max_new_tokens, k, temperature).

Requirements:

  • Use torch.topk to take the K logits with the highest probability
  • Perform softmax normalization on the Top-K subset
  • After sampling, use gather to map indices back to the original vocabulary IDs
# Expected usage
prompt = tokenizer.encode("Tell me a story", return_tensors='pt')
output = generate_topk(model, prompt, 100, k=40, temperature=0.8)
def generate_topk(model, input_ids, max_new_tokens=100, k=40, temperature=1.0): """Top-K sampling: only retain the K tokens with the highest probability""" model.eval() device = next(model.parameters()).device input_ids = input_ids.to(device) for _ in range(max_new_tokens): with torch.no_grad(): logits = model(input_ids) next_logits = logits[:, -1, :] / temperature # take top-k top_logits, top_indices = torch.topk(next_logits, k, dim=-1) # sample on the top-k subset top_probs = torch.softmax(top_logits, dim=-1) sampled_idx = torch.multinomial(top_probs, num_samples=1) # map back to original vocabulary IDs next_token = top_indices.gather(-1, sampled_idx) input_ids = torch.cat([input_ids, next_token], dim=-1) if next_token.item() == model.tokenizer.eos_token_id: break return input_ids

Exercise 4: Top-P (Nucleus) Sampling

Task: Implement generate_topp(model, input_ids, max_new_tokens, p, temperature).

Requirements:

  • Sort the probability distribution in descending order, calculate cumulative probability
  • Key: First use cumsum >= p to mark positions that exceed the nucleus, then shift the mask right by one position — ensuring the first token that reaches or exceeds p is retained, and the nucleus is as small as possible
  • After masking, renormalize the remaining probabilities (ensuring they sum to 1)
  • Sample from the normalized distribution
# Expected usage
prompt = tokenizer.encode("Tell me a story", return_tensors='pt')
output = generate_topp(model, prompt, 100, p=0.9, temperature=0.8)
def generate_topp(model, input_ids, max_new_tokens=100, p=0.9, temperature=1.0): """Top-P (Nucleus) sampling: select the smallest token set whose cumulative probability reaches p""" model.eval() device = next(model.parameters()).device input_ids = input_ids.to(device) for _ in range(max_new_tokens): with torch.no_grad(): logits = model(input_ids) next_logits = logits[:, -1, :] / temperature probs = torch.softmax(next_logits, dim=-1) # sort in descending order sorted_probs, sorted_indices = torch.sort(probs, descending=True, dim=-1) # cumulative probability, mask the tail cumsum = torch.cumsum(sorted_probs, dim=-1) mask = cumsum >= p mask[..., 1:] = mask[..., :-1].clone() mask[..., 0] = False sorted_probs = sorted_probs.masked_fill(mask, 0.0) # renormalize renormalized = sorted_probs / sorted_probs.sum(dim=-1, keepdim=True) # sample sampled_idx = torch.multinomial(renormalized, num_samples=1) next_token = sorted_indices.gather(-1, sampled_idx) input_ids = torch.cat([input_ids, next_token], dim=-1) if next_token.item() == model.tokenizer.eos_token_id: break return input_ids

💡 Why shift the mask right? cumsum >= p marks the first token that reaches the threshold; after shifting the mask right by one position, this token is retained, and the tail tokens after it are removed. This yields the smallest nucleus whose cumulative probability reaches p.

8.7A How to Evaluate Generation Strategies: Diversity is Not Quality, Low Repetition is Not Correctness

When comparing decoding strategies, the most common mistake is to look at only one or two examples, or to report only distinct-n. Generation quality is a multi-objective problem: the same strategy might increase diversity while reducing factuality, format stability, or reproducibility. Rigorous experiments need to fix the task type, sampling parameters, random seed, output length, and evaluation metrics.

What Questions Do the Metrics Answer?

Metric/MethodMain Question AnsweredWhat It Does Not ExplainApplicable Scenario
distinct-1 / distinct-2Output n-gram diversity and repetition degreeFactual correctness, logical coherence, or task completionOpen-ended generation, degenerate repetition analysis
Repetition Rate / self-BLEUWhether the model repeats within or across samplesWhether high repetition is necessarily wrongLong text, story, dialogue generation
exact match / F1 / pass@kWhether tasks with clear answers are completedStyle and preference in open-ended textQA, extraction, code, short math answers
Human Preference or Pairwise RankingWhich output humans preferWhether the reason for preference is factuality, style, or lengthDialogue, summarization, creative writing
Latency and Cost MetricsTTFT, TPOT, tokens/s, cost per 1M tokensSemantic qualityService deployment and strategy tuning

Why Perform a Parameter Sweep

Temperature, top-k, and top-p are not isolated switches. Higher temperature usually increases entropy; lower top-p truncates the tail; their combination can cancel out or amplify each other. A reasonable experiment should compare multiple parameter sets on the same prompt set, for example:

temperature in {0.2, 0.7, 1.0} top_p in {0.8, 0.9, 0.95} max_tokens fixed seeds at least 3 metrics distinct-2, repetition rate, task score, latency

If conclusions are drawn from only one prompt, one seed, and one sample output, it is usually insufficient to determine strategy superiority. For open-ended generation, failure cases should also be saved: repetitive loops, early stopping, going off-topic, factual errors, format corruption, and excessive conservatism are all different failure modes.

The Role of Repetition Penalty

Repetition penalty is not a new probability distribution nor a training objective; it is a processor applied to logits before softmax and sampling. The common rule is: for tokens already present in the prefix, if the logit is positive, divide by the penalty; if the logit is negative, multiply by the penalty, thereby relatively decreasing the probability of these tokens. A correct implementation should apply this transformation before greedy, top-k, or top-p selection, then verify that it actually changes the candidate ranking.

Worked Example: From Logits Processor to Final Sampling Distribution

Given logits [4.0, 3.0, -1.0], generated token ids are [0,2], repetition penalty is 2.0. First, process the repeated tokens: the positive logit of token 0 changes from 4.0 to 2.0, the negative logit of token 2 changes from -1.0 to -2.0, resulting in [2.0,3.0,-2.0]. If top_p=0.8 is then used, after softmax, the probability of token 1 is approximately 0.721, which does not yet reach 0.8; adding token 0 brings the cumulative sum to approximately 0.986, so the nucleus is tokens {1,0}, and token 2 is truncated.

Before final sampling, renormalize on the retained set: the probabilities of token 0 and token 1 are approximately [0.269, 0.731], entropy is approximately 0.582, and the highest probability token is 1. A useful distribution ledger records the processed logits, retained candidates, normalized probabilities, entropy, and current highest-probability token; this turns the statement "sampling parameters change the output" into a checkable probability trace.

Practical Judgment for Strategy Selection

  • Code, Math, Extraction: Prefer low temperature or beam / pass@k; diversity is not the primary goal, correctness and reproducibility are more important.
  • Creative Writing, Brainstorming: Temperature/top-p can be increased, but repetition, topic drift, and safety boundaries must be monitored.
  • Structured Output: Sampling strategies cannot replace schema constraints; prioritize structured decoding or output validation.
  • Online Services: Need to consider both quality and TPOT/throughput; high top-p may increase output length and post-processing failure rate.
🔑
How to Write Experimental Conclusions: Do not write "top-p is best." Instead, write "Under a specific prompt set, length limit, temperature, top-p, random seed, and metric, top-p=0.9 reduces repetition rate compared to greedy but increases output variance; human inspection shows some samples are more divergent."

8.7B Beam Search: Searching Multiple High-Probability Candidates Instead of Taking Only the Single Best Step

Greedy decoding only retains the highest probability token at each step, easily falling into a local optimum early on. Beam search retains num_beams candidate paths. At each step, it expands each path to several next tokens, then sorts by cumulative log probability and retains the top few.

Length Bias

Directly accumulating log probability naturally biases towards shorter sequences because each additional generated token adds another non-positive log probability. In machine translation, summarization, and structured candidate generation, length normalization or length penalty is commonly used to mitigate this issue:

Length-normalized Beam Score

The concepts behind length_normalized_score and beam_search correspond to this process. A compact batch-size-1 implementation is enough to expose the relationship between the beam table, raw score, and normalized score before introducing production batching and early-stop policies.

Worked Example: Two-Step Beam Search with Length Normalization

Let num_beams=2. In the first step, two candidates are expanded from the prompt: the log probability of A is -0.2, and the log probability of EOS is -0.4. If early stopping is allowed, EOS is a complete short answer of length 1; A needs to continue generating.

StepCandidateNew logpCumulative logpLengthalpha=1 Normalized Score
1EOS-0.4-0.41-0.400
1A-0.2-0.21-0.200
2A B-0.3-0.52-0.250
2A C-1.0-1.22-0.600

If only cumulative logp is considered, the short answer EOS with -0.4 is higher than A B with -0.5, so the beam might favor overly short outputs. Using s(y)=logp/|y|, the score for A B is -0.25, which is better than EOS's -0.4. This is the role of length penalty: it does not unconditionally reward long sentences, but reduces the short-sentence bias caused by "adding another negative number for each additional token."

When to Use Beam

  • Suitable for: Tasks with clear objectives like translation, extraction, code candidates, short math answers, schema-constrained output.
  • Not suitable for blind use: In open-ended chat and creative writing, beam may produce high-probability but mediocre, repetitive, or overly short outputs.
  • Must report: Beam size, length penalty, maximum length, whether early stopping is used, and candidate ranking metrics.
🔑
Boundary: Beam search is a search strategy, not a training objective, nor a guarantee of factuality. It can explore high-probability regions more systematically, but if the model's probability inherently favors incorrect answers, beam search will only find high-probability errors more reliably.

8.7C Reasoning Generation: Trading More Test-Time Compute for Higher Accuracy

For mathematics, code, planning, and multi-hop QA, the generation strategy is not just about "which token to choose." The model may need to generate intermediate reasoning first, then select the final answer. Recent reasoning methods can be unified under one question: after training is complete, are you willing to use more test-time compute in exchange for a higher task success rate?

Common Methods

MethodApproachSource of GainCost/Risk
Chain-of-ThoughtLet the model write intermediate steps before giving the answerUnfolds implicit computation into a token sequenceSteps may seem plausible but be factually incorrect
Self-consistencySample multiple reasoning paths, vote on the final answerReduces chance errors from a single pathCost scales roughly with the number of samples; voting only works for aggregatable answers
Best-of-NGenerate N candidates, select using a reward model or ruleTurns generation into search and re-rankingReranker bias may select answers with good style but factual errors
Verifier / CriticTrain or call a separate model to judge if the solution is correctSeparates "solving" from "judging"Insufficient verifier coverage may let similar errors pass
Process supervisionEvaluate each reasoning step instead of just the final answerEasier to locate intermediate errorsHigh annotation cost; step granularity is hard to define

Why It Differs from Ordinary Sampling

Ordinary top-p or temperature mainly controls the entropy of the output distribution; reasoning generation is concerned with the search space. The same problem can generate multiple reasoning paths, each with its own intermediate states and final answer. For tasks with a standard answer, the evaluation focus is not on distinct-n, but on pass@k, majority vote accuracy, verifier accuracy, average output token count, and cost per correct answer. A reasoning evaluation should use 1 - C(n-c,k)/C(n,k) to estimate the probability of drawing at least one correct answer from k samples out of n; it should also define a final-answer extractor, aggregate multiple candidate outputs into a vote distribution, and report both the total generated tokens and the average token count.

Worked Example: pass@k is not c/n

Suppose for the same coding problem, we sample n=10 candidates, of which c=2 pass the tests. If we look at the single-sample success rate, it is c/n = 0.2. But pass@3 asks: if we randomly draw 3 candidates from these 10 to submit, and we succeed if at least one is correct. The failure event is that all 3 drawn are incorrect candidates:

pass@3 for n=10, c=2

So for the same set of candidates, pass@1 is 0.2, and pass@3 is approximately 0.533. This shows that pass@k measures the "probability of hitting at least once given a budget of k attempts," not the average accuracy of a single sample. When reporting reasoning experiments, you must clearly state n, k, sampling parameters, and the average number of generated tokens; otherwise, an accuracy improvement might simply be due to spending more test-time compute.

There is a second boundary: final-answer pass@k does not prove that the reasoning path is correct. A candidate can pass a unit test, match a numeric answer, or be selected by majority vote while containing an invalid chain of thought. Recent RLVR evaluation work therefore argues for checking both the final answer and the reasoning trace when the task requires trustworthy reasoning. In course reports, separate answer correctness, trace validity, and cost per correct-and-valid answer; do not use a higher pass@k alone as evidence that the model has learned better reasoning.

How to Report the Compute Budget

If a method improves accuracy from 60% to 72%, but goes from 1 generation per problem to 32 generations, you cannot just report the accuracy. You should at least report:

  • Number of samples: N, temperature, top-p, max reasoning tokens.
  • Accuracy: single sample, pass@k, majority vote, or reranked top-1.
  • Cost: average generated tokens, TTFT/TPOT, latency per problem, and token cost per problem.
  • Failure types: arithmetic errors, missing conditions, invalid code, incorrect votes, verifier misjudgments.
🔑
Engineering Takeaway: Reasoning ability is not "outputting a longer explanation." Rigorous experiments must prove that more intermediate tokens or more candidates indeed improve the task success rate, and must state how much test-time compute was consumed.

8.7D Thinking Mode, Thinking Budget, and Runtime Routing

Recent reasoning models have made "how much thinking to spend" an inference-time control variable rather than a fixed property of a model. The Qwen3 technical report describes a unified framework with both thinking mode and non-thinking mode, plus a thinking budget mechanism that allocates more or fewer generated reasoning tokens according to task complexity. For engineers, this should be treated as a runtime policy problem: the model, prompt template, max reasoning tokens, candidate count, verifier, latency target, and fallback route jointly define the user-visible behavior.

🔑
Engineering Judgment: Thinking mode is not a magic capability label. It is a controlled allocation of generation tokens and verification work. Senior inference engineers should be able to decide when to spend those tokens, how to stop, and how to prove that the additional compute bought real task success.

8.7E Reasoning Controller: Effort, Hidden Tokens, Incomplete Outputs, and Tool Continuation

Once reasoning becomes a request-level control, the generation system needs a controller rather than a fixed decoding recipe. OpenAI's reasoning model documentation makes this concrete: reasoning.effort trades speed and token usage against deeper reasoning, hidden reasoning tokens occupy context and are billed as output tokens, and a response can become incomplete if the token cap is reached before visible output is produced. Qwen3's thinking-budget design points to the same engineering pattern from the open-model side: the model may support multiple reasoning modes, but the product must decide when each mode is worth its cost.

Figure 8-3: Reasoning controller lifecycle. Effort selection, hidden-token budget, incomplete-response handling, tool continuation, and eval feedback are part of one serving policy.

1. Effort levels are an evaluation knob, not a quality guarantee

Increasing effort is useful only when representative evals show a measurable improvement that justifies latency and cost. Low effort can be better for short tool plans, extraction, and customer-service routing; medium effort is often a balanced starting point; high or xhigh effort should be reserved for hard reasoning, long-horizon research, debugging, security review, or asynchronous agentic workflows. If the prompt has unclear success criteria, weak stopping rules, or unconstrained tool access, more reasoning can create overthinking, unnecessary search, and output regressions.

Medium effort baselineGeneral production workloads with mixed planning and ordinary answer generationWasting tokens on tasks that do not need reasoningPer-slice quality/cost table, not only a global average
Higher effortComplex debugging, math/code reasoning, high-value analysis, long-horizon agent tasksOverthinking, tool loops, context exhaustion, higher queue latencyClear lift on hard slices after controlling for output tokens and retries

2. Hidden reasoning tokens are still capacity and cost

Reasoning tokens may not be visible in the final answer, but they still consume output budget and context space. This creates two engineering traps. First, max_output_tokens must cover both hidden reasoning and visible answer tokens; otherwise, the model can spend the entire budget thinking and return an incomplete response. Second, long prompts, retrieved context, tool traces, and hidden reasoning compete for the same context window, so a long-context RAG request may need a lower reasoning budget than a short math problem.

3. A production reasoning controller needs fallbacks

A robust controller should define what happens when the run is incomplete, the verifier rejects the answer, a tool call times out, or the token budget is exhausted. Typical fallbacks include asking for clarification, returning a partial answer with uncertainty, retrying once with a higher effort or smaller context, routing to a specialized model, or deferring to an asynchronous workflow. The important point is that fallback rules must be evaluated just like decoding parameters: measure recovery success, added latency, and whether the fallback hides systematic model failures.

🔑
Engineering Judgment: Reasoning effort is a control surface for a serving policy, not a universal upgrade switch. Advanced inference engineers tune it with per-slice evals, hidden-token accounting, incomplete-response handling, tool-continuation rules, and fallback metrics.

8.8 DeepSeek MTP – The Theory of Multi-Token Prediction

Multi-Token Prediction (MTP) is a training and inference technique used in DeepSeek-V3. Traditional language models predict only one future token at a time, whereas MTP allows the model to predict multiple future tokens at once.

MTP During Training: Multi-Output Head Architecture

A standard language model has only one output head (LM Head), which maps the last hidden state to vocabulary logits to predict . MTP adds additional output heads (MTP Heads), each predicting a future token at a different offset:

Equation 8.3 MTP Loss

Each MTP Head is a small MLP + LayerNorm, with far fewer parameters than the main Transformer. They share the underlying feature representation, only having independent parameters at the output layer.

Dual Benefits

  • Training Benefit: Denser supervision signals. Each token position must predict not only the next token but also the token after that. This may encourage the model to learn longer-range dependencies. The DeepSeek-V3 report noted the training efficiency gains brought by MTP; the specific magnitude depends on the model, data, and training setup, and the main text does not extrapolate it as a universal ratio.
  • Inference Benefit: During inference, the MTP Head can serve as a draft model for speculative decoding. Because the MTP Head has far fewer parameters than the full Transformer, it can quickly generate candidate tokens, which are then verified by the main model in a single forward pass.
💡
The DeepSeek-V3 report chose (predicting 1 additional token) because larger showed diminishing marginal returns in their experiments. The key takeaway here is understanding that "multiple prediction heads can serve both training and draft generation," not memorizing a specific fixed benefit ratio.

8.9 Programming Exercise 6: Implementing a Simplified Speculative Decoding

The core idea of Speculative Decoding is very clever: use a small model (draft model) to quickly generate candidate tokens, then use a large model (target model) to verify these candidates in parallel. When the rejection sampling condition is met, it can preserve the target model's sampling distribution; the actual speedup depends on the draft model's speed, acceptance rate, batching, and system bottlenecks, with papers and system reports commonly showing improvements of roughly 1.x to several times.

Algorithm Flow

  1. Draft Phase: Use a lightweight draft model (e.g., 100M) to autoregressively generate candidate tokens (typically or 5)
  2. Verification Phase: Feed the candidate tokens + prefix into the target model (e.g., 7B), obtain logits for all positions in one forward pass
  3. Rejection Sampling: Check the probability distributions of the target/draft models at each position – if the divergence is too large, reject and resample from a corrected distribution
  4. Acceptance: The accepted tokens and the resampled token are added to the output sequence together
Equation 8.4 Acceptance/Rejection Criterion

Exercise 6: Implementing Speculative Decoding

Task: Implement speculative_decoding(target_model, draft_model, input_ids, gamma=4, max_new_tokens=100, eos_token_id=None).

Requirements:

  • The draft model uses Top-P sampling to generate gamma candidate tokens
  • Target model verification: compute target probabilities and draft probabilities for all candidate positions in one forward pass
  • Rejection sampling: accept each candidate with probability min(1, p_target / p_draft)
  • When rejected, resample from max(0, p_target - p_draft)
  • If all are accepted, additionally sample one token from the target model
# Expected usage
output, stats = speculative_decoding(
    target_model=model_7b,       # Large model
    draft_model=model_100m,      # Small model
    input_ids=prompt,
    gamma=4,
)
print(stats)  # proposed, accepted, acceptance_rate
def speculative_decoding(target_model, draft_model, input_ids, gamma=4, max_new_tokens=100, eos_token_id=None): """Speculative Decoding: Draft → Verify → Rejection Sampling → Distribution-Preserving Acceleration""" device = next(target_model.parameters()).device prefix = input_ids.to(device) n_orig = prefix.size(1) proposed = 0 accepted_count = 0 while prefix.size(1) - n_orig < max_new_tokens: draft_ids = prefix.clone() with torch.no_grad(): for _ in range(gamma): logits = draft_model(draft_ids) probs = torch.softmax(logits[:, -1, :], dim=-1) next_token = torch.multinomial(probs, 1) draft_ids = torch.cat([draft_ids, next_token], dim=-1) target_logits = target_model(draft_ids) draft_logits = draft_model(draft_ids) draft_tokens = draft_ids[:, -gamma:] proposed += draft_tokens.size(1) target_probs = torch.softmax(target_logits[:, -(gamma + 1):-1, :], dim=-1) draft_probs = torch.softmax(draft_logits[:, -(gamma + 1):-1, :], dim=-1) accepted = [] n_accepted = 0 for t in range(gamma): token = draft_tokens[0, t].item() p = target_probs[0, t, token].item() q = max(draft_probs[0, t, token].item(), 1e-12) if p >= q or torch.rand(1).item() < p / q: accepted.append(token) n_accepted += 1 accepted_count += 1 else: residual = (target_probs[0, t] - draft_probs[0, t]).clamp(min=0) residual = residual / residual.sum() accepted.append(torch.multinomial(residual, 1).item()) break if n_accepted == gamma: extra_probs = torch.softmax(target_logits[:, -1, :], dim=-1) accepted.append(torch.multinomial(extra_probs, 1).item()) accepted = accepted[:max_new_tokens - (prefix.size(1) - n_orig)] prefix = torch.cat([prefix, torch.tensor([accepted], device=device)], dim=-1) if eos_token_id is not None and accepted[-1] == eos_token_id: break stats = { "proposed": proposed, "accepted": accepted_count, "acceptance_rate": accepted_count / proposed if proposed else 0.0, } return prefix, stats

💡 Distribution Preservation Analysis: When the draft/target probabilities, sampling temperature, and rejection sampling implementation all satisfy the paper's assumptions, the final output distribution can be consistent with directly sampling from the target model. The speedup cannot be judged solely by : if each round generates draft tokens, the target model verifies once, and the draft model's per-step cost is times that of the target model's per-step cost, then the time per round is approximately target model step units; the real numerator is the number of tokens actually written as output in that round, which depends on the acceptance rate.

8.9A Worked Example: Acceptance Rate and Speedup Accounting for Speculative Decoding

Let gamma=4, and the time for the draft model to generate one token is times the time for the target model to take one step. In each round, the draft model first proposes 4 tokens, then the target model performs 1 parallel verification. Consider three rounds of generation:

RoundDraft ProposalsAccepted Draft TokensOutput Tokens WrittenExplanation
1445All 4 accepted, target model samples 1 additional token
24233rd draft token rejected, write 2 accepted tokens + 1 correction token
3434Accept 3 then reject 1, write correction token and proceed to next round

Over these three rounds, a total of 12 draft tokens were proposed, 9 were accepted, giving an acceptance rate of 9/12=75%; the actual number of output tokens written is 5+3+4=12. Without speculative decoding, the target model would need 12 autoregressive forward passes. With speculative decoding, the target model only verifies 3 times, but the draft model runs 12 times, so the approximate time is:

speculative decoding rough speedup

This accounting explains two common misconceptions: First, gamma=4 does not guarantee a fixed 4x speedup, because the draft model also takes time, and rejections reduce the number of tokens produced per round. Second, a high acceptance rate does not automatically translate to high end-to-end throughput; real services are also affected by batching, KV cache, memory bandwidth, network, and queuing. Therefore, experimental reports should simultaneously report proposed, accepted, generated_tokens, acceptance_rate, the number of target model verifications, the number of draft model steps, and a rough speedup. A speculative decoding report should calculate these quantities from the per-round acceptance counts and output token counts, then combine them with P95 latency, QPS, draft overhead, additional memory, and quality/distribution checks before deciding whether to enable speculative decoding.

8.10 Context Window and Attention Sink

Context Window is the maximum number of tokens a model can "see" during a single generation. A longer window helps only when the model can still retrieve, prioritize, and use the relevant information inside that window.

  • Long-range Dependencies: A novel's foreshadowing might be revealed hundreds of pages later; a function definition is at the beginning of a file, but its call is at the end. Longer context means better cross-paragraph reasoning.
  • Few-shot Learning: Placing more examples (in-context examples) in the prompt improves model performance. The longer the context, the more examples can be accommodated.
  • Multi-turn Dialogue: Maintaining a complete conversation history. A 128K window can hold approximately 100,000 characters of dialogue history.

Attention Sink

Recent research has discovered an interesting phenomenon—in long sequences, models tend to allocate a large amount of attention weight to the initial tokens (usually <BOS> or sentence-start tokens), even when these tokens are irrelevant to the current generation. This is called the Attention Sink (Xiao et al., 2024).

Cause: During training, the sequence length is limited (e.g., 4K), causing the model to develop a "residual attention bias"—initial tokens absorb excess attention scores. When the sequence length increases during inference, this bias is amplified, and the model wastes significant attention on irrelevant early tokens.

8.10A Generation Strategy Deployment Decision: Putting Quality, Format, Latency, and Cost on the Same Table

The greedy, top-p, beam, self-consistency, speculative decoding, and constrained decoding covered in the course are not a set of mutually exclusive buttons. Real services typically combine them into a generation policy: first determine the output space based on the task, then choose the sampling/search method, and finally use format constraints, stopping conditions, and cost budgets to protect the system.

ScenarioRecommended Starting PointQuality MetricsSystem MetricsPrimary Risk
Code CompletionLow temperature top-p or pass@kunit tests, compile rate, pass@kAverage output tokens, cost per questionGenerates code that looks reasonable but doesn't run

1. First Determine the Output Space

If the output must be JSON, SQL, function calls, or a fixed enumeration, constrained decoding or structured output should be prioritized, rather than relying on a prompt to remind the model "please output only JSON". If the output is an open-ended explanation, then tuning temperature, top-p, and repetition penalty is the main focus. If the task has a verifiable answer, such as code or math, sampling diversity itself is not the goal; multi-sample generation must be converted into correctness rates via pass@k, unit tests, or a verifier.

2. Then Determine Risk Boundaries

Each generation policy should clearly state its failure boundaries. For example, in customer service RAG, low temperature can reduce randomness but cannot guarantee faithful citations; beam search can search for high-probability candidates but cannot fix missed retrievals; self-consistency can reduce the chance of a single inference error but can lead to a majority of wrong votes for systematic misunderstandings; structured decoding can guarantee syntactic validity but cannot guarantee semantic field correctness.

3. Cost Must Be Calculated Per Complete Request

A deployment report cannot just write the model's tokens/s. The end-to-end cost of a request includes prompt tokens, generated tokens, reasoning samples, RAG retrieval/reranking, tool calls, structured decoding overhead, retries, and queuing. A simplified accounting can be described as:

request_cost ~= prompt_tokens + generated_tokens * samples + reranker_calls * reranker_cost + tool_calls * tool_cost + retries * retry_cost

If samples is increased from 1 to 16, pass@k might improve, but P95 latency and token cost will also increase nearly linearly, unless the system performs parallel sampling and has sufficient batch/hardware headroom. Speculative decoding can reduce the number of target model verifications, but it also introduces draft model computation and scheduling complexity.

3A. Reasoning / Test-Time Compute Requires a Budget Decision

After OpenAI o1, DeepSeek-R1, and Kimi k1.5, many systems use "thinking a bit longer" as a quality improvement method: increasing reasoning tokens, sampling multiple solution paths, using a verifier or unit tests for best-of-N selection, or distilling long chains into short outputs. Engineering reports cannot just report accuracy improvements; they must also report samples per question, average output tokens, P95 latency, verifier calls, cost per correct answer, and marginal benefit.

A test-time compute budget report formalizes this judgment into a deployment decision. It compares greedy decoding, self-consistency, best-of-N, verifier reranking, or reasoning model tiers, and outputs quality, token budget, latency, cost, and marginal efficiency signals, along with a recommended strategy for deployment.

report = test_time_compute_budget_report( [ {"name": "greedy", "accuracy": 0.55, "samples_per_prompt": 1, "mean_output_tokens": 120, "latency_ms": 500}, {"name": "self_consistency_5", "accuracy": 0.72, "samples_per_prompt": 5, "mean_output_tokens": 160, "latency_ms": 3200}, {"name": "best_of_16", "accuracy": 0.78, "samples_per_prompt": 16, "mean_output_tokens": 170, "latency_ms": 9000}, ], budgets={ "min_accuracy": 0.70, "max_output_tokens_per_prompt": 1000, "max_latency_ms": 4000, "min_accuracy_gain_per_1k_tokens": 0.05, }, ) assert report["recommended_strategy"] == "self_consistency_5"

The purpose of this budget check is not to replace benchmarks, but to prevent the conclusion of "trading unlimited sampling for scores" from polluting deployment decisions. If best-of-16 has the highest accuracy but exceeds the token and latency budget, a deployment report should state "requires distillation, routing to hard problems, optimizing parallel sampling, or rejecting this strategy", rather than directly using the highest score as the production strategy.

4. Format of a Qualified Strategy Description

A production generation policy should be written in the following form:

Task: Chinese customer-service RAG Output: natural-language answer with cited document ids Decoding: temperature=0.2, top_p=0.9, max_new_tokens=512, stop=["\n\nUser:"] Constraints: answer must include at least one retrieved doc id; JSON only for tool calls Evaluation: faithfulness, answer F1, citation precision/recall, unsafe completion rate System: TTFT/P95 TPOT, generated tokens/request, retry rate, cost/request Known boundary: low temperature reduces variance but does not prove faithfulness
🔑
Engineering Judgment: The generation strategy is not a single hyperparameter, but a combination of task objectives, output constraints, sampling/search, evaluation metrics, and service SLOs. The strategy report should explain why this set of strategies was chosen and within which tasks, loads, and risk boundaries it holds.

8.12 Structured Decoding — JSON Mode and Constrained Generation

In API and Agent scenarios, we often need the LLM to output structured data in a specific format (JSON, YAML), rather than free text. Asking the LLM to "try to output JSON" is not enough — it occasionally outputs incomplete JSON or extraneous text.

Constrained Decoding Principle

Constrained decoding dynamically modifies logits at each sampling step according to the target grammar: it sets the probability of tokens that do not conform to the grammar to -inf, ensuring the generation adheres to the specified format.

Constrained Decoding

Implementation: Token Filtering based on JSON Schema

Taking JSON as an example: compile the JSON Schema into a Deterministic Finite Automaton (DFA). At each decoding step, feed each token in the vocabulary into the DFA to determine if it is legal — only legal tokens can be sampled. For example, when the prefix is {"name":, the next token must be ".

Exercise 7: Implement Token Constraint Mask

Task: Implement apply_token_constraints(logits, allowed_token_ids), setting the logits of illegal tokens in each batch row to -inf.

Requirements:

  • Input logits shape is [batch, vocab]
  • allowed_token_ids can be a set of tokens shared by all samples, or a separate set for each sample
  • Each row must retain at least one legal token; illegal ids or an empty set should raise an error
  • After applying constraints, execute greedy, top-k, or top-p; probabilities are renormalized only over the set of legal tokens
# Example: In the second row, token 2 originally has the highest logit, but the grammar currently only allows 0 or 1
logits = torch.tensor([[5.0, 4.0, 3.0],
                       [1.0, 2.0, 9.0]])
allowed = [[1, 2], [0, 1]]
masked = apply_token_constraints(logits, allowed)
next_token = sample_next_token(masked, strategy="greedy")
# next_token == [[1], [1]]

8.12A Structured Output Reliability: Schema Validity Is Not Semantic Correctness

Structured output should be understood as a contract stack, not a single feature. JSON mode, strict JSON Schema, regex/grammar constraints, tool-call schemas, post-validation, and retry repair each protect a different boundary. OpenAI's Structured Outputs documentation emphasizes schema adherence; vLLM and SGLang expose similar constrained-output options through JSON schema, regex, choices, grammar, or EBNF. These mechanisms make parseability much more reliable, but they do not prove that the selected field values are true, complete, safe, or allowed to be executed.

Figure 8-4: Structured output reliability stack. Constrained decoding protects syntax and schema, while semantic validation and execution guards protect correctness and safety.

1. Choose the constraint level based on the downstream action

2. Validation must be layered after decoding

Even if the decoder produces schema-valid JSON, the application still needs semantic validation. For RAG, document IDs must be among retrieved documents and the cited span must support the answer. For extraction, dates, currencies, units, and entity IDs must match source evidence. For tool calls, arguments must pass permission checks, tenant boundaries, rate limits, and idempotency rules. For code or SQL, the syntax may be valid while the program is unsafe or semantically wrong.

3. Report both schema quality and semantic quality

A structured output experiment should not stop at "JSON validity is 99%." A useful report includes schema pass rate, retry count, latency overhead, semantic field accuracy, unsupported citation rate, tool authorization failures, and post-execution failure rate. If constrained decoding improves parseability but increases latency or hides a systematic extraction error, the release conclusion must mention that boundary.

For senior inference engineers, the correct deployment claim is narrow: structured decoding guarantees the output shape under the supported schema; application validation decides whether the content is true and safe to execute. Confusing these two layers is a common cause of agent incidents: a valid tool-call JSON object can still delete the wrong resource, cite a document that does not support the claim, or submit an unsafe query.

🔑
Engineering Judgment: Use constrained decoding to make the output parseable, then use validators and execution guards to decide whether it is correct, authorized, and safe. Schema adherence is necessary infrastructure, not a substitute for task evaluation.

8.12B Constrained Decoding Engines: CFG, FSM, Schema, and Serving Cost

Structured output is not only a prompting technique. In production inference, it is usually implemented as a serving-time constraint engine: the request provides a schema, regex, choice set, grammar, or tool signature; the backend compiles that contract into a constraint state; every decoding step masks illegal tokens before sampling. OpenAI Structured Outputs, vLLM Structured Outputs, SGLang Structured Outputs, Outlines-style libraries, and guidance-style libraries all follow this broad pattern, although they differ in supported schema features, grammar backend, batching behavior, and latency overhead.

This section complements the previous reliability discussion. Constrained decoding can make the surface format valid, but it does not make the content true. The inference engineer still needs validators, evidence checks, permission checks, and failure metrics after the decoder returns a parseable object.

Figure 8-5: A constrained decoding engine converts a format contract into a parser state and token mask. The serving stack must still handle unsupported schemas, tokenizer edge cases, latency, streaming, and semantic validation.

2. Production failure modes

Failure ModeWhy It HappensPractical Response

3. What to measure before enabling it by default

A production report should include schema_compile_ms, per-token mask overhead, schema pass rate, semantic field accuracy, retry rate, no-legal-token rate, unsupported schema rate, and p95 latency delta versus unconstrained decoding. For high-throughput serving, also report how constrained requests affect continuous batching and KV-cache utilization.

Primary references worth reading next are OpenAI Structured Outputs, vLLM Structured Outputs, SGLang Structured Outputs, and the JSONSchemaBench benchmark for structured-output reliability and efficiency.

🔑
Key Takeaway: Constrained decoding is a serving-time control plane. It converts format reliability into a state-machine problem, but task correctness still depends on validators, evidence, permissions, and bounded failure handling.
