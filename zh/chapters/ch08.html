采样策略、MTP speculative decoding 和结构化输出：把概率变成文本

8.1 自回归生成的数学原则

训练后的模型本质上是 scorer：给定前缀，输出下一个 token 的 logits。生成就是把这个分布一次次转成实际 token。

P(x_1, ..., x_T) = product_t P(x_t | x_

每一步拿到 P(x_t | x_ 后，可以取最大值，也可以按概率采样。把选出的 token 追加到上下文，再进入下一步，直到遇到 EOS、stop sequence 或达到 max_new_tokens。

Prefill 和 Decode

  • Prefill：一次性处理用户 prompt，计算密集，主要影响 TTFT。
  • Decode：每步生成一个 token，读取 KV cache，常受显存带宽和 batch/并发影响，主要影响 TPS/TPOT。

复现实验要固定模型、prompt 集、上下文长度、并发、batching 和 serving 配置，否则 TTFT/TPS 数字不能横向比较。

8.2 为什么 Greedy Decoding 会失败

Greedy decoding 每步选择概率最高的 token：x_t = argmax P(x | x_。它简单、稳定，但有两个常见问题。

重复陷阱

模型一旦进入高概率局部模式，后续 argmax 可能不断选择相似 token，出现循环和重复。这通常来自过尖的分布、上下文中的重复模式、没有随机探索和停止约束太弱。

Exposure Bias

训练时模型看到真实前缀；推理时模型看到自己生成的前缀。一步错误会改变后续分布，错误可能越滚越大。

Temperature 控制 entropy

P_T(x_i) = softmax(z_i / T)
  • T -> 0：退化为 argmax，更确定。
  • T = 1：原始分布。
  • T -> infinity：接近均匀分布，更随机。

8.3 编程练习 1：实现 Greedy Decoding

练习 1：Greedy Decoding

  • 输入 input_ids: [1, seq_len]。
  • 每步取 logits[:, -1, :]。
  • 用 argmax 选下一个 token。
  • 遇到 EOS 提前停止。
def generate_greedy(model, input_ids, max_new_tokens, eos_token_id=None): model.eval() for _ in range(max_new_tokens): with torch.no_grad(): logits = model(input_ids) next_token = logits[:, -1, :].argmax(dim=-1, keepdim=True) input_ids = torch.cat([input_ids, next_token], dim=-1) if eos_token_id is not None and next_token.item() == eos_token_id: break return input_ids

8.4 编程练习 2：实现 Temperature Sampling

Temperature sampling 把 argmax 换成 softmax + multinomial。

练习 2：Temperature Sampling

  • softmax 前用 logits / temperature。
  • 用 torch.multinomial 采样。
  • temperature=0 时退化为 greedy，不要真的除以 0。
def sample_temperature(logits, temperature=1.0): if temperature == 0: return logits.argmax(dim=-1, keepdim=True) probs = torch.softmax(logits / temperature, dim=-1) return torch.multinomial(probs, num_samples=1)

8.5 从 Temperature 到 Top-K / Top-P

Temperature 只改变分布尖锐度，不能去掉长尾低质量 token。Top-K 和 Top-P 会截断低价值尾部，再重新归一化。

Top-K

保留概率最高的 K 个 token。问题是 K 固定：分布很尖时 K 可能太大，分布很平时 K 又可能太小。

Top-P / Nucleus

按概率从高到低累加，保留累计概率达到 p 的最小候选集。它会随分布形状自适应。

例子：同一分布下的候选集

概率为 [0.40, 0.25, 0.15, 0.10, 0.06, 0.04]。top_k=3 固定保留前三个；top_p=0.9 会保留前四个，因为 0.40+0.25+0.15+0.10=0.90。

8.5A Sampling Distribution Ledger

一次生成 step 应该明确处理顺序。推荐顺序：

  1. 拿到 raw logits。
  2. 应用 logit bias、bad words、repetition penalty 等 logits processors。
  3. 应用 temperature。
  4. 应用 top-k/top-p/min-p 等截断。
  5. 重新归一化，再采样。

记录 logprob、entropy、candidate set 时，必须说明它们对应的是哪一个分布：raw logits、processor 后、temperature 后，还是最终截断后。

8.6 编程练习 3-4：Top-K 和 Top-P Sampling

练习 3：Top-K Sampling

def top_k_filter(logits, k): if k is None or k <= 0: return logits values, _ = torch.topk(logits, k, dim=-1) threshold = values[:, [-1]] return logits.masked_fill(logits < threshold, float("-inf"))

练习 4：Top-P Sampling

def top_p_filter(logits, p): sorted_logits, sorted_idx = torch.sort(logits, descending=True, dim=-1) probs = torch.softmax(sorted_logits, dim=-1) cum = probs.cumsum(dim=-1) remove = cum > p remove[..., 1:] = remove[..., :-1].clone() remove[..., 0] = False sorted_logits = sorted_logits.masked_fill(remove, float("-inf")) out = torch.full_like(logits, float("-inf")) return out.scatter(dim=-1, index=sorted_idx, src=sorted_logits)

8.7 编程练习 5：比较四种采样策略

比较 greedy、temperature、top-k、top-p 时，不要只看“哪个更有趣”。至少记录：

  • 重复率：重复 n-gram 或 repeated token 比例。
  • distinct-n：输出 n-gram 的多样性。
  • 平均长度和停止原因：EOS、stop sequence、max token。
  • 人工/任务质量：是否答题正确、是否守格式、是否跑题。

多样性不是质量，低重复也不等于正确。对生产系统，策略选择要绑定任务类型。

8.7A 怎样评估生成策略

参数 sweep 比单点比较更可靠。固定 prompt 集和随机 seed，扫描 temperature、top-p、top-k、repetition penalty，报告质量、重复、长度、成本和延迟。

Repetition penalty 会降低已出现 token 的 logits，能缓解重复，但可能伤害需要复用术语、代码变量或 JSON key 的任务。

8.7B Beam Search：搜索多个高概率候选

Beam search 每步保留多个候选序列，而不是只选单个最佳 token。它适合翻译、摘要等比较封闭的输出空间，但在开放式写作里可能更模板化。

Length Bias

序列概率是 token 概率乘积，越长通常越小，所以 beam search 需要 length normalization，否则容易偏短。

8.7C Reasoning Generation：用更多推理时计算换准确率

reasoning 任务常用更多 test-time compute：self-consistency、best-of-n、verifier/reranker、tree search 或更高 effort。它和普通采样的差别在于：不只是改变一个 token 分布，而是用更多候选、更多中间 token 或外部验证来提高正确率。

pass@k 不是简单的 c/n。当一次生成有多条候选时，报告要写清候选数、采样策略、verifier、token budget 和选择规则。

8.7D Thinking Mode、Thinking Budget 和 Runtime Routing

thinking mode 是推理时预算控制问题。系统可以根据任务复杂度选择不同 effort：快速回答、普通回答、深度推理、工具调用。更高 effort 不保证质量一定更好，但会增加延迟和 token 成本。

生产上需要 routing policy：哪些请求启用 reasoning，预算多少，超时怎么办，格式不完整怎么办，是否允许工具继续补全。

8.7E Reasoning Controller：隐藏 token、未完成输出和工具续写

  • Effort 是评估旋钮：不是质量保证。
  • Hidden reasoning tokens 仍然是成本：要计入上下文、延迟、账单和限额。
  • 需要 fallback：超预算、工具失败、输出不完整、schema 不合法时，系统要有降级路径。

8.8 DeepSeek MTP：Multi-Token Prediction

传统 LM 每个位置只预测下一个 token。MTP 让模型在训练时预测多个未来 token，增加训练信号，也能为 speculative decoding 提供更好的 draft 能力。

收益主要有两类：训练阶段得到更密集的监督；推理阶段可以一次提出多个候选 token，再由主模型验证。

8.9 编程练习 6：简化 Speculative Decoding

Speculative decoding 用一个便宜的 draft model 先生成多个候选 token，再用 target model 批量验证。若候选被接受，就一次前进多个 token；不接受则回退到 target 分布采样。

练习 6：Speculative Decoding

  1. draft model 连续提出 k 个 token。
  2. target model 对这 k 个位置并行算概率。
  3. 按 acceptance rule 接受前缀候选。
  4. 遇到拒绝时，从修正后的 target 分布采样一个 token。

它的目标是加速而不改变目标模型分布。速度收益取决于 draft 质量、acceptance rate、k、batching 和 target model 验证成本。

8.9A Acceptance Rate 和加速账

如果 draft 每轮提 k=4 个 token，平均接受 3 个，那么每次 target 验证能推进约 3 个 token。但真实加速还要扣掉 draft model 成本、额外调度和缓存管理成本。

8.10 Context Window 和 Attention Sink

生成时上下文窗口有限。超过窗口后，要截断、滑窗、摘要、检索，或使用长上下文模型。

Attention Sink

一些早期 token 会吸引大量 attention，像“锚点”一样稳定后续生成。长上下文推理中，保留少量 sink tokens 加上最近窗口，可能比简单丢掉开头更稳。

8.10A 生成策略部署决策

上线策略要把质量、格式、延迟和成本放在同一张表里。

  1. 先确定输出空间：开放文本、短答案、代码、JSON、工具参数。
  2. 再确定风险边界：是否允许随机性、是否需要引用、是否能重试。
  3. 按完整请求算成本：input tokens、output tokens、hidden reasoning tokens、工具调用、重试。
  4. 给出合格策略描述：sampling 参数、stop 规则、fallback、监控指标和评估集。

8.11 本章小结

  • Greedy decoding 简单但容易重复，也暴露 exposure bias。
  • Temperature 控制分布 entropy，T -> 0 时接近 greedy。
  • Top-K 固定保留 K 个候选，Top-P 按累计概率自适应截断。
  • 策略评估要同时看质量、多样性、重复、长度、延迟和成本。
  • Beam search 适合较封闭任务，开放式生成容易变模板。
  • reasoning budget 是推理时成本控制，不是质量保证。
  • MTP 和 speculative decoding 通过 draft + verify 提升推理吞吐，关键指标是 acceptance rate。
  • constrained decoding 能保证 schema 形状，但不能保证语义正确。

8.12 Structured Decoding：JSON Mode 和 Constrained Generation

结构化输出要求模型不仅“说得对”，还要输出可解析格式。Constrained decoding 会把非法 token 的 logits 设为 -inf，只在合法集合上重新归一化。

练习 7：Token Constraint Mask

  • 根据 JSON schema 或状态机计算当前合法 token 集。
  • 非法 token logits 设为 -inf。
  • 采样前重新归一化。

常见工具包括 JSON mode、grammar/CFG decoding、FSM 约束、schema-guided decoding 和后置 validator。

8.12A 结构化输出可靠性：Schema 合法不等于语义正确

JSON 能 parse，只说明格式合法。字段值可能仍然错、单位错、枚举错、引用不存在，或者违反业务规则。

  • 低风险：普通展示字段，可以用宽松 JSON mode。
  • 中风险：写数据库或调用内部 API，需要 schema + business validation。
  • 高风险：支付、医疗、法律、权限变更，需要人工或更强 verifier。

8.12B Constrained Decoding Engines：CFG、FSM、Schema 和成本

约束越强，格式越稳，但服务成本也更高：每步要计算合法 token 集、维护 parser state，并和 tokenizer 边界对齐。

约束类型适合场景风险
FSM固定格式、简单协议表达能力有限
CFG嵌套语法、JSON、SQL 子集实现和 token 对齐复杂
Schema字段类型和结构约束语义仍需后置验证

默认开启前要测：schema pass rate、semantic pass rate、延迟、fallback 频率、工具调用成功率和人工介入率。
