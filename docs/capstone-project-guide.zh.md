# 端到端 LLM 结课项目指南

这份指南把分散在各章的练习收束为一个可审查的工程项目。目标不是训练前沿模型，也不是用一个演示页面代替工程结论；目标是让学习者能把任务、数据契约、模型或流水线、评估证据、服务约束和发布决策连成闭环。英文版见 [End-to-End LLM Capstone Guide](capstone-project-guide.md)。

## 这个结课项目要证明什么

单个练习只能证明一个组件能运行。结课项目应当证明这些组件能在一个明确的决策中协同工作。审阅者应能直接回答下面的问题：

- 系统服务什么用户任务，错误的代价是什么？
- 什么数据进入系统，保留什么状态，允许输出什么？
- 正在与哪个基线比较，只改变了哪个因素？
- 哪些证据支持质量、安全、延迟、成本和可复现性的结论？
- 在什么条件下应拒绝发布、回滚，或交给人工处理？

项目可以使用 CPU 级教学模型、Mock engine 或很小的本地语料。小规模演示的分数不能外推为生产级质量或安全性。

## 选择一个范围受控的项目方向

选择一个主方向。下面三个方向共享同一份证据契约；不同点只在于主问题和最有解释力的基线。

| 方向 | 示例问题 | 最小基线 | 主要复用章节 | 不能当作充分证据的内容 |
|---|---|---|---|---|
| 适配型助手 | SFT 或少量偏好数据能否改善窄场景回答风格，同时不伤害留出任务？ | 基础模型或仅 prompt 的行为 | Ch01、Ch06-Ch09、Ch11 | 训练 loss 下降 |
| 检索 / 工具助手 | 检索、结构化输出或一个受保护工具能否在 SLO 内提高有依据的任务完成率？ | 不检索或自由生成 | Ch01、Ch08、Ch10-Ch11 | JSON 可以解析 |
| 结构化 NLP 流水线 | 对这一输出空间，抽取、重排序或分类是否比自由生成更安全、成本更低？ | decoder-only prompt 或简单规则 | Ch02-Ch03、Ch08、Ch10-Ch11 | 没有字段级错误分析的 token 准确率 |

好的范围应只有一个用户、一类任务、一个数据源或模拟器，以及一个明确的非目标。例如：“针对版本化手册回答政策问题并给出引用；不提供法律建议，也不执行外部动作。” “做一个聊天机器人”还不是可检验的问题。

## 里程碑地图：把章节连成系统

| 里程碑 | 课程关联 | 交付物 | 验收信号 |
|---|---|---|---|
| M0 — 项目契约 | Ch01、Ch11 | 一页任务、风险与度量契约 | 读者能说清允许输入/输出和拒绝条件 |
| M1 — 数据与接口 | Ch01-Ch02、Ch07 | 版本化数据或语料 manifest；tokenizer/template 决策 | 已记录样本 ID、数据切分、token 长度分布和泄漏检查 |
| M2 — 参考路径 | Ch03-Ch06 | 最小正确的前向、检索或抽取路径 | shape、mask 和一个确定性 golden case 通过 |
| M3 — 改进实验 | Ch07-Ch09 | 一个受控的训练、调参或流水线改动 | 基线与变体只在一个已说明的因素上不同 |
| M4 — 评估 | Ch08、Ch11 | 冻结 eval 集、切片表、错误分类 | 质量结论包含样本数、设置、不确定性和失败案例 |
| M5 — 服务与安全 | Ch08、Ch10 | 请求契约、预算、校验与回退 | 压力、畸形输入和不安全动作演练有预期结果 |
| M6 — 发布决策 | Ch07、Ch10-Ch11 | 证据台账、回滚规则和最终报告 | 审阅者能批准、拒绝或提出范围明确的后续工作 |

不要把评估留到 M6。M0 就应建立一个很小的留出集，M2 运行基线，只有在系统变化时才扩充切片。

## M0：先写项目契约，再写代码

新建 `project_contract.md`，至少包含下列字段。重点不是措辞漂亮，而是让每一项都能被判定。

| 字段 | 有用的写法 | 常见的弱写法 |
|---|---|---|
| 用户任务 | “对手册问题返回一条带引用的回答。” | “让回答更好。” |
| 输出契约 | “`answer`、`citations`、`abstain_reason` 必须通过校验；引用必须来自检索语料。” | “返回 JSON。” |
| 非目标 | “不提供法律、医疗建议，不修改账户。” | 不说明范围 |
| 基线 | “Top-1 检索加确定性回答模板。” | “现有模型。” |
| 成功指标 | “冻结测试集上的有引用支持回答准确率。” | “模型看起来有用。” |
| 护栏 / 拒绝规则 | “无来源超过检索阈值时拒答；allow-list 之外的工具一律阻断。” | “提示模型要安全。” |
| 资源边界 | “此机器上端到端 P95 小于 3 秒；请求输入少于 4K token。” | “足够快。” |

契约还应写明每项外部数据或工具的所有者、版本和权限。检索文本和工具返回都应被当作不可信输入，而不是系统指令。

## M1：建立数据、Token 与状态台账

数据台账将 Ch01 的 tokenizer 契约连接到 Ch07 的训练/评估切分，再连接到 Ch10 的上下文预算。至少记录：

- 数据集或语料版本、许可/权限边界、采集日期和内容所有者。
- 稳定样本 ID；train/dev/test 或检索语料/query 切分；精确重复和答案泄漏检查。
- tokenizer、chat template、special token ID、最大长度、截断方向和长度分位数。
- 对 RAG：文档 ID、chunk ID、来源版本、切块规则、embedding 或索引版本，以及删除/更新路径。
- 对工具：schema 版本、权限等级、超时、重试上限，以及结果是否能进入后续模型上下文。

选一个人工检查过的 golden sample，沿完整路径追踪：原始输入 → 渲染后的 prompt 或 query → token ID → mask/position ID → 模型/流水线输出 → 校验器 → 用户可见响应。这样的单样本追踪通常比汇总指标更早暴露接口不一致。

## M2 与 M3：先建立正确的参考路径，再一次只改一个因素

优化前，先让最简单的路径可复现。可复用课程里的检查：

| 层 | 要验证什么 | 对应课程内容 |
|---|---|---|
| Token / template | 同一文本渲染为预期 ID 与 assistant-loss mask | Ch01、Ch02、Ch09 |
| 模型路径 | 固定输入上的 tensor shape、causal 可见性、logit 或 loss parity | Ch03-Ch06 |
| 训练 / 适配 | seed、优化器/调度器状态、checkpoint resume、有效 token 分母 | Ch07、Ch09 |
| 解码 | 输出同时记录采样或 constrained decoding 参数 | Ch08 |
| 流水线 | 检索排名、选中上下文、工具计划、校验结果和回退可追踪 | Ch10 |

每次实验只改变一个主要因素：检索 top-k、切块规则、LoRA rank、数据过滤、解码策略或 batching 策略。若不得不同时改多个配置，应标注为混杂比较，只能用于探索，不能据此做因果结论。

训练类工作应保存完整续训状态：模型、优化器、调度器、scaler（如使用）、RNG 状态、已消费数据/token 位置和配置。服务类工作应将模型、prompt/template、语料/索引、schema 与 engine 配置一同版本化。

## M4：评估要验证主张，不是验证演示

在选择最终设置前冻结测试集。把系统拆成可观测阶段，使每个错误都有位置可查：

| 系统问题 | 最小证据 | 有用的失败切片 |
|---|---|---|
| 答案或抽取是否正确？ | 任务准确率、EM/F1、通过率，或校准过的人类/裁判 rubric | 短/长输入、罕见格式、歧义、多语言输入 |
| RAG 是否找到了证据？ | Recall@k/MRR/nDCG 加检索 chunk ID | 无相关 chunk、陈旧来源、重复 chunk、context packing 丢失位置 |
| 生成是否有依据且结构正确？ | 引用支持率、schema 有效率、语义校验、正确拒答率 | JSON 有效但字段错误、无依据断言、答案不完整、过度拒答 |
| 系统是否可用？ | TTFT、端到端 P50/P95、错误率、单请求成本、token 数 | 长 prompt、并发负载、cache miss、超时、重试风暴 |
| 安全边界是否工作？ | 攻击成功率、正常任务完成率、过度拒绝率、阻断动作率 | 注入文档、越权工具、畸形 schema、冲突策略 |

小评估集也要报告样本数和比较的不确定性。重复 seed、bootstrap 区间，或坦诚说明样本过小不足以支持发布结论，都比把微小变化四舍五入成确定提升更严谨。错误分类至少覆盖数据/接口、检索、生成/建模、校验/工具、安全、服务中的三类。

## M5：把演示变成边界明确的服务

把请求写成契约，而不只是 prompt。至少要说明输入大小上限、认证/租户边界（如适用）、输出 schema、校验器行为、工具 allow-list、超时/重试/取消策略，以及回退或人工接管路径。

在宣布项目完成前，运行下面的演练：

| 演练 | 预期的安全行为 | 要保存的证据 |
|---|---|---|
| 输入缺失或畸形 | 返回机器可读错误，不虚构字段 | 请求、校验结果、响应码 |
| 检索文本中的 prompt injection | 把文本当数据，不扩大权限，不执行嵌入指令 | 检索 chunk、策略决策、trace |
| 无效工具参数 | 执行前阻断，返回可修复的校验反馈 | 解析后的计划、schema/权限结果 |
| 检索失败 / 低置信度 | 拒答、追问或走声明过的 fallback | 阈值、选中的 fallback、用户响应 |
| 更长输入或并发突发 | 准入控制或有意降级，而不是静默失效 | 输入/输出 token、排队时间、TTFT/TPOT/P95 |
| 版本变更后的回归 | 触发预先声明的 gate 时停止发布或回滚 | 基线/变体报告和回滚决策 |

“模型通常会遵守 system prompt”不是控制措施。校验、授权、预算和可观测性必须在模型输出之外存在。

## 证据台账与仓库结构

报告应由小而可检查的产物支撑。下面只是建议结构；关键是每条结论都指向版本化输入和可复现命令。

可以从可复制的[结课项目模板](../capstone-template/)开始。模板提供项目契约、JSONL 记录、决策报告骨架和本地证据校验器；只有替换完所有占位字段后，才将 `*.template.*` 文件重命名为项目文件。

```text
capstone/
  README.md                 # 问题、范围、基线、命令、决策
  project_contract.md       # 输出/风险/资源契约
  configs/                  # 模型、template、索引、服务配置
  data_manifest.jsonl       # ID、版本、切分、权限；不放敏感数据
  eval/                     # 冻结用例、rubric、切片定义
  runs/                     # 不可变运行元数据与汇总指标
  traces/                   # 脱敏的检索/工具/校验 trace
  reports/                  # 基线-变体对比表和失败分析
  scripts/                  # 确定性准备、评估、压测命令
```

每条 run 记录都应包含 commit ID、命令、seed、硬件/runtime、模型/checkpoint、tokenizer/template、数据/索引版本、生成参数、并发/负载形状、起止时间和指标定义。复现材料应脱敏，不能反过来造成隐私泄漏。

## 最终报告与评分标准

如果链接证据台账，最终报告可以控制在 4–8 页。它应回答：问题、范围和非目标、基线、干预、数据/接口契约、评估设计、分切片结果、失败、资源画像、安全/服务控制、决策与局限。

| 维度 | 0 — 缺失 | 1 — 部分具备 | 2 — 证据充分 |
|---|---|---|---|
| 问题和边界 | 演示目标模糊 | 说明了任务，但输出/风险边界不完整 | 任务可判定，给出非目标、错误代价和拒绝规则 |
| 系统正确性 | 只描述组件 | 只有一条 happy path | 有 golden trace 和 shape/schema/validator 检查 |
| 实验设计 | 只有最终分数 | 有基线或指标 | 冻结 eval、受控变量、切片、不确定性、错误分类 |
| 工程证据 | 未版本化截图 | 有部分日志或代码 | 有复现命令、版本、资源指标、回滚 gate |
| 安全和运行 | 只靠 prompt | 只有人工注意事项 | 外部校验、权限、预算、回退和对抗演练 |

最低的合理标准为 7/10，且 **系统正确性**、**实验设计**、**安全和运行** 均不得为 0。演示结果很好但无法说明使用了什么数据/版本/配置的项目，仍然没有完成。

## 完成定义

提交前确认以下陈述都成立：

- 新读者能按文档命令运行基线和最终变体。
- 报告指出一个受控改动，不把混杂比较说成因果关系。
- 冻结评估包含任务相关切片和至少一种已观察到的失败类型。
- 在输出、检索或工具结果可能造成伤害的边界做了校验。
- 延迟/成本结果写明模型、硬件、输入/输出分布、并发和分位数。
- 有具体的拒绝、回滚或人工接管规则，并实际跑过一次演练。
- 结论同时说明证据支持什么，以及不支持什么。

请将本指南与章节作业、[书面推导与概念题库](written-problem-set.md)和[可复算例题包](worked-example-pack.md)结合使用。结课项目不取代这些练习；它用一个端到端的工程判断，证明公式、测试和诊断可以共同支撑决策。
