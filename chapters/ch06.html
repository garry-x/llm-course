6.1 GPT-2 Architecture Overview — Complete Data Flow from Token ID to Logits

After the accumulation of the first 5 chapters, we can finally assemble all components into a complete GPT model. The architecture of GPT-2 small (124M) can be summarized in three main stages:

  1. Embedding Stage: Token Embedding + Position Embedding → Map discrete token IDs to continuous vectors
  2. Computation Stage: 12 layers of Transformer Blocks stacked → Transform representations layer by layer
  3. Output Stage: Final Norm + LM Head → Map hidden states to vocabulary probability distribution

Key Differences Between GPT-2 and Subsequent Models

  • Absolute Position Encoding: GPT-2 uses learnable Position Embeddings (1024 positions × 768 dimensions), not RoPE. RoPE is used by subsequent models like Llama, DeepSeek, etc.
  • LayerNorm with bias: GPT-2 uses LayerNorm with bias in each Block (not RMSNorm)
  • GELU Activation: GPT-2's FFN uses GELU, not SwiGLU
  • Weight Tying: GPT-2 shares weights between Embedding and LM Head

In subsequent chapters, we will see how these design choices were progressively improved by later models.

6.3A From Logits to the Language Modeling Objective: Why the Output Is [batch, seq, vocab]

The forward pass of the complete GPTModel to logits is not the end. The true optimization objective of language model training is: at each position , use the hidden state at the current position to predict the next token . Therefore, logits[:, t, :] does not predict the token at the current position, but rather the categorical distribution of the token at the next position.

Autoregressive Decomposition

During training, the input and labels are typically shifted by one position:

# tokens: [BOS, The, cat, sat, EOS] inputs = tokens[:, :-1] # [BOS, The, cat, sat] labels = tokens[:, 1:] # [The, cat, sat, EOS] logits = model(inputs) # (batch, seq-1, vocab) loss = F.cross_entropy(logits.reshape(-1, vocab), labels.reshape(-1))

The causal_lm_loss_from_logits(logits, input_ids) function in the Ch06 assignment asks students to implement this step: align logits[:, :-1, :] with input_ids[:, 1:], and use ignore_index to skip padding or positions not participating in training. This function should not use logits[:, 1:, :], nor should it let logits[:, t, :] predict input_ids[:, t].

Teacher Forcing and Parallel Training

The training phase does not involve letting the model generate step-by-step and feeding the results back; instead, it uses the true prefix to predict all next tokens in parallel. This is called teacher forcing. It allows a sequence of length to produce classification tasks in a single forward pass, thus fully leveraging GPU parallelism during training. The inference phase is different: the model can only use the tokens it has already generated, progressively expanding the context.

Softmax is Only Needed in the Loss

The LM Head outputs unnormalized scores, logits. During training, logits are passed directly to F.cross_entropy, because it internally computes log-softmax and negative log-likelihood stably:

Next-token Cross Entropy

Do not apply softmax in the model's forward pass before passing to cross entropy; this would sacrifice numerical stability and lead to worse gradients.

Minimal Example of Causal Leakage

If the causal mask is incorrect, the model might see at position , causing the training loss to decrease unusually easily, but this information is unavailable during inference. A simple check is: fix the prefix, only change future tokens, and compare whether the logits at the prefix positions change.

a = torch.tensor([[10, 20, 30, 40]]) b = torch.tensor([[10, 20, 99, 88]]) # first two tokens are the same, future tokens differ logits_a = model(a) logits_b = model(b) assert torch.allclose(logits_a[:, :2], logits_b[:, :2], atol=1e-5)

This test is not a quality evaluation, but a causal structure check: the output at prefix positions must not depend on future tokens. It ties together the causal mask from Chapter 3, the logits shape from Chapter 6, and the next-token loss from Chapter 7 into a single training objective.

6.3B Worked Example: Manually Computing Label Shift and Loss Denominator for a Batch

Assume a batch contains two samples already padded to length 5. Labels corresponding to PAD are represented by -100, which is passed to F.cross_entropy(..., ignore_index=-100) to be skipped:

sample 1 input_ids: [BOS, I, like, NLP, EOS] sample 2 input_ids: [BOS, GPT, EOS, PAD, PAD]

The model forward pass will still output a vocabulary classification distribution for each input position, with shape [2, 5, |V|]. When computing the next-token loss, only the logits from the first 4 positions are used, and they are tasked with predicting the token shifted one position to the right:

logits used: positions 0 1 2 3 sample 1 pred: BOS -> I, I -> like, like -> NLP, NLP -> EOS sample 2 pred: BOS -> GPT, GPT -> EOS, EOS -> PAD, PAD -> PAD labels before ignore: sample 1 labels: [I, like, NLP, EOS] sample 2 labels: [GPT, EOS, PAD, PAD] labels after padding mask: sample 1 labels: [I, like, NLP, EOS] sample 2 labels: [GPT, EOS, -100, -100]

Therefore, the average loss for this batch is divided by 6 valid prediction positions, not 2 * 4 = 8 positions, nor the original sequence length of 2 * 5 = 10 positions. The logits at the last input position have no corresponding "next token" label, and padding positions do not represent a true language modeling objective.

Batch Mean with ignore_index

This is also why, during implementation, one should first construct shift_logits = logits[:, :-1, :] and shift_labels = input_ids[:, 1:], and then change the padding in shift_labels to ignore_index. If logits[:, 1:, :] is used to predict input_ids[:, 1:], the model would be using the hidden state at position t to predict the token at the same position, causing the training objective to degenerate into "parroting the input token", which is no longer autoregressive language modeling.

6.4 Theory: Weight Tying — Which Models Use It, Which Do Not, and Why?

Weight Tying makes the embedding layer (Token Embedding) and the output layer (LM Head) share the same weight matrix. More precisely, the input side uses the -th row vector to represent token ; the output side computes the dot product of the hidden state with each token's row vector to obtain the vocabulary logits. It is not an inverse mapping in the mathematical sense, but rather the same set of token vectors simultaneously serving the dual roles of "input representation" and "output classification prototype".

Weight Tying Sharing Method

Parameter Count Impact

For GPT-2 small: vocabulary size 50257 × d_model 768 = 38.6M parameters—accounting for 31% of the total model's 124M parameters. With tying, these 38.6M parameters are counted only once.

DeepSeek-V3No129K~0.9B< 0.2%

Why Don't Some Large Models Use It?

As the proportion of non-embedding parameters increases, the parameter savings from Weight Tying decrease: in models with tens to hundreds of billions of parameters, is often no longer the dominant factor in the total parameter count. The benefit of an untied LM Head is that input representations and output classifiers can learn separately, which is suitable for architectures requiring larger output capacity, special vocabulary designs, or multi-modal output interfaces. However, this is not a fixed rule: whether to tie depends on vocabulary size, hidden dimension, total parameter scale, training stability, and the engineering trade-offs of the model family.

Impact on Backpropagation

When tied, the gradient for the embedding layer comes from two paths: one from the embedding lookup path, and the other from the LM Head classification path. Token rows used by the input sequence receive gradients from the lookup path; all token rows appearing in the softmax classifier receive gradients through the output distribution. This coupling sometimes acts as a regularizer, and sometimes limits the ability of the input and output sides to adapt to the task independently. It cannot be simply concluded that "small models always benefit, large models always suffer".

Tied LM Head Gradients

Where is the gradient of the cross entropy with respect to the logits: for valid token rows, it is (softmax - one_hot) / N_valid; rows skipped by ignore_index are all zeros. The tied_lm_head_gradients function in the Ch06 assignment requires students to manually compute these three steps and align them with autograd.

6.6 Pre-training Integrity Check – Forward Pass Test

Before training, you need to confirm that the model's forward pass runs correctly. The key check items here are:

Implementation Points

  • Shape Correctness: The entire chain from token IDs of shape (batch, seq_len) to logits of shape (batch, seq_len, vocab_size) is not broken
  • Numerical Stability: Under random initialization, logits are within a reasonable range (mean ~0, standard deviation ~1-10). If NaN or Inf appears, it indicates an initialization problem
  • Gradient Flow: Gradients from the scalar loss of logits to all parameters can be computed correctly (the .grad of each parameter is not None)
  • Parameter Sharing Verification: If tie_weights=True, confirm that lm_head.weight and token_embedding.weight are the same tensor object

A common cause of training failure is an excessively large initialization standard deviation. GPT-2 uses 0.02, but if 0.1 is mistakenly used, the logits of deep networks may produce NaN. This is because the variance of the residual stream accumulates layer by layer over 12 layers:

Variance Accumulation in Residual Stream

When is too large, the accumulation over layers can exceed the representable range of float16. This is also why std=0.02 in _init_weights is not chosen arbitrarily – it comes from the analysis of residual variance in the GPT-2 paper.

6.6A Engineering Perspective: What a Complete GPT Implementation Must Satisfy Simultaneously

After connecting the GPT components, engineering must answer three more questions: whether the training objective aligns with the data, whether the parameter and memory ledger is explainable, and whether autoregressive constraints are followed during inference. Below, we string together the key implementation points of this chapter into an engineering checklist, which corresponds to model design capability, not just unit tests.

1. Tensor Contract: From token id to next-token loss

The forward contract for a complete decoder-only LM is:

input_ids: [B, T] token + pos emb: [B, T, d_model] blocks output: [B, T, d_model] logits: [B, T, vocab_size]

Here, logits[:, t, :] predicts input_ids[:, t+1], not the token at the current position. Therefore, the core alignment for the training loss is:

shift_logits = logits[:, :-1, :] shift_labels = input_ids[:, 1:] loss = F.cross_entropy( shift_logits.reshape(-1, vocab_size), shift_labels.reshape(-1), ignore_index=-100, )

The last input position has no next-token label; padding or positions not participating in training must be excluded using ignore_index. This alignment error usually does not cause a shape error, but it turns the language modeling objective into "parroting the current position token," making it one of the most insidious errors in beginner implementations.

2. Parameter Ledger: Why GPT-2 small is 124,439,808

The parameter count of GPT-2 small is not a number to memorize, but the result of an architectural ledger. The default configuration is V=50257, T_max=1024, d=768, L=12, d_ff=3072. The main parameter items are:

token embedding: V * d position embedding: T_max * d per block attention: d * 3d + 3d bias + d * d + d bias per block MLP: d * 4d + 4d bias + 4d * d + d bias per block LayerNorm: 4d final LayerNorm: 2d LM head: 0 if tied, V * d if untied

Under weight tying, the LM head and token embedding are the same parameter object, counted only once. A training engineer needs to be able to decompose the total parameters into embedding, position embedding, blocks, final norm, and LM head, because optimizer states, checkpoint size, tensor parallelism partitioning, and memory budget all depend on this decomposition.

3. Weight Tying: Shared Parameters Does Not Mean Only Updating the Target Token

When weights are shared, the output layer logits are written as . The cross-entropy gradient first acts on all vocab logits, then propagates back to the hidden states and the embedding matrix:

Tied LM Head Gradient Path

Therefore, rows not corresponding to target tokens also receive output gradients through the softmax probability. Token rows indexed by the input sequence also receive gradients from the embedding lookup path. Understanding this, weight tying is not just about "saving parameters," but a modeling constraint that forces the input token space and the output classifier to share a geometric structure.

4. Causal mask: Inference constraints must hold during training

During training, teacher forcing allows the model to see the entire sequence at once; however, each position within the model can still only attend to past and current positions. The causal constraint for dense attention is:

allowed[i, j] = j <= i scores[~allowed] = -inf weights = softmax(scores)

If the mask direction is reversed or the broadcast dimension is incorrect, the training loss may decrease abnormally smoothly because the model is peeking at future tokens; however, during inference, future tokens do not exist, and quality will collapse. A small sanity check is to fix the prefix, replace future tokens, and verify that the logits for past positions do not change.

5. Initialization and Positional Encoding: GPT-2 formulas cannot be mixed with modern Llama formulas

GPT-2 style implementations use learned absolute position embeddings, LayerNorm, GELU MLP, linear layers with bias, and the common N(0, 0.02) initialization. Llama/DeepSeek style models commonly use RoPE, RMSNorm, SwiGLU, and bias-free projections. Both model types are decoder-only Transformers, but they differ in parameter count formulas, weight loading methods, and training stability details.

If the input length exceeds max_seq_len, learned position embeddings have no corresponding row, and the model must either raise an error or explicitly extend the position table. This is different from the long-context extrapolation problem with RoPE and cannot be explained using the same reasoning.

6. MoE: Total capacity and per-token computation must be discussed separately

MoE replaces the FFN with multiple experts, and a router selects a few experts for each token. For a bias-free SwiGLU expert:

expert_params = 3 * d_model * expert_hidden total_expert_params = (n_routed_experts + shared_experts) * expert_params activated_expert_params_per_token = (top_k + shared_experts) * expert_params capacity_to_compute_ratio = total_expert_params / activated_expert_params_per_token

This shows that "total model parameters" and "activated computation per token" are not the same metric. In inference engineering, MoE also introduces dispatch/combine, cross-device communication, expert load skew, and tail latency. Therefore, a smaller top-k does not necessarily mean faster, and more experts do not necessarily mean better.

6.6B Checkpoint Compatibility: state_dict is not a bag of tensors

After completing GPTModel.forward(), the next step is not just to save model.state_dict(), but to ensure this checkpoint can be accurately restored by another process, another framework, or an inference service. For training and inference engineers, a checkpoint is an architectural contract: it simultaneously binds the configuration, parameter names, tensor shapes, tokenizer, numerical precision, and forward pass semantics.

1. What a deliverable checkpoint should at least contain

A minimal usable package usually consists of more than just a weight file. Taking a decoder-only LM as an example, engineering requires saving the following simultaneously:

config.json # vocab_size, n_layers, n_heads, d_model, max_seq_len, norm/rope/mlp types model.safetensors # or pytorch_model.bin; parameter names, shapes, and dtype tokenizer.json # or tokenizer.model/merges.txt/vocab.json; vocabulary and special tokens generation_config.json # optional; default max_new_tokens, temperature, top_p, eos_token_id, etc. training_args.json # optional; optimizer/scheduler/data split info needed for continued training

Errors may sometimes be exposed during the shape check, and sometimes be delayed as a quality degradation.

3. Why GPT-2 weight transposition is easy to get wrong

PyTorch nn.Linear(in_features, out_features) has a parameter shape of [out_features, in_features], and its forward pass is equivalent to x @ W.T + b. Early HuggingFace implementations of GPT-2 followed OpenAI's Conv1D naming, where internal weights are often stored in the [in_features, out_features] format. Therefore, when loading HF GPT-2 weights into a custom nn.Linear, many projection matrices need to be transposed.

def copy_linear_from_hf(linear, hf_weight, hf_bias=None, transpose=True): src = hf_weight.t() if transpose else hf_weight assert linear.weight.shape == src.shape linear.weight.data.copy_(src) if hf_bias is not None: assert linear.bias.shape == hf_bias.shape linear.bias.data.copy_(hf_bias) # Example: HF GPT-2 c_attn is a fused QKV projection # If the custom implementation also uses nn.Linear(d_model, 3 * d_model), transpose=True is usually needed

The transposition decision cannot be made by looking at the name alone; you must check the shapes and forward formulas on both sides. If your custom implementation splits Q, K, V into three separate linear layers, you must first slice the HF fused QKV weight along the output dimension into three parts, then copy them to q_proj/k_proj/v_proj respectively.

4. Diagnostic order for logit parity

After loading pretrained weights, the most practical equivalence check is to compare logits for the same batch of input_ids. The recommended diagnostic order starts with the items least affected by training randomness:

  1. Confirm both sides use the same tokenizer and that input_ids are identical.
  2. Confirm config is completely identical: number of layers, hidden size, number of heads, context length, LayerNorm eps, whether bias is used, position encoding type.
  3. Confirm both sides are in eval() mode and dropout is disabled.
  4. Confirm all missing_keys and unexpected_keys for loaded parameters can be explained; do not treat missing key projection layers as harmless warnings.
  5. Compare hidden states layer by layer: after embedding, after each block, after final norm, after lm head, to find the first position with a large error.
  6. If the error starts after embedding, prioritize checking token embedding, position embedding, and position ids; if the error starts after attention, prioritize checking QKV order, mask direction, head reshaping, and projection transposition.

Under fp32, same tokenizer, same architecture, and no dropout, the maximum absolute logit error for GPT-2 small should typically be close to the floating-point rounding error magnitude. If the error reaches 1e-2 or larger, it is usually not "the pretrained model inherently has differences," but rather that the loading mapping or forward implementation still has semantic inconsistencies.

🔑
Section Conclusion: The maturity of a model implementation is reflected in its ability to bridge the gap between "can run with random initialization" and "can equivalently load pretrained weights." The former verifies shapes and basic gradients; the latter verifies the complete model contract formed by naming, layout, position encoding, normalization, mask, tokenizer, and numerical mode.

6.6C Modern Decoder Architecture Ledger: Reading Dense, MoE, Long-Context, and Multimodal Model Reports

The GPT-2 implementation in this chapter is a clean pedagogical baseline. Modern industrial model reports, however, rarely describe a model with only n_layers, n_heads, and d_model. They usually combine dense or sparse FFN, RoPE variants, GQA/MLA-style attention compression, active-versus-total parameter accounting, multimodal token fusion, thinking-budget controls, and checkpoint/export constraints. A high-level engineer must be able to translate these claims into a concrete resource and compatibility ledger.

For example, DeepSeek-V3 reports a 671B-total-parameter MoE model with 37B activated parameters per token, MLA attention, auxiliary-loss-free load balancing, and multi-token prediction. Qwen3 mixes dense and MoE models and treats thinking/non-thinking behavior as a unified inference-time control problem. Llama 4 moves further by combining MoE, native multimodality, long-context design, and teacher-model distillation. These are not isolated features; each changes the tensor contract, training run ledger, and serving constraints.

1. Do not compare model size with a single number

Dense and MoE models use parameter counts differently. A dense 70B model stores and activates roughly the same set of parameters for every token. A sparse MoE model may store hundreds of billions of parameters but activate only a subset per token. This means total parameters, active parameters, KV cache, communication cost, and latency must be reported separately.

6.8 Theory: MoE Architecture — Decoupling Capacity and Per-Token Computation

Mixture of Experts (MoE) is the most important extension beyond the GPT-2 architecture for LLMs. It replaces the FFN in each layer with multiple "expert" networks, activating only a few of them per token, achieving sparse activation—a large total parameter count but a forward pass computation cost dependent only on the activated parameters.

Core Design of DeepSeekMoE

MoE Forward Computation

Taking the common DeepSeekMoE-style design as an example, the course focuses only on transferable structural points:

  • Multiple Routed Experts: The router selects the top-K experts for each token, rather than activating all experts.
  • Shared Experts: All tokens pass through a small number of shared experts to preserve a general transformation path.
  • Fine-grained Experts: A single expert is typically smaller than a dense FFN, but the total number of experts is larger.
  • Core Issues: Total parameter count determines the capacity upper bound; the number of activated experts per token determines the single forward pass computation cost; load balancing determines throughput stability.

Why Does Activating Only a Few Experts Still Work?

The core assumption of MoE is conditional computation: different tokens do not need to pass through exactly the same FFN parameters. The router selects experts based on the hidden state, directing different input patterns—such as mathematical symbols, code snippets, entity names, and format tokens—to different parameter subsets. Whether this assumption holds depends on the training data, the router's representational capacity, expert capacity, and load balancing; if the router collapses onto a few experts, MoE degenerates into an expensive and unstable approximation of a dense model.

🔑
Core Trade-off: MoE uses more total parameters to increase capacity and top-K sparse activation to control per-token computation. However, it introduces additional problems: router training, expert load balancing, cross-device communication, and intra-batch token dispatch; these system costs must be analyzed together with the capacity benefits.

The moe_parameter_budget in the assignment will formalize this trade-off into a computable parameter budget. If each expert is a bias-free SwiGLU FFN, the parameter count for a single expert is 3 * d_model * d_ff, corresponding to the three linear matrices: gate, up, and down. The total expert parameters for MoE need to multiply by all routed experts and shared experts; the activated expert parameters per token only multiply by the top_k routed experts and shared experts. The router itself also contains d_model * n_experts gating parameters. This clearly distinguishes "model capacity comes from all experts" from "single token computation only passes through a few experts".

Worked Example: Manual Calculation of MoE Parameter Budget and Load Bias

Consider a toy MoE layer using d_model=16, d_ff=32, 8 routed experts, top_k=2, and 1 shared expert. Each bias-free SwiGLU expert has 3 * 16 * 32 = 1536 parameters: the gate, up, and down matrices.

Now consider load balancing. If 4 experts receive token counts [80, 10, 5, 5] in a batch, the ideal load is 0.25, and the actual loads are [0.80, 0.10, 0.05, 0.05]. The dynamic bias update bias += lr * (target_load - load_ratio), with lr=0.04, gives an increment for expert 0 of 0.04 * (0.25 - 0.80) = -0.022, and for expert 2 of 0.04 * (0.25 - 0.05) = 0.008. Therefore, overloaded experts are less likely to be selected in the next step, while underloaded experts are more likely to enter the top-K; this is precisely the directional requirement of AuxLossFreeBalancer.update.

Router Probabilities, Top-k Normalization, and Gradient Path

The MoE router first projects the hidden state of each token into expert logits:

Router logits and probabilities

Top-k routing only retains the experts with the highest probabilities. Since the probability mass of the discarded experts no longer participates in the output, the retained weights must be re-normalized:

Top-k mixture weights

This step explains why the assignment checks top_weights.sum(dim=-1) == 1. Without re-normalization, the expert outputs for different tokens would be scaled by different residual probability masses, causing the model's residual stream scale to vary with the router distribution, making training more unstable.

Worked Example: MoE Load-Balancing Loss Considers Both Probability and Assignment

Suppose the router probabilities for 4 tokens across 3 experts are [0.7,0.2,0.1], [0.6,0.3,0.1], [0.1,0.8,0.1], and [0.2,0.2,0.6], and the top-1 selected expert ids are [0,0,1,2]. The actual assignment proportion load is [0.5,0.25,0.25]; the average router probability is [0.4,0.375,0.225]. The Switch-style auxiliary loss can be written as n_experts * sum_i load_i * prob_i, so for this example it is 3 * (0.5*0.4 + 0.25*0.375 + 0.25*0.225) = 1.05.

The purpose of this loss is not to make each token activate all experts equally, but to penalize batch-level router collapse. Looking only at top-k counts ignores whether the router softmax is already biased towards a few experts; looking only at the average probability ignores whether the actual top-k assignment is already congested. Therefore, the moe_load_balance_loss in the Ch06 assignment returns load_fraction, mean_router_prob, and the loss simultaneously.

Capacity, Token Dispatch, and Load Imbalance

Parameter budget only answers "how many experts are theoretically activated per token," but does not answer "whether a batch can execute smoothly." In practice, the MoE layer needs to dispatch tokens to experts. If each expert has a capacity of tokens, and expert receives tokens, then:

Expert overflow

Even without explicit token dropping, extreme imbalance will cause the busiest expert to determine the entire layer's latency, while other devices or experts wait idly. Therefore, the quality and throughput of MoE depend not only on top-k, but also on load distribution, capacity factor, expert parallelism communication, and the implementation of dispatch/combine kernels.

This example illustrates that MoE's "fewer activated parameters per token" does not automatically imply "faster training and inference."

class MoERouter(nn.Module): """DeepSeekMoE Router — Top-K Gating""" def __init__(self, d_model, n_experts=256, top_k=8): super().__init__() self.n_experts = n_experts self.top_k = top_k self.gate = nn.Linear(d_model, n_experts, bias=False) def forward(self, x, bias=None): """ x: (batch, seq_len, d_model) bias: (n_experts,) or None — dynamic bias Returns: (weights, indices) each (b, s, top_k) """ logits = self.gate(x) # (b, s, n_experts) if bias is not None: logits = logits + bias.unsqueeze(0).unsqueeze(0) weights = F.softmax(logits, dim=-1) # (b, s, n_experts) # Top-K selection top_weights, top_indices = torch.topk( weights, self.top_k, dim=-1 ) # each (b, s, top_k) # Re-normalize top-k weights top_weights = top_weights / top_weights.sum(dim=-1, keepdim=True) return top_weights, top_indices class AuxLossFreeBalancer: """Aux-Loss-Free Load Balancer — DeepSeek-V3 Uses dynamic bias instead of an auxiliary loss to balance expert load. Decreases bias for overloaded experts (reducing their selection probability), increases bias for underloaded experts. """ def __init__(self, n_experts, bias_update_rate=0.01, bias_clip=0.1): self.n_experts = n_experts self.bias_update_rate = bias_update_rate self.bias_clip = bias_clip self.bias = torch.zeros(n_experts) def update(self, token_counts): """Update bias based on token counts for each expert""" total = token_counts.sum().float() if total == 0: return load_ratio = token_counts / total # Actual load ratio target_load = 1.0 / self.n_experts # Ideal load delta = self.bias_update_rate * (target_load - load_ratio) self.bias += delta self.bias.clamp_(-self.bias_clip, self.bias_clip) def apply_bias(self, gate_logits): """Apply dynamic bias to routing logits""" return gate_logits + self.bias.to(gate_logits.device) def get_load_stats(self, token_counts): """Return load balancing statistics""" total = token_counts.sum().float() load_ratio = token_counts / total if total > 0 else token_counts ideal = 1.0 / self.n_experts return { 'max_load': load_ratio.max().item(), 'min_load': load_ratio.min().item(), 'std_load': load_ratio.std().item(), 'ideal_load': ideal, } # Integration test router = MoERouter(768, n_experts=256, top_k=8) balancer = AuxLossFreeBalancer(256) x = torch.randn(4, 64, 768) # Simulate training steps for step in range(10): weights, indices = router(x, balancer.bias) # Statistics token_counts = torch.zeros(256) token_counts.scatter_add_(0, indices.flatten(), torch.ones_like(indices.flatten())) balancer.update(token_counts) if step in [0, 4, 9]: stats = balancer.get_load_stats(token_counts) print(f"Step {step}: max={stats['max_load']:.6f}, " f"min={stats['min_load']:.6f}, std={stats['std_load']:.6f}")

💡 Dynamic Bias vs. Auxiliary Loss: The auxiliary loss fine-tunes router parameters via gradient backpropagation to balance load, which has low computational cost but introduces an additional hyperparameter α. Dynamic bias does not go through backpropagation—it is a direct adjustment to the routing logits during the forward pass. DeepSeek-V3 reports that, under its training setup, the balancing effect achieved by dynamic bias is comparable to that of the auxiliary loss, and it reduces the need for tuning the auxiliary loss weight; however, actual measurements are still needed for different models and data distributions.
