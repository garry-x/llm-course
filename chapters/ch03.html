3.1 为什么需要注意力？

在 Transformer 出现之前，序列建模的主要工具是 RNN/LSTM。它们的工作原理可以概括为"迭代式压缩"：逐个 token 读取输入，将"已读到的全部信息"压缩到一个固定维度的隐状态 中：

这种设计有两个根本性问题：

  • 顺序计算瓶颈：第 步必须等第 步完成才能开始。无法并行化训练，序列越长越慢。
  • 长程依赖困境：隐状态是一个"信息漏斗"——当信息从位置 1 传到位置 1000 时，中间经历了 999 次非线性变换的"挤压"。梯度要么消失（Vanishing Gradient）要么爆炸（Exploding Gradient），导致模型几乎无法捕捉远程依赖。LSTM 和 GRU 用门控机制做了缓解，但并未从根本上解决问题。

直观想象一下：你读一篇 1000 字的文章来回答一个涉及开头句的问题。RNN 需要从头读到尾，"记忆"要在千步压缩中幸存；而注意力机制让你直接"回看"开头，用检索代替记忆。

🔑
注意力本质：不是记忆，而是检索。每个位置主动从所有位置中查询相关信息，用"软对齐"（soft alignment）的方式聚合全局信息。这避开了顺序压缩的根本缺陷——信息不经过中间层挤压，直接被提取。

3.2 注意力直觉：查询、匹配、聚合

Scaled Dot-Product Attention 可以用一个非常直观的三步流程来理解：

第一步：查询（Query）——"我想找什么？"

序列中每个 token 都会生成一个"问题向量" ，表示"我在这个位置上需要什么信息"。

第二步：匹配（Match）——"谁有我要的信息？"

每个 token 也生成一个"标签向量" ，表示"我有什么信息可以提供"。用 衡量匹配程度 → 这就是注意力分数。

第三步：聚合（Aggregate）——"把信息拿过来"

将匹配分数通过 softmax 归一化到概率分布，然后用这个分布对所有位置携带的"内容向量" 做加权平均：

这个公式就是 Transformer 一切魔力的源头。

💡
为什么叫"自"注意力（Self-Attention）？因为 都来自同一个输入序列 ——每个 token 在自己和自己之间做匹配。而在 Encoder-Decoder Attention 中，Q 来自解码器，K、V 来自编码器，那是"交叉注意力"。

3.3 动手：QKV 线性投影

在实现完整注意力之前，我们先实现三个线性投影。每个 token 的嵌入向量 通过三个不同的权重矩阵投影到查询、键、值空间：

其中 ， 是注意力头维度（通常 ）。

import torch import torch.nn as nn import torch.nn.functional as F class QKVProjection(nn.Module): """Q, K, V 线性投影层。""" def __init__(self, d_model: int, d_k: int): super().__init__() self.W_q = nn.Linear(d_model, d_k, bias=False) self.W_k = nn.Linear(d_model, d_k, bias=False) self.W_v = nn.Linear(d_model, d_k, bias=False) def forward(self, x: torch.Tensor): """ x: (batch, seq_len, d_model) 返回: q, k, v 各 (batch, seq_len, d_k) """ q = self.W_q(x) k = self.W_k(x) v = self.W_v(x) return q, k, v # 例: d_model=768, d_k=64 for 12 头的多头注意力 proj = QKVProjection(d_model=768, d_k=64) x = torch.randn(2, 16, 768) q, k, v = proj(x) print(f"Q: {q.shape}, K: {k.shape}, V: {v.shape}")

输出：

Q: (2, 16, 64), K: (2, 16, 64), V: (2, 16, 64)

每个投影矩阵的形状都是 ，参数量为 。在 GPT-2 124M 中，，所以 QKV 投影共 参数——注意这只是单头的参数；12 个头合起来将乘以 12。

注意 bias=False 这个细节——GPT-2 和 Llama 系列的注意力投影均不使用偏置。原因是后续的 LayerNorm 已经处理了均值偏移，额外的偏置反而增加了不必要的参数量，且实验表明去掉偏置不影响模型质量。

3.4 动手：Scaled Dot-Product Attention

现在我们实现完整的单头缩放点积注意力。整个计算只有不到 10 行核心代码：

class ScaledDotProductAttention(nn.Module): """单头缩放点积注意力。""" def __init__(self, d_k: int, dropout: float = 0.0): super().__init__() self.d_k = d_k self.dropout = nn.Dropout(dropout) def forward(self, q: torch.Tensor, k: torch.Tensor, v: torch.Tensor, mask: torch.Tensor = None): """ q, k, v: (batch, seq_len, d_k) mask: (batch, 1, seq_len) 或 (batch, seq_len, seq_len) — 可选 返回: (batch, seq_len, d_k), 注意力权重 (batch, seq_len, seq_len) """ # 1. 缩放点积: 计算注意力分数 scores = torch.matmul(q, k.transpose(-2, -1)) # (b, seq, seq) scores = scores / (self.d_k ** 0.5) # 缩放: 除以 √d_k # 2. 应用掩码（可选）— 将掩码位置设为 -inf if mask is not None: scores = scores.masked_fill(mask == 0, float('-inf')) # 3. Softmax 归一化 → 注意力权重 attn_weights = F.softmax(scores, dim=-1) # (b, seq, seq) # 4. Dropout attn_weights = self.dropout(attn_weights) # 5. 加权聚合: 用权重对 V 加权求和 output = torch.matmul(attn_weights, v) # (b, seq, d_k) return output, attn_weights # 使用示例 d_k = 64 batch, seq_len = 2, 16 q = torch.randn(batch, seq_len, d_k) k = torch.randn(batch, seq_len, d_k) v = torch.randn(batch, seq_len, d_k) attn = ScaledDotProductAttention(d_k) out, weights = attn(q, k, v) print(f"输出: {out.shape}") # (2, 16, 64) print(f"注意力权重: {weights.shape}") # (2, 16, 16) print(f"第 0 个样本的注意力矩阵:\n{weights[0]}")

这五步——点积 → 缩放 → 掩码 → softmax → 加权聚合——就是整个注意力机制的完整计算。让我们逐行解析：

  • torch.matmul(q, k.transpose(-2, -1))：每对 的点积构成分数矩阵 。
  • scores / sqrt(d_k)：除以 防止点积过大（详见 3.5 节）。
  • masked_fill(mask == 0, -inf)：将不需要的位置设为 ，这样 softmax 后这些位置的权重为 0。
  • F.softmax(scores, dim=-1)：对每个 query，将所有 key 的分数归一化为概率分布。
  • matmul(attn_weights, v)：用注意力权重对 value 做加权平均——这是"信息聚合"步骤。

注意 dropout 应用在 softmax 之后、加权聚合之前——这相当于随机丢弃一些注意力连接，起到正则化作用，防止某些注意力模式过度固化。原版 Transformer 中 dropout = 0.1。

这个缩放因子看似是一个小细节，但它对训练的稳定性至关重要。原因如下：

问题出在后续的 softmax 上。Softmax 对指数级大的输入非常敏感：

如果 的绝对值很大， 会变得极大（或极小）。假设 且某个 token 恰好与自身匹配得分达到 11.3×2 = 22.6（2 个标准差），——这个 token 的 softmax 输出将接近 1，其他所有 token 接近 0。这就是"注意力坍缩"：

  • 梯度变得极小——接近 1 的位置 softmax 导数接近 0
  • 每个 token 只关注自己（或某个单一 token），失去了全局信息聚合能力
  • 训练初期就陷入尖锐的注意力模式，难以恢复

分数稳定在 量级，softmax 不会进入饱和区，梯度保持良好。这就是缩放点积注意力（Scaled Dot-Product Attention）中"Scaled"的含义。

3.6 Softmax 温度与梯度

与 缩放相关的一个概念是 温度（Temperature）。在注意力计算中引入温度参数 ：

温度 控制 softmax 输出的"尖锐程度"：

  • ：softmax 退化为 one-hot（argmax 的软近似）——只关注最相关的 token
  • ：softmax 退化为均匀分布——所有 token 被同等对待
  • ：标准缩放——保持梯度良好的平衡点

在训练中，温度固定为 。但在推理时（第 8 章），我们可以调节生成采样时的温度来控制文本的创造力。

def attention_with_temperature(q, k, v, temp=1.0, mask=None): """带可调温度的注意力——用于演示温度对分布的影响。""" d_k = q.size(-1) scores = torch.matmul(q, k.transpose(-2, -1)) / temp if mask is not None: scores = scores.masked_fill(mask == 0, float('-inf')) attn_weights = F.softmax(scores, dim=-1) return torch.matmul(attn_weights, v), attn_weights # 不同温度下注意力权重分布的熵 d_k = 64 q = torch.randn(1, 1, d_k) k = torch.randn(1, 5, d_k) for temp in [0.1, 1.0, 8.0]: _, weights = attention_with_temperature(q, k, temp=temp * (d_k ** 0.5)) entropy = -(weights * torch.log(weights + 1e-8)).sum(dim=-1).mean() print(f"temp={temp:.1f} | 分布熵: {entropy.item():.3f}")

输出显示：低温时熵低（尖锐分布），高温时熵高（均匀分布）。 缩放的本质就是为注意力设置一个"恰当的温度"，使 softmax 既不坍缩也不过度平滑。

3.7 动手：Causal Mask

在 GPT 这样的自回归（Autoregressive）语言模型中，每个 token 只能"看到"它之前的 token（包括自己），而不能偷看未来的 token。这就是 Causal Mask（因果掩码）——一个上三角矩阵：

def create_causal_mask(seq_len: int, device: str = 'cpu'): """ 创建因果掩码: 上三角全为 0 (被掩码), 下三角 (包括对角线) 为 1 (可见) 返回: (1, 1, seq_len, seq_len) 可广播到 (batch, n_heads, seq_len, seq_len) """ mask = torch.tril(torch.ones(seq_len, seq_len, device=device)) # mask[i,j] = 1 ⇔ j ≤ i (token i 可以看到 token j) return mask.unsqueeze(0).unsqueeze(0) # (1, 1, seq_len, seq_len) seq_len = 4 mask = create_causal_mask(seq_len) print(f"Causal Mask ({seq_len}×{seq_len}):\n{mask[0,0]}") # 在注意力中使用 def causal_attention(q, k, v): d_k = q.size(-1) scores = torch.matmul(q, k.transpose(-2, -1)) / (d_k ** 0.5) mask = create_causal_mask(q.size(-2), q.device) scores = scores.masked_fill(mask == 0, float('-inf')) return F.softmax(scores, dim=-1) @ v q = k = v = torch.randn(1, 4, 64) out = causal_attention(q, k, v) print(f"Causal attention 输出: {out.shape}")

输出：

Causal Mask (4×4): tensor([[1., 0., 0., 0.], [1., 1., 0., 0.], [1., 1., 1., 0.], [1., 1., 1., 1.]]) Causal attention 输出: (1, 4, 64)

关键观察：

  • 第 0 行：[1,0,0,0]——第一个 token 只能看自己
  • 第 1 行：[1,1,0,0]——第二个 token 可以看前两个
  • 第 3 行：[1,1,1,1]——最后一个 token 可以看全部

这是自回归生成的核心约束——每个位置的输出只依赖于它之前（包括它自己）的 token。这也是为什么 GPT 训练不需要标注数据——文本本身就是标签，每个 token 的任务是预测它后面那个 token。

⚠️
常见的困惑：在训练时我们同时输入整个序列，但 causal mask 确保位置 的输出 不会"看到"位置 的输入。这就是所谓的"并行训练 + 自回归约束"——我们利用因果掩码在单次前向传播中完成所有位置的预测，同时保持自回归的正确性。

3.8 注意力权重可视化

注意力权重矩阵 是非常有用的可解释性工具。每一行 表示位置 如何分配对各个位置的注意力：

# 假设训练好的模型，某层的注意力权重 # 展示一个长度为 20 的句子的注意力模式 sentence = ["", "大", "语", "言", "模", "型", "深", "度", "学", "习", "课", "程", "从", "零", "开", "始", "构", "建", "GPT", ""] seq_len = len(sentence) # 模拟训练好的注意力权重（实际应来自真实模型） attn_weights = torch.softmax( torch.randn(seq_len, seq_len) * 0.5 + torch.eye(seq_len) * 2.0 + # 对角线更强（自己关注自己） torch.tril(torch.ones(seq_len, seq_len), -1) * -1e9, # 因果掩码 dim=-1 ) # 分析注意力模式 for i in [0, 5, 10, 15]: top_k = attn_weights[i].topk(3) top_indices = top_k.indices.tolist() top_tokens = [sentence[idx] for idx in top_indices] print(f"位置 {i} ({sentence[i]}) 最关注: {list(zip(top_tokens, [f'{w:.2f}' for w in top_k.values.tolist()]))}")

在实际模型中，注意力权重通常显示出几种典型的模式：

  • 对角线主导：每个 token 最关注自己（占 30-60% 的权重）
  • 邻近偏好：相邻的几个 token 获得更多关注
  • 句法对齐：动词关注它的主语/宾语，代词关注它指代的名词
  • 特殊 token：<BOS> 或 <EOS> 往往被广泛关注

注意力可视化在研究模型中非常常见——通过观察注意力头关注的位置，研究人员可以判断模型是否学会了句法结构、共指消解等语言能力。例如，"it"的注意力会集中在其指代的名词上，即使两者相隔数十个位置——这正是注意力机制解决长程依赖的直观证据。

3.9 O(n²) 挑战与 DeepSeek MLA 预告

标准的 Full Attention 有一个致命弱点：计算复杂度是 O(n²)，其中 是序列长度。为什么？

Q 和 K 的矩阵乘法产生一个 的分数矩阵——每个位置与所有位置都要计算匹配度。当 时，注意力矩阵约 1600 万元素；当 时，约 160 亿——这已经远超 GPU 显存的承载能力。

这对 LLM 意味着什么？

  • 长上下文成本高昂：128K 上下文的模型，注意力计算可能占推理时间的 70% 以上
  • KV Cache 膨胀：推理时需要缓存每个 token 的 K 和 V 向量，序列越长缓存越大
  • 训练受限：O(n²) 的内存使长序列训练在现有硬件上极其昂贵

应对策略百花齐放：

  • 稀疏注意力（Sparse Attention）：只计算部分位置的注意力（如 Sliding Window、Strided Pattern）
  • 线性注意力（Linear Attention）：用核技巧将复杂度降到 O(n)
  • Flash Attention（第 4 章详述）：不改变数学结果，但通过 IO-aware tiling 大幅减少显存读写
  • DeepSeek MLA（Multi-head Latent Attention）：对 KV Cache 进行低秩压缩，将每个 token 的 KV 维度从 压缩到 ——这是第 4 章的压轴内容
🔑
线索预告：标准 MHA 中，KV Cache 占用 = 。以 70B 模型为例（），单序列 KV Cache 约 512 MB。而 MLA 可以将这个值压缩 4-8 倍，是 DeepSeek-V2/V3 支持 128K 上下文的关键技术之一。
