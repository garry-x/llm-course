2.1 From One-Hot to Dense Embeddings — Why Distributed Representations Are Needed

Chapter 1 converted text into a token sequence , where each . The question now is: how to convert this integer into a vector that a neural network can process?

One-Hot Encoding and Its Problems

The most straightforward idea is One-Hot encoding: map token to a vector of length , where the -th bit is 1 and the rest are 0. For example, with a vocabulary size of 5, "apple" (token 3) is encoded as .

But One-Hot has three fundamental problems:

1. Curse of Dimensionality. With a typical vocabulary , each token becomes a 50000-dimensional vector. Storing a batch (32 x 1024) of One-Hot vectors requires floats — about 6.4 GB, completely unacceptable. This is not just a storage problem: in high-dimensional space, distances between all points tend to become equal, and the concept of "near" and "far" disappears (this is a manifestation of the curse of dimensionality in distance metrics).

2. Semantic Flatness. The cosine similarity between any two different One-Hot vectors is 0. "king" and "queen" are as "dissimilar" in One-Hot space as "king" and "apple" — the model cannot extract any semantic relationships from the representation.

3. Extremely Low Parameter Efficiency. Multiplying a One-Hot vector by a weight matrix is equivalent to looking up the -th row. So why not just use a lookup table directly?

Distributed Representations: The Core Breakthrough

The solution is learnable dense embeddings: a matrix (where ) maps tokens to a low-dimensional continuous space:

This is Token Embedding—equivalent to looking up the -th row in a table. The key difference is: embedding vectors are trainable. Through language model training, semantically similar tokens will obtain similar vectors.

Distributional Hypothesis

This is the theoretical foundation of distributed representations—"You shall know a word by the company it keeps" (Firth, 1957). Words with similar meanings appear in similar contexts, so they should have similar vector representations. In the embedding space:

  • "king" and "queen" appear in positions (contexts) with significant overlap → vectors are close
  • "king" and "apple" have contexts that barely overlap → vectors are far apart
  • Analogical reasoning: —common in static word vectors like word2vec/GloVe, but it is an empirical phenomenon formed by the corpus, training objective, and evaluation method, not a mathematical property automatically guaranteed by the embedding layer

SGNS turns co-occurrence into a binary classification training signal

Assume in a certain batch, the dot product of the positive example (center=king, context=queen) is 1.5, and the dot products of two negative samples apple and another random word are -0.4 and 0.2 respectively. The negative log-likelihood for this sample is:

The larger the positive example dot product, the smaller the first term loss; the smaller the negative sample dot product, the smaller the latter two terms loss. Training does not directly provide the label "king and queen are very similar," but through many local context binary classification samples, words that frequently appear in similar contexts obtain similar vectors.

Analogy reasoning is a diagnosis of spatial structure, not a guarantee from the objective function

The meaning of the 3CosAdd query is: subtract the direction related to man from the vector of king, then add the direction related to woman, and finally find the candidate with the highest cosine similarity. This process usually also excludes the input words king, man, woman; otherwise, the nearest neighbor might just be the original word itself.

Successful analogy depends on many conditions: the relevant relationships in the corpus are sufficiently systematic, the vectors are appropriately trained, frequency and polysemy do not dominate the distance, and the evaluation set accepts a single word answer. Therefore, this course treats analogy tasks as a check for "whether the vector space has learned a certain type of linear regularity," rather than as sufficient proof of semantic understanding.

2.1C Embedding Space Diagnostics: Similarity, Anisotropy, and Frequency Effects

After training an embedding matrix, one should not assume the representation is "well-learned" just because the loss has decreased. This course will repeatedly emphasize: the geometric structure of the vector space arises from the training objective and data distribution, and it also brings biases and failure modes. For LLM engineers, embedding diagnostics are early signals for discovering issues with the tokenizer, data mix, initialization, and weight tying.

1. Row norm is not semantic strength

Each row in the embedding matrix E[V, d] corresponds to a token. A common phenomenon after training is that high-frequency tokens have larger row norms because they are updated more often; special tokens, spaces, punctuation, and common subwords may also form distinct clusters. A large norm does not mean greater semantic importance; it often mixes frequency, optimization history, and output layer gradients.

diagnostics: row_norm_mean / row_norm_p95 top_norm_tokens low_norm_tokens norm_by_frequency_bucket special_token_norms

If the norms of newly added special tokens are significantly lower than those of common tokens, it may indicate insufficient training after initialization; if the norm of the padding token is heavily updated, it usually suggests a problem with the loss mask or attention mask.

2. Cosine similarity must first handle scale

Cosine similarity separates vector direction and magnitude, making it a common tool for observing semantic neighbors, analogies, and clusters. In the Ch02 assignment, cosine_similarity_matrix first normalizes each row, then computes normalized @ normalized.T. This step is important: if dot product is used directly, high-norm, high-frequency tokens will dominate the nearest neighbor results.

However, cosine is not a universal explanation. The input embedding of an LLM is only the first layer representation; the final hidden state of the same token in different contexts can be completely different. Therefore, "close in embedding space" only indicates that the initial token representations are similar, not that the model will ultimately treat the two words as synonyms.

3. Anisotropy can degrade nearest neighbors

High-dimensional embedding spaces often exhibit anisotropy: many vectors point towards certain common directions, causing the cosine similarity between any two tokens to be artificially high, and nearest neighbors to be dominated by frequency, format, or common directions. A simple diagnostic is to analyze the distribution of off-diagonal cosine values:

cosine matrix excluding diagonal: mean close to 0 -> space is relatively balanced mean high positive -> possible common direction or frequency dominance p95 very high -> local clusters or repeated tokens too strong

Research often uses mean-centering, principal component removal, or whitening to analyze static word vectors, but in engineering, do not arbitrarily apply these transformations to a training LLM's embedding. The goal of diagnostics is to understand the source of the problem, not to modify model parameters without validation.

4. Analogy success requires systematic relationships, not individual examples

The form of 3CosAdd, v_b - v_a + v_c, assumes that a certain relationship can be approximately represented as a linear direction. For example, gender, capital-country, or morphological changes may form directions in some corpora. However, one successful example does not prove the model understands the relationship; one failed example does not prove the embedding is devoid of semantics. A rigorous approach is to construct a set of queries by relationship type, report top-k accuracy, and check if errors stem from frequency, subword segmentation, polysemy, or evaluation set ambiguity.

Diagnostic ItemWhat it can indicateWhat it cannot indicate
Nearest NeighborsWhether local directions cluster related tokensCannot prove contextual semantic equivalence
Analogy top-kWhether a certain linear relationship exists systematicallyCannot cover reasoning, factuality, or polysemy
Norm DistributionWhether updates for frequency, mask, special tokens are abnormalCannot explain token importance alone
Off-diagonal CosineWhether the space is dominated by a common directionCannot directly provide a fix

5. Engineering checkpoints for LLM embeddings

When training or fine-tuning a model, it is recommended to run embedding diagnostics at least at the following times: after a tokenizer change, after adding special tokens, after enabling/disabling weight tying, before and after SFT starts, and when training loss is abnormal. Pay special attention to:

  • Whether new tokens are being trained: Do special tokens and domain terms have non-zero gradients and reasonable norms?
  • Whether padding leaks: Is the PAD row updated by the loss? Does the attention mask cause the model to attend to padding?
  • Whether language/domain is imbalanced: Are tokens for minority languages or code frequently fragmented and have low norms?
  • Whether weight tying is consistent: When input embedding and LM head are shared, the output gradient updates all softmax rows; when not shared, the two spaces may gradually diverge.
🔑
Section Conclusion: Embeddings are not static semantic dictionaries, but parameter matrices shaped jointly by the training objective, tokenizer, frequency, and output layer. The value of diagnosing the embedding space lies in discovering systemic issues, not in using a single nearest neighbor or analogy example to prove the model "understands language."

2.1D From Static Word Vectors to Contextualized Representations: Why the Same Token Can Have Different Semantics

Word2vec and GloVe learn static vectors: each word or subword in the vocabulary typically corresponds to a fixed vector. The input embedding of a modern Transformer is also a fixed lookup table matrix, but the representation the model actually uses for prediction, classification, or generation is not this initial row vector; it is the hidden state after multiple layers of attention and FFN. This distinction is key to understanding BERT, GPT, RAG rerankers, and in-context learning.

1. Input embedding is just the layer-0 representation

Given a token sequence x_1,...,x_T, the layer-0 representation is usually written as:

Contextual representation

Therefore, E[token] for the same token is identical across all contexts, but h_i^{(l)} changes with the surrounding context, position, attention pattern, and computations from previous layers. The static embedding provides a trainable starting point for the model; the contextualized representation is the model's interpretation of that position within the current sentence.

2. Worked Example: Two Contexts for "bank"

Consider two sentences:

S1: I deposited money at the bank. S2: I sat near the river bank.

If the tokenizer maps both instances of bank to the same token id, then the lookup table yields the exact same E[bank]. The difference comes from the context: in S1, tokens like money and deposited will write financial semantics into the hidden state at the bank position via attention; in S2, tokens like river and near will write geographical/riverbank semantics into the same position. At higher layers, the hidden states of the two bank instances may be far apart, even though their input embeddings are the same row.

3. Why This Changes Evaluation Methods

Static word vectors are commonly evaluated using nearest neighbors, analogies, and word similarity datasets because the model's output is a single word vector space. Contextualized models must be evaluated by task and layer: classification uses the [CLS] or pooled representation, extractive QA uses span logits, and decoder-only LMs use the next-token distribution from the hidden state at each position passed through the LM head. One cannot judge whether the entire LLM understands polysemy, anaphora, or reasoning relationships solely by looking at the nearest neighbors of the input embedding.

ObjectComputation LocationSuitable QuestionsUnsuitable Questions
Input embedding rowE[token]Token initialization, frequency, special tokens, vocabulary geometryActual semantics in the current sentence
Layer hidden stateh_i^{(l)}Contextual semantics, syntactic role, task-relevant featuresCannot interpret model capability independent of layer and task
Output logitsh_i W_{out}Next token distribution, generation behavior, CE lossCannot alone explain intermediate mechanisms causally
Attention weightssoftmax(QK^T)Information routing diagnostics, candidate paths for long-range dependenciesCannot alone prove explanation or causal contribution

4. Engineering Implication: Don't Mistake Embedding Similarity for Final Semantics

In fine-tuning, RAG reranking, or safety evaluation, if the nearest neighbors of a token's input embedding seem reasonable, it only indicates that the initial representation has no obvious anomaly. To truly understand model behavior, one must examine contextual hidden states, logits, task outputs, and failure examples. For instance, a domain term might be close to general words in embedding space but still be correctly extracted after contextualization; conversely, a newly added special token might have a normal embedding norm, yet the model may fail to use it as intended due to errors in the chat template or loss mask.

🔑
Core Judgment: Static word vectors answer "who this word resembles in corpus statistics"; contextualized representations answer "what role this position plays in the current context." LLM engineers need to understand both; otherwise, they can easily conflate embedding diagnostics, attention diagnostics, and final task quality.

2.2A Worked Example: embedding lookup, scaling, and tied LM head

Let vocabulary size V=3, embedding dimension d=2, the embedding matrix is:

E = [[1, 0], [0, 2], [3, 4]]

Input token ids are [[0, 2], [1, 0]], direct lookup yields a tensor of shape [B=2, T=2, d=2]:

E[token_ids] = [[[1, 0], [3, 4]], [[0, 2], [1, 0]]]

Explicit one-hot multiplication is completely equivalent: the shape of one_hot(token_ids) is [2,2,3], and after multiplying with E[3,2], it yields [2,2,2]. Actual models do not construct this large one-hot tensor because lookup is more memory and bandwidth efficient.

If adopting the embedding scaling convention from this chapter's assignment, the output is also multiplied by sqrt(d)=sqrt(2). For example, the original vector for token 2 is [3,4], after scaling it becomes [3sqrt(2),4sqrt(2)]. Scaling changes the numerical scale but does not change the number of parameters; the parameters are still V*d=6.

2.3 Permutation Equivariance—Why Positional Encoding is Indispensable

The embedding layer maps each token to a fixed vector. Without positional encoding, "dog bite" and "bite dog" use the same set of token vectors, only arranged in a different order; the model has no additional signal to indicate which token comes first and which comes last.

This stems from the Permutation Equivariance of Self-Attention. More precisely, the output of Self-Attention satisfies:

Permutation Equivariance

Where is a permutation matrix. Swapping the order of input tokens will also swap the output in the same way—this is equivariant, not invariant. Without positional encoding, the attention layer can model interactions based on token content, but it cannot know the absolute positions or relative order of these tokens.

Why is Self-Attention Permutation Equivariant?

From the formula, the computation of attention scores involves dot products between all pairs of tokens. If the input rows (token order) are permuted, the rows of Q and K are permuted accordingly, but the value of depends only on the content of token and token , not on their positions. Therefore, the output rows are also permuted—the output for each token remains the same, only its position changes.

This means that without positional encoding:

  • "I hit you" and "You hit me" contain the same set of tokens; without positional signals, the model can only see a rearrangement of content and cannot reliably use subject-verb-object order
  • "The 1st token" and "The 1000th token" would be treated identically—no positional decay
  • The model cannot leverage word order, a fundamental syntactic signal

Solution: Add positional signals to the representation of each token to break permutation symmetry. Mainstream approaches fall into two categories:

  • Additive Encoding: Add a position vector to the embedding vector, e.g., Sinusoidal, Learned PE
  • Rotary Encoding: Modify the attention computation to encode position in Q/K, e.g., RoPE
🔑
Intuition: The embedding layer gives each token an "ID card" (semantic identity), and positional encoding gives each token a "seat number" (its position in the sequence). Adding them together forms the complete input representation.

For position 0, all even dimensions are 0 and all odd dimensions are 1, which corresponds exactly to row0[0::2] == 0 and row0[1::2] == 1 in the assignment tests. Higher-frequency dimensions quickly distinguish adjacent positions; lower-frequency dimensions change very little over short distances and are used to carry longer-range positional changes.

Why is it designed this way?

Multi-scale frequency: Low dimensions (small ) use high frequencies—sin/cos waveforms change quickly, encoding large differences between adjacent positions, suitable for encoding local positional information. High dimensions (large ) use low frequencies—change slowly, suitable for encoding global positional information. This design mimics Fourier features: by encoding positions onto basis functions of different frequencies, the model can learn to selectively attend to high-frequency or low-frequency dimensions.

import torch import torch.nn as nn import math class SinusoidalEncoding(nn.Module): """Sinusoidal Positional Encoding — fixed function, no trainable parameters.""" def __init__(self, d_model, max_len=8192): super().__init__() pe = torch.zeros(max_len, d_model) pos = torch.arange(0, max_len, dtype=torch.float).unsqueeze(1) # Calculate frequencies: 1 / (10000^(2i/d)) i = torch.arange(0, d_model, 2).float() div_term = torch.exp(i * (-math.log(10000.0) / d_model)) pe[:, 0::2] = torch.sin(pos * div_term) # Even columns pe[:, 1::2] = torch.cos(pos * div_term) # Odd columns self.register_buffer("pe", pe.unsqueeze(0)) def forward(self, x): return x + self.pe[:, :x.size(1), :] pe = SinusoidalEncoding(512) x = torch.randn(2, 128, 512) out = pe(x) print(f"Output shape: {out.shape}") print(f"Trainable parameters: {sum(p.numel() for p in pe.parameters())}") # Output: (2, 128, 512), Trainable parameters: 0

💡 Purpose of register_buffer: Registers pe as a persistent buffer—it will be saved with the model in the checkpoint (.state_dict()), and automatically moved by .to(device), but does not update gradients during training. This allows the encoding table to be "carried" with the model without affecting backpropagation.

2.9A Position Encoding Engineering Selection: Training Length, Extrapolation, and Serving Cost

After learning to implement Sinusoidal and RoPE, an engineering question remains: given a model, training length, and serving scenario, how should one choose the position mechanism? Position encoding is not a decorative item. It determines how the model distinguishes order, how it handles tokens beyond the training length, how position indices continue in the KV cache, and the quality and cost boundaries of long-context serving.

Position Mechanism Comparison

MethodHow Position Enters the ModelAdvantagesMain Limitations
Learned absolute PEAdded to input embeddingSimple to implement, strong expressiveness within training lengthPosition table only learns a fixed length; extrapolation to longer positions is usually unreliable
Sinusoidal PEAdded to input embeddingNo trainable parameters, defined for any position, clear frequency structure"Definable" does not guarantee the model can stably extrapolate after training
RoPERotation of Q/K in AttentionDot product naturally carries relative displacement, widely adopted by decoder-only LLMsLong context requires handling frequency extrapolation, base/scale, and training distribution differences
ALiBiDistance bias added to Attention logitsExplicitly encourages nearby positions, strong intuition for long sequence extrapolation, simple cachingExpression form is relatively limited; may not be interchangeable with RoPE/existing checkpoints
YaRN / NTK-aware scalingAdjusts RoPE frequency or position scaleCommonly used to extend the context window of RoPE modelsRequires re-evaluation of quality; claiming long-context capability just by changing config is not valid

Three Levels of Long-Context Extrapolation

Many model configurations specify max_position_embeddings=32768 or larger, but engineers need to distinguish three levels:

  1. Mathematically computable: The position function can generate sin/cos or bias for longer positions; forward pass does not error.
  2. Numerically stable: Angles, dtype, cache, softmax, and attention kernel remain stable under long sequences.
  3. Task-effective: The model can actually use distant information in long documents, cross-section references, needle retrieval, long code, or multi-turn dialogues.

The first level is easiest to satisfy; the third is the hardest. Both Sinusoidal and RoPE can generate values for longer positions, but whether the model correctly utilizes these positions depends on the pre-training length distribution, the density of long-range dependencies in the data, the attention training signal, post-training data, and prompt organization during inference.

RoPE Engineering Ledger

When using RoPE, at least the following configurations must be recorded; otherwise, different implementations may find it difficult to reproduce the behavior of the same checkpoint:

Configuration ItemMeaningWhy It Affects Results
head_dimRotation dimension for each attention headMust rotate in pairs; odd dimensions or partial rotary changes implementation conventions
base / thetaFrequency base, e.g., 10000 or model-specific valueDetermines the phase change speed for different dimensions
max_positionPre-computed cos/sin cache lengthExceeding cache during serving requires extension or regeneration
position_idsActual position indices passed for each tokenPacked sequences, prefix cache, and incremental decoding all depend on it being correctly continuous

Worked Example: RoPE Cache and Incremental Decoding Positions

Assume a prompt length of 5. During the prefill phase, K/V is computed and cached for token positions 0..4. When generating the first new token, its position should be 5; the second new token is 6. If, during incremental decoding, the position_id for the new token is incorrectly set to 0 at each step, then the Q/K rotation treats all generated tokens as the same position, destroying the relative displacement structure of the attention scores. The model may still appear to run, but long-generation quality and repetition behavior will be significantly abnormal.

This is also why prefix cache must manage both the "content prefix" and the "position prefix." When reusing the KV cache of the same system prompt, the position indices of subsequent tokens in a new request must follow the shared prefix. If retrieval segments or tool results of different lengths are inserted, old positions cannot be blindly reused.

How to Evaluate Position Mechanisms

Changes to position encoding should not be evaluated based on a single short-text PPL. It is recommended to evaluate at least three groups:

  • Short-context retention: Whether PPL within training length, and regular QA/code/math metrics degrade.
  • Long-context retrieval: Needle-in-a-haystack, cross-section references, long-document QA, reported by answer position bucket.
  • Serving metrics: TTFT for long prompts, KV cache memory, cos/sin cache size, throughput changes under batching.
🔑
Core judgment: The learning objective of position encoding is not "to let the model know which token number it is," but to allow attention to stably distinguish order and distance during training and inference. Long-context capability must be jointly demonstrated by the position mechanism, training distribution, attention implementation, and task evaluation.

2.11 Introduction to In-Context Learning: Why Prompts Affect Generation

Having completed Embedding and RoPE, we can re-understand the prompt from inside the model: it is not a "script" independent of the model, but a conditional sequence that has already been tokenized, embedded, and augmented with position information. An autoregressive language model learns the conditional distribution \(p(x_t \mid x_{<t})\) at each step, so the content, order, and relative positions of preceding tokens all change the probability distribution of the next token.

From Token Sequence to Conditional Distribution

Let the input sequence be \(x_1,\ldots,x_n\). After the embedding layer, each token obtains a semantic vector; after position encoding, the model can also distinguish "examples first, question later" from "question first, examples later." Therefore, if the same task description changes the example order, label format, or boundary symbols, the conditional sequence the model sees is different, and the final generation distribution will also differ.

🔑
Key connection: The effect of a prompt first comes from the representation layer: token embedding provides content semantics, position encoding provides sequential structure; the attention mechanism in Chapter 3 will further explain how the model selectively reads this contextual information during generation.

Zero-shot, Few-shot, and In-Context Learning

Zero-shot provides only the task instruction, without input-output examples; the model relies on language patterns and task descriptions learned during pre-training to make predictions. Few-shot places several examples in the context; the model parameters are not updated, but the example format, label space, and reasoning pattern enter the conditional sequence, influencing the generation of subsequent tokens.

This type of capability is often called in-context learning. Its distinction from fine-tuning is important: fine-tuning updates parameters via gradients, writing task information into the weights; in-context learning only provides conditional information within the current sequence and does not persist as new model parameters after leaving this context.

Position Issues in Long Contexts

A long-context model is not simply about "making the window larger." When the sequence length expands, the model needs to stably handle relative positions over longer distances, preventing important information from failing due to position extrapolation, attention dilution, or disorganized context. Methods like RoPE, ALiBi, and YaRN all revolve around a core issue: how to enable the model to reliably distinguish the relative distance and order relationship between tokens in longer sequences.
