2.1 从独热编码到稠密嵌入——为什么需要分布式表示

第1章将文本转换为 token 序列 ，其中每个 。下一个问题是如何将每个整数 token ID 转换为神经网络可以处理的向量。

独热编码及其问题

最直接的想法是独热编码：将 token 映射为长度为 的向量，其中第 位为1，其余为0。例如，词表大小为5时，"apple" (token 3) 编码为 。

但独热编码有三个根本问题：

1. 维度灾难。 典型词表 时，每个 token 变成50000维向量。存储一批 (32 x 1024) 独热向量需要 个浮点数——约6.4 GB，完全不可接受。这不只是存储问题：在高维空间中，所有点之间的距离趋于相等，“近”和“远”的概念消失了（这是距离度量中维度灾难的一种表现）。

2. 语义扁平性。 任意两个不同独热向量之间的余弦相似度都是0。“king”和“queen”在独热空间中与“king”和“apple”一样“不相似”——模型无法从表示中提取任何语义关系。

3. 参数效率极低。 将独热向量乘以一个 权重矩阵等价于查找第 行。那么为什么不直接使用查找表呢？

分布式表示：核心突破

标准解决方案是可学习的稠密嵌入：一个矩阵 （其中 ）将 token 映射到一个低维连续空间：

这就是Token Embedding——等价于在表中查找第 行。关键区别在于：嵌入向量是可训练的。通过语言模型训练，语义相似的 token 将获得相似的向量。

分布假设

这是分布式表示的理论基础——"You shall know a word by the company it keeps" (Firth, 1957)。含义相近的词出现在相似的上下文中，因此它们应该具有相似的向量表示。在嵌入空间中：

  • "king" 和 "queen" 出现在大量重叠的位置（上下文）中 → 向量接近
  • "king" 和 "apple" 的上下文几乎不重叠 → 向量相距很远
  • 类比推理： ——在 word2vec/GloVe 等静态词向量中常见，但这是由语料、训练目标和评估方法共同形成的经验现象，并非嵌入层自动保证的数学性质
🔑
参数量：嵌入矩阵 。在 GPT-2 124M 中，，嵌入层约有3860万参数——占整个模型参数的31%！嵌入矩阵通常与 LM Head（输出投影）通过权重绑定（weight tying）共享，以进一步复用参数。

2.1A 从 word2vec/GloVe 到 LLM 的 Token Embedding

本课程从词向量开始，因为它们展示了一个核心思想：语义不是人工编写的规则，而是可以从上下文统计中学习到的。现代 LLM 中的 nn.Embedding 不再像静态词向量那样独立训练，但它仍然继承了相同的表示学习问题：如何将离散 token 映射到一个可微分、可比且可组合的连续空间。

word2vec 的 skip-gram 目标是：给定中心词 ，预测窗口内的上下文词 。负采样版本区分“真实的上下文词”和“随机噪声词”：

从窗口共现到损失：为什么这是一个可计算的练习

skip-gram 的训练样本来自滑动窗口。给定 token 序列 [0, 1, 2, 1] 和窗口大小为 1，中心词 1 将相邻 token 0、2 和 2 计为上下文；这形成了一个有向共现矩阵 。Ch02 练习中的 build_cooccurrence_matrix 函数要求学习者将这个统计过程写成代码。

在常见假设下，SGNS 学到的点积与移位 PMI 密切相关：负样本数 越大，目标点积向下移位 越多。Ch02 练习中的 shifted_pmi_matrix(cooccurrence, negative_samples) 函数要求学习者从共现矩阵显式计算这个统计量；对于未共现的对 ，PMI 无定义，函数返回 -inf。只有在使用 PPMI 时，才会进一步截断为非负值。

类比推理是对空间结构的诊断，而非目标函数的保证

3CosAdd 查询 的含义是：从 king 的向量中减去与 man 相关的方向，再加上与 woman 相关的方向，最后找出余弦相似度最高的候选词。这个过程通常还会排除输入词 king、man、woman；否则最近邻很可能就是原词本身。

成功的类比依赖许多条件：语料中相关关系足够系统化，向量训练适当，词频和多义性不主导距离，以及评估集接受单一词答案。因此，本课程将类比任务视为对“向量空间是否学到某种线性规律”的检查，而非语义理解的充分证据。

2.1C 嵌入空间诊断：相似性、各向异性与频率影响

训练完嵌入矩阵后，不要仅仅因为损失下降就认为表示已经“学得很好”。本课程反复强调，向量空间的几何结构来自训练目标和数据分布，同时也承载着偏差和失败模式。对于大语言模型工程师而言，嵌入诊断是发现分词器、数据混合、初始化和权重绑定问题的早期信号。

1. 行范数不等于语义强度

嵌入矩阵 E[V, d] 中的每一行对应一个 token。训练后一个常见现象是高频 token 的行范数更大，因为它们更新得更频繁；特殊 token、空格、标点符号和常见子词也可能形成不同的簇。大的范数并不意味着更大的语义重要性；它通常混合了频率、优化历史和输出层梯度。

diagnostics: row_norm_mean / row_norm_p95 top_norm_tokens low_norm_tokens norm_by_frequency_bucket special_token_norms

如果新添加的特殊 token 的范数显著低于常见 token，可能表明初始化后训练不足；如果 padding token 的范数被大量更新，通常意味着损失掩码或注意力掩码存在问题。

2. 余弦相似度必须先处理尺度问题

余弦相似度将向量方向和大小分开，是观察语义邻居、类比和簇的常用工具。在 Ch02 作业中，cosine_similarity_matrix 首先对每一行进行归一化，然后计算 normalized @ normalized.T。这一步很重要：如果直接使用点积，高范数、高频的 token 会主导最近邻结果。

然而，余弦相似度并非万能解释。大语言模型的输入嵌入只是第一层表示；同一个 token 在不同上下文中的最终隐藏状态可能完全不同。因此，“在嵌入空间中接近”仅表明初始 token 表示相似，并不表示模型最终会将这两个词视为同义词。

3. 各向异性会降低最近邻质量

高维嵌入空间通常表现出各向异性 (anisotropy)：许多向量指向某些共同方向，导致任意两个 token 之间的余弦相似度被人为抬高，并且最近邻被频率、格式或共同方向主导。一个简单的诊断是分析非对角线余弦值的分布：

cosine matrix excluding diagonal: mean close to 0 -> 空间相对平衡 mean high positive -> 可能存在共同方向或频率主导 p95 very high -> 局部簇或重复 token 过强

研究工作中常用均值中心化、主成分去除或白化来分析静态词向量，但在工程实践中，不要随意将这些变换应用于正在训练的大语言模型的嵌入。诊断的目的是理解问题来源，而不是在未经验证的情况下修改模型参数。

4. 类比推理的成功需要系统性关系，而非个别例子

3CosAdd 的形式 v_b - v_a + v_c 假设某种关系可以近似表示为线性方向。例如，性别、首都-国家或形态变化可能在某些语料中形成方向。然而，一个成功的例子并不能证明模型理解了该关系；一个失败的例子也不能证明嵌入缺乏语义。严谨的方法是按关系类型构造一组查询，报告 top-k 准确率，并检查错误是否源于频率、子词分割、多义词或评估集歧义。

诊断项目可以指示什么不能指示什么
最近邻 (Nearest Neighbors)局部方向是否聚集了相关 token不能证明上下文语义等价
类比 top-k (Analogy top-k)某种线性关系是否系统性地存在不能涵盖推理、事实性或一词多义
范数分布 (Norm Distribution)频率、掩码、特殊 token 的更新是否异常不能单独解释 token 重要性
非对角线余弦 (Off-diagonal Cosine)空间是否被共同方向主导不能直接提供修复方法

5. 大语言模型嵌入的工程检查点

在训练或微调期间，应在分词器变更后、添加特殊 token 后、更改权重绑定后、SFT 开始前后以及训练损失看起来异常时运行嵌入诊断。需要特别关注：

  • 新 token 是否在训练：特殊 token 和领域术语是否具有非零梯度和合理的范数？
  • padding 是否有泄露：PAD 行是否被损失更新？注意力掩码是否导致模型关注 padding？
  • 语言/领域是否不平衡：少数语言或代码的 token 是否经常被碎片化且范数较低？
  • 权重绑定是否一致：当输入嵌入和 LM head 共享时，输出梯度会更新所有 softmax 行；当不共享时，两个空间可能逐渐偏离。
🔑
本节要点：嵌入不是静态的语义词典，而是由训练目标、分词器、频率和输出层共同塑造的参数矩阵。诊断嵌入空间的价值在于发现系统性问题，而不是使用单个最近邻或类比示例来证明模型“理解语言”。

2.1D 从静态词向量到上下文表示：为什么同一 token 可以有不同的语义

Word2vec 和 GloVe 学习静态向量：词汇表中的每个词或子词通常对应一个固定向量。现代 Transformer 的输入嵌入也是一个固定的查找表矩阵，但模型实际用于预测、分类或生成的表示并不是这个初始行向量，而是经过多层注意力机制和前馈网络后的隐藏状态。这一区别是理解 BERT、GPT、RAG 重排序器和上下文学习的关键。

2. 工作示例："bank" 的两种上下文

考虑两个句子：

S1: I deposited money at the bank. S2: I sat near the river bank.

如果 tokenizer 将两个 bank 实例映射到相同的 token id，那么查找表会返回完全相同的 E[bank]。差异来自上下文：在 S1 中，像 money 和 deposited 这样的 token 会通过注意力机制将金融语义写入 bank 位置的隐藏状态；在 S2 中，像 river 和 near 这样的 token 会将地理/河岸语义写入同一位置。在较高层中，两个 bank 实例的隐藏状态可能相差很远，尽管它们的输入嵌入是同一行。

3. 这如何改变评估方法

静态词向量通常使用最近邻、类比和词相似度数据集进行评估，因为模型的输出是一个单一的词向量空间。上下文模型必须按任务和层进行评估：分类使用 [CLS] 或池化表示，抽取式问答使用跨度 logits，仅解码器 LM 使用每个位置隐藏状态经过 LM head 后得到的下一个 token 分布。不能仅通过查看输入嵌入的最近邻来判断整个大语言模型是否理解多义性、指代关系或推理关系。

4. 工程启示：不要将嵌入相似性误认为最终语义

在微调、RAG 重排序或安全评估中，如果一个 token 的输入嵌入的最近邻看起来合理，那仅表明初始表示没有明显的异常。要真正理解模型行为，必须考察上下文隐藏状态、logits、任务输出和失败案例。例如，一个领域术语在嵌入空间中可能接近通用词，但在上下文化之后仍然能被正确提取；相反，一个新添加的特殊 token 可能有正常的嵌入范数，但由于聊天模板或 loss mask 的错误，模型可能无法按预期使用它。

🔑
工程要点： 静态词向量回答的是“这个词在语料库统计中像谁”；上下文表示回答的是“这个位置在当前上下文中起什么作用”。大语言模型工程师需要理解两者，否则容易混淆嵌入诊断、注意力诊断和最终任务质量。

2.3 排列等变性——为什么位置编码不可或缺

嵌入层将每个 token 映射为一个固定向量。没有位置编码时，"dog bite" 和 "bite dog" 使用同一组 token 向量，只是排列顺序不同；模型没有任何额外信号来区分哪个 token 在前、哪个在后。

这源于自注意力（Self-Attention）的排列等变性（Permutation Equivariance）。更准确地说，自注意力的输出满足：

排列等变性

其中 是排列矩阵。交换输入 token 的顺序也会以相同方式交换输出——这是等变，而不是不变。没有位置编码，注意力层只能根据 token 内容建模交互，却无法知道这些 token 的绝对位置或相对顺序。

为什么自注意力具有排列等变性

从公式来看，注意力分数 的计算涉及所有 token 对之间的点积。如果输入行（token 顺序）被排列，Q 和 K 的行也会相应地被排列，但 的值只取决于 token 和 token 的内容，而不依赖于它们的位置。因此输出行也会被排列——每个 token 的输出保持不变，只是其位置发生了变化。

这意味着没有位置编码时：

  • "I hit you" 和 "You hit me" 包含相同的 token 集合；没有位置信号，模型只能看到内容的重新排列，无法可靠地利用主-谓-宾语序
  • "第1个 token" 和 "第1000个 token" 会被同等对待——不存在位置衰减
  • 模型无法利用词序这一基本句法信号

解决方案： 为每个 token 表示添加位置信号，打破排列对称性。常见方法分为两类：

  • 加法编码（Additive Encoding）： 在嵌入向量上加一个位置向量，例如 Sinusoidal、Learned PE
  • 旋转编码（Rotary Encoding）： 修改注意力计算，在 Q/K 中编码位置，例如 RoPE
🔑
直觉理解： 嵌入层给每个 token 一张“身份证”（语义身份），位置编码给每个 token 一个“座位号”（在序列中的位置）。二者相加就构成了完整的输入表示。

  • forward(x) 返回 x + pe[:, :seq_len, :]（在 batch 维度上进行广播）

import torch import torch.nn as nn import math class SinusoidalEncoding(nn.Module): """Sinusoidal Positional Encoding — fixed function, no trainable parameters.""" def __init__(self, d_model, max_len=8192): super().__init__() pe = torch.zeros(max_len, d_model) pos = torch.arange(0, max_len, dtype=torch.float).unsqueeze(1) # Calculate frequencies: 1 / (10000^(2i/d)) i = torch.arange(0, d_model, 2).float() div_term = torch.exp(i * (-math.log(10000.0) / d_model)) pe[:, 0::2] = torch.sin(pos * div_term) # Even columns pe[:, 1::2] = torch.cos(pos * div_term) # Odd columns self.register_buffer("pe", pe.unsqueeze(0)) def forward(self, x): return x + self.pe[:, :x.size(1), :] pe = SinusoidalEncoding(512) x = torch.randn(2, 128, 512) out = pe(x) print(f"Output shape: {out.shape}") print(f"Trainable parameters: {sum(p.numel() for p in pe.parameters())}") # Output: (2, 128, 512), Trainable parameters: 0

import torch import torch.nn as nn import math class RoPE(nn.Module): """Rotary Position Embedding.""" def __init__(self, d_model, max_len=8192, base=10000.0): super().__init__() # Compute frequencies theta_i theta = torch.pow(base, -torch.arange(0, d_model, 2).float() / d_model) pos = torch.arange(max_len).float().unsqueeze(1) # (max_len, 1) angles = pos * theta.unsqueeze(0) # (max_len, d_model/2) self.register_buffer("cos", torch.cos(angles).unsqueeze(0).unsqueeze(0)) self.register_buffer("sin", torch.sin(angles).unsqueeze(0).unsqueeze(0)) def _rotate_half(self, x): """Negate the second half of x and swap with the first half.""" x1, x2 = x.chunk(2, dim=-1) return torch.cat((-x2, x1), dim=-1) def _apply_rotary(self, x, cos, sin): x1, x2 = x.chunk(2, dim=-1) return torch.cat((x1 * cos - x2 * sin, x1 * sin + x2 * cos), dim=-1) def forward(self, q, k, pos_ids=None): batch, n_heads, seq_len, d_head = q.shape # Get cos/sin for the corresponding positions; pos_ids used for packed sequences or incremental decoding if pos_ids is None: cos = self.cos[:, :, :seq_len, :d_head//2] sin = self.sin[:, :, :seq_len, :d_head//2] else: cos = self.cos[:, :, pos_ids, :d_head//2] sin = self.sin[:, :, pos_ids, :d_head//2] # Apply rotation q_out = self._apply_rotary(q, cos, sin) k_out = self._apply_rotary(k, cos, sin) return q_out, k_out # Verify norm preservation rope = RoPE(d_model=64) q = torch.randn(2, 8, 10, 64) q_rot, _ = rope(q, q) assert torch.allclose(q.norm(dim=-1), q_rot.norm(dim=-1), atol=1e-5) print("Norm preserved ✅")

def verify_rope_relative_property(d_model=64, seq_len=8): """验证 RoPE 的点积只依赖于相对位置。""" rope = RoPE(d_model=d_model) # 构造一个固定的 q 和 k（不含位置信息） q = torch.randn(1, 1, 1, d_model) # (1, 1, 1, d) k = torch.randn(1, 1, 1, d_model) # 扩展到 seq_len 个位置：每个位置上的 q/k 向量相同 q_expanded = q.expand(1, 1, seq_len, -1) # (1, 1, seq_len, d) k_expanded = k.expand(1, 1, seq_len, -1) # 应用 RoPE：相同向量在不同位置被旋转不同角度 q_rot, k_rot = rope(q_expanded, k_expanded) # 计算所有 (m, n) 对的点积 scores = torch.matmul(q_rot, k_rot.transpose(-2, -1)) # (1, 1, seq_len, seq_len) scores = scores.squeeze() # (seq_len, seq_len) print(f"注意力分数矩阵（{seq_len}x{seq_len}）：") # 显示到小数点后4位 for row in scores: print(" " + " ".join(f"{v:.4f}" for v in row)) # 验证 Toeplitz 性质：向右下移动一步后，重叠区域应完全相等。 toeplitz_ok = torch.allclose(scores[1:, 1:], scores[:-1, :-1], atol=1e-5) if toeplitz_ok: print("\n✅ 相对位置性质成立：Q_m^T K_n 只依赖于 (m-n)!") print("分数矩阵是 Toeplitz 矩阵。") else: print("\n❌ Toeplitz 验证失败：同一偏移对角线上的值不一致。") # 不检查距离的单调衰减：对于固定的 q/k，偏移分数由相对旋转相位决定。 print("\n不同偏移的分数可以是非单调的；这里验证的是 Toeplitz 性质。") return scores scores = verify_rope_relative_property(d_model=64, seq_len=8)

💡 实验结论： 旋转后的 Q 和 K 的点积构成一个 Toeplitz 矩阵——每个固定偏移对角线上的值均相等。这意味着注意力分数只依赖于 token 之间的相对位移。不同的距离对应不同的相位组合，但对于任何固定的 q/k，分数并不需要随 单调递减；长上下文性能仍然取决于训练分布、频率设计和缩放策略。

2.9A 位置编码工程选择：训练长度、外推与推理成本

在实现了 Sinusoidal 和 RoPE 之后，仍有一个工程问题：给定模型、训练长度和推理场景，应该如何选择位置机制？位置编码不是装饰品。它决定了模型如何区分顺序、如何处理超出训练长度的 token、位置索引如何在 KV cache 中延续，以及长上下文推理的质量和成本边界。

位置机制对比

方法位置如何进入模型优势主要限制
可学习的绝对位置编码加到输入嵌入中实现简单，训练长度内表达力强位置表只学到固定长度；外推到更长位置通常不可靠
Sinusoidal 位置编码加到输入嵌入中无可训练参数，可为任意位置定义，频率结构清晰“可定义”并不能保证模型训练后能稳定外推
RoPE在 Attention 中对 Q/K 进行旋转点积自然携带相对位移，被纯解码器 LLM 广泛采用长上下文需要处理频率外推、base/scale 以及训练分布差异
ALiBi向 Attention logits 加入距离偏置显式鼓励邻近位置，对长序列外推有强烈直觉，缓存简单表达形式相对有限；可能与 RoPE/现有 checkpoint 不可互换
YaRN / NTK-aware 缩放调整 RoPE 频率或位置尺度常用于扩展 RoPE 模型的上文窗口需要重新评估质量；仅通过修改配置就声称长上下文能力是不成立的

长上下文外推的三个层次

许多模型配置指定 max_position_embeddings=32768 或更大，但仅凭这个数字并不能保证长上下文能力。区分三个层次：

  1. 数学上可计算：位置函数能为更长的位置生成 sin/cos 或偏置；前向传播不报错。
  2. 数值稳定：角度、dtype、cache、softmax 和 attention kernel 在长序列下保持稳定。
  3. 任务有效：模型能实际利用长文档、跨段落引用、大海捞针检索、长代码或多轮对话中的远程信息。

第一层最容易满足，第三层最难。Sinusoidal 和 RoPE 都能为更长的位置生成数值，但模型能否正确利用这些位置取决于预训练长度分布、数据中长距离依赖的密度、注意力训练信号、训练后数据以及推理时的 prompt 组织。

RoPE 工程记录表

使用 RoPE 时，至少必须记录以下配置；否则，不同的实现可能难以复现同一个 checkpoint 的行为：

配置项含义为何影响结果
head_dim每个注意力头的旋转维度必须成对旋转；奇数维度或部分旋转会改变实现惯例
base / theta频率基数，例如 10000 或模型特定值决定不同维度的相位变化速度
max_position预计算的 cos/sin 缓存长度推理期间超出缓存需要扩展或重新生成
position_ids每个 token 传入的实际位置索引打包序列、前缀缓存和增量解码都依赖它正确连续
scaling缩放策略，如线性、动态 NTK、YaRN改变远程相位；可能改善长上下文但也可能损害短上下文
dtype/devicecos/sin 缓存的精度和设备低精度导致长位置时的三角函数误差；CPU/GPU 传输影响速度和稳定性

工作示例：RoPE 缓存与增量解码位置

假设 prompt 长度为 5。在 prefill 阶段，为 token 位置 0..4 计算并缓存 K/V。生成第一个新 token 时，其位置应为 5；第二个新 token 为 6。如果在增量解码过程中，新 token 的 position_id 每一步都被错误地设为 0，那么 Q/K 旋转会将所有生成的 token 视为同一位置，破坏了 attention score 的相对位移结构。模型可能看起来仍能运行，但长序列生成质量和重复行为将显著异常。

这也是为什么前缀缓存必须同时管理“内容前缀”和“位置前缀”。在复用相同系统 prompt 的 KV cache 时，新请求中后续 token 的位置索引必须跟随共享的前缀。如果插入了不同长度的检索片段或工具结果，则不能盲目复用旧位置。

如何评估位置机制

对位置编码的改动不应仅用单个短文本 PPL 来评估。至少评估三组：

  • 短上下文保持：训练长度内的 PPL 以及常规 QA/代码/数学指标是否下降。
  • 长上下文检索：大海捞针、跨段落引用、长文档 QA，按答案位置桶报告。
  • 推理指标：长 prompt 的 TTFT、KV cache 内存、cos/sin 缓存大小、批处理下的吞吐量变化。
🔑
核心判断：位置编码的学习目标不是“让模型知道它是第几个 token”，而是让 attention 在训练和推理中稳定区分顺序和距离。长上下文能力必须由位置机制、训练分布、注意力实现和任务评估共同展示。

2.9B RoPE 缩放与长上下文发布：从配置更改到能力证据

扩展 RoPE 模型的上下文长度不仅仅是修改 max_position_embeddings。诸如 YaRN、NTK-aware scaling 和 LongRoPE 之类的方法会调整 RoPE 频率将 token 索引映射到相位的方式，通常还需要在更长序列上进行持续训练或微调。DeepSeek-V3 报告是一个有用的工业案例：在预训练之后，它应用了 YaRN 并利用额外的训练阶段逐步扩展上下文。工程结论是长上下文能力是一个发布程序，而不是一个配置标志。

图 2-2：RoPE 缩放发布生命周期。一个长上下文声明需要缩放配置、持续训练证据、短上下文保持、长位置评估以及服务成本测量。

1. 不同的 RoPE 缩放方法以不同方式改变相位映射

直观问题是相位不匹配。RoPE 使用多个频率将位置索引映射为旋转。当在 4K 或 8K 上训练的模型被要求读取 128K token 时，许多位置会产生训练分布之外的相位模式。缩放方法试图压缩、拉伸或选择性调整这个相位映射，使得长位置不那么偏离分布，同时短位置仍然可用。

方法族工程思路优势风险
仅更改 max_position为更长位置生成 cos/sin 缓存而不更改频率映射前向传播可以运行更长不能证明模型可以使用远距离上下文；可能静默失败
线性位置插值通过缩放位置 ID 将长位置映射回较短训练范围简单且通常是良好的基线会压缩局部距离并损害短上下文行为
NTK-aware / 动态缩放修改频率基数或随上下文长度增长缩放改善长序列相位行为而不更改架构实现特定；必须精确记录以便检查点兼容
YaRN-style 缩放使用更结构化的 RoPE 扩展方案，通常配合较短持续训练被多个长上下文系统使用的实用上下文扩展需要在原始和目标上下文长度上验证
LongRoPE-style 搜索与渐进扩展搜索非均匀插值并分阶段扩展，同时调整短上下文目标是很大的窗口，同时尽量保留原始能力活动部件更多；发布证据必须分离缩放、训练和评估效果

2. 持续训练数据必须与长上下文声明匹配

一个在 128K 上通过大海捞针测试的模型，不一定学会了长文档推理。大海捞针测试主要证明在受控位置下检索特定事实的能力。真正的长上下文工作负载包括多跳引用、重复实体、表格、代码仓库、会议记录、工具追踪和不相关的干扰项。因此，持续训练或微调数据应该既包含长序列，也包含需要使用远距离信息的任务，而不仅仅是长填充或拼接的不相关文档。

3. 发布报告必须绑定配置、检查点和服务引擎

长上下文行为既依赖于模型检查点也依赖于推理栈。发布包应记录 RoPE base/theta、缩放类型和因子、原始训练长度、持续训练长度、打包序列的位置 ID 策略、前缀缓存兼容性、注意力内核、量化格式以及最大服务上下文。如果这些在训练、评估和部署之间有任何不同，长上下文声明可能无法迁移。

例如，量化服务引擎如果以不同于训练框架的方式解释 RoPE 缩放，可能能通过短提示但在接近目标上下文长度时失败。类似地，前缀缓存可以重用共享提示的 K/V，但前提是位置 ID 和插入的检索/工具段被一致地重建。这就是为什么 Ch02 的位置账本直接连接到 Ch03 注意力掩码、Ch07 打包训练数据、Ch08 长上下文生成以及 Ch10 服务 SLO。

🔑
工程判断：一个长上下文模型发布必须分别证明四件事：模型可以计算更长位置，训练使其暴露于有用的长上下文信号，评估显示按位置桶的质量，以及服务引擎能在成本和延迟限制下提供上下文长度。

2.11 上下文学习简介：为什么提示会影响生成

学完 Embedding 和 RoPE 后，我们可以从模型内部重新理解 prompt：它并不是独立于模型的"脚本"，而是一段已经经过分词、嵌入并添加了位置信息的条件序列。自回归语言模型每一步学习条件分布 \(p(x_t \mid x_{<t})\)，因此前续 token 的内容、顺序和相对位置都会改变下一个 token 的概率分布。

从 token 序列到条件分布

设输入序列为 \(x_1,\ldots,x_n\)。经过嵌入层后，每个 token 获得语义向量；经过位置编码后，模型也能区分"先示例后问题"还是"先问题后示例"。因此，如果同一任务描述改变了示例顺序、标签格式或边界符号，模型看到的条件序列就不同，最终的生成分布也会不同。

🔑
关键联系： prompt 的效果首先来自表示层：token embedding 提供内容语义，position encoding 提供序列结构；第 3 章的注意力机制将进一步解释模型如何在生成过程中有选择地读取这些上下文信息。

零样本、少样本与上下文学习

零样本仅提供任务指令，不提供输入-输出示例；模型依赖预训练阶段学到的语言模式和任务描述进行预测。少样本在上下文中放置若干示例；模型参数不更新，但示例的格式、标签空间和推理模式进入了条件序列，影响后续 token 的生成。

这类能力通常称为上下文学习。它与微调的重要区别在于：微调通过梯度更新参数，将任务信息写入权重；上下文学习仅在当前序列中提供条件信息，离开该上下文后不会作为新模型参数持久化。

长上下文中的位置问题

长上下文模型并非简单"把窗口放大"。当序列长度扩展时，模型需要稳定处理更远距离的相对位置，防止重要信息因位置外推、注意力稀释或上下文混乱而失效。RoPE、ALiBi、YaRN 等方法都围绕一个核心问题：如何让模型在更长序列中可靠地区分 token 之间的相对距离和顺序关系。

本章需要掌握的建模视角

💡
1. prompt 是条件序列 — 它首先经过 tokenizer、embedding 和 position encoding 处理
2. 少样本即上下文学习 — 示例改变当前条件分布，但不更新参数
3. 顺序与边界很重要 — 相同 token 的不同排列形成不同的模型输入
4. 长上下文依赖位置外推 — 位置编码决定模型能否稳定理解长距离关系
5. 注意力是下一章的核心 — 它解释模型如何读取、比较和聚合上下文 token
