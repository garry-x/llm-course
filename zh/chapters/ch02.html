2.1 从 One-Hot 到 Dense Embedding：为什么需要分布式表示

第 1 章把文本变成 token 序列 [t_1, t_2, ..., t_n]，每个 t_i 是一个整数 ID。下一步的问题是：怎样把这个整数变成神经网络能处理的向量。

One-Hot 编码的问题

最直接的方式是 one-hot：vocabulary size 为 V，token t 就是长度为 V 的向量，第 t 位为 1，其余为 0。比如 vocab size 为 5，token 3 可以写成 [0,0,0,1,0]。

问题也很明显：

  • 维度太高：常见 vocabulary 动辄 50K，batch 里显式存 one-hot 会非常浪费。
  • 语义太平：任意两个不同 one-hot 向量的 cosine similarity 都是 0，king 和 queen 并不比 king 和 apple 更近。
  • 计算没有必要：one-hot 乘以 V x d 矩阵，本质就是取矩阵第 t 行。既然如此，直接查表就行。

Dense Embedding 的核心突破

标准做法是使用可训练的 dense embedding 矩阵 E ∈ R^{V x d}，其中 d 远小于 V。token t 的向量就是 E[t, :]。这就是 nn.Embedding 做的事：不是把 ID 变成巨大的 one-hot，而是用 ID 查一行可训练向量。

Distributional Hypothesis

分布式表示的直觉来自一句经典说法：一个词的意思，可以从它经常和谁一起出现看出来。上下文相似的词，应该有相似的向量。word2vec/GloVe 这类静态词向量，就是把这种语料统计变成向量空间结构。

🔑
参数量：Embedding 参数量是 V * d。GPT-2 124M 中，V=50257、d=768，embedding 层约 3860 万参数。很多模型会把 embedding 矩阵和 LM head 共享，也就是 weight tying。

2.1A 从 word2vec/GloVe 到 LLM Token Embedding

课程先讲词向量，是因为它能说明一个核心思想：语义不是手写规则，而是可以从上下文统计里学出来。现代 LLM 的 nn.Embedding 不再是单独训练好的静态词向量，但它仍然面对同一个表示学习问题：怎样把离散 token 映射到可微、可比较、可组合的连续空间。

Skip-gram with Negative Sampling

word2vec 的 skip-gram 目标是：给定中心词，预测窗口里的上下文词。Negative sampling 版本把真实上下文当正样本，把随机采样词当负样本，用二分类式 loss 来训练向量。

SGNS 思路
让 center 和真实 context 的 dot product 变大； 让 center 和 negative samples 的 dot product 变小。

Ch02 作业中的 build_cooccurrence_matrix、skipgram_negative_sampling_loss 和 sgns_center_gradient，就是把“上下文统计变成训练信号”写成可运行代码。

PMI 与 shifted PMI

如果 X_ij 是中心词 i 和上下文词 j 的共现次数，PMI 可以写成：

PMI(i,j) = log( X_ij * X_** / (X_i* * X_*j) ) SPMI(i,j) = PMI(i,j) - log(K)

SGNS 学到的 dot product 和 shifted PMI 有紧密关系。负样本数 K 越大，目标 dot product 的零点会整体下移。这里的重点不是背公式，而是理解：词向量空间来自共现统计和训练目标。

GloVe：从全局共现矩阵学习

GloVe 直接从全局 co-occurrence matrix 出发，用加权最小二乘拟合 log X_ij。它和 skip-gram 的入口不同，但都在利用语料里的共现结构。

2.1B 例子：从共现计数到 PMI、SGNS 和 analogy

假设只有三个词：king、queen、apple。滑动窗口统计得到的共现表里，king 和 queen 经常互为上下文，而 king 和 apple 几乎只是偶然碰到。

PMI 衡量的是：两个词一起出现的频率，是否高于随机独立情况下的预期。SGNS 则把这个共现关系变成一个个正负样本，让真实上下文 dot product 变大，让噪声样本 dot product 变小。

analogy 里的 v_king - v_man + v_woman 是在检查向量空间里是否形成了稳定的线性方向。它是一个诊断工具，不是“模型理解性别、王权和语义关系”的充分证明。

2.1C Embedding 空间诊断：相似度、各向异性和频率效应

训练出 embedding 矩阵后，不要因为 loss 降了就默认表示空间没问题。向量空间由训练目标和数据分布共同塑造，也会带着频率、tokenizer 和初始化带来的偏差。

1. 行范数不等于语义强度

高频 token 被更新得更多，row norm 往往更大。special token、空格、标点和常见 subword 也可能形成特殊簇。范数大不等于“更重要”，它混合了频率、优化历史和输出层梯度。

2. cosine similarity 需要先处理尺度

cosine similarity 会消掉向量模长的影响，适合看局部邻居和 analogy。但它只能说明初始 embedding 空间里的相近关系，不能证明模型在上下文中会把两个词当同义词。

3. 各向异性会污染最近邻

高维 embedding 空间可能出现 common direction，导致任意两个 token 的 cosine 都偏高，nearest neighbor 被频率或格式支配。工程上可以看非对角 cosine 分布，判断空间是否被某些公共方向主导。

4. LLM embedding 的工程检查点

  • 新增 token 是否真的被训练到，梯度和范数是否合理。
  • PAD token 是否被 loss 错误更新，mask 是否泄漏。
  • 少数语言、代码、特殊符号是否被 tokenizer 切得过碎。
  • weight tying 前后，输入 embedding 和输出分类空间是否一致。

2.1D 从静态词向量到上下文化表示：为什么同一个 token 可以有不同语义

word2vec 和 GloVe 学到的是静态向量：vocabulary 里的每个词或 subword 通常只有一个固定向量。现代 Transformer 的输入 embedding 也是固定查表矩阵，但模型真正用于预测、分类或生成的表示，不是这个初始行向量，而是多层 attention 和 FFN 之后的 hidden state。

给定序列 x_1,...,x_T，第 0 层表示可以写成 h_i^{0} = E[x_i] + p_i。经过多层 Transformer 后，h_i^{l} 会依赖上下文、位置、attention pattern 和前面层的计算。

例子：两个语境里的 bank

S1: I deposited money at the bank. S2: I sat near the river bank.

如果两个 bank 被映射到同一个 token ID，那么查表得到的 E[bank] 完全相同。差异来自上下文：money、deposited 会把金融语义写进 hidden state；river、near 会把河岸语义写进去。

🔑
工程结论：静态词向量回答“这个词在语料统计里像谁”；上下文化表示回答“当前位置在当前上下文里扮演什么角色”。不要用 input embedding 的最近邻，直接判断整个 LLM 是否理解多义词或推理关系。

2.2 编程练习 1：实现 TokenEmbedding

PyTorch 的 nn.Embedding(V, d) 本质是一个可训练查表操作。如果 embedding 矩阵是 E ∈ R^{V x d}，token ID i 的 one-hot 向量乘以 E，等价于直接取 E[i]。

练习 1：TokenEmbedding Module

任务：实现 TokenEmbedding，并写一个 embedding_lookup_as_matmul，用显式 one-hot 矩阵乘法复现 embedding_weight[token_ids]。

  • 输入 shape：[batch, seq_len]。
  • 输出 shape：[batch, seq_len, d_model]。
  • 内部使用 nn.Embedding(vocab_size, d_model)。
class TokenEmbedding(nn.Module): def __init__(self, vocab_size, d_model): super().__init__() self.embed = nn.Embedding(vocab_size, d_model) self.d_model = d_model def forward(self, x): return self.embed(x) * math.sqrt(self.d_model)

本章练习采用乘以 sqrt(d_model) 的缩放约定，用来显式观察 embedding 数值尺度怎样进入后续层。真实模型会把初始化、LayerNorm、残差缩放和训练稳定性一起设计，不是所有 decoder-only 模型都必须这么写。

2.2A 例子：embedding lookup、缩放和 tied LM head

设 vocabulary size 为 3，embedding dim 为 2：

E = [[1, 0], [0, 2], [3, 4]] token_ids = [[0, 2], [1, 0]] E[token_ids] = [[[1,0], [3,4]], [[0,2], [1,0]]]

显式 one-hot 乘法完全等价：one_hot(token_ids) 的 shape 是 [2,2,3]，乘以 E[3,2] 后得到 [2,2,2]。实际模型不会构造巨大 one-hot，因为查表更省显存和带宽。

如果 input embedding 和 output LM head 共享权重，hidden state h 的 logits 就是 h E^T。这能减少大 vocabulary 下的参数量，也让输入 token 空间和输出分类空间共享同一组向量。

2.3 置换等变性：为什么必须有位置编码

Embedding 层只给每个 token 一个向量。没有位置编码时，dog bite 和 bite dog 用的是同一组 token 向量，只是顺序换了。Self-Attention 只看内容之间的相似度，本身不知道谁在前、谁在后。

更准确地说，Self-Attention 具有 permutation equivariance：输入 token 顺序被同样置换，输出也会同样置换。它不是完全不变，但它没有额外信号区分绝对位置或相对顺序。

  • “I hit you”和“You hit me”包含同一批词，但语义完全不同。
  • 第 1 个 token 和第 1000 个 token 如果没有位置信号，会缺少位置差异。
  • 语序是句法核心信号，模型必须拿到它。

解决方法是给 token representation 加入位置：可以加一个位置向量，也可以像 RoPE 一样在 attention 的 Q/K 上编码位置。

2.4 Sinusoidal 位置编码：多尺度频率直觉

原始 Transformer 使用固定的 Sinusoidal Positional Encoding。它用不同频率的 sin/cos 函数，为每个位置生成一个固定向量。

PE[pos, 2i] = sin(pos / 10000^(2i/d)) PE[pos, 2i+1] = cos(pos / 10000^(2i/d))

低维使用较高频率，变化快，适合区分相邻位置；高维使用较低频率，变化慢，适合承载更长距离的位置变化。这个设计类似 Fourier features：用多种频率的基函数表示位置。

当 d_model=4 时，维度 0/1 使用 sin(pos) 和 cos(pos)，维度 2/3 使用 sin(pos/100) 和 cos(pos/100)。位置 0 的偶数维为 0，奇数维为 1，这正是很多实现测试会检查的性质。

Sinusoidal 没有可训练参数，也可以对任意长度计算。不过“函数定义得出来”不等于模型一定能在训练长度之外稳定工作，长上下文外推还要看训练数据、attention 机制和评估证据。

2.5 编程练习 2：实现 Sinusoidal Positional Encoding

练习 2：SinusoidalEncoding Module

  • 继承 nn.Module，内部没有可训练参数。
  • 生成 [max_len, d_model] 的位置编码矩阵。
  • 用 register_buffer 保存，使它进入 checkpoint，也能跟着模型移动到 GPU。
  • forward(x) 返回 x + pe[:, :seq_len, :]。
class SinusoidalEncoding(nn.Module): def __init__(self, d_model, max_len=8192): super().__init__() pe = torch.zeros(max_len, d_model) pos = torch.arange(max_len, dtype=torch.float).unsqueeze(1) i = torch.arange(0, d_model, 2).float() div_term = torch.exp(i * (-math.log(10000.0) / d_model)) pe[:, 0::2] = torch.sin(pos * div_term) pe[:, 1::2] = torch.cos(pos * div_term) self.register_buffer("pe", pe.unsqueeze(0)) def forward(self, x): return x + self.pe[:, :x.size(1), :]

2.6 深入 RoPE：用旋转编码相对位置

RoPE（Rotary Position Embedding）是 Llama、Mistral、DeepSeek 等模型常用的位置编码方式。它不把位置向量加到 embedding 上，而是在 attention 层对 Q 和 K 做旋转，让点积天然带上相对位置信息。

问题设定

设位置 m 的 query 是 q_m，位置 n 的 key 是 k_n。我们希望 attention score 依赖 m-n，也就是相对位置，而不是只看绝对位置。

核心：旋转矩阵

RoPE 把向量每两个维度看成一组二维平面，然后按位置给它旋转。位置越大，旋转角越大；不同维度组使用不同频率。

q'_m = R_m q_m k'_n = R_n k_n (q'_m)^T k'_n = q_m^T R_m^T R_n k_n = q_m^T R_{n-m} k_n

关键一步是 R_m^T R_n = R_{n-m}。旋转矩阵的转置等于反向旋转，所以先反转 m 再旋转 n，等价于旋转 n-m。因此 RoPE 的点积自然依赖相对位置。

RoPE 的几个性质

  • 相对位置：点积里出现的是 n-m。
  • 范数保持：旋转矩阵是正交矩阵，不改变向量 L2 norm。
  • 只作用于 Q/K：RoPE 改变“谁 attend 谁”，不直接改变被读出的 V 内容。
  • 不等于无限外推：RoPE 给了相对位置归纳偏置，但长上下文稳定性仍需要训练和评估证据。

2.7 编程练习 3：实现 RoPE

练习 3：RoPE Module

  • 输入 Q/K shape：[batch, n_heads, seq_len, d_head]。
  • 预先计算 cos/sin table，并能 broadcast 到 Q/K。
  • 按二维 pair 做旋转：[x1*cos - x2*sin, x1*sin + x2*cos]。
  • 返回 q_rot, k_rot，shape 不变。
def apply_rope(x, cos, sin): x1 = x[..., 0::2] x2 = x[..., 1::2] out = torch.empty_like(x) out[..., 0::2] = x1 * cos - x2 * sin out[..., 1::2] = x1 * sin + x2 * cos return out

2.8 验证：RoPE 的相对位置性质

理论上，RoPE 旋转后的点积只依赖相对距离。实验上可以构造两组位置 (m,n) 和 (m+k,n+k)，它们的相对距离相同。把同一组 q/k 向量旋转到这两个位置后，点积应该近似相等。

score_1 = dot(rope(q, pos=m), rope(k, pos=n)) score_2 = dot(rope(q, pos=m+k), rope(k, pos=n+k)) assert torch.allclose(score_1, score_2, atol=1e-5)

这个实验检查的是位置机制的数学性质，不是模型长上下文能力。长上下文能力还需要真实任务、训练长度、cache 实现和服务端位置 ID 一起验证。

2.9 编程练习 4：验证 RoPE 相对位置性质

这一练习要求你写一个小测试：固定 q/k 内容，选择不同绝对位置但相同相对距离，比较旋转后的 dot product。通过测试后，你应该能解释为什么关键等式是 R_m^T R_n = R_{n-m}。

2.9A 位置编码工程选择：训练长度、外推和服务成本

机制位置进入方式优点风险
Learned absolute PE查表加到 embedding简单，训练内长度效果稳定长度外推差，max length 固定
Sinusoidal固定函数加到 embedding无参数，可计算任意位置不保证长上下文任务稳定
RoPE旋转 Q/K点积天然含相对位置，主流 LLM 常用长上下文 scaling 需要额外验证
ALiBi给 attention score 加距离 bias外推直觉强，简单表达能力和训练设置有关

服务端还要处理 position id 和 KV cache：prefill 阶段写入每个 token 的 K/V，decode 阶段新 token 必须使用正确的下一个 position。如果 position id 和 cache 长度错位，模型会读到“看起来存在但位置语义不对”的缓存。

2.9B RoPE Scaling 与长上下文发布

RoPE scaling 不是把配置里的最大长度改大就完事。不同方法会改变位置到相位的映射方式，比如 NTK-aware scaling、YARN、LongRoPE 等。它们要和 continued training 数据、评估集、服务引擎和发布报告一起绑定。

  • 配置要清楚：base、scaling factor、max position、训练长度和推理长度不能只写一个“支持 128K”。
  • 数据要匹配：如果 continued training 没有长上下文样本，配置外推很难单独支撑能力声明。
  • 评估要分层：区分短上下文保持、长距离检索、多跳推理、长文摘要、代码仓库级任务。
  • 服务要验证：position id、prefix cache、sliding window、chunked prefill 都会影响结果。

2.10 本章小结

  • Embedding 是用 token ID 查可训练向量，不需要显式构造 one-hot。
  • word2vec/GloVe 说明了“上下文统计可以塑造语义空间”。
  • LLM 的 input embedding 只是 layer-0 表示，真正语义来自上下文化 hidden state。
  • Self-Attention 本身没有顺序信号，位置编码是必需的。
  • Sinusoidal 用多尺度 sin/cos 函数表示位置；RoPE 用旋转 Q/K 让点积依赖相对位置。
  • 长上下文能力不能只靠改配置声明，必须绑定训练、评估和服务实现证据。

2.11 In-Context Learning 入门：为什么 prompt 会影响生成

Decoder-only LM 看到的是 token 序列，输出的是下一个 token 的条件分布。prompt 的作用，是改变当前上下文里的 token 序列，从而改变模型在每个位置的 hidden state 和 logits。

Zero-shot、Few-shot 和 In-Context Learning

Zero-shot 是直接给任务描述；few-shot 是在 prompt 里放几个示例；in-context learning 是模型在不更新参数的情况下，根据上下文里的模式调整输出。这里没有发生梯度更新，变化来自上下文计算。

长上下文中的位置问题

示例放在哪里、目标问题离示例多远、位置编码是否稳定、KV cache 是否一致，都会影响生成质量。这也是为什么本章的位置表示会一路影响到 Ch08 生成和 Ch10 推理服务。
