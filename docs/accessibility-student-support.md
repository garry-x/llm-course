# Accessibility and Student Support Guide

本指南用于把课程从“材料完整”推进到“可被不同背景学生公平参与”。它不替代学校正式政策；教师应按本校教务、残障服务、心理健康和数据保护规则填写具体联系人、流程和截止时间。

## 适用范围

适用于所有课程活动：讲授、讨论课、Office Hours、编程作业、书面题、阅读复盘、同伴 review、训练工程 capstone、推理工程 capstone、项目展示和复核。

## 学术便利安排

学生如有正式学术便利安排，应尽早联系教师或课程邮箱。课程团队应在不降低学习目标的前提下提供合理调整，例如：

- 延长作业或考试时间。
- 调整展示形式，例如现场展示、录制展示或一对一问答。
- 提供可访问格式的讲义、作业说明和评分 rubric。
- 允许辅助技术参与阅读、编码、记录和展示。
- 为长期健康、家庭或突发情况提供个案处理流程。

处理原则：

- 不要求学生在公开讨论区披露个人健康或身份信息。
- 只向需要执行便利安排的教师或助教共享必要信息。
- 便利安排不得要求学生绕过核心学习目标，例如必须能解释自己的代码、推导和实验结论。

## 课程材料可访问性

发布课程材料时应检查：

| 材料 | 最低要求 |
|------|----------|
| HTML 章节 | 标题层级清楚，链接文字有意义，图像有 `alt` 或邻近文字说明 |
| Markdown handout | 表格有表头，命令块可复制，长链接有可读标题 |
| 公式 | 公式附近有文字解释关键变量、shape 或边界条件 |
| 图示 | 不只依赖颜色表达差异，必要时补充文字说明 |
| 视频或录屏 | 若用于正式评分，应提供字幕、讲义或文字摘要 |
| 作业说明 | 明确输入输出、运行命令、评分项、隐藏测试类别和复核方式 |

每次发布前至少运行：

```bash
.venv/bin/python verify_course.py
```

教师还应抽查一章、一份作业 README 和一个 capstone README，确认屏幕阅读器或纯文本阅读时仍能理解任务。

## 学生支持渠道

课程应提供至少三类支持渠道：

| 渠道 | 适合问题 | 响应目标 |
|------|----------|----------|
| 公开讨论区 | 概念、公开测试、环境配置、阅读问题 | 1-2 个工作日 |
| 私密消息或课程邮箱 | 个人情况、便利安排、评分争议、协作披露 | 2 个工作日 |
| Office Hours | Debug、shape 推理、项目设计、复现失败 | 按周安排 |

教师应在开课第一周说明：

- 哪些问题应发公开区，哪些应私下联系。
- 助教能看哪些代码，不能看哪些代码。
- 隐藏测试、他人提交和评分脚本不得共享。
- 紧急健康或安全问题应联系学校正式支持渠道，而不是只等待课程组回复。

## 学习困难与干预

课程团队应使用低风险信号识别学生是否需要支持：

- prerequisite diagnostic 未达到 Ready 或 Borderline。
- 连续两次作业公开测试无法本地运行。
- quick check 或 exit ticket 反复显示同一基础概念困难。
- capstone milestone 缺少可复现命令、数据说明或日志。
- 学生在复核或 Office Hours 中暴露明显环境/时间管理障碍。

建议干预：

- 分配补救材料，例如 [数学与 PyTorch 先修复习](math-prerequisites.md)。
- 安排一次短 Office Hours 定位最小复现。
- 要求提交简化版里程碑，先证明数据、命令和指标闭环。
- 对涉及个人情况的问题转入私密渠道并按学校政策处理。

## 公平评分与便利边界

合理便利安排改变访问方式，不改变评分目标。评分时保持以下边界：

- 可以延长时间、调整提交形式或允许辅助技术。
- 不降低必须掌握的核心能力，例如 shape 推理、代码解释、实验复现和来源审计。
- 不泄露隐藏测试、标准答案或其他学生提交。
- 若便利安排影响同伴 review 或团队项目，应明确每位成员的贡献记录。

## 团队项目支持

团队项目应在 proposal 和最终报告中说明：

- 团队规模和成员角色。
- 每位成员的具体贡献。
- 是否有外部协作者或另一门课共享项目。
- 若成员因便利安排、健康或突发情况改变贡献，应私下通知教师并在报告中保持事实准确。

教师可以要求团队补交贡献说明、复现记录或一对一问答，以确认评分公平。

## 记录与反馈闭环

不记录学生个人敏感信息到公开材料。课程运行后只把聚合证据写入 [Course Operations and Improvement Log](course-operations-log.md)，例如：

- 作业环境问题数量。
- Office Hours 高频问题。
- 可访问性问题类别。
- 材料需要补充的说明。
- 因便利安排暴露出的流程缺口。

若同一问题影响多名学生，应更新相关章节、作业 README、rubric、Office Hours 模板或 syllabus。
