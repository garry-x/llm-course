10.1 推理瓶颈分析——Roofline 模型视角

你已经构建了模型、训练了它、微调了对齐。现在，是时候让它在实际应用中运行了。推理优化就是让模型跑得更快、更便宜、支持更长的上下文。

推理分为两个阶段，各自的瓶颈不同：

  • Prefilling（预填充）：处理整个 prompt。大量矩阵乘法可并行——计算密集（compute-bound）。瓶颈在 GPU 算力（FLOPS）。
  • Decoding（解码）：逐 token 生成。每步只处理 1 个 token——带宽密集（memory-bound）。GPU 大部分时间在等待将参数从 HBM 搬运到计算单元。瓶颈在内存带宽。

Roofline 模型

Roofline 模型是一个可视化工具，横轴是算术强度（Arithmetic Intensity = FLOPs / Bytes），纵轴是性能（FLOPs/s）。Prefilling 的算术强度高，位于 Roofline 的"计算受限区"（右上斜线区域）；Decoding 的算术强度极低，位于"带宽受限区"（水平平台区域）。

这解释了为什么增加 batch size 对 Decoding 帮助很大——更大的 batch 提高了算术强度，将操作推回计算受限区。这也解释了为什么 KV Cache、量化、GQA 等技术对 Decoding 特别有效——它们都减少了内存搬运量。

10.2 KV Cache——消除 Decoding 的重复计算

在自回归生成中，每一步计算注意力时，我们都在重复计算之前所有 token 的 Key（K）和 Value（V）矩阵。KV Cache 的核心思想是将之前步的 K 和 V 缓存起来，避免重复计算。

10.3 编程练习 1：实现 KV Cache

练习 1：带 KV Cache 的自注意力层

任务：实现 AttentionWithCache，支持增量解码时的 KV 缓存。

要求：

  • 若无 cache（首步），正常计算所有位置的 K、V 并缓存
  • 若有 cache（后续步），拼接历史 K/V 与新计算的 K/V
  • 返回 (output, new_cache)
  • 实现因果掩码时注意 的情况（ 但 ）

import torch import torch.nn as nn import torch.nn.functional as F class AttentionWithCache(nn.Module): """支持 KV Cache 的自注意力层""" def __init__(self, d_model, n_heads, dropout=0.1): super().__init__() assert d_model % n_heads == 0 self.d_model = d_model self.n_heads = n_heads self.d_head = d_model // n_heads self.q_proj = nn.Linear(d_model, d_model, bias=False) self.k_proj = nn.Linear(d_model, d_model, bias=False) self.v_proj = nn.Linear(d_model, d_model, bias=False) self.o_proj = nn.Linear(d_model, d_model, bias=False) self.dropout = nn.Dropout(dropout) def forward(self, x, cache=None): """ x: (batch, seq_len, d_model) — 当前步输入 cache: {"k": Tensor, "v": Tensor} """ batch, seq_len, _ = x.shape # 投影 QKV Q = self.q_proj(x).view(batch, seq_len, self.n_heads, self.d_head) K = self.k_proj(x).view(batch, seq_len, self.n_heads, self.d_head) V = self.v_proj(x).view(batch, seq_len, self.n_heads, self.d_head) Q = Q.transpose(1, 2) # (batch, n_heads, seq_len, d_head) K = K.transpose(1, 2) V = V.transpose(1, 2) # KV Cache: 拼接历史 K/V if cache is not None: K = torch.cat([cache["k"], K], dim=2) V = torch.cat([cache["v"], V], dim=2) new_cache = {"k": K, "v": V} # 注意力（Q 和 K/V 可能长度不同） scale = self.d_head ** 0.5 scores = (Q @ K.transpose(-2, -1)) / scale mask = self._causal_mask(K.size(2), Q.size(2), x.device) scores = scores.masked_fill(mask == 0, float('-inf')) weights = torch.softmax(scores, dim=-1) weights = self.dropout(weights) out = weights @ V out = out.transpose(1, 2).contiguous().view(batch, seq_len, -1) out = self.o_proj(out) return out, new_cache def _causal_mask(self, L_kv, L_q, device): """因果掩码：支持 L_kv > L_q（已有缓存时）""" mask = torch.triu(torch.ones(L_q, L_kv, device=device), diagonal=1) == 0 if L_kv > L_q: mask = mask[:, -L_kv:] # 只保留最后 L_q 列 return mask # Decoding 循环 cache = None x = embed(torch.tensor([[bos_id]])) for step in range(max_len): out, cache = attn(x, cache=cache) logits = lm_head(out[:, -1, :]) token_id = torch.multinomial(torch.softmax(logits, dim=-1), 1) x = embed(token_id)

def kv_cache_memory_analysis(n_layers, n_kv_heads, d_head, seq_len, batch_size=1, dtype_bytes=2): """ 分析 KV Cache 的显存占用 Returns dict with cache_per_layer_mb, total_cache_gb, etc. """ # 每层: K 和 V 各一份 cache_per_layer = 2 * seq_len * n_kv_heads * d_head * dtype_bytes total_cache = cache_per_layer * n_layers d_model = n_kv_heads * d_head # 粗略估计: 每层 ~8 * d_model^2 个参数 (Q,K,V,O + FFN1,FFN2) params_per_layer = 8 * d_model * d_model total_params = params_per_layer * n_layers total_params_gb = total_params * dtype_bytes / (1024**3) result = { "cache_per_layer_mb": cache_per_layer / (1024**2), "total_cache_gb": total_cache / (1024**3), "total_params_gb": total_params_gb, "cache_to_params_ratio": (total_cache / (1024**3)) / total_params_gb, } return result # GPT-2 XL (MHA: n_kv_heads = n_heads) gpt2 = kv_cache_memory_analysis(12, 12, 64, 1024) print(f"GPT-2 XL: KV Cache {gpt2['total_cache_gb']:.2f} GB " f"(参数: {gpt2['total_params_gb']:.1f} GB)") # Llama 3 8B (GQA: n_kv_heads=8, n_heads=32) llama = kv_cache_memory_analysis(32, 8, 128, 8192) print(f"Llama 3 8B: KV Cache {llama['total_cache_gb']:.2f} GB " f"(参数: {llama['total_params_gb']:.1f} GB)") # DeepSeek-V4 (MLA: 等效 n_kv_heads=1) v4 = kv_cache_memory_analysis(60, 1, 128, 128000) # 标准 GQA (kv_heads=8) 对比 v4_gqa = kv_cache_memory_analysis(60, 8, 128, 128000) print(f"DeepSeek-V4 MLA: KV Cache {v4['total_cache_gb']:.2f} GB " f"(标准 GQA: {v4_gqa['total_cache_gb']:.1f} GB)")

💡 MLA 为何如此有效？标准 MHA 每头独立 K/V，GQA 分组共享。MLA 更进一步——将 K/V 投影到低维潜空间（latent space），缓存压缩后的向量而非完整 K/V。DeepSeek-V2 声称将 KV Cache 减少了 8-16x。V4 的 CSA+HCA 在此基础上再次压缩注意力计算。

10.5 DeepSeek V4 CSA+HCA——压缩注意力实现 1M 上下文

DeepSeek-V4 引入了两种新型注意力机制：压缩滑动注意力（CSA）和超压缩全局注意力（HCA）。

CSA（压缩滑动注意力）

传统滑动窗口的窗口大小固定（如 4096）。CSA 对窗口内的 K/V 进行 4x 压缩——用一个小的 MLP 压缩器将 4 个相邻 token 的 K/V 压缩为 1 个表示。这使滑动窗口的"有效感知范围"扩大 4 倍，计算量和 KV Cache 不变。

公式 10.3 CSA

HCA（超压缩全局注意力）

将全局序列的 K/V 压缩 128 倍——每 128 个 token 的 K/V 被平均池化为 1 个表示。这使得模型能在 1M 上下文上执行全局注意力，计算量仅相当于标准注意力的 1/128。

公式 10.4 HCA

V4 注意力架构全景

V4 同时使用三种注意力，通过可学习的门控组合 ：

  • 局部精确注意力 — 最近 4096 tokens，标准注意力保证精度
  • CSA — 覆盖 16K 范围（4x 窗口扩充）
  • HCA — 覆盖整个 1M 上下文（128x 压缩）
🔑
效果：总计算量仅为标准 1M 上下文注意力的 27%，使 1M 上下文在 H800 GPU 上实际可用。

# 预期: INT8 量化误差约 0.5-2%
def quantize_per_channel(tensor): """ 对称 per-channel INT8 量化 对输出维度（最后一维）的每个 channel 独立计算 scale """ # tensor shape: (out_features, in_features) # 对每个 output channel 计算 scale abs_max = tensor.abs().max(dim=-1, keepdim=True).values # (out, 1) scales = abs_max / 127.0 scales = scales.clamp(min=1e-8) # 防止除零 quantized = torch.round(tensor / scales).clamp(-127, 127) quantized = quantized.to(torch.int8) return quantized, scales.squeeze(-1) # (out,), (out,) def dequantize(quantized, scales): """反量化：x_hat = x_q * s""" return quantized.float() * scales.unsqueeze(-1) def nrmse(original, reconstructed): """归一化均方根误差 (%)""" mse = ((original - reconstructed) ** 2).mean() std = original.std() return (torch.sqrt(mse) / std).item() * 100 # 测试 torch.manual_seed(42) weight = torch.randn(4096, 4096) * 0.02 q_w, s = quantize_per_channel(weight) dq_w = dequantize(q_w, s) print(f"INT8 量化误差: {nrmse(weight, dq_w):.2f}%") # GPTQ/AWQ 原理简述 # GPTQ: 使用 Hessian 矩阵加权量化误差，逐列优化 # AWQ: 识别 1% 的重要通道 (salient channels)，用缩放因子保护 # 两者都在 INT4 下达到接近 FP16 的精度

💡 SmoothQuant 的技巧：激活值中的 outlier 是量化的主要挑战。SmoothQuant 通过在数学上等价地将激活值的难度"迁移"到权重上——对激活除以 s，对权重重乘 s——使两者都更容易量化。这个技巧在 INT8 量化中被广泛采用。

10.7 FlashAttention——IO 感知的注意力加速

FlashAttention（Dao et al., 2022）是近年来最重要的注意力加速技术。它不改变数学结果，而是通过 IO-aware 算法 大幅减少 GPU HBM 访问次数。

标准注意力的问题：HBM 瓶颈

标准注意力将 写入 HBM，读取算 softmax，再将 写回。这是 的 HBM 读写——当 L=128K 时，约 32GB 的 HBM 流量（仅存储一个中间矩阵）。

FlashAttention 的三个关键思想

  • Tiling（分块）：将 QKV 分块到 SRAM（片上缓存，~192KB/A100）中计算，避免中间矩阵写入 HBM
  • Recomputation（重计算）：反向传播时不存储注意力权重，重新计算——以计算换内存
  • Online Softmax：分块计算 softmax 的算法——不需要先算所有 S 再做 softmax，每处理一个块就更新 softmax 统计量
HBM 访问量对比 (L=4096, d=128): 标准注意力: ~4L² + 3Ld = ~68M 字节 FlashAttention: ~4Ld = ~2M 字节 加速比 ≈ L / 4d = 8x! 当 L=128K: 加速比 ≈ 250x!
🔑
FlashAttention-3：利用 Hopper 架构的 Tensor Core 和异步拷贝，在 H100 上达到超过 60% 的理论 FLOPS 利用率。这一代将 FP8 支持与 FlashAttention 结合，进一步加速。

10.8 编程练习 4：实现最小 RAG 流水线

RAG（检索增强生成）是解决 LLM 知识局限和幻觉的最实用方案。它给模型"开卷考试"——不需要把所有知识存储于参数中，只需知道如何阅读检索到的文本。

练习 4：RAG 检索 + 生成流水线

任务：实现 SimpleRAG 类，包含文档分块、嵌入、检索、增强生成的全流程。

要求：

  • add_document(text) 将文档分块（每块 512 tokens，64 tokens 重叠），计算每块的嵌入向量
  • retrieve(query, top_k=3) 查询编码后做余弦相似度搜索，返回最相关的 top_k 块
  • query(question) 检索 → 上下文注入 prompt → LLM 生成回答
  • 支持稠密检索（嵌入相似度），用 np.dot 和 np.linalg.norm 实现余弦相似度

# 如果文档中无相关信息，应明确说明"未找到相关信息"
import numpy as np from typing import List, Tuple class SimpleRAG: """简化的 RAG 系统：分块 → 嵌入 → 检索 → 增强生成""" def __init__(self, embed_model, llm, chunk_size=512, overlap=64): self.embed_model = embed_model self.llm = llm self.chunk_size = chunk_size self.overlap = overlap self.documents = [] self.embeddings = None def add_document(self, text: str): """分块并嵌入""" tokens = text.split() chunks = [] start = 0 while start < len(tokens): end = min(start + self.chunk_size, len(tokens)) chunk = ' '.join(tokens[start:end]) chunks.append(chunk) start += self.chunk_size - self.overlap self.documents.extend(chunks) chunk_embs = [self.embed_model.encode(c) for c in chunks] embs = np.array(chunk_embs) if self.embeddings is None: self.embeddings = embs else: self.embeddings = np.concatenate([self.embeddings, embs], axis=0) def retrieve(self, query: str, top_k: int = 3) -> List[Tuple[str, float]]: """余弦相似度检索""" query_emb = self.embed_model.encode(query) scores = (self.embeddings @ query_emb) / ( np.linalg.norm(self.embeddings, axis=1) * np.linalg.norm(query_emb) ) top_indices = np.argsort(scores)[-top_k:][::-1] return [(self.documents[i], scores[i]) for i in top_indices] def query(self, question: str) -> str: """检索 + 生成""" context_chunks = self.retrieve(question, top_k=3) context = "\n\n".join( [f"[文档 {i+1}]: {chunk}" for i, (chunk, _) in enumerate(context_chunks)] ) prompt = f"""基于以下文档内容回答问题。 文档内容： {context} 问题：{question} 请基于文档内容给出准确、简洁的回答。如果文档中没有相关信息，请明确说明。""" return self.llm.generate(prompt)

💡 进阶 RAG 技巧：1) Hybrid Search = 稠密检索 + BM25 稀疏检索，通过 RRF 合并排序；2) Reranking：检索 Top-50 后用交叉编码器重排，只取 Top-3；3) Query Rewriting：检索前让 LLM 改写查询（消歧、指代消解）。这些技巧可以将 RAG 的命中率从 60% 提升到 90%+。

10.9 Agent 模式——从生成到行动

Agent 是 LLM 的终极应用形态——模型不再是"一次性回答一个问题"，而是自主执行多步骤任务，包括使用工具、规划步骤、反思结果、修正错误。

Agentic Loop: Plan → Act → Observe → Reflect

Agent 的核心循环：

  1. Plan（规划）：将任务分解为子步骤。如"我需要先搜索 XX，得到结果后分析，再执行下一步"
  2. Act（行动）：生成工具调用（如搜索、执行代码、读写文件）。工具调用输出为结构化 JSON。
  3. Observe（观察）：接收工具执行结果。这可能是一次搜索的结果、一段代码的输出、或一个 API 的响应。
  4. Reflect（反思）：基于观察结果调整计划。如果结果不符合预期，Agent 应能修正策略。

Tool Use 的实现

模型生成特殊格式的输出（JSON），框架解析后调用外部函数。典型的工具定义包括工具名、描述、参数 schema（类似 OpenAI Function Calling 格式）。框架收到模型输出后，解析 JSON 并执行对应函数，将结果作为消息返回给模型。

安全挑战

赋予模型"行动能力"大幅增加了安全风险。核心防护：

  • 最小权限原则：Agent 只获得完成任务所需的最小工具集
  • 人类确认环：高影响操作（写文件、执行命令）需要人类批准
  • 输入消毒：对工具输出进行安全检查
💡
Claude Code 案例：Claude Code 是一个基于 Claude 的自主编码 Agent。它读取 PR 描述 → 搜索代码库 → 读取文件 → 执行命令 → 编写代码 → 提交 PR。每个步骤中 Agent 都可能"反思"前一步的结果并调整策略。生成循环预测的是下一个 token，Agentic Loop 决定的是下一个行动。

💡 预期观察：KV Cache 通常能加速 Decoding 5-10x（从 O(L²) 到 O(L)）；INT8 量化能额外加速 1.5-2x（更少的内存搬运）。注意 KV Cache 会增加显存使用（缓存了 K/V），而量化减少显存使用（权重缩小 4 倍）。

10.11 DeepSeek V2→V4 演进路线图——"删减"哲学

回顾 DeepSeek 从 V2 到 V4 的演变，可以看到一个清晰的"删减"哲学：

版本核心创新"删掉"了什么
V2MLA删 KV — 将 K/V 压缩到低维潜空间，Cache 减少 8-16x
V3MoE + MTP删参数 — 稀疏激活，总参量大但每步只激活 10%
R1GRPO删 Critic — 组内白化替代价值网络
V4CSA + HCA + OPD删注意力 — 压缩/稀疏化注意力计算，1M 上下文 27% FLOPs

每一代的创新都遵循同一个模式：找到当前架构中最浪费的部分，然后系统性地优化它。这个趋势的终点可能是完全不用"标准" Transformer——如 Mamba 等状态空间模型试图彻底干掉注意力机制。
