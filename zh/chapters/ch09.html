9.1 为什么预训练不够：从续写文本到助手行为

预训练模型擅长续写文本，但不一定会按用户意图回答问题。预训练目标是“预测下一个 token”，不是“帮用户完成任务”。Alignment 要补上这个行为差距。

HHH 目标

  • Helpful：按用户意图完成任务。
  • Honest：减少幻觉，知道不知道时要说明。
  • Harmless：避免有害输出。
技术数据类型是否需要 RL解决什么
SFTinstruction-response否学格式、角色、基础任务行为
RLHF/PPO偏好 + reward model + RL是用奖励优化行为
DPOchosen/rejected 偏好对否直接优化偏好，不单独训练 reward model
GRPOrollout group + reward是强化推理，省去 critic

9.1A 后训练控制循环

现代后训练不是几个技巧堆叠，而是一个控制循环：选 prompt，生成或标注候选，打分，按约束更新 policy，再在真实推理预算下评估。

信号教会模型什么不能单独证明什么工程检查
SFT demonstrations格式、工具调用、拒答风格答案比其他候选更优assistant mask、chat template、held-out slice
DPO preference pairs相对偏好chosen 一定客观正确长度偏置、标注分歧、安全/能力回归
RLVR/RFT reward探索能通过测试或 grader 的答案grader 没有漏洞false positive、reward hacking probe
GRPO/DAPO/GSPO长推理下的策略优化更长 reasoning 一定更值长度 P95、clip fraction、成本/正确率

9.2 SFT：核心是 Loss Masking

Supervised Fine-Tuning 用人工或高质量模型写好的 instruction-response 对继续训练模型。关键差异是：loss 只算 assistant response，不算 user prompt。

做法是把 prompt 部分 label 设为 -100，让 F.cross_entropy(..., ignore_index=-100) 跳过。

<|im_start|>user Please explain backpropagation.<|im_end|> <|im_start|>assistant Backpropagation is an algorithm...<|im_end|>

chat template 是模型接口的一部分。模板不一致，训练和推理看到的 token 分布就不一致。

9.3 编程练习 1：实现 SFT Training Loop

练习 1：SFT Training

  • SFTDataset 接受 {"instruction": "...", "response": "..."}。
  • 用 chat template 拼完整序列。
  • prompt 部分 label 设为 -100，response 保留真实 token id。
  • sft_step 执行 forward -> shift logits -> CE loss。
  • collate 时做 dynamic padding。
def sft_step(model, batch, optimizer): input_ids = batch["input_ids"].to(model.device) labels = batch["labels"].to(model.device) logits = model(input_ids) shift_logits = logits[:, :-1, :].contiguous() shift_labels = labels[:, 1:].contiguous() loss = F.cross_entropy( shift_logits.view(-1, shift_logits.size(-1)), shift_labels.view(-1), ignore_index=-100, ) loss.backward() optimizer.step() optimizer.zero_grad(set_to_none=True) return loss

9.3A SFT label mask 和有效 token 分母

多轮对话里，只有 assistant 回复参与 loss。user、system、工具返回、padding 通常都要 mask 掉。平均 loss 的分母是有效 assistant tokens，不是序列总长度。

9.3B SFT 数据管线

  • Chat template：训练、评估、推理必须一致。
  • 多轮 mask：只训练 assistant 回合，工具状态要按任务定义处理。
  • Packing/truncation：提升 token 利用率，但不能破坏对话边界和 label mask。
  • 数据混合：格式、代码、数学、安全、工具调用比例会直接改变模型行为。
  • 污染检查：评估集相似样本不能泄漏进 SFT。

9.4 Bradley-Terry 偏好模型和 RLHF 三部曲

偏好数据通常是同一 prompt 下的 chosen/rejected。Bradley-Terry 模型把“chosen 胜过 rejected”的概率写成 reward 差的 sigmoid：

P(chosen > rejected) = sigmoid(r_chosen - r_rejected)

传统 RLHF 三步：SFT 得到初始助手；训练 reward model；用 PPO 等 RL 方法优化 policy，同时用 KL 约束不要偏离 reference 太远。

9.5 编程练习 2：实现 LoRA

LoRA 冻结原权重 W，只训练低秩增量 BA：

W' = W + alpha/r * (B @ A)

练习 2：LoRA Layer

  • 冻结 base Linear。
  • 新增 lora_A: [r, in] 和 lora_B: [out, r]。
  • forward = base(x) + scaling * lora_B(lora_A(x))。
  • 支持把 LoRA 应用到 attention/MLP 指定模块。
class LoRALinear(nn.Module): def __init__(self, base, r=8, alpha=16, dropout=0.0): super().__init__() self.base = base for p in self.base.parameters(): p.requires_grad = False self.lora_A = nn.Linear(base.in_features, r, bias=False) self.lora_B = nn.Linear(r, base.out_features, bias=False) self.scaling = alpha / r self.dropout = nn.Dropout(dropout) nn.init.zeros_(self.lora_B.weight) def forward(self, x): return self.base(x) + self.scaling * self.lora_B(self.lora_A(self.dropout(x)))

9.5A LoRA/PEFT 工程账

  • 省的是训练状态：base 权重不训练，就不需要为它保存 gradient 和 optimizer states。
  • rank 越高容量越大：但显存和过拟合风险也增加。
  • target modules 要选对：常见是 Q/V projection、attention output、MLP；不是随便全塞。
  • alpha 和 dropout：控制增量尺度和正则。
  • merge inference：上线时可以把 LoRA 增量合并进 base，也可以多 adapter 动态服务。

9.6 DPO：为什么不需要单独 Reward Model

DPO 从偏好数据直接优化 policy，使 chosen 相对 rejected 的 logprob 改善幅度大于 reference model。直观上，它比较的是“当前模型相对参考模型，更偏向 chosen 多少”。

loss = -log sigmoid(beta * [(log pi(y_w)-log pi_ref(y_w)) - (log pi(y_l)-log pi_ref(y_l))])

reference model 很关键：它限制 policy 不要为了赢偏好对而大幅漂移。

9.6A 偏好数据质量

DPO/RLHF 对数据偏差很敏感。常见问题包括：chosen 更长、格式更花、标注者偏好不一致、安全拒答混进能力任务、评估集泄漏。

训练前应统计 chosen/rejected 长度、语言、主题、toxicity、format、标注分歧和来源。不要等 loss 下降后才发现模型只学会“写长一点”。

9.6B Reward 和 Verifier 设计

不同 reward 回答的问题不同：

reward 类型适合风险
人类偏好主观质量、帮助性、安全性长度/风格偏置、标注噪声
单元测试/判题器代码、数学、可验证任务过拟合测试、reward hacking
模型 judge规模化评估judge 偏见、和 policy 共谋
工具环境回报agent 任务状态记录复杂、信用分配难

9.7 编程练习 3：实现 DPO Loss

练习 3：DPO Loss

  • 计算 policy 对 chosen/rejected completion 的 sequence logprob。
  • 计算 reference model 的对应 logprob。
  • 用 DPO 公式得到 batch loss。
def dpo_loss(pi_w, pi_l, ref_w, ref_l, beta=0.1): pi_logratios = pi_w - pi_l ref_logratios = ref_w - ref_l logits = beta * (pi_logratios - ref_logratios) return -F.logsigmoid(logits).mean()

9.8 GRPO：Group Relative Policy Optimization

GRPO 面向推理 RL：对同一个 prompt 采样一组回答，用组内 reward 做相对标准化，不训练单独 critic。

advantage_i = (reward_i - mean(group_rewards)) / std(group_rewards)

它减少了 value model 复杂度，适合可验证 reward 的 reasoning 任务。DeepSeek-R1 系列展示了这种路线能让模型出现更长、更探索式的推理行为。

GRPO vs PPO

PPO 通常需要 policy、reference、reward model、critic/value model；GRPO 用 group relative advantage 替代 critic，系统更轻，但依赖 rollout 质量和 reward 可靠性。

9.8A GRPO 组内白化例子

同一 prompt 采样 4 个回答，reward 是 [1.0, 0.5, 0.0, -0.5]。先减均值，再除标准差。这样优化信号看的是组内相对好坏，而不是绝对 reward 尺度。

9.8B 从 GRPO 到 DAPO / GSPO

推理 RL 的工程 recipe 会加入动态采样、长度控制、clip-higher、token-level 或 sequence-level ratio、overlong reward shaping 等细节。它们的目标不是换名词，而是稳定长 CoT 训练、控制长度膨胀和减少无效 rollout。

9.8C 从研究到可部署证据链

reasoning RL 不能只报 benchmark 分数。还要报告：平均输出长度、P95 长度、pass rate、cost per correct answer、失败类型、拒答/安全回归、工具调用成功率和服务延迟。

9.8D Post-Training RL 系统：Rollout、权重同步和 stale policy

RL 系统包括 policy 训练进程、rollout workers、reward/verifier、reference model、数据队列和评估。异步 rollout 能提高吞吐，但样本可能来自旧 policy，产生 staleness。

最小 run ledger 要记录：policy version、prompt id、sampling 参数、reward version、reference version、rollout 时间、是否 stale、是否通过 verifier。

9.8E Agentic RL：从一个答案到一条轨迹

工具进入 RL 后，样本不再是一段文本，而是一条 trajectory：观察、动作、工具调用、工具返回、状态变化、最终答案和 reward。

  • 要记录工具 schema、调用参数、返回值、错误、重试和环境状态。
  • reward 可能来自最终结果，也可能来自中间状态。
  • 评估要 trajectory-aware，不能只看最终文本。

9.9 编程练习 4：PPO Clipped Objective 和 GRPO Advantage

练习 4：GRPO Advantage + Clipped Loss

  • 对每个 prompt 的 rollout group 做 reward whitening。
  • 计算新旧 policy 的 ratio。
  • 实现 clipped surrogate objective。
  • 加入 KL penalty 或 reference 约束。
adv = (rewards - rewards.mean(dim=1, keepdim=True)) / (rewards.std(dim=1, keepdim=True) + 1e-6) ratio = torch.exp(new_logprob - old_logprob) obj1 = ratio * adv obj2 = torch.clamp(ratio, 1 - eps, 1 + eps) * adv loss = -torch.min(obj1, obj2).mean()

9.10 编程练习 5：比较 DPO 和 GRPO

在 toy preference dataset 上比较两类方法：

  • DPO：用固定 chosen/rejected pairs，离线、稳定、实现简单。
  • GRPO：需要 on-policy rollout 和 reward，更接近推理任务，但系统复杂。

比较指标不只看训练 loss，还要看 win rate、长度变化、KL、reward hacking、输出成本和失败类型。

9.10A 对齐评估协议

单个 win rate 不够。最小评估矩阵应包含：

切片看什么
任务能力数学、代码、写作、检索、工具调用
安全拒答边界、越狱、敏感主题
格式JSON、引用、tool call schema
成本输出长度、reasoning tokens、延迟
回归预训练能力和 SFT 能力是否下降

9.10B Preference Optimization 训练日志

DPO 至少记录：chosen/rejected logprob、reference logprob、log-ratio、loss、KL proxy、长度、win rate。PPO/GRPO 还要记录 ratio、clip fraction、advantage 分布、reward 均值/方差、rollout pass rate。

9.11 On-Policy Distillation、Rejection Sampling 和可调推理预算

传统蒸馏容易把 teacher 的偏差和过时行为固化。On-policy distillation 让 student 或当前 policy 参与生成，再筛选高质量样本，形成更贴近部署分布的数据。

Rejection sampling 要记录生成模型、采样参数、过滤器、通过率和拒绝原因。可调推理预算则把快答、普通答、深度 reasoning 变成 runtime 策略，而不是单一模型能力声明。

9.12 编程练习 6：加载 LoRA Adapter 并合并推理

练习 6：LoRA Merge Inference

  • 加载 base model 和 LoRA adapter。
  • 确认 adapter target modules 和 base 架构匹配。
  • 把 delta_W = scaling * B @ A 合并进 base weight。
  • 比较 merge 前后同一输入 logits 是否一致。
def merge_lora_linear(layer): delta = layer.scaling * (layer.lora_B.weight @ layer.lora_A.weight) layer.base.weight.data += delta.to(layer.base.weight.dtype) return layer.base

9.13 Alignment Tax

Alignment tax 指对齐后某些原有能力、速度或自由度下降。常见表现：回答更保守、拒答变多、输出更长、代码能力下降、开放写作变模板、推理成本增加。

这不是“对齐不好”，而是要把收益和代价放在同一个评估矩阵里。

9.13A RLVR / RFT：可验证奖励不是万能奖励

RLVR/RFT 适合有可靠 grader 的任务：数学答案、代码测试、形式化证明、工具环境结果。它不适合主观写作、开放建议、价值判断这类难以验证的任务。

先验证 reward，再训练 policy。grader 的 false positive、false negative、覆盖范围和可被利用程度，都要进入报告。

9.14 本章小结

  • SFT 用 response-only loss 学助手格式和基础任务行为。
  • chat template、label mask、packing 和数据混合会直接影响 SFT 行为。
  • LoRA 冻结 base，只训练低秩增量，主要节省梯度和 optimizer states。
  • DPO 用 chosen/rejected pairs 直接优化偏好，不需要单独 reward model，但依赖 reference model 和偏好数据质量。
  • GRPO 用组内 reward 相对标准化做推理 RL，减少 critic 复杂度。
  • DAPO/GSPO/RLVR/RFT 的核心不是名词，而是 reward、rollout、约束和评估是否可靠。
  • Agentic RL 要记录完整 trajectory，而不是只看最终文本。
  • Alignment evaluation 要同时看能力、安全、格式、成本和回归。
  • Alignment tax 是工程取舍，必须量化。
