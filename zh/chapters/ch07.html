7.1 语言建模就是最大似然估计 MLE

LLM 的核心训练目标是 next-token prediction。统计上，它是在做 Maximum Likelihood Estimation (MLE)：让模型给真实训练序列更高概率。

自回归分解

p(x_1, ..., x_T) = product_t p(x_t | x_

这不是近似，而是概率链式法则。自回归模型把联合概率拆成一连串“给定前缀预测当前 token”的条件概率。

从 likelihood 到 cross entropy

最大化训练数据的 log-likelihood，等价于最小化 negative log-likelihood：

NLL = - mean_t log p_theta(x_t | x_

模型最后输出 vocabulary logits，再通过 softmax 参数化离散分布。工程实现里用 F.cross_entropy，它内部会稳定地做 log_softmax。

KL 视角

MLE、最小化 cross entropy、最小化 KL(p_data || p_theta) 是同一个目标的三种说法，因为数据分布的 entropy 对模型参数是常数。

7.2 Perplexity

Perplexity (PPL) 是 NLL 的指数：

PPL = exp(NLL)

可以粗略理解成模型在每个位置平均有多不确定。这个解释只是帮助记忆，不要把 PPL 当成候选 token 数、事实准确率或指令跟随能力。

影响因素为什么影响 PPL报告时要写清
Tokenizertoken 粒度改变序列长度和平均 NLLtokenizer 和 vocab size
Dataset/split新闻、代码、对话、书籍难度不同数据集、split、过滤规则
Context length长上下文和截断策略会改变条件信息窗口长度、stride、mask 规则
Eval 实现BOS/EOS、padding、sliding window 会改变 loss命令、seed、loss mask

PPL 只能说明模型对给定 token 序列的概率拟合，不直接说明 factuality、安全性或生成质量。

7.3 编程练习 1：实现 DataLoader

训练的第一步是把文本变成模型能吃的 batch：

Text -> Tokenizer -> Token IDs -> fixed-length chunks -> Batch -> Model

每个样本返回 (x, y)，其中 y 是 x 右移一位的目标。

练习 1：Building a DataLoader

  • TextDataset.__init__ 读文件并一次性 tokenize 全语料。
  • __getitem__ 返回长度为 block_size 的 x 和右移一位的 y。
  • create_dataloader 支持 batch_size、shuffle、pin_memory、drop_last。
  • 不要在 __getitem__ 里重复 tokenize。
class TextDataset(Dataset): def __init__(self, data_path, tokenizer, block_size=1024): text = open(data_path, "r", encoding="utf-8").read() self.tokens = tokenizer.encode(text) self.block_size = block_size def __len__(self): return (len(self.tokens) - 1) // self.block_size def __getitem__(self, idx): start = idx * self.block_size end = start + self.block_size x = torch.tensor(self.tokens[start:end], dtype=torch.long) y = torch.tensor(self.tokens[start + 1:end + 1], dtype=torch.long) return x, y

7.3A 数据配比、Token Budget 和 Scaling Laws

真实预训练不是“跑几个 epoch”。LLM 通常用 token budget 描述训练量，因为语料巨大、样本会 pack 成连续 token 流，而且不同数据源按采样比例混合。

global_batch_tokens = micro_batch * seq_len * grad_accum * data_parallel training_steps = ceil(total_tokens / global_batch_tokens)

例如 micro_batch=4、seq_len=2048、grad_accum=8、data_parallel=16，每个 optimizer step 消耗 1,048,576 tokens。训练 20B tokens 约需 19,074 steps。

AdamW 内存账

训练内存不是只有参数。bf16 参数 2 bytes、bf16 gradient 2 bytes、AdamW 的 m/v fp32 各 4 bytes，合计每参数约 12 bytes。ZeRO-1 会 shard optimizer states，ZeRO-2 进一步 shard gradients，ZeRO-3/FSDP 会 shard 参数、梯度和 optimizer states。

数据混合

不同数据源的 token 质量不同。代码、数学、网页、书籍、论坛、合成数据的采样比例不是“数据量比例”，而是优化信号。比例调错，loss 仍然会下降，但模型能力会偏。

7.3B 预训练数据管线：质量、去重、隐私和采样比例

数据管线通常是：raw corpus -> extract -> normalize -> filter -> deduplicate -> shard -> tokenize -> sample。每一步都可能改变模型能力。

  • 过滤：去掉乱码、低质量页、重复模板、恶意内容，但不要把某些真实语域全过滤掉。
  • 去重：能减少 memorization 和 contamination，但过度去重会删除合法重复，例如代码模板、诗歌结构、法律条款。
  • 隐私和凭证：PII、API key、密码、私有仓库片段要进入训练 ledger，不能只靠“模型应该学不会”。
  • 版权风险：训练数据来源、许可、过滤规则和保留证据要可审计。

数据清洗不是一次性任务，而是优化循环：看训练曲线、评估切片、bad cases 和 contamination 报告，再调整过滤和混合比例。

7.3C 合成数据和蒸馏：把生成变成可控训练信号

合成数据在不同阶段作用不同：预训练可以补稀缺格式，SFT 可以扩 instruction coverage，RL/post-training 可以产生偏好或验证信号。它不是“越多越好”。

  • Rejection sampling：先生成多条，再用规则、模型或测试筛选，不是魔法监督。
  • 蒸馏：teacher 模型提供答案、reasoning trace 或偏好信号，student 学到的是 teacher 的行为分布和偏差。
  • 合成数据 ledger：要记录 generator、prompt 模板、采样参数、过滤器、通过率、版本和评估切片。

7.4 Cross-Entropy 的数值稳定性

Cross entropy 直接算 softmax 容易 overflow，因为 exp(logit) 可能非常大。稳定写法是 log-sum-exp trick：

logsumexp(z) = m + log(sum(exp(z - m))), where m = max(z)

F.cross_entropy 已经处理了这些细节。手写时不要先 softmax 再 log。

7.4A Cross-Entropy 梯度到底更新了什么

对 logits 的梯度是：

dL/dz = softmax(z) - one_hot(target)

目标 token 的 logit 被推高，其他 token 的 logit 按模型当前概率被压低。若使用 ignore_index，被 mask 的位置梯度全为 0。

Label smoothing 和 calibration

Label smoothing 不把目标分布当成绝对 one-hot，可以减少过度自信。Calibration 关心模型说“90%”时，真实正确率是否接近 90%。ECE 会把预测按置信度分桶，比较每桶 confidence 和 accuracy 的差距。

7.5 编程练习 2：手写 Cross-Entropy Loss

练习 2：Manual Cross-Entropy

  • 输入 logits [B,T,V] 和 targets [B,T]。
  • 实现 stable log-softmax。
  • 支持 ignore_index=-100。
  • 输出 mean loss，并和 F.cross_entropy 对齐。
def manual_cross_entropy(logits, targets, ignore_index=-100): V = logits.size(-1) flat_logits = logits.reshape(-1, V) flat_targets = targets.reshape(-1) valid = flat_targets != ignore_index z = flat_logits[valid] y = flat_targets[valid] m = z.max(dim=-1, keepdim=True).values log_probs = z - m - torch.log(torch.exp(z - m).sum(dim=-1, keepdim=True)) return -log_probs[torch.arange(y.numel(), device=y.device), y].mean()

7.6 学习率调度：Warmup + Cosine Decay

为什么需要 warmup

训练初期参数和 optimizer state 还不稳定，直接用大学习率容易让 loss spike 或 NaN。warmup 让学习率从 0 或很小值线性升到目标值。

为什么常用 cosine decay

训练后期需要更细的更新。cosine decay 比 step decay 更平滑，减少学习率突然跳变带来的曲线抖动。

lr(step) = max_lr * step / warmup_steps if step < warmup lr(step) = min_lr + 0.5*(max_lr-min_lr)*(1+cos(pi*progress)) otherwise

7.7 编程练习 3：实现 AdamW

AdamW 的关键是 decoupled weight decay：权重衰减不混进梯度的一阶/二阶矩估计里，而是单独作用在参数上。

练习 3：AdamW Optimizer

  • 维护 m 和 v。
  • 做 bias correction。
  • 参数更新前后正确应用 decoupled weight decay。
m = beta1 * m + (1 - beta1) * grad v = beta2 * v + (1 - beta2) * grad.pow(2) m_hat = m / (1 - beta1 ** step) v_hat = v / (1 - beta2 ** step) param.mul_(1 - lr * weight_decay) param.addcdiv_(m_hat, v_hat.sqrt().add(eps), value=-lr)

7.8 编程练习 4：实现 Warmup + Cosine Scheduler

练习 4：Learning Rate Scheduler

  • warmup 阶段线性升高。
  • 之后 cosine decay 到 min_lr。
  • 训练日志里同时记录 step、lr、consumed tokens。
def get_lr(step, warmup_steps, max_steps, max_lr, min_lr): if step < warmup_steps: return max_lr * step / warmup_steps progress = (step - warmup_steps) / max(1, max_steps - warmup_steps) progress = min(1.0, progress) coeff = 0.5 * (1.0 + math.cos(math.pi * progress)) return min_lr + coeff * (max_lr - min_lr)

7.9 Muon Optimizer：矩阵正交化为什么有用

Muon 的核心观察是：很多权重是矩阵，优化更新也可以按矩阵结构处理。对更新方向做近似正交化，可以让不同输出方向的更新更均衡，减少某些方向过度放大。

它不是 AdamW 的简单替代。比较时要看 loss、tokens/s、显存、稳定性、调参成本和任务切片。Muon 对矩阵参数更自然，对 embedding、bias、norm scale 这类参数通常还要用 AdamW 或特殊规则。

7.10 FP8/FP4 混合精度训练

低精度训练的目标是降低显存和带宽压力，提高吞吐，但不能把所有 tensor 都盲目降精度。

  • E4M3：mantissa 更长，精度更好，动态范围较小，适合部分 activation/weight。
  • E5M2：动态范围更大，精度更低，适合 gradient 等范围更不稳定的 tensor。
  • Block-wise scaling：按 block 记录 scale，让低精度值覆盖局部范围。
  • FP4 QAT：更激进，需要量化感知训练和更严格验证。

7.10A 低精度训练 ledger：FP8、MXFP8、NVFP4

低精度选择不能只看 bit width，要看 tensor 角色、硬件支持和误差敏感度。

内容常见策略原因
Matmul activations/weights可用 FP8/MXFP8/NVFP4吞吐和带宽收益最大
Norm 统计量保留 fp32 或更高累计精度均值/方差/RMS 对误差敏感
Softmax/logits关键部分保高精度防 overflow/underflow 和 NaN
Optimizer states常保 fp32 或专门压缩长期累计误差会影响收敛

低精度验证要同时看质量、稳定性和吞吐：loss 曲线、eval PPL、NaN 率、gradient norm、tokens/s、MFU 和恢复能力都要记录。

7.11 编程练习 5：完整训练循环

练习 5：Training Loop

  • 从 DataLoader 取 batch。
  • 前向得到 logits，计算 cross entropy。
  • 反向传播，做 gradient clipping。
  • optimizer step，scheduler 更新 lr。
  • 记录 loss、PPL、lr、grad_norm、tokens/s。
  • 定期保存完整 checkpoint。
for step, (x, y) in enumerate(loader): x, y = x.to(device), y.to(device) with torch.autocast(device_type="cuda", dtype=torch.bfloat16, enabled=use_amp): logits = model(x) loss = F.cross_entropy(logits.view(-1, logits.size(-1)), y.view(-1)) scaler.scale(loss).backward() scaler.unscale_(optimizer) grad_norm = torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm) scaler.step(optimizer) scaler.update() optimizer.zero_grad(set_to_none=True) lr = get_lr(step, warmup_steps, max_steps, max_lr, min_lr)

7.11A 一个 optimizer step 的 ledger

一次 step 至少要记录：global step、consumed tokens、micro batch、gradient accumulation、data shard、loss、lr、grad norm、tokens/s、显存峰值、checkpoint id。没有这些信息，resume 后曲线跳了也很难判断原因。

7.11B 怎样读训练日志

先区分三类问题：

  • 优化问题：loss spike、grad norm 爆、lr 不合适、初始化或精度错误。
  • 数据问题：重复、污染、编码错误、数据切片质量差。
  • 系统问题：tokens/s 降、dataloader 卡、通信等待、checkpoint 写入阻塞。

grad_norm 偶尔有尖峰不一定坏，但持续升高、和 loss spike 同步、或 resume 后突然换分布，都要查。

7.11C 训练异常 Runbook

  1. 先定位异常发生在哪个 step、哪个 token 区间、哪个 data shard。
  2. 把 loss spike 分成一次性数据异常、可恢复优化震荡、不可恢复 NaN/Inf。
  3. NaN/Inf 最短路径：查 logits、loss、grad、norm 输出、mask、lr、AMP scaler。
  4. resume 后曲线不连续，先查 checkpoint 是否包含 model、optimizer、scheduler、scaler、RNG、dataloader state。
  5. throughput 下降不等于模型变慢，先拆 dataloader、通信、checkpoint、thermal throttling。
  6. 每个修复都要做最小实验，不要同时改三四个变量。

7.13 DualPipe Pipeline Parallelism：减少 pipeline bubbles

传统 pipeline parallelism 会把模型层切到不同设备上，但 micro-batch 在流水线里有填充和排空阶段，设备会等待，这叫 pipeline bubble。

DualPipe 通过双向调度，把 forward/backward 的不同阶段更紧密地交错，减少等待时间。它解决的是大规模训练调度效率问题，不改变模型数学函数。判断它是否值得用，要看模型深度、micro-batch 数、通信拓扑、activation memory 和实现复杂度。

7.15 分布式训练：从单 GPU 到千卡集群

先问两个问题：模型和训练状态是否放得下？放得下以后，硬件是否跑得高效？

每卡显存账

参数通常不是唯一大头。要同时算 parameters、gradients、optimizer states、activations、temporary buffers、communication buffers 和 fragmentation。

四类并行

策略做什么主要代价
Data Parallel每卡一份模型，分 batch梯度 AllReduce
Tensor Parallel切矩阵乘法层内通信
Pipeline Parallel按层切模型pipeline bubble 和调度复杂度
Expert ParallelMoE experts 分布到不同设备token dispatch/combine 和负载倾斜

ZeRO 和 FSDP

ZeRO/FSDP 的核心是 shard 模型状态。ZeRO-1 shard optimizer states，ZeRO-2 再 shard gradients，ZeRO-3/FSDP 进一步 shard parameters。省显存的同时，会引入 ReduceScatter、AllGather 等通信成本。

MFU

Model FLOPs Utilization 衡量训练系统是否真正用上了硬件算力。tokens/s 下降时，要拆解是计算、通信、dataloader、checkpoint 还是调度问题。

7.15A Training Run Plan：把实验变成可恢复工程过程

  1. 先定义训练目标：预训练、继续训练、领域适配，还是后训练前的 warm start。
  2. 用 token progress 管理训练，不用“epoch 口号”。
  3. checkpoint 保存训练状态，不只是模型权重。
  4. eval cadence 服务决策，不要为了频繁 eval 打断训练吞吐。
  5. 分阶段扩容：单卡 sanity -> 小规模多卡 -> 全量集群。
  6. failure recovery 也要算进成本模型。

7.15B 工业训练系统检查

  • 数据：来源、过滤、去重、隐私、污染、采样比例要可追溯。
  • 并行策略：必须绑定模型结构和集群拓扑，不是随便选 DDP/FSDP/TP/PP。
  • Checkpoint：同时覆盖恢复和发布，能证明 resume 后曲线连续。
  • 评估：决定继续训练、调数据、降学习率、回滚还是停止。
