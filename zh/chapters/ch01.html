1.1 Tokenizer 的定位：从文本接口到工程代价

Tokenizer 是 LLM 的第一层接口。它将原始字符串转换为 token ID，同时决定了序列长度、嵌入矩阵大小、上下文窗口利用率以及服务计费指标。本章把 tokenization 当作训练和推理共用的模型约束（model contract），而不是一个预处理细节。

  • 训练侧：更改 tokenization 规则会改变 token 分布、batch 内 token 数量、嵌入参数量以及数据质量诊断结果。
  • 推理侧：同一文本的 token 数量影响上下文容量、KV Cache 内存、延迟及调用成本。
  • 发布侧：检查点、聊天模板、特殊 token 和 tokenizer 文件必须一起版本化管理，否则模型接口将不匹配。

下面我们用最简的 BPE 实现来说明核心步骤：统计 pair 频率、按优先级合并、保存合并表，编码和解码用同一套规则。理解了这套流程后，对比 WordPiece、Unigram LM 和 byte-level tokenizer 等变体时，就能看到它们的工程权衡。

1.3 什么是 Token——信息论视角

大语言模型不直接处理字符或字符串，它处理的是 token——文本的最小不可分割单元。每个 token 对应一个整数 ID，模型看到的是一串整数，而不是文本。

从信息论视角看 Tokenization

假设一段文本的信息量为 比特。如果将它表示为 n 个 token，每个 token 平均携带 比特信息。tokenizer 的目标是：给定词汇表大小 ，最小化每段文本所需的 token 数量。这源自 率失真理论（Rate-Distortion Theory） 中的经典权衡：

  • 更大的词汇表（大 V）：每个 token 携带的信息多，序列较短，但嵌入矩阵 参数量大。
  • 更小的词汇表（小 V）：参数更少，但序列更长，计算量和推理延迟也随之增加。

为什么不直接使用单词？——Zipf 定律

自然语言遵循 Zipf 定律：按频率排序的第 个单词的频率与 成正比。这意味着大约一半的单词只出现一次——如果使用"单词"级别的词汇表，稀有词会带来严重的稀疏问题。此外，词级词汇表无法处理词表外词（OOV）。

子词 Tokenization 的核心想法是：在"字符"和"单词"之间找到一种中间粒度。常见词（如 "the"、"and"）保留为完整 token，罕见词被分解为更小的子词（例如 "tokenization" → "token" + "ization"），任何 OOV 词都可以回退到字符甚至字节级别。

BPE 的字节级设计

GPT 系列的 BPE tokenizer 在 字节 级别操作——基础词汇表是 256 个字节值（0x00-0xFF），而不是字符。这意味着无论文本是英文、中文、阿拉伯语还是 Emoji，底层都统一表示为字节序列。UTF-8 编码将 Unicode 字符映射为 1-4 个字节，BPE 在这个字节序列上进行合并。其好处是：任何字符串均可被 tokenize，不存在 OOV。

🔑
关键数据： GPT-4 的 cl100k_base 词汇表约 10 万 token，Llama 3 约 12.8 万 token。中/日等 CJK 语言通常消耗的 token 数量是英文的 2-3 倍，因为每个汉字约需要 1.5-2 个 token，而每个英文 token 约覆盖 3.3 个字符。

1.4 BPE 作为数据驱动的压缩：从频率中归纳符号

BPE（字节对编码）由 Gage（1994）提出用于通用数据压缩，后来由 Sennrich 等人（2016）引入神经机器翻译。注意，BPE 不是 NLP 专用技巧，而是一种适用于文本建模的通用压缩策略。

核心观察：频繁模式 = 冗余

在任何字节序列中，某些相邻的字节对会反复出现。例如，在英文文本中，字节序列 "t" + "h" 出现频率非常高。BPE 的策略是：把这个高频对合并成一个新符号，从而缩短序列长度。每次合并等价于发现一个重复模式并将其抽象为一个更高级的符号。

算法流程（抽象层）

BPE 训练

每次合并操作会使序列长度减少 （该对出现的次数）。经过 次合并后，总序列长度显著缩短，而词表从 256 增长到 。

为什么最频繁的对是合理的合并

压缩角度：假设我们有一个 token 序列，并希望用一个新的合并符号替换一个相邻对。替换后序列变短，且替换规则是确定性的。合并最频繁的对相当于在当前步骤最大化序列长度减少量。这是一种贪心启发式策略，并不保证全局最优的合并顺序；实际效果还取决于语料库、预分词设置和词表预算。

信息论角度：频繁共现的相邻符号对通常值得词汇化，但原始频率并不等同于互信息。高频对可能只是高频单字节组合的结果。因此，BPE 的选择标准更准确地说，是局部压缩增益，而非 PMI 或语义关联的最优估计。

一个具体例子

Corpus: "low lower lowest" UTF-8 byte sequence: l, o, w, _, l, o, w, e, r, _, l, o, w, e, s, t Initial vocabulary (8 characters): l, o, w, _, e, r, s, t Round 1: "l"+"o" appears 3 times → merged into token 256 ("lo") Sequence: lo, w, _, lo, w, e, r, _, lo, w, e, s, t Round 2: "lo"+"w" appears 3 times → merged into token 257 ("low") Sequence: low, _, low, e, r, _, low, e, s, t Round 3: "e"+"r" appears 2 times → merged into token 258 ("er") Sequence: low, _, low, er, _, low, e, s, t Vocabulary evolution: 8 → 9 → 10 → 11 symbols

注意： "e"+"r" 只在第3轮被合并，因为前两轮中 "l"+"o" 和 "lo"+"w" 的频率更高。合并顺序完全由频率决定——这就是 BPE 唯一的“学习信号”。

工作示例：每次合并后必须重新计算配对统计

实现 BPE 时最常见的错误是在第一轮配对统计后重复使用旧计数。BPE 统计的对象是当前 token 序列，不是原始字符串；一次合并会改变相邻关系，下一轮必须重新运行 _get_stats。

1.5 编程练习 1：统计相邻 token 对频率

BPE 训练的第一步：获取当前 token 序列，统计所有相邻对的出现次数。这些统计结果决定每次迭代时合并哪个相邻对。

🖥️ 练习 1：实现 _get_stats

任务：实现函数 _get_stats(ids)，接收一个整数列表（token ID 序列），返回一个字典，包含所有相邻对的频率。

要求：

  • 输入 ids 是 list[int]
  • 输出 dict[tuple[int, int], int]，键是 (左 token, 右 token)，值是该相邻对出现的次数
  • 相邻对不重叠：对于序列 [1,2,3,4]，考虑 (1,2), (2,3), (3,4)
  • 可以使用 collections.Counter 或手动字典
# Expected usage
assert _get_stats([1, 2, 3, 1, 2]) == {(1,2): 2, (2,3): 1, (3,1): 1}

# Empty or single-element sequences should return an empty dictionary
assert _get_stats([]) == {}
assert _get_stats([42]) == {}
from collections import Counter def _get_stats(ids): """统计所有相邻 token 对的频率。""" counts = Counter() for pair in zip(ids, ids[1:]): counts[pair] += 1 return counts # 一行版本： def _get_stats(ids): return dict(Counter(zip(ids, ids[1:])))

💡 zip(ids, ids[1:]) 是生成相邻对的 Python 惯用写法。它创建一个大小为 2 的滑动窗口迭代器，不产生额外中间列表，清晰表达“相邻”的含义。

1.6 深入分析：频率选择的压缩启发式方法

上一节实现了 _get_stats。那下一个问题是：为什么 BPE 总是合并频率最高的 pair？

压缩视角

每次合并都会把序列中所有出现的 替换为一个新符号 。替换后，序列长度减少 （ 是该 pair 出现的频率），词汇表增加 1。选择最频繁的 pair，相当于每一步都最大化边际压缩比 。这种贪心启发式方法简单有效；它不能保证全局最优，实际效果取决于语料库、目标词汇量、预分词（pre-tokenization）和打破平局的规则。

与 Huffman 编码的对比

Huffman 编码自底向上构建最优前缀码——它合并两个频率最低的节点。而 BPE 合并频率最高的 pair。二者的关键区别在于：

  • Huffman 构建的是编码——每个符号都有一个唯一的二进制码。合并低频节点，高频符号就能获得较短的码字。
  • BPE 构建的是词汇表——序列中相邻的符号被抽象成一个新符号。合并高频 pair，最常见的组合就被“词汇化”了。

两者都通过重复合并来构建层次结构，但合并准则服务于不同的最终目标。

频率 vs. 互信息

两个 token 的点互信息（Pointwise Mutual Information, PMI）定义为：

点互信息

高频不一定意味着高 PMI——例如，在英语中 "t" + "h" 很常见，但这只是因为 "t" 和 "h" 各自都常见。真正“有意义”的组合，是那些频率远高于各部分独立概率乘积的 pair。标准 BPE 实现直接使用原始频率而非 PMI，这可能会对高频但弱相关的 pair（如 " " + "t"）产生次优的合并。这是 BPE 已知的局限性之一。

💡
改进思路：一些分词器训练准则在选择时会将候选 segment 的边际收益或似然变化纳入考虑，而不只看原始频率。WordPiece 通常使用与 segment 概率相关的得分来减少对高频但弱相关组合的偏向；它和 PMI 有相似的直觉，但不能简单地说“等价于 PMI”。

1.7 编程练习 2：实现词对替换

找到最高频词对后，训练循环需要把序列中所有非重叠出现的该词对替换为合并后的新 token。这是 BPE 训练的核心步骤。

🖥️ 练习 2：实现 _merge

任务：实现函数 _merge(ids, pair, new_id)，将整数序列 pair 中所有出现的 ids 替换为 new_id。

要求：

  • 输入 ids 是 list[int]，pair 是 tuple[int, int]，new_id 是 int
  • 返回一个新列表，不修改原列表
  • 替换不重叠：例如将 (1,2) 在 [1,2,1,2] 中替换为 [new_id, new_id]，而不是 [new_id, 1, 2]
  • 单趟完成，时间复杂度
  • 不能使用 str.replace 或正则表达式
# Expected usage
assert _merge([1, 2, 3, 1, 2], (1, 2), 99) == [99, 3, 99]
assert _merge([1, 1, 1], (1, 1), 99) == [99, 1]   # Non-overlapping replacement
assert _merge([5, 6], (1, 2), 99) == [5, 6]        # No match
def _merge(ids, pair, new_id): """用新 id 替换序列中所有指定的相邻词对""" new_ids = [] i = 0 while i < len(ids): # 检查当前位置是否匹配词对 if (i < len(ids) - 1 and ids[i] == pair[0] and ids[i+1] == pair[1]): new_ids.append(new_id) i += 2 # 跳过词对的第二个元素（非重叠） else: new_ids.append(ids[i]) i += 1 return new_ids

💡 为什么用 while 而不是 for？ 因为循环体根据是否匹配需要跳过 1 或 2 步，while 配合手动索引可以灵活控制步长，for 循环做不到这一点。

1.8 BPE 训练循环：从字节到词表

有了 _get_stats 和 _merge，BPE 训练就是一个简单的迭代过程。每次迭代：

  1. 统计当前序列中所有相邻对的频率
  2. 找到最高频的对
  3. 记录合并规则
  4. 在序列中执行合并
  5. 将新 token 的字节表示存入词表

这个过程会产生两个数据结构：

  • vocab（dict[int, bytes]）：从 token id 到原始字节序列的映射——用于解码
  • merges（dict[tuple[int,int], int]）：从对到新 token id 的映射——用于编码

词表的构建：新 token 的字节表示就是其组成 token 的字节拼接。例如，如果 token 256 = "lo"（由 token 108='l' 和 token 111='o' 合并而成），那么 vocab[256] = vocab[108] + vocab[111]。这个递归定义保证了 vocab 中每个条目都能回溯到原始字节。

🔑
重要性质：词表是无损的——任何 token 都可以还原为原始字节序列。这是 BPE 解码无需模型或概率的数学基础。

示例：BPE 合并跟踪与 token 节省

以 abababa 为例，初始字节序列长度为 7。第一轮中，相邻对中 (a,b) 和 (b,a) 都出现 3 次；本章实现按扫描顺序选择了 (a,b)，创建 256="ab"，经过非重叠替换后，序列从 [a,b,a,b,a,b,a] 变为 [256,256,256,a]，长度 7→4，节省 3 个 token。第二轮中，(256,256) 出现 2 次，创建 257="abab"，序列变为 [257,256,a]，长度 4→3，再节省 1 个 token。

Ch01 作业中的 bpe_training_trace(text, vocab_size, max_steps) 函数要求记录每一步的 pair、count、new_id、before_length、after_length 和 tokens_saved。这有助于区分两点：BPE 的压缩来自高频相邻对的非重叠替换；每次合并后必须重新计算对统计，不能复用旧的频率表。

复杂度分析

如果语料包含 个 token，需要 次合并：

  • 每次合并需要扫描整个序列（）来统计频率并执行替换
  • 总复杂度
  • 实际训练中 ，而 可达数亿——因此原始 BPE 训练可能很慢

优化方法：使用优先队列（堆）维护对频率，每次合并后只更新受影响的局部区域。GPT-4 使用的 tiktoken 库采用高度优化的并行版本。

🖥️ 练习3：BPE 训练函数

任务：实现函数 train(text, vocab_size)，它在输入文本上训练 BPE 词汇表并返回 (vocab, merges)。

要求：

  • 输入 text 是 str，vocab_size 是 int（目标词汇大小，≥ 256）
  • 基础词汇表包含 256 个字节 token（vocab[i] = bytes([i])）

def train(text, vocab_size): """Train a BPE tokenizer on the input text, returning (vocab, merges).""" # Initialization: byte sequence + base vocabulary ids = list(text.encode("utf-8")) vocab = {i: bytes([i]) for i in range(256)} merges = {} num_merges = vocab_size - 256 for i in range(num_merges): # 1. Count frequencies stats = _get_stats(ids) if not stats: break # Sequence has only one token, cannot continue # 2. Find the highest frequency pair pair = max(stats, key=stats.get) # 3. Assign new token id new_id = 256 + i # 4. Record the merge rule merges[pair] = new_id # 5. Build the byte representation of the new token vocab[new_id] = vocab[pair[0]] + vocab[pair[1]] # 6. Apply the merge ids = _merge(ids, pair, new_id) # (Optional) Print status every 1000 steps if (i + 1) % 1000 == 0: print(f" Merge {i+1}: Sequence length = {len(ids)}") return vocab, merges

💡 vocab 的递归构建：vocab[new_id] = vocab[pair[0]] + vocab[pair[1]] 利用已存储在 vocab 中的字节表示——pair[0] 和 pair[1] 可能是原始字节 token，也可能是之前合并产生的 token。递归定义保证了解码的正确性。

1.10 编码与解码——从规则到完整的 Tokenizer

训练完成后，我们使用 merges 规则对新文本进行编码，并使用 vocab 将 token 序列解码回文本。

解码：简单且完全可逆

解码就是查找并拼接：对每个 token ID，从 vocab 中取出对应的字节串，拼接起来，再用 UTF-8 解码成文本：

因为 vocab 中的每个 token 对应一个确定的字节序列，解码只是一个查找、拼接和 UTF-8 解码的过程，不需要搜索或模型——结果总是唯一的。这也是 BPE 相比其他 tokenization 方案的一个优势。

编码：按合并等级进行确定性的贪婪合并

编码将新文本转换成 token ID 序列。做法是：从字节序列开始，反复应用合并规则，每次优先选择训练顺序中最早出现的合并（即优先级最高）。

具体方法：

  1. 将文本编码为 UTF-8 字节序列，转换成整数列表
  2. 反复检查所有相邻对——如果某对在 merges 字典中，找到最早被合并的那一对（即具有最小的 new_id）
  3. 应用该合并并更新序列
  4. 继续直到无法再合并任何对

寻找“最早合并”的技巧：min(stats, key=lambda p: merges.get(p, float('inf')))。如果某对不在 merges 中，则返回 inf —— 只应用学习到的合并规则。

💡
直觉：训练中较早合并（new_id 较小）的对对应更基础的子词模式，它们在训练语料中更早被词汇化。编码时优先应用这些规则，保证同一输入用相同合并规则总能得到相同的切分。

示例：编码不会基于频率重新学习

假设训练后获得以下合并规则，其中数字越小表示在训练中添加越早、优先级越高：

合并等级对新 token字节含义
256('l', 'o')lo首先学习到的常见片段 lo
257('lo', 'w')low然后合并了 lo 和 w 为 low
258('e', 'r')er最后学习到的后缀 er

现在编码新文本 "lower"。初始字节序列为 [l, o, w, e, r]，相邻对包括 (l,o)、(o,w)、(w,e)、(e,r)。其中，(l,o) 和 (e,r) 出现在 merges 中；尽管两者在该新文本中都只出现一次，但必须选择等级更早（优先级更高）的 (l,o) → lo：

[l, o, w, e, r] -> apply rank 256: (l, o) -> lo [lo, w, e, r] -> apply rank 257: (lo, w) -> low [low, e, r] -> apply rank 258: (e, r) -> er [low, er]

如果在编码时重新计算当前输入的频率或临时选择“看起来最长”的片段，就会破坏训练时保存的合并优先级，导致同一个 vocabulary 在不同实现上产生不一致的切分。因此，课程作业中的 encode 函数使用 merges.get(pair, inf) 来找到当前可用对中的最小等级，而不是重新训练一个 local tokenizer。

1.11 编程练习 4-5：实现完整的编码与解码

🖥️ 练习 4：实现 decode

任务：实现 decode(ids, vocab)，把 token ID 序列转成文本。

要求：

  • 输入 ids 为 list[int]，vocab 为 dict[int, bytes]
  • 返回 str，用 .decode('utf-8', errors='replace') 处理无效字节序列
  • 将无效字节替换为 �
# Expected usage (requires training first)
vocab, merges = train("hello hello hello", vocab_size=260)
decoded = decode([104, 101, 256], vocab)
print(decoded)  # Assuming 256="he" → should output "he"
def decode(ids, vocab): """Decode a sequence of token IDs into a text string.""" tokens = b"".join(vocab[idx] for idx in ids) return tokens.decode("utf-8", errors="replace")

💡 解码只需两行 Python！BPE 训练复杂，但解码简单。推理时只需要一张查找表和一次 UTF-8 解码，效率很高。

🖥️ 练习 5：实现 encode

任务：实现 encode(text, merges)，把文本编码成 token ID 序列。

要求：

  • 输入 text 为 str，merges 为 dict[tuple[int,int], int]
  • 返回 list[int]
  • 从 UTF-8 字节序列开始，反复应用 merges 中优先级最高的合并
  • 用 min(stats, key=lambda p: merges.get(p, float('inf'))) 找优先级最高的 pair
  • 当 merges 中不存在任何 pair 时停止
# Expected usage
vocab, merges = train("low " * 100 + "lower " * 80 + "lowest " * 60, vocab_size=280)
text = "lowest lower"
ids = encode(text, merges)
print(f"Encoded {text!r} → {ids}")

# Verification loop: encode → decode should yield the original text
decoded = decode(ids, vocab)
assert decoded == text, f"Round-trip mismatch: {decoded!r} != {text!r}"
def encode(text, merges): """Encode text into a sequence of token IDs.""" ids = list(text.encode("utf-8")) while len(ids) >= 2: # Count frequencies of adjacent pairs in the current sequence stats = _get_stats(ids) # Find the pair with the highest priority (earliest merged, i.e., smallest new_id) pair = min(stats, key=lambda p: merges.get(p, float("inf"))) # If the highest priority pair is not in merges → stop if pair not in merges: break # Apply the merge new_id = merges[pair] ids = _merge(ids, pair, new_id) return ids

💡 编码是贪心的：每轮只应用一个合并，即可用合并等级最高的那个（训练时最早添加的）。合并等级来自训练时保存的合并顺序；编码按这个固定优先级反复合并相邻 pair，而不是根据当前输入重新算频率，也不等价于通用的最长匹配。

1.12A 如何评估分词器

BPE 只是第一步。实际训练 LLM 时，分词器会影响序列长度、嵌入参数数量、上下文窗口利用率、推理延迟和多语言覆盖。词汇量不是简单的“越大越好”或“越小越好”；它涉及数据分布、模型规模和部署成本之间的权衡。

小实验

从英文、中文、代码、数学公式和表情/混合文本中各选 100 个样本。用不同词汇量大小或不同分词器对它们编码，报告平均 token 数、P95 token 数、往返是否成功，以及嵌入参数数量 。如果中文的平均 token 数是英文的 2–3 倍，那么相同的 8K 上下文对中文用户提供的有效文本容量就会更少；这不是模型能力问题，而是分词器成本问题。

第一章作业中的 tokenizer_report(tokenizer, texts, vocab_size, d_model) 是这个实验的最小版本：它报告 avg_tokens、p95_tokens、tokens_per_character、round_trip_success_rate 和 embedding_params。进一步的 tokenizer_group_report(tokenizer, groups, vocab_size, d_model) 报告按英文、中文、代码、数学或表情分组的 token 开销，并计算最高与最低 tokens_per_character 之间的差异。学习者应当用相同的文本集合比较不同词汇量大小，而不是只报告“编码/解码有效”或单一的全局平均值。

示例演练：从分词器报告中观察成本

假设同一个分词器编码了 4 段文本，长度如下：

文本组字符数token 数tokens/字符
英文2060.30
中文8121.50
代码16100.625
表情/混合10141.40

全局总 token 数为 6+12+10+14=42，每段文本的平均 token 数为 avg_tokens=10.5，排序后的长度为 [6,10,12,14]，教学版本的 P95 可取最大值 14。总字符数为 54，因此 tokens_per_character=42/54≈0.778。如果词汇量大小是 V=50,000 和 d_model=768，那么仅输入嵌入矩阵就有 50,000 * 768 = 38.4M 个参数。

分组的成本也不能忽视：中文 1.50 与英文 0.30 相比，差异为 1.50/0.30=5.0。这意味着相同的 8K token 上下文能容纳的中文字符远少于英文字符。评估分词器时，应当同时考虑压缩率、往返测试和嵌入参数，而不仅仅是词汇量大小。

🔑
工程要点： 分词器是模型架构的一部分。它决定了“文本长度”对模型意味着什么，也决定了嵌入、注意力和 KV Cache 的实际成本。

1.12B Tokenizer 是模型接口契约：特殊 Token、模板与检查点

很多训练事故不是 Transformer 代码写错了，而是 tokenizer 协议没对齐：训练和推理解码用的角色 token 不一样，加了新特殊 token 之后没扩展嵌入层，填充 token 被当成普通文本学，或者评估集用了不同的 tokenizer 计算上下文长度。对 LLM 训练和推理工程师来说，tokenizer 不只是预处理工具，它本身就是模型检查点的一部分。

1. 特殊 token 必须具有稳定的语义

现代 LLM 通常至少需要下面这些特殊 token 或等效协议：

类型功能错误的后果
BOS/EOS标记序列开始与结束，控制生成停止模型不停止、过早停止、或将结束 token 作为内容输出
PAD批量填充，通常不参与 loss 计算填充被当作可预测文本训练，loss 分母和注意力掩码错误
角色 token区分 system/user/assistant/tool模型混淆说话者，SFT 掩码无法可靠定位响应区间
工具/函数 token标记工具调用、参数 JSON 或返回值结构化调用边界不稳定，推理服务难以解析
保留 token为未来扩展预留 ID后续扩展需要重新训练或迁移嵌入层

特殊 token 不应被普通 BPE 合并“意外”切分，应视为不可分割的原子 token。否则，字符串 <|assistant|> 可能被拆成多个普通片段，模型难以学习稳定的边界，安全过滤和工具解析也会变脆弱。

2. 聊天模板决定了 SFT 的监督边界

同一段对话，通常先在 tokenizer 层序列化成模板文本，再转成 token ID。在 SFT 中，标签掩码依赖这个模板来定位 assistant 的响应：

... 请解释梯度裁剪。 梯度裁剪用于限制梯度范数...

如果训练流水线用 <assistant>，而推理服务用 Assistant:，模型看到的角色边界就不一样。短期内可能只是格式漂移，长期则会影响多轮对话、拒绝边界、工具调用，以及 DPO/GRPO 中的序列 logprob 对齐。第 9 章的 SFT 数据流水线会沿用这个原则：模板、tokenizer 和标签掩码必须共同定义训练目标。

3. 修改词表等同于修改模型形状

词表大小 V 决定了 token 嵌入层和 LM head 的形状。如果预训练模型用 V=50,257，之后加了 100 个特殊 token，嵌入层就要从 [50257, d_model] 变成 [50357, d_model]。这不只是改个配置文件，而是检查点兼容性问题：

  • 输入嵌入层：新行需要初始化。常见做法是随机初始化或用相似 token 的均值。
  • LM head：如果用了权重绑定，LM head 会随嵌入层一起扩展；如果不共享，输出投影也必须同步扩展。
  • 旧检查点：加载时会出现形状不匹配，必须显式处理新行，不能靠抑制错误来忽略。
  • 旧数据：旧 token ID 的语义必须保持不变。重新排列 token ID 会破坏所有现有权重。

因此，生产模型通常冻结现有 token ID，只把新 token 追加到词表末尾。如果需要完全替换 tokenizer，通常意味着要继续预训练或至少做较长的适应训练，因为整个嵌入空间的几何结构已经变了。

4. Tokenizer 版本必须与模型一起发布

一个可复现的模型包必须包含 tokenizer 产物（tokenizer.json 或 SentencePiece 模型）、特殊 token 映射、聊天模板、归一化规则以及模型配置。只发布 model.safetensors 不足以复现模型行为。加载模型时，推理服务还应检查：

config.vocab_size == tokenizer.vocab_size special token ids match model config chat template version matches SFT/DPO training pad/eos/bos ids are explicit round-trip tests pass for multilingual and tool-call samples

这些检查不是形式主义。如果 tokenizer 改变了中文、代码缩进、JSON 引号或表情符号的分割方式，模型的上下文长度、生成概率和停止行为都会改变，进而影响到第 6 章的权重加载、第 9 章的序列 logprob 以及第 10 章的 token 成本计算。

🔑
本节要点：Tokenizer 定义了模型能读写的外部接口。BPE 合并规则、特殊 token、聊天模板和词表大小必须和模型一起做版本管理；否则，训练 loss、对齐数据、检查点加载和推理服务面对的就不是同一个任务。

1.12C 聊天模板与多模态接口契约：一条消息就是一个 token 契约

生产环境中的 tokenizer 不只是把纯文本转成 token ID。它还负责把结构化的对话拼成一条连续的序列：系统指令、用户消息、助手回答、工具调用、工具结果、多模态占位符、停止 token 和训练掩码区间。同一个消息对象，在预训练适配器、SFT（监督微调）、偏好训练、评估和在线推理中，必须序列化出完全相同的 token 边界。如果做不到，模型的训练接口和服务接口就不一致，等于用不同的接口训练和提供服务。

最安全的做法是把聊天模板看成一个小型编译器，从结构化消息编译到 token ID。编译器需要回归测试。下面这些检查应该放在 tokenizer 发布的旁边，而不是放在一个部署前没人看的独立 notebook 里。

工作流对齐检查需要检查什么
SFT序列化一个多轮样本，解码带掩码的标签。只有助手回答 token 应该贡献监督损失，除非训练目标另有说明。
DPO/GRPO 对数概率用完全相同的 prompt 前缀重新 tokenize chosen 和 rejected 答案。Prompt token 必须一致；比较序列对数概率时，只有答案 span 可以不同。
工具调用把工具调用和工具结果经过序列化、tokenize、生成、解析做往返测试。函数名、参数 JSON 和结果边界在流式和截断中都要保持不变。
多模态输入用图像或音频占位符 tokenize 文本，再与处理器输出对比。模态占位符的数量和位置必须与模态编码器的特征匹配。
流式与停止生成直到 EOS、一个停止字符串和一个解析完成的状态。运行时不能过早截断有效的 JSON，也不能把 EOS/控制 token 泄露给客户端。

所以，修改 tokenizer 就等于一次接口迁移。只有发布产物包含合并规则、特殊 token ID、聊天模板、归一化、处理器占位符和兼容性测试时，才算安全。对训练工程师来说，它定义了损失边界。对推理工程师来说，它定义了解析器边界、工具执行边界和多模态 token 预算。把模板当成简单的 UI 格式化，是导致模型行为静默漂移最快的方式之一。

1.12D 多语言分词器训练：从语料比例到词元成本

对于真实世界的大语言模型，分词器不是在英文语料上随便训练的组件；它是连接语言、文字系统和领域的数据接口，决定不同语言在相同上下文窗口内能表达多少信息，并影响训练批次中的有效文本量、推理计费、KV Cache 内存以及长上下文用户体验。多语言模型尤其需要把分词器训练当作一项数据工程决策。

1. 语料比例改变合并规则

BPE 只关注训练语料中相邻符号的频率。如果英文占分词器训练语料的 95%，而中文、阿拉伯语、印地语、代码和数学公式只占很小一部分，那么高频合并会优先采用英文模式。其他语言更常回退到字节或短片段，表现为更高的词元数和更低的上下文容量。相反，盲目地对一种小语种进行过采样，可能会牺牲高流量语言和领域文本的压缩率。

设计项工程问题推荐做法典型失败场景
语言采样不同语言都能获得足够的合并预算吗？根据目标用户和训练数据设置采样上下限，而不是直接根据原始网页量进行采样低资源语言被拆成字节；8K 上下文只能容纳很短文本
领域采样代码、数学、表格、JSON 是否得到合理压缩？为代码缩进、常见符号、LaTeX、结构化标记保留代表性样本代码空格和标点过于碎片化；工具调用 JSON 容易超出长度限制
Unicode 规范化视觉上相同的字符是否映射到同一序列？明确定义 NFC/NFKC、大小写、全角/半角、空白字符策略，并在训练和推理之间保持一致相同文本因规范化方式不同产生不同词元，影响去重、评估和缓存
字节回退未见过的字符能否无损表示？保留字节级回退或等效机制，确保任何 Unicode 字符都能被编码稀有字符、emoji、混合文字输入导致 OOV 或不可逆的解码
特殊词元预算对话、工具、多模态占位符是否需要原子边界？为 system、role、tool、图像/音频占位符以及未来扩展预留词元后续添加词元会改变 embedding 形状，需要模型重新适应

2. 使用 fertility 衡量不同语言的词元负担

多语言分词器用 fertility 衡量一个自然单元被拆分成多少词元。对于英文，可以看 tokens/word（词元/词）；对于中文、日文、泰文等无空格或弱空格语言，tokens/character（词元/字符）或 tokens/byte（词元/字节）更稳定。本课程建议同时报告三种指标：

  • tokens/character（词元/字符）：衡量相同字符长度下的上下文代价，适用于中英文对比。
  • tokens/byte（词元/字节）：衡量 UTF-8 多字节文字是否过度回退，适用于跨文字系统对比。
  • P95 token length（P95 词元长度）：衡量长尾样本是否会在 packing、RAG 上下文或工具参数中导致膨胀。

假设同一分词器对 1000 个英文字符平均产生 300 个词元，对 1000 个中文字符平均产生 1500 个词元。在 8K token 的上下文中，英文可容纳约 26,666 个字符，而中文可容纳约 5,333 个字符。这一差异直接影响长文本问答、RAG 分块大小、缓存命中率和用户感知的成本。

实操示例：为什么分词器训练不应仅依据原始数据量

假设分词器训练预算为 100 亿字符，原始候选数据由 80 亿英文、10 亿中文、8 亿代码和 2 亿其他语言组成。如果直接按原始比例训练，英文配对将主导前几万个合并；如果对所有语言均匀采样，则会导致高流量语言损失过多压缩率。一种更稳妥的方法是先定义服务目标，再设定采样比例：

  1. 为主要服务语言设置最低覆盖率，例如英文、中文和代码各自至少占分词器训练样本的一定比例。
  2. 对低资源语言使用温度采样，避免被网页数量完全淹没，同时也要避免小语料过度重复。
  3. 保留字节回退，使未明确覆盖的字符仍能被无损编码。
  4. 训练多个候选分词器，并使用同一套多语言/代码/工具调用样本来比较 tokens/character、round-trip（往返一致性）、P95 长度和 embedding 参数量。
  5. 做最终选择时，同时考虑训练成本和推理成本：词元减少 10% 可能意味着注意力/KV 成本降低约 10%，但更大的词表会增加 embedding 和 LM head 参数。

3. 分词器决策传导至训练和推理

多语言分词器的影响不只局限于第 1 章。第 7 章的数据打包（data packing）按词元数而非字符数切分批次；第 9 章的 SFT/DPO 在词元序列上计算损失和 log probability；第 10 章的 KV Cache、TTFT、TPOT 和计费都按 token 进行。某个语言的 fertility 越高，意味着相同文本消耗更多的训练 FLOPs、更多的 KV Cache 和更高的推理延迟。

🔑
工程要点：不能将分词器视为一个“预处理细节”。对于训练和推理工程师而言，分词器是数据分布、模型参数、上下文容量、服务成本和多语言体验之间的通用接口。
