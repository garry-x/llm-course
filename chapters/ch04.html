4.1 From Single-Head to Multi-Head — Why Isn't One Head Enough?

Chapter 3 implemented single-head self-attention: one -dimensional Q/K/V space responsible for all matching patterns. But relationships in language are multifaceted — a single token needs to match other tokens from multiple "perspectives" simultaneously.

Concrete Scenario: Why Isn't One Head Enough?

Consider the sentence "The cat sat on the mat because it was comfortable."

When the model processes the word "it", it needs to answer multiple questions:

  • Syntactic question: "it" is a singular pronoun; its antecedent should syntactically be a singular noun — this requires a specific matching rule
  • Semantic question: "it" should refer to the "most plausible" thing — the cat sat on the mat, so it could be "the cat" or "the mat" — this requires a different matching rule
  • Positional question: The noun before "it" is "mat", and further back is "cat" — the model needs to distinguish tokens at position 5 and position 1

Single-head attention can only use one -dimensional space to encode all these complex matching rules. This is insufficient — it's like asking a single judge to preside over civil, criminal, and administrative cases simultaneously with only one set of criteria.

The Solution of Multi-Head Attention: Parallel Subspaces

The core idea of Multi-Head Attention (MHA) is: split the input -dimensional space into subspaces of dimension , where each head independently computes attention in its own subspace, and finally all head outputs are concatenated and fused through the output projection :

Multi-Head Attention Formula

Key Insight: Parameter Equivalence

The total number of parameters in multi-head attention is exactly equal to that of single-head attention. Because :

Parameter Equivalence

Worked Example: MHA Does Not Get Stronger by Adding Parameters

Let d_model=8, using n_heads=2, then the dimension of each head is d_k=4. If single-head attention projects Q/K/V to 8 dimensions, it requires 3 * 8 * 8 = 192 QKV parameters; each of the two heads has 3 * 8 * 4 = 96 parameters, totaling 2 * 96 = 192.

SchemeQ/K/V Projection SpaceQKV Parameter CountAttention Score Shape
Single Head1 8-dimensional space3 * 8 * 8 = 192[T, T]
2-Head MHA2 4-dimensional spaces2 * 3 * 8 * 4 = 192[2, T, T]

Both have the same number of parameters, but MHA produces two attention maps in different projection spaces, then concatenates the two 4-dimensional head outputs back to 8 dimensions and mixes them with W_O. Its representational advantage comes from "multiple learnable similarity functions working in parallel," not from simply adding parameters.

In other words, multi-head attention is not about "getting bigger," but about reorganizing the existing parameter budget into multiple smaller subspaces. Each subspace learns a different attention pattern, and the final is responsible for integrating these scattered pieces of information together.

This is a classic embodiment of "Divide and Conquer" in deep learning—decomposing a large problem into multiple smaller, more specialized sub-problems solved in parallel.

🔑
Why same parameters but better results? Each head has independent Q/K/V projections and can learn different similarity functions; the output projection then recombines these subspace results. MHA's benefit comes from representational decomposition and learnable mixing, not from "more heads, more parameters."

import torch import torch.nn as nn import torch.nn.functional as F class MultiHeadAttention(nn.Module): """Multi-Head Attention — Fused QKV Projection + Efficient Head Separation""" def __init__(self, d_model, n_heads, dropout=0.1): super().__init__() assert d_model % n_heads == 0 self.d_model = d_model self.n_heads = n_heads self.d_k = d_model // n_heads self.W_qkv = nn.Linear(d_model, 3 * d_model, bias=False) self.W_o = nn.Linear(d_model, d_model, bias=False) self.dropout = nn.Dropout(dropout) def forward(self, x, mask=None): # x: (batch, seq_len, d_model) batch, seq_len, _ = x.shape # 1. Fused projection + separate Q/K/V qkv = self.W_qkv(x) # (b, s, 3*d) qkv = qkv.reshape(batch, seq_len, 3, self.n_heads, self.d_k) q, k, v = qkv.unbind(dim=2) # each (b, s, h, d_k) # 2. Move head dimension forward: (b, h, s, d_k) q = q.transpose(1, 2) k = k.transpose(1, 2) v = v.transpose(1, 2) # 3. Scaled dot-product attention scores = torch.matmul(q, k.transpose(-2, -1)) / (self.d_k ** 0.5) if mask is not None: scores = scores.masked_fill(mask == 0, float('-inf')) attn_weights = F.softmax(scores, dim=-1) attn_weights = self.dropout(attn_weights) out = torch.matmul(attn_weights, v) # (b, h, s, d_k) # 4. Concatenate heads out = out.transpose(1, 2).contiguous() out = out.reshape(batch, seq_len, self.d_model) # 5. Output projection out = self.W_o(out) return out, attn_weights

💡 Why fuse the QKV projection? GPUs are more efficient with large matrix multiplications—a single (d, 3d) matrix multiplication is faster than the sum of three (d, d) matrix multiplications because the GPU's parallel compute units can better utilize larger matrix blocks. This is a common "fuse then separate" pattern in engineering practice.

4.3 Deep Dive: Why Can Multiple Heads Learn Different Representation Subspaces?

Initialization-Driven Differentiation

In practice, the key reason multiple heads can learn different patterns lies in random initialization. The Q/K/V projection matrices for each head start with different random values. During training, different initializations cause each head to optimize along different gradient directions—like starting from different points to climb different mountains.

Mathematical Perspective: Gradient Differences Across Heads

Assume the initial projection matrices for two heads are and , and . Their outputs and will also differ. Through the loss function's gradient, —the parameter update directions are different, causing them to gradually diverge.

What If All Heads Learn Exactly the Same Thing?

This is theoretically possible; in practice, it is usually unlikely to remain exactly the same for long because:

  • Random initialization breaks symmetry
  • The training data flow (gradients) for different heads is naturally different—because each head sees a different attention distribution
  • Even if two heads are similar early in training, small differences are amplified by the stochasticity of SGD

In fact, research by Michel et al. (2019) shows that a significant proportion of attention heads are redundant after training (removing them does not affect performance). This means the actual "effective number of heads" required by the model may be much less than the designed number—which is one of the theoretical bases for GQA and MLA compressing the number of KV heads.

Competition and Cooperation Between Heads

Interestingly, there is a competitive relationship between attention heads. If one head has already captured a certain syntactic pattern, there is no "marginal benefit" for other heads to capture the same pattern. This encourages each head to discover different patterns—similar to "Negative Correlation Learning," a classic ensemble learning strategy.

🔑
Essence: The success of multi-head attention stems from "divide and conquer + differentiated initialization." The total parameter count remains unchanged, but by splitting a large space into multiple smaller subspaces, each subspace can focus more on specific matching patterns. This is equivalent to using multiple "narrow experts" instead of one "generalist" under a fixed parameter budget.

4.3A Multi-Head Is Not a Simple Ensemble: Representational Decomposition, Redundancy, and Compression Space

"Multiple heads learn different patterns" is the correct intuition, but it is not rigorous enough. MHA does not average the votes of several complete models; rather, within a fixed width, it decomposes the matching and aggregation of attention into multiple narrower subspaces, and then recombines them with the output projection .

Decomposed Form of MHA

Expressiveness comes from "different similarity functions"

Each head has its own , so it computes similarity in a different projection space. One head might make parentheses, indentation, or delimiters similar, another might make subjects and predicates similar, and yet another might specialize in local patterns near the current position. The key point is: the outputs of these heads are not the final answer, but intermediate representations for the next layer to continue processing.

Why head redundancy occurs

After training, not every head is equally important. There are three categories of reasons:

  • Task demand is less than design capacity. If the current layer only needs a few major routing patterns, extra heads may learn similar functions.
  • The output projection can re-weight. can weaken the contribution of certain heads; a head's attention may appear active, but that doesn't mean it has a strong impact on the logits.
  • Adjacent layers can compensate. After removing or compressing some heads, subsequent layers can sometimes still recover sufficient information, so performance degradation may not appear immediately.

How this explains GQA and MLA

The basic assumption of GQA is: not every query head must have its own independent K/V representation. Multiple query heads can share a set of K/Vs, still posing different queries through their respective Q projections. MLA goes further, compressing K/V information into a low-dimensional latent, then reconstructing the required representation when used. Both leverage the same fact: the nominal number of heads in the model design is not equal to the amount of independent K/V information that must be retained during inference.

PhenomenonTheoretical ExplanationEngineering Consequence

⚠️
Don't treat the number of heads as a direct indicator of capability. More heads provide finer subspace decomposition, but can also increase redundancy; fewer KV heads improve inference efficiency, but may lose retrieval diversity. Rigorous comparison must simultaneously examine loss, downstream tasks, long-context behavior, throughput, and KV Cache.

4.5 MHA vs MQA vs GQA—The Evolution of KV Cache

The Big Problem During Inference: KV Cache

In autoregressive generation, only one new token is generated per step, but it needs to compute attention with all historical tokens. If the K and V of all historical tokens were recomputed at every step, the computation would be . Therefore, we cache the K and V vectors of historical tokens—this is the KV Cache.

MQA: The Most Aggressive Compression

MQA (Shazeer, 2019) makes all Q heads share the same set of K and V. The KV Cache is reduced to of MHA. The cost is that the diversity of KV is completely lost—all attention heads can only retrieve information from the same K, V space.

GQA: A Compromise Between Quality and Cache Cost

GQA (Ainslie et al., 2023) divides the Q heads into groups, each sharing one K/V head. Llama 2 70B uses 8 KV heads (), reducing the KV Cache by 8 times, while retaining some K/V diversity.

The gqa_head_mapping(n_heads, n_kv_heads) function in the Ch04 assignment expresses this grouping relationship as an explicit list. If n_heads=8, n_kv_heads=2, then every 4 Q heads share one KV head, resulting in the mapping [0,0,0,0,1,1,1,1]; if n_kv_heads=1, it's MQA; if n_kv_heads=n_heads, it degenerates back to standard MHA. During implementation, repeat_kv_heads merely expands the [B,H_{kv},T,D] view to the semantics of [B,H_q,T,D] before the attention computation; the KV Cache still only needs to store H_{kv} copies of K/V.

The empirical benefits of GQA depend on the task, model scale, the n_heads/n_kv_heads ratio, and the training method. Experiments by Ainslie et al. show that with appropriate grouping, it is possible to significantly reduce the KV cache and bandwidth pressure during the decoding phase while maintaining quality close to MHA; however, if n_kv_heads is too small, K/V diversity may be lost. Post-training conversion can also obtain GQA initialization by averaging or reorganizing K/V heads from an MHA checkpoint, followed by continued fine-tuning.

💡
Learning Focus: MQA/GQA do not change the self-attention formula, but rather change "how many groups of independent K/V are cached." Quality issues arise from reduced representation capacity after sharing K/V, while system benefits come from reduced cache and memory bandwidth during the decode phase.

If the original K/V cache shape is [B, H_kv, T, D] = [1, 2, 3, 4], then before attention computation, repeat_kv_heads(k, n_rep=4) can be used to expand it to [1, 8, 3, 4], matching Q's [1, 8, 3, 4]. The key point is: the cache itself still only stores 2 KV heads; the repeat is a semantic expansion during computation, not a 4x replication of the cache memory.

This is clearer with a smaller tensor: if H_kv=2, n_rep=3, and the two KV heads are denoted A and B, then the repeated head sequence is:

cache heads: [A, B] # stores 2 KV heads compute heads: [A, A, A, B, B, B] # aligns with 6 Q heads

Here, copies of A share the same K/V content; during backpropagation, gradients from multiple query heads are aggregated onto the corresponding KV projection. If [A,A,A,B,B,B] were also persistently written into the KV cache during inference, the main memory benefit of GQA would be lost.

4.5B Training Attention and Inference KV Cache Are Not the Same Layout Problem

When students first implement MHA/GQA, they usually only care whether the [B, H, T, D] tensor can compute the correct output. However, training engineering and inference engineering use the same set of weights to solve two different problems: the training phase requires high-throughput processing of entire sequences, while the inference phase requires appending new tokens to the historical cache at each step. Understanding this distinction is key to explaining why GQA/MLA are particularly important for the decode phase.

1. Training Phase: Parallel Computation on Full Sequences

During the training or prefill phase, a complete sequence of length T is processed at once. The standard implementation projects Q/K/V into:

q: [B, H_q, T, D] k: [B, H_kv, T, D] v: [B, H_kv, T, D]

At this point, all Q/K/V values come from the hidden states of the current batch. In MHA, H_q = H_kv; in GQA, H_q > H_kv, and K/V are expanded by groups to match the number of query heads before computing attention; in MLA, a low-dimensional latent is stored first, and then the equivalent K/V effect is restored using a decompression matrix or matrix absorption when computing scores. The main pressure during training comes from large matrix multiplications, the T x T computation of attention scores, and activation memory.

2. Decode Phase: Only One New Token Per Step

At step t of autoregressive decode, only the latest token is input, and the query length is typically 1:

new_q: [B, H_q, 1, D] new_k/new_v:[B, H_kv, 1, D] cache_k/v: [B, H_kv, t, D] attn score: [B, H_q, 1, t]

At this point, the attention score is not a large T x T matrix, but rather each request performs a read from the historical cache. The bottleneck in decode is often HBM bandwidth and cache read/write, not single-step FLOPs. The direct benefit of GQA/MLA is reducing the volume and read bandwidth of cache_k/v; they do not magically reduce all model weights, nor do they automatically improve the full throughput of prefill.

3. Cache Layout Determines Whether the System Can Be Scheduled Efficiently

In a real inference service, requests within a batch often have different lengths. If each request stores its KV cache as a single contiguous tensor, short requests waste space, long requests cause fragmentation, and releasing and reusing memory becomes difficult. Inference engines typically split the cache into blocks/pages and maintain a mapping from "logical token position → physical cache block". Chapter 10's PagedAttention will cover scheduling in detail; for now, remember: GQA/MLA determine how much KV content each token needs to store, while PagedAttention determines how this content is placed in memory and reused across different requests.

4.5C Converting an MHA Checkpoint to GQA: Compression Is Not Simply Changing the Config

In practical engineering, a common need arises: you have an MHA checkpoint, but the inference service bottleneck is the KV Cache, so you want to change the model from H_q=H_kv to GQA with H_q > H_kv. This process cannot simply change n_kv_heads in the config. The K/V projection weights, optimizer states, cache layout, and quality evaluation all need to be realigned.

1. How to Fold the Weights

Suppose the original model has H_q=8 K/V heads, and you want to convert it to H_kv=2. Every 4 original K/V heads will become 1 shared KV head. The most common initialization method is to average the original K/V projections by group:

old K heads: [K0, K1, K2, K3, K4, K5, K6, K7] new K heads: [mean(K0..K3), mean(K4..K7)] old V heads: [V0, V1, V2, V3, V4, V5, V6, V7] new V heads: [mean(V0..V3), mean(V4..V7)]

The Q projection and output projection W_o usually keep the original number of heads, because the goal of GQA is to reduce the K/V representation, not the diversity of query heads. After conversion, the forward pass first obtains K/V of shape [B,H_kv,T,D], then aligns it to [B,H_q,T,D] via repeat_kv_heads for attention.

2. Why Uptraining Is Necessary

Group averaging is just an initialization. Originally, each query head could access an independent K/V space; after conversion, multiple query heads share the same K/V, reducing information routing capacity. A short period of continued training is usually needed to allow Q, K, V, and subsequent layers to adapt to the constraint of shared K/V. This phase is often called uptraining or conversion fine-tuning.

3. Optimizer and Checkpoint Cannot Be Directly Reused

After conversion, the shape of the K/V projection changes, and the corresponding AdamW m/v in the original optimizer state will no longer have the same shape. The safe approach is to treat the converted model as a new checkpoint: retain the mappable weights, reinitialize the optimizer state for the parts whose shape changed, and clearly specify n_heads, n_kv_heads, head grouping, RoPE settings, and cache layout in the config. Otherwise, when resuming training or loading for inference, the state_dict keys might match but the semantics will be inconsistent.

conversion: MHA H_q=32,H_kv=32 -> GQA H_q=32,H_kv=8 init: group-mean K/V projection, Q/O unchanged training: uptraining tokens, lr schedule, data mixture, validation loss quality: short-context PPL, long-context retrieval, code/math/chat slices serving: KV cache GB/request, P50/P95 TPOT, throughput at target concurrency boundary: tasks where quality regressed or cache was not the bottleneck

This is also the connection point between this chapter and Chapter 10: the architectural compression of GQA is only considered a complete engineering loop when it is measured in an inference workload. If the quality degradation outweighs the serving benefit, or if the bottleneck is actually in prefill, network, RAG retrieval, or sampling post-processing, then reducing KV heads may not be the correct optimization.

🔑
Engineering Judgment: GQA is an architectural choice that trades K/V representation capacity for decode cache and bandwidth benefits. Training engineers must be able to convert an MHA checkpoint into a GQA initialization that can be further trained; inference engineers must be able to demonstrate that the cache, TPOT, and concurrency benefits truly offset the quality risk.

def repeat_kv_heads(kv, n_rep): """kv: [B, H_kv, T, D] -> [B, H_kv * n_rep, T, D]""" if n_rep == 1: return kv return kv.repeat_interleave(n_rep, dim=1) class GroupedQueryAttention(nn.Module): """Grouped Query Attention — KV heads < Q heads""" def __init__(self, d_model, n_heads, n_kv_heads, dropout=0.1): super().__init__() assert d_model % n_heads == 0 assert n_heads % n_kv_heads == 0 self.d_model = d_model self.n_heads = n_heads self.n_kv_heads = n_kv_heads self.d_k = d_model // n_heads self.n_rep = n_heads // n_kv_heads self.W_q = nn.Linear(d_model, n_heads * self.d_k, bias=False) self.W_k = nn.Linear(d_model, n_kv_heads * self.d_k, bias=False) self.W_v = nn.Linear(d_model, n_kv_heads * self.d_k, bias=False) self.W_o = nn.Linear(d_model, d_model, bias=False) self.dropout = nn.Dropout(dropout) def forward(self, x, mask=None): batch, seq_len, _ = x.shape q = self.W_q(x).reshape(batch, seq_len, self.n_heads, self.d_k) k = self.W_k(x).reshape(batch, seq_len, self.n_kv_heads, self.d_k) v = self.W_v(x).reshape(batch, seq_len, self.n_kv_heads, self.d_k) q = q.transpose(1, 2) # (b, n_heads, s, d_k) k = k.transpose(1, 2) # (b, n_kv_heads, s, d_k) v = v.transpose(1, 2) k = repeat_kv_heads(k, self.n_rep) # (b, n_heads, s, d_k) v = repeat_kv_heads(v, self.n_rep) scores = torch.matmul(q, k.transpose(-2, -1)) / (self.d_k ** 0.5) if mask is not None: scores = scores.masked_fill(mask == 0, float('-inf')) attn_weights = F.softmax(scores, dim=-1) attn_weights = self.dropout(attn_weights) out = torch.matmul(attn_weights, v) out = out.transpose(1, 2).contiguous() out = out.reshape(batch, seq_len, self.d_model) out = self.W_o(out) return out, attn_weights

4.7 DeepSeek MLA—Compressing KV into Latent Space

Limitations of GQA

GQA reduces the number of KV heads from to , but each KV head still stores a -dimensional vector. Can we compress the individual KV vectors themselves? This is the core innovation of DeepSeek MLA.

Core Idea of Low-Rank Decomposition

MLA (Multi-head Latent Attention) treats KV storage as a low-rank approximation problem:

Matrix Absorption—Reducing Explicit Decompression Overhead

The true cleverness of MLA lies in inference: the decompression matrix can be absorbed into the Q projection , because:

Matrix Absorption

This means during inference:

  • When computing Q for each layer, first apply the transformation to Q; in engineering implementation, this can be fused with the Q projection, thereby reducing the extra overhead of explicit decompression, but still incurring the cost of projection, RoPE branching, and attention computation
  • K can be directly dot-producted with the latent vector without decompression
  • The KV Cache only needs to store the latent vector

The mla_absorbed_attention_scores(q, latent_kv, up_k_weight) function in the Ch04 assignment requires students to verify this equivalence. The input q has shape [B,H,T_q,D], the latent cache is [B,T_k,L], and the K decompression matrix weight is [H*D,L]. The reference test first explicitly computes K = W_{UK} c and obtains qK^T, then reshapes W_{UK} by head into [H,D,L] and computes (q W_{UK}) c^T; the two must be element-wise equal. This step illustrates that MLA's cache savings come from the associativity of linear algebra, not from making K/V information "disappear for free."

Note: the cache is a single latent vector, not one each for K and V.

Worked Example: How RoPE Encodes Relative Position into the Dot Product

RoPE treats every two dimensions as a 2D plane and rotates Q and K according to the token position. Suppose at a certain frequency, advancing 1 position corresponds to a rotation of 30°, and before rotation q = [1, 0], k = [1, 0]. If the query is at position 5 and the key is at position 2, they are rotated by 150° and 60° respectively, and the dot product equals the cosine of the angle between them, 90°:

2D RoPE dot product

If the entire text segment is shifted right by 10 positions, the query becomes position 15 and the key becomes position 12. Both absolute rotation angles change, but the relative difference remains 3 positions, and the dot product is still cos(90°)=0. This is the core property of RoPE: the attention score can perceive relative distance, rather than just memorizing absolute position indices.

This property comes from the orthogonality and additivity of the rotation matrix: R_m^T R_n = R_{n-m}. It is precisely because the rotation depends on positions m, n that it cannot be arbitrarily interchanged with MLA's fixed decompression matrix. The content vector can undergo matrix absorption, but the position rotation part usually needs to be retained and computed separately.

The Necessity of Decoupled RoPE

MLA faces a key problem: the RoPE rotation itself is a linear orthogonal transformation, but it depends on the token position; this position-dependent rotation generally cannot be interchanged with the fixed decompression matrix, nor can it be uniformly absorbed into the same Q projection. If RoPE is applied to the decompressed K, the matrix absorption for the content part is no longer directly valid.

DeepSeek's solution is decoupled RoPE: split K into two parts—the content part (via low-rank compression/decompression) and the RoPE part (projected separately, without compression). During inference, only the content part participates in matrix absorption, while the RoPE part computes its dot product contribution separately.

🔑
Significance of MLA: In standard MHA, KV Cache accounts for 30-40% of total GPU memory (even higher for long contexts). MLA reduces KV Cache by approximately 93%, making the inference cost of DeepSeek-V2/V3 only about 1/7 that of a similarly sized Llama 3. This is one of the core technological pillars of the "cost-effective" DeepSeek series.

4.8 Programming Exercise 4: Implementing a Simplified MLA

🖥️ Exercise 4: Simplified Multi-head Latent Attention

Task: Implement a simplified MLA module (omit the RoPE decoupling part, focus on KV compression).

Requirements:

  • Q Projection: Standard nn.Linear(d_model, n_heads * d_k)
  • KV Low-Rank Compression: First use W_dkv: nn.Linear(d_model, d_latent) to compress KV into the latent space
  • KV Decompression: Use W_uk and W_uv (both nn.Linear(d_latent, n_heads * d_k)) to decompress into K and V for each head
  • Attention computation is the same as MHA (scaled dot-product + softmax)
  • forward returns (output, attn_weights, latent), where latent is the compressed latent vector, simulating the content of the KV Cache
  • Verification: MLA caches only one shared latent vector d_latent per token (not 2 * d_latent—K/V are decompressed from the same latent vector; much smaller than MHA's 2 * n_heads * d_k)
# Expected usage
mla = MLA(d_model=4096, n_heads=64, d_latent=512)

out, weights, latent = mla(x)
assert out.shape == (2, 16, 4096)
assert latent.shape == (2, 16, 512)

# KV Cache size comparison
mha_cache_per_token = 2 * 64 * (4096 // 64)   # = 8192
mla_cache_per_token = 512                       # single shared latent vector c^KV (not 2×)
print(f"Compression ratio: {mha_cache_per_token / mla_cache_per_token:.0f}x")  # 16x
class MLA(nn.Module): """Simplified Multi-head Latent Attention — Low-Rank KV Compression""" def __init__(self, d_model, n_heads, d_latent, dropout=0.1): super().__init__() assert d_model % n_heads == 0 self.d_model = d_model self.n_heads = n_heads self.d_k = d_model // n_heads self.d_latent = d_latent # Q Projection self.W_q = nn.Linear(d_model, n_heads * self.d_k, bias=False) # KV Low-Rank Compression (shared compression matrix) self.W_dkv = nn.Linear(d_model, d_latent, bias=False) # KV Decompression self.W_uk = nn.Linear(d_latent, n_heads * self.d_k, bias=False) self.W_uv = nn.Linear(d_latent, n_heads * self.d_k, bias=False) self.W_o = nn.Linear(n_heads * self.d_k, d_model, bias=False) self.dropout = nn.Dropout(dropout) def forward(self, x, mask=None): batch, seq_len, _ = x.shape n_heads, d_k, d_latent = self.n_heads, self.d_k, self.d_latent # Q Projection q = self.W_q(x).reshape(batch, seq_len, n_heads, d_k) q = q.transpose(1, 2) # (b, n_heads, s, d_k) # KV: Compress -> Decompress latent = self.W_dkv(x) # (b, s, d_latent) k = self.W_uk(latent).reshape(batch, seq_len, n_heads, d_k) v = self.W_uv(latent).reshape(batch, seq_len, n_heads, d_k) k = k.transpose(1, 2) v = v.transpose(1, 2) # Standard Attention scores = torch.matmul(q, k.transpose(-2, -1)) / (d_k ** 0.5) if mask is not None: scores = scores.masked_fill(mask == 0, float('-inf')) attn_weights = F.softmax(scores, dim=-1) attn_weights = self.dropout(attn_weights) out = torch.matmul(attn_weights, v) out = out.transpose(1, 2).contiguous() out = out.reshape(batch, seq_len, self.d_model) out = self.W_o(out) return out, attn_weights, latent # latent is used to simulate KV Cache

💡 In a real MLA implementation, the decompression matrix is absorbed into the Q projection during inference, so the actual "decompression" operation is not needed at each step. For pedagogical clarity, we explicitly perform the complete compression->decompression flow here.

Starting from single-head attention, you have implemented three generations of multi-head attention variants along the evolutionary path of attention mechanisms:

  • ✅ Multi-Head Attention (MHA) — Parallel subspaces, parameter count equivalent to single-head but with stronger representational power

  • ✅ KV Cache Quantization Comparison — Under 128K context: MHA 7GB vs GQA 0.9GB vs MLA 0.25GB
