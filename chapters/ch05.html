5.1 Decoder-Only Architecture Overview — The Transformer Block as the Main Reusable Unit

Many mainstream autoregressive LLMs (such as the language models in the GPT, Llama, and DeepSeek series) adopt a Decoder-only architecture: there is no separate encoder; it mainly consists of stacked Transformer Blocks. Still, note that encoder-only, encoder-decoder, or hybrid architectures also exist in NLP and multimodal systems; this chapter focuses on the decoder-only route. Each Block contains two core sublayers:

  • Self-Attention Layer — Allows tokens to exchange information (completed in Chapter 4)
  • Feed-Forward Neural Network (FFN) — Performs a nonlinear transformation independently for each token

Each sublayer is wrapped with residual connections and layer normalization. The overall form (Pre-Norm specification) is:

Equation 5.1 Transformer Block (Pre-Norm)

Why are two different sublayers needed? Attention provides "information retrieval" — each token aggregates information from the entire sequence. The FFN provides "information processing" — each token performs a complex nonlinear transformation after receiving information. Without the FFN, the model would be just a linear weighted summation system with extremely limited expressiveness.

The output dimension of each Block remains , which allows the model to stack many layers. However, whether the depth can be trained stably also depends on the normalization position, initialization, residual scaling, optimizer, learning rate schedule, and data scale.

5.1A Worked Example: Gradient Paths of Pre-Norm vs. Post-Norm

Use a scalar residual block to approximate the difference in normalization position. Let the sublayer be f and the normalization be n. Pre-Norm is written as y = x + f(n(x)), and Post-Norm is written as y = n(x + f(x)). After local linearization:

Local gradient factors

Given a three-layer toy block: sublayer slopes are [0.2, -0.1, 0.3], and the local slope of the norm is approximately 0.5. The gradient factor for each layer in Pre-Norm is:

Pre-Norm: [1 + 0.2*0.5, 1 - 0.1*0.5, 1 + 0.3*0.5] = [1.10, 0.95, 1.15] product = 1.20175

In Post-Norm, the norm is applied directly after the residual output:

Post-Norm: [0.5*(1 + 0.2), 0.5*(1 - 0.1), 0.5*(1 + 0.3)] = [0.60, 0.45, 0.65] product = 0.1755

This toy example explains why Pre-Norm is more conducive to gradient propagation in deep layers: an explicit 1 is retained in the residual path. However, it is not a sufficient guarantee for infinitely deep networks; the Jacobian of a real Transformer is a high-dimensional matrix, also influenced by attention, FFN, initialization, learning rate, residual scaling, and the optimizer. Layer Normalization is the cornerstone of Transformer training stability. Unlike BatchNorm, LayerNorm normalizes along the feature dimension and does not depend on batch size—this is crucial for variable-length sequences and autoregressive models.

Task: Implement the LayerNorm class without using nn.LayerNorm.

Requirements:

  • Learnable parameters gamma (scale, initialized to all ones) and beta (shift, initialized to all zeros)
  • eps=1e-6 to prevent division by zero
  • Support arbitrary input shapes: normalization is performed on the last dimension
  • Use unbiased=False for variance calculation
  • Verification: The output should be consistent with nn.LayerNorm (random input, error < 1e-5)
# Expected usage
ln = LayerNorm(dim=768)
x = torch.randn(2, 16, 768)
y = ln(x)
assert y.shape == x.shape

# Comparison with PyTorch official
ref = nn.LayerNorm(768, eps=1e-6)
ref.load_state_dict({'weight': ln.gamma, 'bias': ln.beta})

import torch import torch.nn as nn class LayerNorm(nn.Module): """Layer Normalization — normalizes along the feature dimension (without nn.LayerNorm)""" def __init__(self, dim, eps=1e-6): super().__init__() self.eps = eps self.gamma = nn.Parameter(torch.ones(dim)) self.beta = nn.Parameter(torch.zeros(dim)) def forward(self, x): # Calculate mean and variance (along the last dimension) mean = x.mean(dim=-1, keepdim=True) var = x.var(dim=-1, keepdim=True, unbiased=False) # Normalize x_norm = (x - mean) / torch.sqrt(var + self.eps) # Affine transformation return x_norm * self.gamma + self.beta # Verification ln = LayerNorm(768) x = torch.randn(2, 16, 768) y = ln(x) ref = nn.LayerNorm(768, eps=1e-6) ref.load_state_dict({'weight': ln.gamma, 'bias': ln.beta}) y_ref = ref(x) print(f"Max error: {(y - y_ref).abs().max().item():.2e}") assert torch.allclose(y, y_ref, atol=1e-5)

💡 Note on var(unbiased=False): LayerNorm uses population variance (dividing by N, not N-1), which differs from the sample variance in statistics. Using unbiased=False ensures consistency with the implementation of nn.LayerNorm.

5.3 Theory: Pre-Norm vs. Post-Norm — How Normalization Position Affects Training Stability

Whether the normalization layer is placed at the input or output of the sublayer has a decisive impact on training dynamics. This is a key design decision repeatedly validated during the scaling of LLMs.

Post-Norm (Original Transformer)

The original Transformer (Vaswani et al., 2017) used Post-Norm:

Post-Norm

Normalization occurs after the residual connection. This means the output of the residual branch (the output of Attention/FFN) can interfere with the scale of the original information in the residual stream. In deep networks, this interference accumulates continuously, leading to gradient instability.

Pre-Norm (Modern LLM Standard)

Pre-Norm places normalization at the input of the sublayer:

Pre-Norm

Key advantage: The residual stream remains unnormalized. Information can travel along the residual path across layers with less direct suppression from the normalization layer, and the gradient path is shorter and more direct. Pre-Norm generally makes it easier to train deeper networks; the actual achievable depth still depends on initialization, optimizer, residual scaling, data, and implementation details.

Training Dynamics Analysis

More formally, consider the residual stream of an L-layer Pre-Norm Transformer:

Variance Propagation in the Residual Stream

In Post-Norm, normalization occurs after the residual connection, so (forced constant by normalization), but the variance of each sublayer's output, , requires careful control. In Pre-Norm, the variance of the residual stream increases layer by layer (this is normal—deeper layers need a larger representation range) rather than being forcibly truncated.

A minimal hand-calculation model linearizes each sublayer into a local slope f' and the norm into a local slope n'. The approximate gradient factor for a Pre-Norm residual layer is 1 + f' n', because the identity shortcut bypasses the norm; the approximate factor for Post-Norm is n'(1 + f'), where the norm is applied directly after the shortcut. The residual_gradient_path_factors in the Ch05 assignment uses this scalar model for students to manually calculate multi-layer gradient factors; it is a local diagnostic, not a complete proof of training stability for arbitrary depths.

Key Conclusion: The success of Pre-Norm can be attributed to "normalization isolation"—the input of each sublayer is independently normalized, while the residual branch retains an unnormalized information channel. This aids gradient propagation in deep networks, but it is not a guarantee of stability for arbitrary depths and learning rates. Post-Norm places normalization after the residual connection, typically requiring more careful handling of initialization, warmup, and residual scaling. 5.4 Programming Exercise 2: Implement RMSNorm—The Choice of Llama/DeepSeek

RMSNorm (Zhang & Sennrich, 2019) removes the mean-centering step of LayerNorm, retaining only the RMS scaling. It is based on a simple observation: mean-centering has a negligible impact on training quality, while the computational savings (about 15%) are significant.

Equation 5.3 RMSNorm

Worked Example: RMSNorm Does Not Center the Mean

Take a 4-dimensional vector x=[1, 2, 2, 3], ignoring eps and gamma for now. RMSNorm computes the mean of squares:

Manual RMS Calculation

Therefore, the output is approximately [0.471, 0.943, 0.943, 1.414]. Its RMS equals 1, but its mean is still approximately 0.883, not 0. LayerNorm would first subtract the mean 2.0 and then divide by the standard deviation; RMSNorm preserves the original vector direction, only scaling the length to a stable magnitude. This is why the homework test checks that the output RMS is close to 1, while also confirming that the output mean is not necessarily 0.

RMSNorm Input Gradient

The second term is the coupling term most easily missed in RMSNorm backpropagation: each feature affects the shared RMS denominator, so the gradient for one dimension depends not only on its own upstream gradient. The rms_norm_input_gradient in the Ch05 homework requires students to write this formula as tensor operations and align it with PyTorch autograd.

🖥️ Exercise 2: RMSNorm Module

Task: Implement RMSNorm — perform only RMS scaling, removing mean centering.

Requirements:

  • The only learnable parameter is gamma (initialized to all ones), no beta
  • eps=1e-6 to prevent division by zero
  • Compute rms = sqrt(mean(x^2) + eps), then x / rms * gamma
  • Verify: The output norm (L2 norm) of RMSNorm(x) is approximately equal to the norm of gamma
  • Verify: The output direction of RMSNorm is the same as the input — it only changes the vector's length, not its angle
# Expected usage
rms = RMSNorm(dim=768)
x = torch.randn(2, 16, 768)
y = rms(x)
assert y.shape == x.shape

# Verify direction invariance
cos_sim = (x * y).sum(dim=-1) / (x.norm(dim=-1) * y.norm(dim=-1))
assert torch.allclose(cos_sim, torch.ones_like(cos_sim), atol=1e-6)
# Direction invariance means cos_sim ≈ 1.0
class RMSNorm(nn.Module): """RMS Layer Normalization — Llama / DeepSeek standard""" def __init__(self, dim, eps=1e-6): super().__init__() self.eps = eps self.gamma = nn.Parameter(torch.ones(dim)) def forward(self, x): # RMS = sqrt(mean(x^2)) rms = torch.sqrt(x.pow(2).mean(dim=-1, keepdim=True) + self.eps) return x / rms * self.gamma def extra_repr(self): return f"dim={self.gamma.shape[0]}, eps={self.eps}" # Verify direction invariance rms = RMSNorm(768) x = torch.randn(2, 16, 768) y = rms(x) cos_sim = (x * y).sum(dim=-1) / (x.norm(dim=-1) * y.norm(dim=-1)) print(f"Direction Cosine (Mean): {cos_sim.mean().item():.6f}") # Should be ≈ 1.0 # Verify output norm ≈ sqrt(dim) * gamma print(f"Output Vector Norm (Mean): {y.norm(dim=-1).mean().item():.4f}") print(f"sqrt(dim) * gamma_norm: {(768 ** 0.5) * rms.gamma.norm().item():.4f}")

5.5 Theory: FFN as Key-Value Memory — How Models Store Knowledge

The FFN is more than just a "nonlinear transformation." From an interpretability perspective, Geva et al. (2021) revealed the essence of the FFN: a large-scale key-value memory.

Revisiting the FFN

The computation of a standard FFN:

Equation 5.4 FFN Rewritten as Key-Value Storage

Each term in this summation reveals a "key-value pair":

  • Key: detects whether the input matches a certain pattern. The ReLU gating determines whether it activates — only matching keys produce a non-zero output
  • Value: is the output "knowledge fragment" corresponding to key i
  • Storage Capacity: is the number of key-value pairs. GPT-2 small has 3072 key-value pairs, GPT-3 175B has 4 × 12288 = 49152 per layer

Hierarchical Distribution of Knowledge

FFNs in different layers store different types of knowledge:

  • Lower Layers (1-4): Syntactic patterns — detecting POS tags, phrase boundaries, shallow syntactic structures
  • Middle Layers (5-8): Factual knowledge — "Paris is the capital of France", "water boils at 100°C"
  • Higher Layers (9-12): Long-range dependencies and reasoning — pronoun resolution, multi-step reasoning patterns

Implications for Model Editing

This perspective directly inspired model editing methods. ROME (Meng et al., 2022) leverages the key-value storage property of FFNs by locating the FFN neurons that store a specific fact and directly modifying their value vectors to update factual knowledge in the model — without retraining.

💡
This means: The model's knowledge is primarily concentrated in the FFN weights. Attention plays the role of "retrieval coordinator" — deciding which key-value pairs to retrieve information from. This also explains why increasing the FFN dimension (more key-value pairs) improves model capacity more than increasing the attention dimension.

5.5A Mechanistic Interpretability: From "Looks Correlated" to "Changing It Alters the Output"

Having viewed the FFN as a key-value memory, a natural question is: how do we determine whether a particular neuron, head, layer, or residual stream direction is actually involved in a specific behavior? This course distinguishes between three types of questions: whether information is present in the representation, whether the model uses this information, and whether changing this information changes the output.

Four Common Methods

MethodCore ApproachCan AnswerCannot Solely Explain
ProbingUse a linear/small model to predict POS, syntax, entity, etc., labels from hidden statesWhether a certain type of information is readable from the representationWhether the model actually uses that information during generation
Logit AttributionProject the output of a residual stream or neuron onto the unembedding directionWhich components directly push up or down certain tokensIndirect pathways and rewriting effects from subsequent layers
Activation PatchingReplace the activation of a certain layer in a perturbed sample with that from a clean sampleWhether the activation at a specific position/layer is sufficient to restore the behaviorWhether only this pathway is responsible for the behavior
AblationZero out, replace, or mask the output of a specific head/neuron/layerHow much performance or behavior changes when a component is removedWhether the model compensates through redundant pathways

A Minimal Case: Subject-Verb Agreement

Given the prompt: The keys to the cabinet ..., the model should favor are over is. An interpretability experiment could be designed as follows:

  1. Construct a contrast: Change the subject from plural keys to singular key, keeping the intervening phrase to the cabinet unchanged.
  2. Locate layers and positions: Compare the residual stream at the subject token, the prepositional phrase token, and the prediction position.
  3. Perform patching: Replace the activation at a specific layer and position from the plural sample into the singular sample, and observe whether the logit difference between are and is is restored.
  4. Then perform ablation: Mask the candidate head or FFN neuron and observe whether the agreement logit difference decreases.

Boundaries of Conclusions

High probe accuracy only shows that the hidden state contains linearly readable information. If patching restores the output, this activation has a causal effect on the behavior for this prompt set. If ablation degrades performance, the component matters for the current task. These conclusions still depend on prompt distribution, layer position, replacement method, and evaluation metric, so they cannot be generalized to "all of the model's syntactic ability is in this head."

🔑
Engineering Takeaway: Interpretability is not about drawing a heatmap. A rigorous approach is to first define a behavioral metric, such as the target logit difference, and then use contrastive samples, patching, and ablation to determine which internal states have an effect on that behavior.

5.5B The Spectrum of Interpretability Methods: From Reading Representations to Intervening on Behavior

Mechanistic interpretability is not a single technique, but a set of methods with increasing evidence strength. In this course, the priority is to articulate what question each method answers, what controls are needed, and how far the conclusions can be pushed. A useful ordering is: first observe representations, then perform attribution, then causal intervention, and only then discuss steering or model editing.

1. Method Hierarchy and Conclusion Strength

LevelRepresentative MethodRequired InputSupported ConclusionMain Risk
Representation ReadingProbing, logit lens, tuned lensHidden states, labels or target tokensWhether syntactic, factual, or next-token tendencies can be read from a layer's representationReadability does not imply actual use by the model
Local AttributionLogit attribution, path attribution, activation normComponent output, unembedding, target logit differenceWhich components directly push up/down the target tokenIgnores rewriting by subsequent layers and nonlinear interactions
Causal InterventionActivation patching, causal tracing, head/neuron ablationClean/corrupted prompt, replacement position, behavioral metricWhether a certain activation is sufficient to restore or disrupt the target behaviorOut-of-distribution intervention, replacement method may produce spurious effects

2. Logit lens looks at "what answer the intermediate layer already resembles"

The logit lens method directly multiplies the residual stream of a certain layer by the final unembedding matrix to observe which tokens the intermediate representation already favors. If layer 8 already ranks Paris in the top few positions on a country-capital question, it indicates the intermediate representation already contains some answer tendency; if subsequent layers change it to another token, it indicates the subsequent layers are still rewriting or integrating information.

Logit lens

Here, target can be the correct token, and foil is the contrast token. For example, in subject-verb agreement, use the logit difference between are and is. This metric is more stable than "looking at the top-1 token" because it directly corresponds to the research question.

3. Causal tracing requires clean/corrupted contrast

Interpretability experiments should avoid looking at a single prompt. A more robust setup is to construct a pair of inputs: a clean prompt that triggers the correct behavior, and a corrupted prompt that destroys some key information. Then, patch the activation from a certain layer and position of the clean run into the corrupted run, and see if the target behavior is restored.

  1. Define behavioral indicators: For example logit(Paris) - logit(Rome), or logit(are) - logit(is).
  2. Construct minimal perturbations: Only change country names, subject number, entity names, or fact slots; keep other tokens as unchanged as possible.
  3. Scan layers and positions: Patch the residual stream, attention output, and MLP output separately.
  4. Report recovery amount: Instead of just saying "has an effect," report what proportion of the clean-corrupted gap was recovered.

If patching the MLP output of a certain layer recovers the target logit difference, it indicates that this activation has a causal effect on behavior for that prompt pair; it still does not prove that all similar facts are stored in the same location, nor that editing that location has no side effects.

Worked Example: Upgrading Conclusions from Attribution to Patching

Suppose the model sees The capital of France is and the target token is Paris, while the contrast token is Rome. You find that the MLP at layer 10 has a high logit attribution for Paris. This only indicates it directly writes to the residual stream in the direction of Paris. If you further patch the MLP output of layer 10 from the clean prompt into the corrupted prompt The capital of Italy is, and observe that Paris rises significantly relative to Rome, only then can you say that this activation is sufficient to carry the France→Paris information. If after ablating the same MLP, the France prompt can still answer correctly, it suggests the model may have redundant pathways.

4. From Interpretation to Intervention Must Evaluate Side Effects

Activation steering and model editing are tempting because they seem to directly change model behavior. But engineering-wise, both target success rate and side effects must be evaluated:

  • Locality: Does editing France→Paris incorrectly affect Italy, Germany, or other facts?
  • Generalization: Does it still work under different phrasings, languages, or context lengths?
  • Capability retention: Do math, code, long context, and refusal boundaries degrade?
  • Rollback capability: Can the intervention be versioned, compared, and undone?
🔑
Engineering Takeaway: The ultimate value of interpretability is not "naming neurons," but decomposing model behavior into testable hypotheses. The closer you get to intervention and editing, the more you must report side effects, generalization, and rollback boundaries.

GPT-2 uses FFN with GELU activation. Many modern decoder-only LLMs (such as Llama, Mistral, and DeepSeek series) have switched to SwiGLU or similar GLU variants—a gated activation function that achieves selective information passing through three weight matrices.

🖥️ Exercise 3: Implementing FFN (GELU) and SwiGLU

Task: Implement the FFN (GELU) and SwiGLU modules.

Requirements (FFN):

  • Two linear layers: d_model → d_ff → d_model
  • Use GELU activation in the middle (use F.gelu)
  • Default d_ff = 4 * d_model (GPT style)

Requirements (SwiGLU):

  • Three linear layers: w1 (gate), w2 (value), w3 (output)
  • Default d_ff = int(8/3 * d_model)—ensures total parameter count matches 4x FFN
  • Use F.silu activation for the gate branch
  • Forward: w3(F.silu(w1(x)) * w2(x))

# Parameter counts should be roughly the same

assert abs(ffn_params - swiglu_params) / ffn_params < 0.05
import torch.nn.functional as F class FFN(nn.Module): """Standard FFN (GELU) — GPT-2 style""" def __init__(self, d_model, d_ff=None): super().__init__() d_ff = d_ff or d_model * 4 self.fc1 = nn.Linear(d_model, d_ff, bias=True) self.fc2 = nn.Linear(d_ff, d_model, bias=True) def forward(self, x): return self.fc2(F.gelu(self.fc1(x))) class SwiGLU(nn.Module): """Gated SwiGLU — Llama / DeepSeek style""" def __init__(self, d_model, d_ff=None): super().__init__() d_ff = d_ff or int(d_model * 8 / 3) # Keep parameter count consistent with 4x FFN self.w1 = nn.Linear(d_model, d_ff, bias=False) # Gate self.w2 = nn.Linear(d_model, d_ff, bias=False) # Value self.w3 = nn.Linear(d_ff, d_model, bias=False) # Output def forward(self, x): gate = F.silu(self.w1(x)) # (b, s, d_ff) value = self.w2(x) # (b, s, d_ff) return self.w3(gate * value) # (b, s, d_model) # Parameter count comparison ffn = FFN(768) swiglu = SwiGLU(768) print(f"FFN parameter count: {sum(p.numel() for p in ffn.parameters()):,}") print(f"SwiGLU parameter count: {sum(p.numel() for p in swiglu.parameters()):,}") print(f"Ratio: {sum(p.numel() for p in swiglu.parameters()) / sum(p.numel() for p in ffn.parameters()):.3f}")

💡 Why SwiGLU is often adopted: the gating mechanism allows the model to selectively pass information. When the gate value is close to 0, information in that dimension is blocked. Compared with GELU's "all neurons activate, but to different degrees" behavior, SwiGLU provides more flexible token-level control. SwiGLU performs well in many Transformer reports, but the specific PPL improvement depends on model scale, data, width balancing, training budget, and implementation details; it should not be treated as a fixed gain.

5.6A Why Modern LLMs Prefer SwiGLU: Gating, Parameter Budget, and Expressiveness

Simply memorizing SwiGLU as w3(silu(w1(x)) * w2(x)) is not enough. What truly matters is that the FFN is no longer just "project, activate, then project again," but instead splits each hidden dimension into a gate branch and a value branch, allowing the model to decide at the token level which intermediate features should pass through.

What Gated Multiplication Changes

In GELU FFN, the output of the intermediate neuron is primarily determined by a single path . SwiGLU introduces element-wise multiplication:

Element-wise gating in SwiGLU

This effectively allows a pattern detector to control whether another value channel is written to the residual stream. Compared to standard activations, gated multiplication makes it easier to express interactions like "only use feature B if condition A holds." Such conditional interactions are common in language modeling: code indentation, bracket closure, entity attributes, coreference relations, and format constraints are not isolated features but are meaningful only under specific contextual conditions.

Difference between SwiGLU and MoE

SwiGLU is a dense FFN: every token passes through the same set of matrices, with soft gating applied along the hidden dimension. MoE is a sparse FFN: each token activates only a few experts, with a router selecting among them. Both introduce "selective computation," but at different granularities:

MechanismWhere selection occursActivation per tokenPrimary benefitPrimary risk
SwiGLUWithin a single FFN's dimensionsAll matrices participate in computationEnhanced conditional interaction under the same parameter budgetActivation memory and dense computation still exist
MoEAcross multiple expertsOnly top-k experts selectedTotal parameters increase significantly while activated parameters remain controllableLoad balancing, communication, and routing stability are more complex
🔑
Design Conclusion: SwiGLU is not simply "stronger because it has one more matrix." It uses gated multiplication to increase token-conditional representation under a roughly similar parameter budget; MoE, at the cost of higher system complexity, scales FFN capacity across multiple sparse experts.

5.6C Parameters, FLOPs, and Activation Memory of a Single Transformer Block

Simply writing TransformerBlock.forward is not enough. In training engineering, one must be able to estimate why a block is expensive: parameters mainly come from projection matrices, FLOPs mainly from matrix multiplications and the quadratic term of attention, and training activation memory stores Q/K/V, FFN intermediate values, and attention scores.

FLOPs and Activation Memory

The estimate_block_resources function in the course assignment uses the common matrix multiplication convention: one [m,k] @ [k,n] operation is approximately FLOPs. It separately estimates the main FLOPs for QKV projection, attention score, attention value mixing, output projection, and the three linear layers of SwiGLU.

Activation memory estimation includes the main tensors: Q/K/V, the gate/value intermediate activations of SwiGLU, attention scores, and the residual stream. The size of the attention scores is , which is why training with long sequences is much more expensive than with short sequences. FlashAttention can avoid fully materializing the attention matrix, but standard pedagogical implementations save it explicitly.

Activation checkpointing involves a trade-off: it does not change the parameter count or the final mathematical function, but saves fewer intermediate activations during the forward pass and recomputes these activations during the backward pass. The activation_checkpointing_tradeoff function in the Ch05 assignment uses a simplified model to estimate: if the checkpoint fraction is c, approximately c * activation_bytes is saved, while incurring an additional recomputation cost of c * forward_flops during backpropagation.

Worked Example: Resource Budget of a Pedagogical Block

Assume B=2, T=128, d_model=768, n_heads=12, d_ff=2048, using fp16/bf16 activations (2 bytes). Here d_head=64. The main parameter count for a single Pre-Norm + MHA + SwiGLU block is:

Q/K/V/O projections4 * d_model^22,359,296
SwiGLU FFN3 * d_model * d_ff4,718,592

Total7,079,424

Under the same configuration, the main forward FLOPs are approximately 3.72B: QKV projection is about 0.91B, attention score and value mixing are each about 0.05B, output projection is about 0.30B, and the three linear layers of SwiGLU are about 2.42B. For short sequences with T=128, the FFN is already the dominant computational term; when T increases, the T^2 term of attention quickly becomes significant.

The main activation memory is approximately 4.63 MiB, of which attention scores account for B * H * T^2 * 2 bytes = 0.75 MiB. If half of these activations are checkpointed, the main activations saved during the forward pass can be reduced by about 2.31 MiB; the cost is an additional recomputation of about 1.86B FLOPs during the backward pass, increasing the simplified training FLOPs from 3 * forward to approximately 1.17x. This is the essence of activation checkpointing: it does not change the model, but trades extra computation for training memory.

The key design is "separating the normalization path from the residual path"—information flows unobstructed along the residual path (no non-linearity, no normalization), and RMSNorm is only applied to the input of each sublayer.

5.7A Block Implementation Stability Ledger: Residuals, Initialization, Dropout, and Precision

The mathematical form of a Transformer Block is very short, but whether training is stable depends on many implementation details. A course-level implementation can first write Pre-Norm + RMSNorm + Attention + SwiGLU; an engineering-level implementation must also explain how residual branches are initialized, where dropout is placed, what precision the norm uses, whether bias is kept, how checkpointing is chosen, and how these choices affect the loss curve, gradient norm, and throughput.

1. Control the Initial Perturbation of Residual Branches

Pre-Norm preserves a direct gradient path, but if each sublayer writes too large an update to the residual stream from the start, deep stacking can still become unstable. Common practices include smaller initialization, depth-dependent scaling of residual projections, or using a learnable residual gate. Abstractly written:

Residual update scale

can be a fixed scaling, an implicit result of the initialization strategy, or an explicit gate. Its role is not to change the model's representational capacity, but to control the magnitude of each layer's perturbation to the residual stream during early training. If grad_norm spikes frequently in the first few hundred steps, inspect the residual output scale and initialization in addition to the learning rate and data anomalies.

2. Dropout Placement Affects Training and Inference Equivalence

Pedagogical code often places dropout after the attention/FFN output and before the residual addition:

x = x + dropout(attention(norm1(x), mask)) x = x + dropout(ffn(norm2(x)))

The meaning of this position is: during training, randomly drop part of the sublayer's update written to the residual stream; during inference, dropout must be turned off. If dropout is placed after the residual, it will randomly destroy the identity path; if dropout is used on attention probabilities, it randomly discards the contribution of keys, which has a different effect from output dropout. If a training report compares ablations, it must clearly state the dropout position and probability.

3. Norm and softmax often require higher precision

In bf16/fp16 training, matrix multiplications can use low-precision tensor cores, but normalization statistics, softmax denominators, and certain accumulations are best done in fp32, then cast back to the model dtype. Otherwise, two types of issues can arise:

  • RMSNorm/LayerNorm statistical error: Low-precision sum of squares and mean accumulate errors over large dimensions, affecting scale stability.
  • Attention softmax overflow or underflow: Overly large logits, inappropriate mask sentinel values, or fully masked rows can produce NaN.

Many high-performance kernels handle these details internally; hand-written teaching implementations must explicitly choose which tensors are accumulated in higher precision.

4. Bias strategy must be consistent with the architecture convention

Modern decoder-only LLMs often omit bias in QKV, output projection, and SwiGLU linear layers, and use RMSNorm instead of LayerNorm with beta. This is not a hard theorem, but a set of engineering conventions: reduce parameters and kernel branches, and cooperate with the scale control of Pre-Norm/RMSNorm. If reproducing a specific checkpoint, the bias strategy must be consistent with the original model; otherwise, the parameter count, state_dict keys, and output distribution will all mismatch.

5. Activation checkpointing is chosen per sublayer

Checkpointing does not have to be a "one-size-fits-all" approach for the entire block. A more reasonable choice depends on which part of the activation is most expensive:

SublayerPrimary ActivationsWhen to Prioritize CheckpointingCost
AttentionQ/K/V, attention scores or kernel internal statesLong sequences, attention score memory dominatesRecompute QKV, scores, softmax/value mixing
SwiGLU FFNGate/value intermediate activationsShort sequences but large d_ff, FFN activations dominateRecompute three linear layers and SiLU/gate multiplication
Full block layerAll intermediate states of both sublayersExtreme memory pressure, maximum savings neededMost recomputation, tokens/s drop more significantly

Therefore, the activation_checkpointing_tradeoff in Ch05 is just the first step; real training needs to combine a profiler to look at the activation bytes, forward time, and backward recompute time for each sublayer.

6. How to locate block-level anomalies

Engineering Takeaway: The engineering quality of a Transformer Block is not just about whether the forward pass returns the correct shape. A reliable implementation should be able to explain residual update norm, norm precision, dropout position, bias strategy, and checkpointing decisions, and be able to locate the specific sublayer when loss or throughput is abnormal.

5.8 Theory: Gradient Check — Why Verify Backpropagation?

When implementing complex custom operators, hand-written backward functions may contain mathematical errors. Even a single sign error can cause the model to "train" for a long time without converging.

Mathematical Principle of Finite Differences

PyTorch's torch.autograd.gradcheck uses Central Difference to approximate the numerical gradient:

Why gradcheck matters

Consider a manually implemented backward function with a sign error:

# Wrong backward: should be dx = grad * w, but written as dx = grad / w # gradcheck will detect the difference between numerical and analytical gradients > atol

This type of error is easy to find in simple modules, but in complex custom operators (like Triton kernels for FlashAttention), manually deriving and implementing the backward pass is a common debugging step.

⚠️
Important Note: gradcheck requires two additional forward passes for each parameter dimension. For large models, this is extremely expensive. Therefore: use float64 (finite differences are sensitive to rounding errors in float32); verify with very small inputs (batch=2, seq=8, dim=64); set eps=1e-4 and atol=1e-3.

from torch.autograd import gradcheck class LayerNormFunction(torch.autograd.Function): """Custom LayerNorm — manual forward + backward for gradcheck verification""" @staticmethod def forward(ctx, x, gamma, beta, eps=1e-6): mean = x.mean(-1, keepdim=True) var = x.var(-1, keepdim=True, unbiased=False) inv_std = torch.rsqrt(var + eps) x_hat = (x - mean) * inv_std ctx.save_for_backward(x_hat, inv_std, gamma) return x_hat * gamma + beta @staticmethod def backward(ctx, grad_output): x_hat, inv_std, gamma = ctx.saved_tensors n = x_hat.shape[-1] grad_x_hat = grad_output * gamma grad_x = inv_std / n * ( n * grad_x_hat - grad_x_hat.sum(dim=-1, keepdim=True) - x_hat * (grad_x_hat * x_hat).sum(dim=-1, keepdim=True) ) reduce_dims = tuple(range(grad_output.dim() - 1)) grad_gamma = (grad_output * x_hat).sum(dim=reduce_dims) grad_beta = grad_output.sum(dim=reduce_dims) return grad_x, grad_gamma, grad_beta, None # Test on small inputs (float64) x = torch.randn(2, 8, 64, dtype=torch.float64, requires_grad=True) gamma = torch.ones(64, dtype=torch.float64, requires_grad=True) beta = torch.zeros(64, dtype=torch.float64, requires_grad=True) test = gradcheck( LayerNormFunction.apply, (x, gamma, beta, 1e-6), eps=1e-4, atol=1e-3 ) print(f"LayerNorm gradient check: {test}") # Verify our own LayerNorm module ln = LayerNorm(64).double() x = torch.randn(2, 16, 64, dtype=torch.float64, requires_grad=True) test_module = gradcheck( lambda x: ln(x).sum(), x, eps=1e-4, atol=1e-3, raise_exception=False ) print(f"LayerNorm module gradient check: {test_module}")

5.10 Theory: Learnable Hyper-Connections — Birkhoff Constraint and Sinkhorn-Knopp Iteration

Hyper-Connections is a class of designs that attempt to extend traditional residual connections: instead of each layer only having a fixed 1:1 residual addition with its adjacent layer, they learn more flexible inter-layer connection weights. To avoid excessive signal amplification introduced by learnable connection matrices, some cutting-edge designs constrain the connection matrix to structured sets like the Birkhoff polytope.

Limitations of Traditional Residual Connections

The residual connection assumes a fixed 1:1 contribution ratio for each layer. However, different layers may contribute differently to the final output — some layers may only need to fine-tune the representation from the previous layer, while others may need to modify it significantly.

Birkhoff Polytope (Doubly Stochastic Matrix)

Learnable hyper-connections can use a connection matrix to control the strength of inter-layer connections, but directly learning can lead to gradient amplification. A common constraint idea is to make close to a doubly stochastic matrix within the Birkhoff polytope.

Birkhoff Polytope Definition

Doubly stochastic matrices have a useful spectral norm upper bound property, often written as (which holds under the corresponding assumptions). This can constrain the amplification effect of inter-layer connections, reducing the risk of gradient amplification; actual training stability also depends on residual scaling, optimizer, data distribution, and implementation details.

Sinkhorn-Knopp Iteration

The algorithm for projecting an arbitrary matrix onto the Birkhoff polytope is as follows:

Sinkhorn-Knopp Iteration

Alternate between row normalization and column normalization. When conditions such as positive matrix elements and feasible target margins are met, Sinkhorn-Knopp converges to a scaled matrix satisfying the given row/column margins; convergence speed and numerical stability depend on matrix conditioning, temperature/scaling, and the number of iterations. In practice, a small number of iterations is often used to obtain an approximately doubly stochastic structure, but it cannot be guaranteed that 10 iterations are sufficient for arbitrary inputs.

def sinkhorn_knopp(M, num_iter=10): """Sinkhorn-Knopp iteration projecting onto the Birkhoff polytope""" M = torch.exp(M - M.max(dim=-1, keepdim=True).values) M = M / (M.sum(dim=-1, keepdim=True) + 1e-6) for _ in range(num_iter): M = M / (M.sum(dim=-1, keepdim=True) + 1e-6) M = M / (M.sum(dim=-2, keepdim=True) + 1e-6) return M

Multi-head hyper-connections can use multiple independent connection matrices to capture different types of inter-layer dependencies, and then merge the connection results. The course focuses on the general mathematical structure of such methods: connection strength is learnable, but requires non-negativity, row/column normalization, residual scaling, or other constraints to control signal amplification. Specific benefits must be measured empirically under a given model, depth, optimizer, initialization, and training data; numbers from a single technical report cannot be treated as universal constants.

🔑
Intuition: The alternating normalization of Sinkhorn-Knopp iteration is closely related to entropy-regularized optimal transport. The resulting connection weights approximately satisfy probability distribution constraints along both the row and column dimensions, making the amount of information each layer "proposes" and "receives" more controlled. This helps explain why this constraint might improve deep training dynamics, but it cannot independently guarantee complete training stability.
