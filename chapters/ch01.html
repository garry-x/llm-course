1.1 Tokenizer Positioning: From Text Interface to Engineering Cost

The tokenizer is the first interface layer of an LLM. It converts raw strings into token IDs and also determines sequence length, embedding matrix size, context window utilization, and service billing metrics. This chapter treats tokenization as a model contract shared by training and inference, not as a preprocessing detail.

  • Training side: Changing tokenization rules changes token distribution, batch token count, embedding parameter count, and data quality diagnostics.
  • Inference side: The token count of the same text affects context capacity, KV Cache memory, latency, and invocation cost.
  • Release side: Checkpoints, chat templates, special tokens, and tokenizer files must be versioned together; otherwise, the model interface will mismatch.

The implementation below keeps only the minimal interpretable kernel of BPE: count pair frequencies, merge by priority, save the merge table, and use the same rules for encoding and decoding. Once this pipeline is clear, WordPiece, Unigram LM, and byte-level tokenizers become easier to compare as engineering trade-offs.

1.3 What a Token Is — An Information-Theoretic Perspective

LLMs do not directly process characters or strings; they process tokens—the smallest indivisible units of text. Each token corresponds to an integer ID, and the model sees a sequence of integers, not text.

Tokenization from an Information Theory Perspective

Assume the information content of a piece of text is bits. If we represent it with n tokens, each token carries an average of bits of information. The goal of a tokenizer is: given a vocabulary size , minimize the number of tokens required for each piece of text. This is precisely the classic trade-off in Rate-Distortion Theory:

  • Larger vocabulary (large V): Each token has high information density, sequences are short, but the embedding matrix has a large number of parameters.
  • Smaller vocabulary (small V): Fewer parameters, but sequences are longer, increasing computation and inference latency.

Why Not Use Words Directly? — Zipf's Law

Natural language follows Zipf's law: the frequency of the -th most frequent word is proportional to . This means about half of all words appear only once—if a "word"-level vocabulary is used, rare words cause severe sparsity issues. Additionally, word-level vocabularies cannot handle out-of-vocabulary (OOV) words.

The insight of Subword Tokenization is: find an intermediate granularity between "characters" and "words". Common words (e.g., "the", "and") are kept as complete tokens, rare words are decomposed into smaller subwords (e.g., "tokenization" → "token" + "ization"), and any OOV word can fall back to the character or even byte level.

Byte-level Design of BPE

The BPE tokenizer of the GPT series operates at the byte level—the base vocabulary is 256 byte values (0x00-0xFF), not characters. This means whether the text is English, Chinese, Arabic, or Emoji, the underlying representation is uniformly processed as a byte sequence. UTF-8 encoding maps Unicode characters to 1-4 bytes, and BPE performs merges on this byte sequence. The benefit is: any string can be tokenized, with no OOV.

🔑
Key Data: GPT-4's cl100k_base vocabulary has ~100K tokens, Llama 3 has ~128K tokens. CJK languages like Chinese/Japanese typically consume 2-3 times more tokens than English, because each Chinese character requires about 1.5-2 tokens, while each English token covers about 3.3 characters.

1.4 The Essence of the BPE Algorithm: Symbol Induction Driven by Data Compression

BPE (Byte Pair Encoding) was proposed by Gage (1994) for general-purpose data compression and introduced to neural machine translation by Sennrich et al. (2016). The key to understanding BPE is: it is not an NLP-specific method, but a general compression strategy.

Core Observation: Frequent Patterns = Redundancy

In any byte sequence, certain adjacent byte pairs appear repeatedly. For example, in English text, the byte sequence "t" + "h" appears with very high frequency. BPE's strategy is: merge this high-frequency pair into a new symbol, thereby shortening the sequence length. Each merge is equivalent to discovering a repeated pattern and abstracting it into a higher-level symbol.

Algorithm Flow (Abstract Layer)

BPE Training

Each merge operation shortens the sequence length by (the occurrence count of that pair). After merges, the total sequence length is significantly reduced, while the vocabulary grows from 256 to .

Why is Merging the Most Frequent Pair Reasonable?

From a compression perspective: Suppose we have a token sequence and want to replace a certain adjacent pair with a new merged symbol. After replacement, the sequence length is shortened, and the replacement rule is deterministic. Merging the most frequent pair is equivalent to maximizing the sequence length compression ratio at each step. This is a greedy heuristic that does not guarantee finding the globally optimal merge sequence; practical effectiveness also depends on the corpus, pre-tokenization, and vocabulary budget.

From an information theory perspective: Frequently co-occurring adjacent symbol pairs are often worth lexicalizing, but raw frequency is not equivalent to mutual information. High-frequency pairs might simply result from the combination of high-frequency single bytes. Therefore, BPE's selection criterion is more accurately described as local compression gain, not an optimal estimate of PMI or semantic association.

A Concrete Example

Corpus: "low lower lowest" UTF-8 byte sequence: l, o, w, _, l, o, w, e, r, _, l, o, w, e, s, t Initial vocabulary (8 characters): l, o, w, _, e, r, s, t Round 1: "l"+"o" appears 3 times → merged into token 256 ("lo") Sequence: lo, w, _, lo, w, e, r, _, lo, w, e, s, t Round 2: "lo"+"w" appears 3 times → merged into token 257 ("low") Sequence: low, _, low, e, r, _, low, e, s, t Round 3: "e"+"r" appears 2 times → merged into token 258 ("er") Sequence: low, _, low, er, _, low, e, s, t Vocabulary evolution: 8 → 9 → 10 → 11 symbols

Note: "e"+"r" is only merged in round 3 because "l"+"o" and "lo"+"w" had higher frequencies in the first two rounds. The order of merges is entirely determined by frequency—this is BPE's only "learning signal".

Worked Example: Pair Statistics Must Be Recalculated After Each Merge

The most common mistake when implementing BPE is to reuse old counts after the first round of pair statistics. BPE's counting target is not the original string, but the current token sequence; one merge changes adjacency relationships, and the next round must re-run _get_stats.

1.6 Deep Dive: The Compression Heuristic of Frequency Selection

The previous section implemented _get_stats. The next question is: Why does BPE always merge the most frequent pair?

Compression Perspective

Each merge replaces all occurrences of in the sequence with a new symbol . After replacement, the sequence length decreases by (where is the frequency of that pair), and the vocabulary increases by 1. Choosing the most frequent pair means maximizing the marginal compression ratio at each step. This is a simple and effective greedy heuristic; it does not guarantee global optimality, and practical effectiveness depends on the corpus, target vocabulary size, pre-tokenization, and tie-breaking rules.

Comparison with Huffman Coding

Huffman coding builds an optimal prefix code from the bottom up—it merges the two lowest-frequency nodes. BPE, however, merges the highest-frequency pair. The difference is:

  • Huffman builds a code—each symbol has a unique binary code. Merging low-frequency nodes gives high-frequency symbols short codes.
  • BPE builds a vocabulary—adjacent symbols in the sequence are abstracted into a new symbol. Merging high-frequency pairs causes the most common combinations to be "lexicalized".

Both build a hierarchy through repeated merging, but the merging criterion serves different ultimate goals.

Frequency vs. Mutual Information

The Pointwise Mutual Information (PMI) of two tokens is defined as:

Pointwise Mutual Information

High frequency does not necessarily mean high PMI—for example, "t" + "h" is very common in English, but this is because "t" and "h" are each common individually. A truly "meaningful" combination is a pair whose frequency far exceeds the product of the independent probabilities of its components. Standard BPE implementations use raw frequency directly rather than PMI, which can produce suboptimal merges for high-frequency but weakly correlated pairs (e.g., " " + "t"). This is one of the known limitations of BPE.

💡
Improvement Idea: Some tokenizer training criteria incorporate the marginal benefit or likelihood change of candidate segments into the selection, rather than only looking at raw frequency. WordPiece commonly uses a score related to segment probability to reduce the bias toward high-frequency but weakly correlated combinations; it shares a similar intuition with PMI, but cannot be simply stated as "equivalent to PMI".

1.7 Programming Exercise 2: Implementing Pair Replacement

After finding the highest-frequency pair, the training loop must replace all non-overlapping occurrences in the sequence. This is the core operation of BPE training.

🖥️ Exercise 2: Implement _merge

Task: Implement the function _merge(ids, pair, new_id), which finds all occurrences of pair in the integer sequence ids and replaces them with new_id.

Requirements:

  • Input ids is list[int], pair is tuple[int, int], new_id is int
  • Return a new list[int], do not modify the original list
  • Replacement is non-overlapping: merging (1,2) in [1,2,1,2] produces [new_id, new_id], not [new_id, 1, 2]
  • Complete in a single pass, complexity
  • Cannot use str.replace or regular expressions
# Expected usage
assert _merge([1, 2, 3, 1, 2], (1, 2), 99) == [99, 3, 99]
assert _merge([1, 1, 1], (1, 1), 99) == [99, 1]   # Non-overlapping replacement
assert _merge([5, 6], (1, 2), 99) == [5, 6]        # No match
def _merge(ids, pair, new_id): """Replace all specified adjacent pairs in the sequence with the new id.""" new_ids = [] i = 0 while i < len(ids): # Check if the current position matches the pair if (i < len(ids) - 1 and ids[i] == pair[0] and ids[i+1] == pair[1]): new_ids.append(new_id) i += 2 # Skip the second element of the pair (non-overlapping) else: new_ids.append(ids[i]) i += 1 return new_ids

💡 Why use while instead of for? Because the loop body needs to skip 1 or 2 steps depending on whether there is a match. Using for i in range(len(ids)) does not allow flexible control of the step size within the loop body. while + manual indexing is the clearest implementation.

1.8 BPE Training Loop: From Bytes to Vocabulary

With _get_stats and _merge, BPE training is a simple iterative process. Each iteration:

  1. Count the frequencies of all adjacent pairs in the current sequence
  2. Find the highest frequency pair
  3. Record the merge rule
  4. Apply the merge in the sequence
  5. Store the byte representation of the new token in the vocabulary

This process produces two data structures:

  • vocab (dict[int, bytes]): Mapping from token id to the original byte sequence—used for decoding
  • merges (dict[tuple[int,int], int]): Mapping from pair to new token id—used for encoding

How the vocabulary is built: The byte representation of a new token is the concatenation of the bytes of its constituent tokens. For example, if token 256 = "lo" (merged from token 108='l' and token 111='o'), then vocab[256] = vocab[108] + vocab[111]. This recursive definition ensures that every entry in the vocab can be traced back to the original bytes.

🔑
Important Property: The vocab is lossless—any token can be reduced back to the original byte sequence. This is the mathematical basis for BPE decoding requiring no model or probability.

Worked Example: BPE merge trace and token savings

Take abababa as an example, the initial byte sequence length is 7. In the first round, among the adjacent pairs, both (a,b) and (b,a) appear 3 times; the implementation in this chapter selects (a,b) based on the current scan order, creates 256="ab", and after non-overlapping replacement, the sequence changes from [a,b,a,b,a,b,a] to [256,256,256,a], length 7→4, saving 3 tokens. In the second round, (256,256) appears 2 times, creates 257="abab", the sequence becomes [257,256,a], length 4→3, saving another 1 token.

The bpe_training_trace(text, vocab_size, max_steps) function in the Ch01 assignment requires recording the pair, count, new_id, before_length, after_length, and tokens_saved for each step. This allows students to distinguish two things: BPE compression comes from non-overlapping replacement of high-frequency adjacent pairs; pair statistics must be recalculated after each merge, and the old frequency table cannot be reused.

1.10 Encoding and Decoding—From Rules to a Complete Tokenizer

After training is complete, we use the merges rules to encode new text, and use vocab to decode the token sequence back into text.

Decoding: Simple and Perfectly Reversible

Decoding is lookup and concatenation: for each ID in the token ID sequence, retrieve the corresponding byte string from vocab, concatenate them, and decode using UTF-8:

Since each token in vocab is a deterministic concatenation of byte sequences, decoding requires no search or model—it always produces a unique result. This is a key advantage of BPE compared to other tokenization schemes.

Encoding: Deterministic Greedy Merging by Merge Rank

Encoding requires converting new text into a sequence of token IDs. The strategy is: start from the byte sequence, repeatedly apply merge rules, each time prioritizing the merge that appeared earliest in training order (i.e., the highest priority).

Specific approach:

  1. Encode the text into a UTF-8 byte sequence, convert to a list of integers
  2. Repeatedly check all adjacent pairs — if a pair exists in the merges dictionary, find the pair that was merged earliest (i.e., with the smallest new_id)
  3. Apply that merge and update the sequence
  4. Continue until no pair can be merged

Trick for finding the "earliest merge": min(stats, key=lambda p: merges.get(p, float('inf'))). If a pair is not in merges, return inf — only learned merge rules are applied.

💡
Intuition: Pairs merged earlier during training (smaller new_id) represent "more basic" patterns — they were lexicalized earlier in the training corpus. During encoding, these fixed rules are applied first, ensuring the same set of merges always produces the same segmentation for the same input.

Worked Example: Encoding does not re-learn based on frequency

Assume the following merge rules are obtained after training, where a smaller number indicates earlier addition and higher priority during training:

merge rankpairnew tokenbyte meaning
256('l', 'o')loFirst learned common fragment lo
257('lo', 'w')lowThen merged lo and w into low
258('e', 'r')erFinally learned suffix er

Now encode the new text "lower". The initial byte sequence is [l, o, w, e, r], with adjacent pairs (l,o), (o,w), (w,e), (e,r). Among these, (l,o) and (e,r) appear in merges; even though both appear only once in this new text, the earlier-ranked (l,o) → lo must be chosen:

[l, o, w, e, r] -> apply rank 256: (l, o) -> lo [lo, w, e, r] -> apply rank 257: (lo, w) -> low [low, e, r] -> apply rank 258: (e, r) -> er [low, er]

If the encoding phase re-computed the frequency of the current input or temporarily selected the "longest-looking" fragment, it would break the merge priority saved during training, causing the same vocabulary to produce inconsistent segmentations across different implementations. Therefore, the encode function in the course assignment uses merges.get(pair, inf) to find the minimum rank among currently available pairs, rather than re-training a local tokenizer.

🖥️ Exercise 5: Implement encode

Task: Implement encode(text, merges) to encode text into a sequence of token IDs.

Requirements:

  • Input text is str, merges is dict[tuple[int,int], int]
  • Return list[int]
  • Start from the UTF-8 byte sequence, repeatedly apply the highest priority merge from merges
  • Use min(stats, key=lambda p: merges.get(p, float('inf'))) to find the highest priority pair
  • Stop when no pair exists in merges
# Expected usage
vocab, merges = train("low " * 100 + "lower " * 80 + "lowest " * 60, vocab_size=280)
text = "lowest lower"
ids = encode(text, merges)
print(f"Encoded {text!r} → {ids}")

# Verification loop: encode → decode should yield the original text
decoded = decode(ids, vocab)
assert decoded == text, f"Round-trip mismatch: {decoded!r} != {text!r}"
def encode(text, merges): """Encode text into a sequence of token IDs.""" ids = list(text.encode("utf-8")) while len(ids) >= 2: # Count frequencies of adjacent pairs in the current sequence stats = _get_stats(ids) # Find the pair with the highest priority (earliest merged, i.e., smallest new_id) pair = min(stats, key=lambda p: merges.get(p, float("inf"))) # If the highest priority pair is not in merges → stop if pair not in merges: break # Apply the merge new_id = merges[pair] ids = _merge(ids, pair, new_id) return ids

💡 Greedy nature of encoding: Only one merge is applied per round — the one with the highest available merge rank (earliest added during training). The merge rank comes from the merge order saved during training; encoding repeatedly merges adjacent pairs according to this fixed priority, rather than re-computing frequencies from the current input, and is not equivalent to general longest-match matching.

1.12A How to Evaluate a Tokenizer

Implementing BPE is only the first step. When actually training an LLM, the tokenizer affects sequence length, embedding parameter count, context window utilization, inference latency, and multilingual coverage. A vocabulary is not simply "bigger is better" or "smaller is better"; it involves trade-offs between data distribution, model scale, and deployment cost.

A Small Experiment

Select 100 samples each of English, Chinese, code, mathematical formulas, and emoji/mixed text. Encode them with different vocab sizes or different tokenizers, and report the average token count, P95 token count, whether round-trip is successful, and the embedding parameter count . 1.12B Tokenizer is the Model Interface Contract: Special Tokens, Templates, and Checkpoints

Many training incidents are not caused by errors in Transformer code, but by inconsistencies in the tokenizer protocol: role tokens differ between training and inference, embeddings are not extended after adding new special tokens, padding tokens are learned as normal text, or the evaluation set calculates context length using a different tokenizer. For LLM training and inference engineers, the tokenizer is not just a preprocessing tool, but a part of the model checkpoint.

1. Special tokens must have stable semantics

Modern LLMs typically require at least the following special tokens or equivalent protocols:

TypeFunctionConsequences of Error
BOS/EOSMark sequence start and end, control generation stopModel doesn't stop, stops prematurely, or outputs the end token as content
PADBatch padding, usually not involved in lossPadding is trained as predictable text, loss denominator and attention mask are wrong
Role tokenDistinguish system/user/assistant/toolModel confuses speakers, SFT mask cannot reliably locate response spans
Tool/Function tokenMark tool calls, parameter JSON, or return valuesStructured call boundaries are unstable, inference service struggles to parse
Reserved tokenReserve IDs for future extensionsSubsequent extensions require retraining or migrating embeddings

Special tokens should not be "accidentally" segmented by ordinary BPE merges, but should be treated as indivisible atomic tokens. Otherwise, the string <|assistant|> might be split into multiple ordinary fragments, making it difficult for the model to learn stable boundaries, and making safety filtering and tool parsing fragile.

2. Chat template determines the supervision boundary for SFT

The same conversation is typically serialized into template text at the tokenizer layer before being converted to token IDs. During SFT, the label mask relies on this template to locate the assistant response:

... Please explain gradient clipping. Gradient clipping limits the gradient norm...

If the training pipeline uses <assistant>, but the inference service uses Assistant:, the role boundaries the model sees change. Short-term, this might just appear as format drift; long-term, it can affect multi-turn conversations, refusal boundaries, tool calls, and sequence log probability alignment in DPO/GRPO. The SFT data pipeline in Ch09 will continue to use the principles here: the template, tokenizer, and label mask must jointly define the training objective.

3. Modifying the vocabulary equals modifying the model shape

The vocabulary size V determines the shape of the token embedding and LM head. If a pre-trained model uses V=50,257 and later 100 special tokens are added, the embedding needs to change from [50257, d_model] to [50357, d_model]. This is not a minor change in a configuration file, but a checkpoint compatibility issue:

  • Input embedding: New rows need initialization, common practices are random initialization or using the mean of similar tokens.
  • LM head: If weight tying is used, the LM head expands with the embedding; if not shared, the output projection must also be expanded synchronously.
  • Old checkpoint: Loading will result in a shape mismatch; new rows must be explicitly handled, not ignored by suppressing errors.
  • Old data: The semantics of old token IDs must remain unchanged; rearranging token IDs would break all existing weights.

Therefore, production models typically freeze existing token IDs and only append new tokens at the end of the vocabulary. If a complete tokenizer replacement is needed, it usually implies the need for continued pre-training or at least a longer adaptation training, as the geometric structure of the entire embedding space has changed.

4. The tokenizer version must be released together with the model

A reproducible model package must include the tokenizer artifact (tokenizer.json or a SentencePiece model), special token mapping, chat template, normalization rules, and model configuration. Releasing only model.safetensors is insufficient to reproduce model behavior. When loading a model, the inference service should also check:

config.vocab_size == tokenizer.vocab_size special token ids match model config chat template version matches SFT/DPO training pad/eos/bos ids are explicit round-trip tests pass for multilingual and tool-call samples

These checks are not formalities. If the tokenizer changes how Chinese, code indentation, JSON quotes, or emojis are segmented, the model's context length, generation probability, and stopping behavior will all change; this will propagate all the way to weight loading in Chapter 6, sequence log probabilities in Chapter 9, and token costs in Chapter 10.

🔑
Section Takeaway: The tokenizer defines the discrete interface the model can read and write. BPE merge rules, special tokens, chat templates, and vocab size must be versioned together with the model; otherwise, the training loss, alignment data, checkpoint loading, and inference service are not looking at the same task.

1.12C Chat Template and Multimodal Interface Ledger: One Message, One Token Contract

A production tokenizer does more than convert plain text into token IDs. It also defines how a structured conversation becomes a single sequence: system instructions, user messages, assistant answers, tool calls, tool results, multimodal placeholders, stop tokens, and masked training spans. The same message object must serialize to the same token boundary in pre-training adapters, SFT, preference training, evaluation, and online inference. If it does not, the model is being trained and served through different interfaces.

Therefore, a tokenizer change is an interface migration. It is safe only when the release artifact includes merge rules, special token IDs, chat template, normalization, processor placeholders, and compatibility tests. For training engineers, this defines the loss boundary. For inference engineers, it defines the parser boundary, tool execution boundary, and multimodal token budget. Treating the template as simple UI formatting is one of the fastest ways to create silent model behavior drift.

1.12D Multilingual Tokenizer Training: From Corpus Ratio to Token Cost

For real-world LLMs, the tokenizer is not something to train casually on an English corpus; it is a data interface across languages, scripts, and domains. It determines how much information different languages can express within the same context window, and it affects effective text volume in a training batch, inference billing, KV Cache memory, and long-context user experience. Multilingual models especially need to treat tokenizer training as a data engineering decision.

1. Corpus ratio changes merge rules

BPE only looks at the frequency of adjacent symbols in the training corpus. If English accounts for 95% of the tokenizer training corpus while Chinese, Arabic, Hindi, code, and mathematical formulas occupy only a small share, high-frequency merges will prioritize English patterns. Other languages will more often fall back to bytes or short fragments, which appears as higher token counts and reduced context capacity. Conversely, blindly oversampling a minority language can sacrifice compression for high-traffic languages and domain text.

Design ItemEngineering ProblemRecommended PracticeTypical Failure
Language SamplingCan different languages all obtain sufficient merge budget?Set sampling upper/lower limits based on target users and training data, rather than sampling directly according to raw web page volumeLow-resource languages are split into bytes; an 8K context can only hold very short text
Domain SamplingAre code, math, tables, JSON reasonably compressed?Reserve representative samples for code indentation, common symbols, LaTeX, structured markupCode spaces and punctuation are too fragmented; tool call JSON is more likely to exceed length limits
Unicode NormalizationAre visually identical characters mapped to the same sequence?Clearly define NFC/NFKC, case, full-width/half-width, whitespace character strategies, and maintain consistency between training and inferenceThe same text produces different tokens due to different normalization, affecting deduplication, evaluation, and caching
Byte FallbackCan unseen characters be represented losslessly?Retain byte-level fallback or an equivalent mechanism to ensure any Unicode character can be encodedRare characters, emojis, mixed-script inputs cause OOV or irreversible decoding
Special Token BudgetDo chat, tool, multimodal placeholders need atomic boundaries?Reserve tokens for system, role, tool, image/audio placeholders, and future extensionsSubsequently adding tokens changes the embedding shape, requiring model re-adaptation

2. Use fertility to measure the token burden of different languages

Multilingual tokenizers commonly use fertility to describe how many tokens a natural unit is split into. For English, you can look at tokens/word; for languages without spaces or with weak spaces like Chinese, Japanese, and Thai, tokens/character or tokens/byte is more stable. The course recommends reporting three types of metrics simultaneously:

  • tokens/character: Measures the context cost for the same character length, suitable for Chinese-English comparison.
  • tokens/byte: Measures whether UTF-8 multi-byte scripts are excessively falling back, suitable for cross-script comparison.
  • P95 token length: Measures whether long-tail samples will blow up in packing, RAG context, or tool parameters.

Assume the same tokenizer produces an average of 300 tokens for 1,000 English characters and 1,500 tokens for 1,000 Chinese characters. In an 8K token context, English can accommodate approximately 26,666 characters, while Chinese can accommodate approximately 5,333 characters. This difference directly affects long-text Q&A, RAG chunk size, cache hit rate, and user-perceived cost.

Worked Example: Why tokenizer training should not follow raw volume alone

Suppose the tokenizer training budget is 10B characters, and the raw candidate data consists of 8B English, 1B Chinese, 0.8B code, and 0.2B other languages. If trained directly according to the raw proportions, English pairs will dominate the first tens of thousands of merges; if all languages are uniformly sampled, it will cause high-traffic languages to lose too much compression rate. A safer approach is to first define the service goals, then set the sampling ratio:

  1. Set minimum coverage for major service languages, e.g., English, Chinese, and code each account for at least a certain proportion of the tokenizer training samples.
  2. Use temperature sampling for low-resource languages to avoid being completely overwhelmed by web page volume, while also avoiding excessive duplication of small corpora.
  3. Retain byte fallback so that characters not explicitly covered can still be encoded losslessly.
  4. Train multiple candidate tokenizers and compare tokens/character, round-trip, P95 length, and embedding parameter count using the same set of multilingual/code/tool-call samples.
  5. When making the final choice, consider both training cost and inference cost: 10% fewer tokens might mean 10% less attention/KV cost, but a larger vocabulary increases embedding and LM head parameters.

3. Tokenizer decisions propagate to training and inference

The impact of a multilingual tokenizer is not confined to Chapter 1. Data packing in Chapter 7 cuts batches by token count, not character count; SFT/DPO in Chapter 9 calculates loss and log probabilities on token sequences; KV Cache, TTFT, TPOT, and billing in Chapter 10 all occur per token. A higher fertility for a language means the same text consumes more training FLOPs, more KV Cache, and higher inference latency.

🔑
Engineering Takeaway: The tokenizer cannot be treated as a "preprocessing detail." For training and inference engineers, the tokenizer is the common interface between data distribution, model parameters, context capacity, service cost, and multilingual experience.
