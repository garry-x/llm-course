第 4 章 多头注意力与 MLA

从 MHA 到 GQA，再到 DeepSeek MLA：注意力为什么要压缩 KV Cache

预计阅读：100 分钟 · 12 小节 · 5 个编程练习 · 5 个概念题

怎么学这一章：先学 MHA，再把 GQA/MLA 看成压缩方案

💡

  • 本章产出：MultiHeadAttention、GQA 和简化 MLA。
  • 完成标准：你能画出“Q head 很多，但 K/V head 更少”时 cache 如何变小。
  • 练习产出：实现 MHA/GQA/MLA，写出参数量和 KV Cache 大小对比表，并说明不同 head 配置的 ablation 该怎么做。核心实现可用 Ch04 Multi-Head Attention Assignment Tests 检查。

4.1 从单头到多头：为什么一个 head 不够

Ch03 的单头 self-attention 只有一个 d_k 维 Q/K/V 空间，所有匹配规则都挤在同一个 similarity function 里。但语言关系不是单一维度。一个 token 往往要同时处理语法、语义和位置线索。

比如句子 The cat sat on the mat because it was comfortable.。处理 it 时，模型至少要问三件事：

  • 语法：it 是单数代词，先行词通常也要在语法上匹配。
  • 语义：到底是 cat 舒服，还是 mat 舒服，要看事件关系。
  • 位置：mat 更近，cat 更远，位置差异不能丢。

单头 attention 只能用一个空间同时编码这些规则，容易互相挤占。Multi-Head Attention (MHA) 的做法是把 d_model 维表示拆成 n_heads 个小空间，每个 head 单独做 attention，最后把所有 head 的结果拼回去，再用 W_o 混合。

head_i = Attention(X W_Qi, X W_Ki, X W_Vi) MultiHead(X) = Concat(head_1, ..., head_h) W_o

参数量为什么没变

如果 d_k = d_model / n_heads，每个 head 的 Q/K/V projection 更小，所有 head 加起来仍然是 3 * d_model * d_model。MHA 的优势不是“堆了更多参数”，而是把同一份参数预算拆到多个可学习子空间里。

例子：2 个 head 不等于 2 倍参数

设 d_model=8、n_heads=2，每个 head 的 d_k=4。单头 Q/K/V 投到 8 维，参数量是 3 * 8 * 8 = 192。两个 head 各自投到 4 维，总参数量是 2 * 3 * 8 * 4 = 192。

方案投影空间QKV 参数量attention score
单头1 个 8 维空间192[T,T]
2-head MHA2 个 4 维空间192[2,T,T]
🔑
记法：MHA 是固定参数预算下的表示分解。不同 head 可以学不同匹配模式，W_o 负责把这些子空间的信息重新混合。

4.2 编程练习 1：实现 MultiHeadAttention

实际实现里通常不会给每个 head 单独写三组 Linear，而是用一个 fused QKV projection：nn.Linear(d_model, 3 * d_model)，再 reshape 成 head 结构。

练习 1：MultiHeadAttention Module

  • 输入 x: [batch, seq_len, d_model]。
  • 用一个 W_qkv 得到 fused Q/K/V。
  • reshape 成 [B,T,3,H,Dh]，再拆成 Q、K、V。
  • 把 head 维移到 batch 后面：[B,H,T,Dh]。
  • 实现 scaled dot-product attention，支持可选 mask。
  • 拼回 [B,T,D]，经过 W_o 输出。
class MultiHeadAttention(nn.Module): def __init__(self, d_model, n_heads, dropout=0.1): super().__init__() assert d_model % n_heads == 0 self.d_model = d_model self.n_heads = n_heads self.d_k = d_model // n_heads self.W_qkv = nn.Linear(d_model, 3 * d_model, bias=False) self.W_o = nn.Linear(d_model, d_model, bias=False) self.dropout = nn.Dropout(dropout) def forward(self, x, mask=None): B, T, _ = x.shape qkv = self.W_qkv(x) # [B,T,3D] qkv = qkv.reshape(B, T, 3, self.n_heads, self.d_k) q, k, v = qkv.unbind(dim=2) # each [B,T,H,Dh] q, k, v = q.transpose(1, 2), k.transpose(1, 2), v.transpose(1, 2) scores = torch.matmul(q, k.transpose(-2, -1)) / (self.d_k ** 0.5) if mask is not None: scores = scores.masked_fill(mask == 0, float("-inf")) weights = torch.softmax(scores, dim=-1) weights = self.dropout(weights) out = torch.matmul(weights, v) # [B,H,T,Dh] out = out.transpose(1, 2).contiguous().reshape(B, T, self.d_model) return self.W_o(out), weights

这里最容易错的是 transpose 后忘记 contiguous()。如果 tensor 的内存布局不连续，直接 view 可能报错或得到错误解释；教学代码里用 reshape 更稳，但你仍然要知道 layout 发生了变化。

4.3 多个 head 为什么会学到不同子空间

MHA 没有手工规定“第 1 个 head 学语法，第 2 个 head 学位置”。分化来自训练过程本身。

  • 初始化不同：每个 head 的 Q/K/V 权重随机初始化不同，一开始看到的相似度空间就不同。
  • 梯度不同：不同 head 的 attention map 不同，取回的 V 不同，反向传播时收到的梯度也不同。
  • W_o 会选择性使用 head：下游 loss 会奖励有用的 head 输出，没用或重复的 head 权重会被压低。

从数学上看，每个 head 都有自己的 W_Qi 和 W_Ki。attention score 近似是 x_a^T W_Qi W_Ki^T x_b。不同 head 的 W_Qi W_Ki^T 不同，就相当于不同的双线性匹配函数。

如果所有 head 学成一样会怎样

如果多个 head 完全重复，MHA 会退化成“同一张 attention map 的复制”。这不会让模型崩掉，但浪费 head 预算。真实大模型里确实会出现 head 冗余，这也是后面 GQA、head pruning 和 MLA 能成立的一部分原因。

4.3A 多头不是简单 ensemble

多头 attention 看起来像“多个小模型投票”，但它不是普通 ensemble。普通 ensemble 通常是多个独立模型各自输出预测，再做平均；MHA 的 head 共享同一层输入，输出会被 concat，然后由 W_o 学习混合。

这种结构的表达力来自两点：

  • 每个 head 有自己的 Q/K/V 投影，能形成不同 attention pattern。
  • W_o 不是平均，而是一个可训练线性混合，可以把不同 head 的 value 信息重新组合。

head 之间既可以分工，也可能冗余。GQA 和 MLA 的压缩思路正是利用这个观察：Q 的多样性很重要，但 K/V 的独立份数不一定必须和 Q head 一样多。

4.4 编程练习 2：验证参数量等价

这一节的目标不是背公式，而是用代码证明：在 n_heads * d_k = d_model 时，多头拆分不会增加 QKV 参数量。

练习 2：Parameter Count Checker

  • 写一个函数，输入 d_model 和 n_heads。
  • 计算单头 QKV 参数量：3 * d_model * d_model。
  • 计算 MHA QKV 参数量：n_heads * 3 * d_model * (d_model // n_heads)。
  • 再加上 output projection：d_model * d_model。
def mha_param_count(d_model, n_heads): assert d_model % n_heads == 0 d_k = d_model // n_heads single_qkv = 3 * d_model * d_model mha_qkv = n_heads * 3 * d_model * d_k output_proj = d_model * d_model return { "single_qkv": single_qkv, "mha_qkv": mha_qkv, "output_proj": output_proj, "total_attention": mha_qkv + output_proj, }

注意：很多实现把 Q/K/V 融合成一个 W_qkv: [d_model, 3*d_model]。这只是计算更高效，参数量和三组 Linear 一样。

4.5 MHA、MQA、GQA：KV Cache 的演化

训练时，我们通常一次处理整段序列。推理 decode 时，每步只来一个新 query，但它要看所有历史 token 的 K/V。为了避免每步重复算历史 token 的 K/V，系统会把它们存进 KV Cache。

问题是：KV Cache 会随 batch_size * n_layers * seq_len * n_kv_heads * d_k 增长。长上下文和大 batch 下，decode 阶段常常不是算力先满，而是显存和带宽先满。

机制Q headsK/V headscache 特点常见取舍
MHAHH每个 Q head 有自己的 K/V质量强，cache 最大
MQAH1所有 Q head 共享一组 K/Vcache 最小，K/V 多样性损失大
GQAHG一组 Q heads 共享一组 K/V在质量和 cache 之间折中

MQA 是 GQA 的特例：n_kv_heads=1。MHA 可以看成 n_kv_heads=n_heads。GQA 把中间档补上了，例如 64 个 Q heads 只保留 8 个 K/V heads。

4.5A 例子：GQA 映射和 cache 计算

设 d_model=4096、n_heads=32，则 d_k=128。MHA 每个 token 每层要缓存：

K: 32 * 128 V: 32 * 128 total = 2 * 32 * 128 = 8192 floats

如果用 GQA，n_kv_heads=8，每 4 个 Q heads 共享一个 K/V head：

K: 8 * 128 V: 8 * 128 total = 2 * 8 * 128 = 2048 floats

cache 直接缩小 4 倍。decode 时仍然有 32 个 Q heads，但 K/V cache 只存 8 组。实现上常见做法是把 K/V 在计算 attention 前 repeat 到 Q head 数，或让 kernel 直接理解 Q/KV head ratio。

def repeat_kv(x, n_rep): # x: [B, n_kv_heads, T, Dh] B, G, T, Dh = x.shape x = x[:, :, None, :, :].expand(B, G, n_rep, T, Dh) return x.reshape(B, G * n_rep, T, Dh)

4.5B 训练 attention 和推理 KV Cache 不是同一个 layout 问题

训练阶段关心的是一次 forward 里怎么高效构造 [B,H,T,T] 的 attention。推理 decode 阶段关心的是历史 K/V 怎么存、怎么追加、怎么被新 query 读取。

  • 训练：Q/K/V 都来自当前 batch 的完整序列，shape 通常是 [B,H,T,Dh]。
  • prefill：输入一段 prompt，计算并写入整段 K/V cache。
  • decode：每步只有 T_q=1 的新 query，读取历史 T_cache 的 K/V。

所以 GQA/MLA 的价值主要体现在推理 cache 和 decode bandwidth，而不是把训练 attention 的 O(T^2) 直接改成线性。

4.5C 把 MHA checkpoint 转成 GQA

如果已经有一个 MHA checkpoint，可以通过合并 K/V heads 得到 GQA 初始化。常见做法是把每组里的 K/V projection 权重做平均，Q 和 W_o 保持不变，然后继续 finetune。

MHA K heads: [H, D, Dh] group size: H / G GQA K heads: mean over each group -> [G, D, Dh]

这不是无损转换。平均会丢掉部分 K/V 多样性，所以通常需要额外训练把质量拉回来。GQA 论文里的核心经验也是：直接从头训练或转换后继续训练，质量会明显好于粗暴共享。

4.6 编程练习 3：实现 GQA

GQA 和 MHA 的主要区别是：Q 有 n_heads 个 head，K/V 只有 n_kv_heads 个 head。要求 n_heads % n_kv_heads == 0。

练习 3：GroupedQueryAttention

  • W_q 输出 n_heads * d_k。
  • W_k 和 W_v 输出 n_kv_heads * d_k。
  • attention 计算前，把 K/V repeat 到 Q head 数，或让 kernel 支持 GQA。
  • 返回 out 和 attn_weights。
class GroupedQueryAttention(nn.Module): def __init__(self, d_model, n_heads, n_kv_heads, dropout=0.1): super().__init__() assert d_model % n_heads == 0 assert n_heads % n_kv_heads == 0 self.d_model = d_model self.n_heads = n_heads self.n_kv_heads = n_kv_heads self.d_k = d_model // n_heads self.n_rep = n_heads // n_kv_heads self.W_q = nn.Linear(d_model, n_heads * self.d_k, bias=False) self.W_k = nn.Linear(d_model, n_kv_heads * self.d_k, bias=False) self.W_v = nn.Linear(d_model, n_kv_heads * self.d_k, bias=False) self.W_o = nn.Linear(d_model, d_model, bias=False) self.dropout = nn.Dropout(dropout) def forward(self, x, mask=None): B, T, _ = x.shape q = self.W_q(x).reshape(B, T, self.n_heads, self.d_k).transpose(1, 2) k = self.W_k(x).reshape(B, T, self.n_kv_heads, self.d_k).transpose(1, 2) v = self.W_v(x).reshape(B, T, self.n_kv_heads, self.d_k).transpose(1, 2) k = repeat_kv(k, self.n_rep) v = repeat_kv(v, self.n_rep) scores = torch.matmul(q, k.transpose(-2, -1)) / (self.d_k ** 0.5) if mask is not None: scores = scores.masked_fill(mask == 0, float("-inf")) weights = torch.softmax(scores, dim=-1) out = torch.matmul(self.dropout(weights), v) out = out.transpose(1, 2).contiguous().reshape(B, T, self.d_model) return self.W_o(out), weights

4.7 DeepSeek MLA：把 KV 压到 latent space

GQA 减少 K/V head 数，但每个 token 仍然要缓存显式 K 和 V。Multi-head Latent Attention (MLA) 更进一步：先把输入压到一个低维 latent 向量 c^KV，cache 里只存这个 latent；需要 K/V 时再通过上投影矩阵还原。

c_t^KV = W_DKV x_t k_t = W_UK c_t^KV v_t = W_UV c_t^KV

如果 d_model=7168、n_heads=64、d_k=112，MHA 每 token 要缓存 2 * 64 * 112 = 14336 个 float。若 MLA 的 d_latent=512，每 token 只缓存 512 个 float，元素数约缩小 28 倍。

低秩分解的核心

MLA 可以理解成对 K/V projection 做低秩分解：先降维，再升维。降维后的 latent 是实际缓存内容，升维矩阵是模型参数，不随 sequence length 增长。

Matrix absorption：为什么推理时不一定要显式还原 K

如果 attention score 是 q^T k，而 k = W_UK c，那么：

q^T (W_UK c) = (W_UK^T q)^T c

也就是说，K 的上投影可以吸收到 Q 侧。推理时可以让 query 先乘上吸收后的矩阵，再直接和 latent cache 做点积，避免每步把所有历史 latent 显式展开成完整 K。

为什么需要 decoupled RoPE

RoPE 是依赖位置的旋转变换，通常不能随便和固定的上投影矩阵交换顺序。内容部分可以走 latent compression 和 matrix absorption；位置旋转部分要单独保留。DeepSeek 的做法是把 K 拆成 content part 和 RoPE part：content part 可压缩，RoPE part 单独投影、单独参与点积。

🔑
别把 MLA 理解成免费午餐：它显著减少 KV Cache，但会增加结构复杂度，对 kernel 支持、RoPE 处理和训练稳定性都有要求。它解决的是长上下文推理里的存储和带宽问题。

4.8 编程练习 4：实现简化 MLA

教学版先忽略 decoupled RoPE，只实现“KV 压缩成一个共享 latent，再还原成 K/V”的主流程。

练习 4：Simplified Multi-head Latent Attention

  • W_q 正常生成 Q。
  • W_dkv 把输入压成 d_latent。
  • W_uk 和 W_uv 从同一个 latent 还原 K 和 V。
  • forward 返回 (output, attn_weights, latent)，其中 latent 模拟 KV Cache 内容。
class MLA(nn.Module): def __init__(self, d_model, n_heads, d_latent, dropout=0.1): super().__init__() assert d_model % n_heads == 0 self.d_model = d_model self.n_heads = n_heads self.d_k = d_model // n_heads self.d_latent = d_latent self.W_q = nn.Linear(d_model, n_heads * self.d_k, bias=False) self.W_dkv = nn.Linear(d_model, d_latent, bias=False) self.W_uk = nn.Linear(d_latent, n_heads * self.d_k, bias=False) self.W_uv = nn.Linear(d_latent, n_heads * self.d_k, bias=False) self.W_o = nn.Linear(d_model, d_model, bias=False) self.dropout = nn.Dropout(dropout) def forward(self, x, mask=None): B, T, _ = x.shape q = self.W_q(x).reshape(B, T, self.n_heads, self.d_k).transpose(1, 2) latent = self.W_dkv(x) # [B,T,d_latent]，这就是要缓存的内容 k = self.W_uk(latent).reshape(B, T, self.n_heads, self.d_k).transpose(1, 2) v = self.W_uv(latent).reshape(B, T, self.n_heads, self.d_k).transpose(1, 2) scores = torch.matmul(q, k.transpose(-2, -1)) / (self.d_k ** 0.5) if mask is not None: scores = scores.masked_fill(mask == 0, float("-inf")) weights = torch.softmax(scores, dim=-1) out = torch.matmul(self.dropout(weights), v) out = out.transpose(1, 2).contiguous().reshape(B, T, self.d_model) return self.W_o(out), weights, latent

真实 MLA 会把 K 上投影吸收到 Q 侧，并单独处理 RoPE 通道。这里显式还原 K/V，是为了让压缩和还原关系更容易看清。

4.9 编程练习 5：比较 MHA、GQA、MLA 的 KV Cache

这一节把结构差异落到数字上。只比较“每 token 每层的 cache 元素数”时：

MHA: 2 * n_heads * d_k GQA: 2 * n_kv_heads * d_k MLA: d_latent

练习 5：KV Cache Size Analyzer

  • 输入 d_model, n_heads, n_kv_heads, d_latent, seq_len。
  • 输出 MHA/GQA/MLA 的 per-token floats 和总字节数。
  • 扩展版再乘上 batch_size * n_layers * dtype_bytes，避免把 per-layer 数字误当整机显存。
def compute_kv_cache_size(d_model, n_heads, n_kv_heads, d_latent, seq_len, dtype_bytes=4): d_k = d_model // n_heads mha_per_token = 2 * n_heads * d_k gqa_per_token = 2 * n_kv_heads * d_k mla_per_token = d_latent def gb(elements): return elements * seq_len * dtype_bytes / (1024 ** 3) return { "MHA": (mha_per_token, gb(mha_per_token)), "GQA": (gqa_per_token, gb(gqa_per_token)), "MLA": (mla_per_token, gb(mla_per_token)), "GQA_vs_MHA": mha_per_token / gqa_per_token, "MLA_vs_MHA": mha_per_token / mla_per_token, } stats = compute_kv_cache_size( d_model=7168, n_heads=64, n_kv_heads=8, d_latent=512, seq_len=131072 )

按上面的配置，MHA 每 token 是 14336 floats，GQA 是 1792 floats，MLA 是 512 floats。128K context、float32、单层估算大约是：MHA 7.0 GB，GQA 0.9 GB，MLA 0.25 GB。

4.10 本章小结

  • MHA 把一个表示空间拆成多个 head，参数量基本不变，但多了多个可学习的匹配子空间。
  • 多个 head 会因为初始化、attention pattern、梯度和 W_o 混合而自然分化；但真实模型里也会出现 head 冗余。
  • MHA、MQA、GQA 的关键差异是 K/V head 数。GQA 在质量和 KV Cache 之间做折中。
  • 训练 attention 的 layout 和推理 KV Cache 的 layout 是两个问题。GQA/MLA 主要优化 decode 阶段的 cache 和带宽。
  • MLA 把 K/V 压成共享 latent cache，通过低秩分解和 matrix absorption 减少显式 K/V 存储。
  • RoPE 是位置相关变换，不能简单吸收到同一套低秩矩阵里，所以真实 MLA 需要 decoupled RoPE。
