1.3 Token 是什么？为什么需要分词

LLM 不直接处理原始文本，而是处理 token 序列。分词（Tokenization）是将原始文本转化为 token 序列的过程——它是 LLM 流水线的第一步和最后一步。

一个 token 可能是：

  • 一个完整的常见词（如 "the"、"是"、"hello"）
  • 一个词的一部分（如 "playing" → "play" + "ing"）
  • 一个标点符号（如 "."、"。"）
  • 一个特殊控制符号（如 <|endoftext|>）
🔑
为什么分词策略至关重要：词表大小直接影响嵌入层参数量、softmax 计算量、多语言支持能力、推理效率（token 越多越慢越贵）。GPT-4 的 cl100k_base 分词器有约 10 万 token，Llama 3 有 12.8 万 token。

对于中文用户来说，分词效率尤其重要——一个中文字通常需要 1.5-2 个 token，而一个英文词只需要约 0.75 个 token。这意味着同样的内容，中文的 token 消耗大约是英文的 2-3 倍。

1.4 BPE 算法原理

Byte Pair Encoding (BPE) 是 GPT 系列最核心的分词算法。它源自数据压缩领域（Gage, 1994），后被 Sennrich 等人（2016）引入 NLP。

BPE 的核心思想非常简单——反复合并最频繁出现的相邻符号对：

  1. 初始化词表为所有基础字符（包括空格、标点等，通常 256 个）
  2. 统计训练语料中所有相邻符号对的出现频率
  3. 合并最高频的符号对，作为新符号加入词表
  4. 重复 2-3，直到词表达到目标大小（如 50,257）

让我们通过一个具体例子理解：

语料: "low lower lowest" 初始词表: l, o, w, _, e, r, s, t (每个字符一个 token) 第1轮合并: "l" + "o" → "lo" (出现 3 次) ← 最高频 第2轮合并: "lo" + "w" → "low" (出现 3 次) ← 最高频 第3轮合并: "e" + "r" → "er" (出现 3 次) ← 最高频 词表演进: 8字符 → +lo → +low → +er = 11个符号

编码时，采用贪心最长匹配：从当前文本位置出发，尝试匹配词表中最长的已知 token，匹配成功则输出该 token 并前进相应长度。

1.5 动手：BPE 训练实现

下面实现 BPE 训练的核心逻辑。我们将维护一个词表和一个合并规则表：

from collections import Counter class BPETokenizer: def __init__(self): self.vocab = {} # token_id → bytes self.merges = {} # (int,int) → int (新 token id) self.special_tokens = {} def _get_stats(self, ids): """统计所有相邻 token 对的频率""" counts = Counter() for pair in zip(ids, ids[1:]): counts[pair] += 1 return counts def _merge(self, ids, pair, new_id): """将序列中所有指定对替换为新 id""" new_ids = [] i = 0 while i < len(ids): if i < len(ids) - 1 and ids[i] == pair[0] and ids[i+1] == pair[1]: new_ids.append(new_id) i += 2 else: new_ids.append(ids[i]) i += 1 return new_ids def train(self, text, vocab_size): # 初始化：每个字节就是一个 token (0-255) ids = list(text.encode('utf-8')) self.vocab = {i: bytes([i]) for i in range(256)} num_merges = vocab_size - 256 for i in range(num_merges): stats = self._get_stats(ids) if not stats: break pair = max(stats, key=stats.get) new_id = 256 + i ids = self._merge(ids, pair, new_id) self.merges[pair] = new_id self.vocab[new_id] = self.vocab[pair[0]] + self.vocab[pair[1]] print(f"训练完成: 词表大小 {len(self.vocab)}")

核心逻辑只有约 30 行代码。_get_stats 统计相邻对频率，_merge 执行合并，train 循环迭代直到达到目标词表大小。整个过程就是不断找最高频对然后合并。

1.6 动手：BPE 编码与解码

有了词表和合并规则，编码和解码就很直接了：

def encode(self, text): """将文本编码为 token id 序列""" ids = list(text.encode('utf-8')) while len(ids) >= 2: stats = self._get_stats(ids) # 找到在合并规则中优先级最高的相邻对 pair = min(stats, key=lambda p: self.merges.get(p, float('inf'))) if pair not in self.merges: break # 没有可合并的对 ids = self._merge(ids, pair, self.merges[pair]) return ids def decode(self, ids): """将 token id 序列解码回文本""" tokens = b''.join(self.vocab[idx] for idx in ids) return tokens.decode('utf-8', errors='replace')

编码过程：从字节序列开始，反复应用学到的合并规则，直到没有可合并的相邻对。注意我们按合并规则被学习的顺序（优先级）来应用——先被合并的规则优先级更高。这与训练时的贪心策略一致。

解码过程：非常简单——将每个 token id 映射回对应的字节序列，拼接后用 UTF-8 解码即可。BPE 的一个巨大优势就是解码完全可逆——不需要任何额外的规则或词典。

1.7 验证：用我们的 Tokenizer 处理中英文

让我们测试实现的正确性：

# 用小语料训练一个微型 tokenizer train_text = "low " * 100 + "lower " * 80 + "lowest " * 60 tokenizer = BPETokenizer() tokenizer.train(train_text, vocab_size=300) # 测试编码 text = "lowest lower" ids = tokenizer.encode(text) print(f"编码: {ids}") print(f"Token 数: {len(ids)}") # 测试解码 decoded = tokenizer.decode(ids) print(f"解码: {decoded}") print(f"还原正确: {decoded == text}") # 测试中文 cn_text = "大语言模型是人工智能的重要分支" cn_ids = tokenizer.encode(cn_text) print(f"中文编码: {cn_ids}") print(f"中文 Token 数: {len(cn_ids)} — 注意，未训练的中文会退化为单字节 token")

预期输出：对于训练语料中的词汇（如 "lowest"），tokenizer 会将其编码为少数几个 token；对于未见过的中文文本，由于没有学到中文字符的合并模式，会退化为逐字节编码——这正是 BPE 在未训练语言上的预期行为。

1.8 主流分词器对比

算法方向选择策略代表模型
BPEbottom-up (合并)最高频相邻对GPT-2/3/4
WordPiecebottom-up (合并)最大似然增益BERT
Unigramtop-down (剪枝)删除低概率 tokenXLNet、ALBERT
SentencePiece多种 (空格归一化)BPE 或 UnigramLlama、T5

SentencePiece 的一个关键创新是将空格视为普通字符（用 ▁ U+2581 表示），使分词完全语言无关——不需要针对中文、日语、韩语等无空格语言做特殊处理。Llama 系列全部使用 SentencePiece BPE。

GPT-4 的 cl100k_base 词表约 10 万，基于 tiktoken 库实现，是 BPE 的优化变体——它在训练时使用了更大的语料和更精细的预处理规则。

⚠️
BPE 的常见陷阱：同一个词在不同上下文中可能被切分为不同的 token 序列（如 "low" 和 "lower" 中 "low" 的切分可能不同），导致模型需要学习同义词的多种表示。这也是为什么有研究在探索 tokenization-free 模型。

1.9 特殊 Token 与 Token 效率

除了从数据中学到的 token，分词器还需要一些特殊 token来表示控制信息：

特殊 Token含义典型用法
<BOS> / <s>序列开始 (Beginning of Sequence)标记每条文本的起始
<EOS> / </s>序列结束 (End of Sequence)标记生成停止；训练中分隔不同文档
<PAD>填充 (Padding)将不等长序列补齐到相同长度
<UNK>未知词 (Unknown)BPE 通常不需要（所有字节都在词表中）
<|endoftext|>文档分隔 (GPT 风格)GPT-2/3/4 用来分隔训练文档
<|user|>, <|assistant|>对话角色标记ChatML 格式（ChatGPT/Claude）

Token 效率直接影响 API 成本和推理速度。不同语言和领域的内容有不同的 token 效率：

语言/内容近似 Token/字 比说明
英文~0.3 token/字最经济；每个 token 约 3.3 个英文字符
中文~0.5-0.8 token/字Llama 3 改善明显，但仍不如英文
代码 (Python)~0.5 token/字缩进和符号是单独的 token
