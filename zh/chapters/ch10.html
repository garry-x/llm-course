10.1 推理瓶颈：用 Roofline 先判断受什么限制

模型训练好之后，真正难的是把它稳定、便宜、低延迟地跑在线上。推理通常分两段：

  • Prefill：一次处理完整 prompt，大矩阵乘法多，容易并行，通常更接近 compute-bound。
  • Decode：一次只生成一个 token，需要反复读权重和 KV，通常更接近 memory-bound。

Roofline 模型把问题讲得很直白：横轴是 arithmetic intensity，也就是每读一个 byte 能做多少 FLOPs；纵轴是实际性能。prefill 的计算密度高，decode 的计算密度低，所以 decode 往往被显存带宽卡住。增大 batch、压缩 KV、量化权重、GQA/MLA 都是在提高有效计算密度或减少搬数据。

10.2 KV Cache：别每步都重算历史 K/V

自回归生成时，第 t 步只多了一个新 token。没有 KV Cache 时，模型会反复为历史 token 计算 K 和 V。KV Cache 的做法是把历史 K/V 存起来，新步只算新 token 的 K/V，再和缓存拼起来做 attention。

KV Cache 更新

代价也很明确：省计算，花显存。单请求 KV Cache 大小大致是 2 * L * n_layers * n_kv_heads * d_head * bytes。注意这里是 n_kv_heads，不是 query heads。GQA、MLA、KV quantization 的意义，就是把这笔常驻账压小。

10.3 编程练习 1：实现带 KV Cache 的 Attention

练习 1：Self-Attention + KV Cache

任务：实现 AttentionWithCache.forward(x, cache=None)，支持增量 decode。

  • 第一次没有 cache，正常计算当前序列 K/V 并缓存。
  • 后续每步只输入一个 token，把新 K/V 拼到历史 K/V 后面。
  • 返回 (output, new_cache)。
  • 注意 causal mask：decode 时 L_Q=1，但 L_KV=history+1。
class AttentionWithCache(nn.Module): def forward(self, x, cache=None): B, T, _ = x.shape q = self.q_proj(x).view(B, T, self.n_heads, self.d_head).transpose(1, 2) k = self.k_proj(x).view(B, T, self.n_heads, self.d_head).transpose(1, 2) v = self.v_proj(x).view(B, T, self.n_heads, self.d_head).transpose(1, 2) if cache is not None: k = torch.cat([cache["k"], k], dim=2) v = torch.cat([cache["v"], v], dim=2) scores = q @ k.transpose(-2, -1) / math.sqrt(self.d_head) mask = self.causal_mask(q_len=q.size(2), kv_len=k.size(2), device=x.device) scores = scores.masked_fill(~mask, float("-inf")) out = torch.softmax(scores, dim=-1) @ v out = out.transpose(1, 2).contiguous().view(B, T, -1) return self.o_proj(out), {"k": k, "v": v}

def kv_cache_memory_analysis(n_layers, n_kv_heads, d_head, seq_len, batch_size=1, dtype_bytes=2): per_layer = batch_size * 2 * seq_len * n_kv_heads * d_head * dtype_bytes total = per_layer * n_layers return { "cache_per_layer_mb": per_layer / 1024**2, "total_cache_gb": total / 1024**3, }

10.5 稀疏与压缩注意力：长上下文不是免费午餐

MLA 主要压缩 KV Cache，但完整 attention 的计算仍然随上下文长度增长。长上下文系统常见两条路：

  • 稀疏 attention：每个 query 只看 top-k 或窗口内的候选 token，把 O(n^2) 近似降到 O(nk)。
  • 压缩 attention：先把若干相邻 token 压成代表项，再在代表项上选择或做 dense attention。

风险也要说清：压缩会丢细粒度证据，稀疏可能漏掉关键 token。代码定位、精确引用、表格单元格、多跳事实这类任务，最容易暴露问题。

10.5A 线性注意力、SSM 与混合序列架构

另一条前沿路线不是“少看一些 token”，而是“不再为每层保存完整逐 token KV 表”。Linear Attention、Mamba/SSM、retention 类模型会把历史信息折叠进更紧凑的状态。

架构推理时保存什么适合什么主要风险
Full attention所有历史 K/V精确回忆、代码、引用KV 和 attention 成本高
Sparse/compressed窗口、top-k 或压缩摘要多数 token 无关的长文档重要证据被跳过或压没
Linear attention递推状态 S_t流式、超长序列任意内容寻址弱一些
SSM/Mamba选择性 recurrent state线性序列伸缩kernel、训练配方和精确召回要单独验证
Hybridattention + linear/SSM + GQA/MLA + MoE现代长上下文模型组件多，回归面更大

读这类论文或报告时，别只看“替代 Transformer”这种标题。要问三件事：保存了什么状态，还保留哪些精确 lookup 路径，在哪些任务切片上证明质量没掉。

编程练习 3：对称 per-channel INT8 量化

def quantize_per_channel(x): abs_max = x.abs().max(dim=-1, keepdim=True).values scale = (abs_max / 127.0).clamp(min=1e-8) q = torch.round(x / scale).clamp(-127, 127).to(torch.int8) return q, scale.squeeze(-1) def dequantize(q, scale): return q.float() * scale.unsqueeze(-1)

这个练习只让你看清 scale、round、clamp 和 dequantization 如何产生误差。真实上线还要看校准、kernel、任务回归和回滚策略。

10.7 FlashAttention：不改数学，改数据搬运

标准 attention 会把 QK^T 和 softmax 后的概率矩阵写进 HBM，再读回来继续算。长序列时，这个中间矩阵非常贵。FlashAttention 的核心是 IO-aware：用 tiling 把 Q/K/V 分块搬进片上 SRAM，用 online softmax 在块内累计统计量，避免把完整 L * L 注意力矩阵落到 HBM。

  • Tiling：分块进 SRAM 计算。
  • Online softmax：按块更新 softmax 统计，不用先 materialize 全矩阵。
  • Recomputation：训练反传时少存中间量，必要时重算。

工程判断：FlashAttention 是 exact attention 的更好实现，不是近似注意力。实际加速取决于 GPU、dtype、head dim、序列长度和 kernel 支持。

10.8 编程练习 4：实现最小 RAG

RAG 解决的是模型知识过期、私有知识缺失和部分幻觉问题。它像开卷考试：先检索外部文档，再把证据塞进 prompt。但检索命中、上下文拼装、引用一致性和回答忠实度都要单独评估。

练习 4：Retrieve + Generate

  • add_document(text)：切 chunk，计算 embedding。
  • retrieve(query, top_k=3)：用 cosine similarity 找 top-k。
  • query(question)：检索、拼上下文、让 LLM 基于证据回答。
class SimpleRAG: def retrieve(self, query, top_k=3): q = self.embed_model.encode(query) scores = (self.embeddings @ q) / ( np.linalg.norm(self.embeddings, axis=1) * np.linalg.norm(q) ) ids = np.argsort(scores)[-top_k:][::-1] return [(self.documents[i], scores[i]) for i in ids] def query(self, question): chunks = self.retrieve(question) context = "\n\n".join([c for c, _ in chunks]) return self.llm.generate(f"基于以下资料回答，资料不足就说明不足：\n{context}\n\n问题：{question}")

10.8A RAG 评估：先判断错在哪一层

RAG 答错时，不要马上怪模型。可能是 chunk 切坏了，embedding 没召回，reranker 排错，context packing 把关键证据截掉，或者 generation 没忠实使用证据。

组件要问的问题常用指标常见修法
Chunking答案证据是否完整留在 chunk 里answer coverage、长度分布按标题/段落切、增大 overlap、保留 metadata
Embeddingquery 和相关文档是否靠近InfoNCE、Recall@k、MRR、nDCG领域对比学习、query rewrite、BM25 hybrid
Retrievaltop-k 是否含足够证据context recall、hit rate调 top-k、去重、过滤低分
Rerank最有用 chunk 是否排前面nDCG@k、top-1 relevancecross-encoder rerank
Context packing证据是否真的进入 prompttruncation rate、token usage压缩上下文、按相关性和新鲜度排序
Generation回答是否忠实于证据accuracy、faithfulness、citation consistency要求基于上下文回答，证据不足就说不足

Recall@k 看有没有找回证据，MRR 看第一个有用证据排多早，nDCG 看高价值证据是否排在前面。线上还要看检索耗时、rerank 耗时、prompt token、answer token、用户追问率和人工接管率。

10.9 Agent：从生成文本到执行动作

Agent 不只是一次性回答，而是通过 plan、act、observe、reflect 的循环做多步任务。模型提出工具调用，系统负责解析、校验、执行并把观察结果返给模型。

  1. Plan：把任务拆成步骤。
  2. Act：生成结构化工具调用。
  3. Observe：读取工具结果。
  4. Reflect：根据结果调整下一步。

安全底线：最小权限、高风险操作需要确认、工具输出要当作不可信数据处理。

10.9A Tool / MCP 工程协议

工具调用不是“模型想调就调”。模型输出只是候选动作，真正执行前必须过 schema、权限、预算、观察隔离和失败恢复。

阶段系统要做什么
Tool registry声明工具名、用途、参数 schema、timeout、权限级别和返回格式。
Model proposal模型基于上下文提出候选调用。
Schema validation检查工具是否存在、字段是否齐全、类型和 enum 是否正确。
Policy check按用户、租户、任务和风险级别判断是否允许执行。
Execution带 timeout、重试、幂等和审计日志执行。
Observation injection把结果作为不可信观察写回模型，而不是新指令。
Final answer要求引用证据、说明不足、记录 trace。

MCP 把外部工具、资源和 prompt 接进 AI 应用，也把数据泄露、远程工具信任、server-initiated sampling、越权文件范围这些风险带进来。工程上要有 allowlist、权限/同意、隐私脱敏、roots 限制、预算和 observation isolation。

report = validate_tool_call_plan( registry, [{"name": "search_docs", "arguments": {"query": "rag metrics", "top_k": 4}}], budgets={"max_calls": 2, "allowed_risks": ["read_only"]}, ) assert report["decision"] == "execute_tool_calls"

10.9B Context Engineering：上下文是有限资源

RAG、Agent、长对话、长文档问答都会把大量信息塞进 context。Context engineering 的核心不是写更长 prompt，而是决定哪些 token 进入当前窗口，哪些放外部存储，哪些压缩成摘要，哪些工具结果只保留引用句柄。

  • Token budget：分别统计 system、tool schema、history、retrieved context、observation、summary、output。
  • Evidence retention：做 head/middle/tail、citation id 和 summary fidelity 检查。
  • Cost / latency：比较长上下文、RAG packing、压缩、清理工具结果后的 P95 TTFT 和成本。
  • Memory safety：跨会话记忆必须有来源、权限、更新时间、删除路径。
  • Observation isolation：网页、文件、工具输出和 MCP resource 都是数据，不是系统指令。

10.9C Prompt Injection 与工具数据边界

RAG 文档、网页、邮件、上传文件、OCR、工具输出和 MCP resources 都会变成 token。模型本身分不清“可信指令”和“不可信内容”，所以系统必须在模型外建立边界。

实用规则：外部内容可以提供证据，但不能变成 policy；工具结果可以报告事实，但不能授权下一次高风险调用；工具描述可以说明接口，但不能偷要私有上下文。

风险面控制手段
检索内容带注入指令标记为 untrusted data，回答时只引用事实，不继承指令。
工具输出诱导越权下一步工具调用仍重新过权限和预算。
远程工具请求私密数据脱敏、拒绝越界参数、需要用户确认。
模型生成危险动作执行前 schema + policy + human approval。

10.10 编程练习 5：推理速度 Benchmark

Benchmark 要分清 TTFT、TPOT、吞吐、并发、prompt/output 长度、P50/P95/P99。只报平均 tokens/s 没用，因为用户感受到的是排队、首 token、流式间隔和失败率。

def benchmark(generate_fn, prompts): rows = [] for prompt in prompts: t0 = time.perf_counter() events = generate_fn(prompt, stream=True) first, n = None, 0 for token in events: now = time.perf_counter() if first is None: first = now n += 1 t1 = time.perf_counter() rows.append({ "ttft_ms": (first - t0) * 1000, "total_ms": (t1 - t0) * 1000, "tokens": n, "tokens_per_s": n / max(t1 - first, 1e-6), }) return rows

10.10A LLM 评估方法：一个数字不能代表模型

评估链路至少要写清：任务是什么，数据切片是什么，指标代表什么，baseline 是谁，差异是否显著，失败样例是什么，结论能否推广到线上流量。平均分只能做摘要，不能替代按场景拆开的判断。

10.11 vLLM：生产级推理引擎

vLLM 的核心价值是把 LLM 推理当作服务系统来做，而不是只跑一个模型 forward。

  • PagedAttention：像虚拟内存一样管理 KV blocks，减少连续大块显存碎片。
  • Continuous batching：请求动态进出 batch，避免 decode 阶段 GPU 空转。
  • Prefix cache：相同系统 prompt、工具 schema、长前缀可以复用 prefill 结果。
  • Admission control：不能只看并发数，还要看 active KV tokens、max batched tokens 和输出长度预算。

读 vLLM 性能数据时，要看 prompt 长度分布、输出长度分布、并发、GPU 型号、dtype、模型、max_num_seqs、max_num_batched_tokens、cache 命中率和 P95/P99。

10.12A MoE Serving：专家并行、负载均衡和 KV 账本

MoE 推理不是“参数多但每次只激活一小部分”这么简单。线上还要处理专家分布不均、跨卡通信、expert parallelism、EPLB、router 热点、token dispatch 和 KV Cache 账本。评估 MoE 服务要同时看激活专家数、专家负载、通信时间、decode TPOT、tail latency 和 OOM 边界。

10.13 生产推理服务蓝图：从模型到 OpenAI-compatible API

一个可上线的推理服务不只是 model.generate()。至少要有：

  • API：支持 chat/completions、stream、tool calls、structured output、error code、request id。
  • Observability：分层记录 metrics、traces 和必要内容样本，注意隐私和采样。
  • SLO：TTFT、TPOT、total latency、availability、error rate、timeout、cost。
  • Routing：不要所有请求都打最大模型；按难度、风险、成本和 SLO 做路由或 cascade。
  • Capacity：按 prompt/output 分布、active KV tokens、GPU 显存、decode 速率做准入控制。
  • Rollback：量化、LoRA、engine、prompt、tool schema 都要能回滚。

上线检查要写出：模型版本、tokenizer、chat template、engine 参数、量化格式、最大上下文、fallback、灰度策略、监控指标、告警阈值和失败样例。

10.13A 推理压测报告：从 workload 到瓶颈归因

  1. 描述 workload：prompt/output 长度、并发、到达率、RAG/tool 比例、长上下文比例。
  2. 记录同一组指标：TTFT、TPOT、tokens/s、GPU 利用率、显存、active KV、queue time、error。
  3. 用容量公式解释曲线：不要只贴图，要说为什么拐点出现。
  4. P/D 分离：prefill 和 decode 分到不同资源池时，要分别估算 worker pool 容量和 KV transfer 成本。
  5. 把症状映射到动作：排队高、OOM、TPOT 高、TTFT 高、cache miss 高，对应不同优化。

10.13B 模型发布包：从 checkpoint 到可回滚服务

训练 checkpoint 和推理导出 checkpoint 是两回事。发布包至少要包含：base model、adapter/merge 说明、tokenizer、chat template、generation config、量化配置、engine 配置、评估报告、兼容性检查、灰度计划、回滚路径和负责人。

Canary 不是把离线分数好看的模型直接放大流量。灰度期间要看质量、延迟、错误、拒答、工具调用、成本和用户行为。发现过载前要主动降级：限流、缩短上下文、切小模型、关闭高成本工具或降低 top-k。

10.13C Prefill/Decode Disaggregation

prefill 偏 compute-bound，decode 偏 memory-bound。高并发长上下文场景下，可以把 prefill 和 decode 拆成两个资源池。这样能提高资源匹配，但也引入新边界：KV Cache transfer、跨节点延迟、失败恢复、调度公平性和监控拆分。

报告必须分开写 prefill latency、KV transfer latency、decode TPOT 和总延迟，否则很容易把 transfer 问题误判成模型慢。

10.13D 集群级推理编排

当服务扩大到多模型、多租户、多 GPU 池，问题会从“某个 engine 多快”变成“整个 serving fabric 如何调度”。控制面要处理模型副本、热更新、KV 位置、prefix cache 命中、租户配额、路由、fallback、灰度和隔离。

10.13E KV-aware Routing 与 Cache Hierarchy

长上下文和 shared prompt 多的业务里，router 不能只看哪台 GPU 空。它还要知道 prefix/KV 在哪里、cache 是否命中、迁移成本是多少、请求是否能复用已有状态。Cache hierarchy 是成本模型，不只是存储功能。

如果业务 prompt 很短、重复前缀少、并发低，那么 KV-aware 基础设施可能不划算。先用数据证明 cache 命中和延迟收益，再上复杂控制面。

10.13F Model Router 与 Cascade Evaluation

模型路由至少有四类问题：按任务选模型、按难度升级模型、按安全风险选路径、按成本/SLO 做降级。评估时必须和“全部走强模型”的 baseline 比，而不是只看小模型自己多便宜。

结果说明动作

10.14 Triton：GPU Kernel 编程入门

Triton 让工程师用 Python 风格写 block-level GPU kernel，介于纯 PyTorch 和手写 CUDA 之间。LLM 里很多性能问题来自 memory layout、fusion、tiling 和 dtype。Triton 的价值是快速写出适配特定 shape 的 kernel，比如 matmul、RMSNorm、activation、attention 辅助算子。

入门时先理解：program id 对应一个 block，block 内用向量化 load/compute/store；性能来自把数据块按合适方式搬进片上内存，并减少不必要的中间写回。

10.15 GPU 内存层级：为什么需要 Triton

GPU 很快，但 HBM 不是无限快。GEMM、attention、norm、activation 的性能经常取决于数据是否重复从 HBM 搬来搬去。Kernel fusion 把多个小操作合到一个 kernel，减少中间张量写回；tiling 把热数据留在 SRAM/register 里重复使用。Roofline 视角能帮助你判断问题是算力不够，还是内存带宽/IO 卡住。

10.16 Engram：类海马体外部记忆案例

外部记忆系统把部分长期信息放在模型上下文之外，需要时再检索回来。它和 RAG 类似，但更强调跨会话、事件、用户状态和可更新记忆。区别在于：RAG 多用于文档证据，外部记忆还要处理写入、遗忘、权限、新鲜度和污染。

10.17 编程练习 6：实现 LSH Bucket Retrieval

LSH 用随机超平面把向量映射到桶里，相似向量更可能落到同一桶。它适合建立近似检索直觉：少扫一些候选，换取可能漏召回的风险。

class RandomHyperplaneLSH: def __init__(self, dim, n_planes): self.planes = np.random.randn(n_planes, dim) self.buckets = {} def hash(self, x): bits = (self.planes @ x > 0).astype(int) return "".join(map(str, bits)) def add(self, item_id, vector): key = self.hash(vector) self.buckets.setdefault(key, []).append((item_id, vector)) def query(self, vector): return self.buckets.get(self.hash(vector), [])

10.17A Speculative Decoding：什么时候值得开

Speculative decoding 用小 draft model 先提出多个 token，再由大模型一次验证。它能减少大模型 decode 步数，但不是必赚。要看 draft 接受率、小模型成本、验证 kernel、batch 干扰、输出质量和工程复杂度。接受率低时，系统可能更慢。

10.19 推理优化总结：你要能诊断什么

  • TTFT 高：看 prefill、queue、prefix cache、RAG/tool、network。
  • TPOT 高：看 decode、batch、权重带宽、KV 读写、speculative 接受率。
  • OOM：看 active KV tokens、batch、context、KV dtype、fragmentation。
  • P99 爆：看长尾请求、tool timeout、rerank、cache miss、调度饥饿。
  • 成本高：看路由、量化、prompt token、输出长度、RAG top-k、工具 schema。
  • 质量掉：按任务切片看量化、RAG 召回、上下文截断、模型路由和安全策略。

10.20 多模态 LLM：视觉和多模态输入

常见架构是 vision encoder 把图像变成视觉 token，再通过 projection 接到 LLM。关键设计包括：图像分辨率、视觉 token 数、投影层、视觉信息进入语言模型的位置、训练阶段和数据混合。

  • 两阶段训练：先做视觉-语言对齐，再做 instruction tuning。
  • 目标不同：alignment、instruction following、OCR、定位、推理、视频理解不是一回事。
  • 成本不同：高分辨率图像会带来大量视觉 token，直接影响 prefill 和显存。
  • 失败模式：幻觉物体、读错文字、空间关系错、图文证据不一致。

多模态实验要按 OCR、图表、空间关系、细粒度识别、视频时序、跨模态引用分别报告，不能只用一个总分。

10.20A 实时多模态与长上下文服务

实时语音/视频服务多了四类状态边界：音视频流状态、部分转写、低延迟回复、跨模态上下文。长上下文优化和实时优化不是一回事。长上下文关注 prefill、KV 和证据保留；实时关注端到端延迟、流式稳定、打断、同步和丢包恢复。

10.21 本章小结

  • Prefill 和 decode 的瓶颈不同，优化前要先分段诊断。
  • KV Cache 用显存换计算，是推理系统的核心账本。
  • 量化、GQA/MLA、PagedAttention、prefix cache 都是在改显存、带宽或复用率。
  • FlashAttention 是 IO-aware exact attention，不是近似算法。
  • RAG 的质量要拆成检索、排序、上下文拼装、引用和生成。
  • Agent/tool/MCP 必须有 schema、权限、预算、观察隔离和审计。
  • vLLM 这类引擎解决的是服务调度、KV 管理和吞吐，不只是模型 forward。
  • 生产上线要把质量、延迟、成本、安全、回滚放进同一份发布报告。
  • 多模态和实时服务把 token 成本、状态边界和评估切片进一步放大。
