11.1 Why an LLM Course Still Needs to Study Classic NLP

This course's classic neural NLP topic connects from word vectors, RNNs, dependency parsing, seq2seq attention, Transformers, and BERT all the way to large language models. Skipping the intermediate models leads to a common misconception: that all NLP problems should be rewritten as prompt-based generation. In reality, many production systems combine different model families: BERT or DeBERTa for reranking, bi-encoders for retrieval embeddings, encoder-only heads for extractive QA, and decoder-only LLMs for answer generation.

The focus of this chapter is not to memorize old model names, but to master three stable concepts:

  • Inductive Bias: RNNs recurse over time, parsers output trees, seq2seq separates source and target sequences, BERT reads context bidirectionally.
  • Objective Functions: Causal LM predicts the next token, MLM computes loss only at masked positions, seq2seq maximizes conditional generation probability, parsers maximize structural actions or head/label probabilities.
  • Evaluation Metrics: BLEU, ROUGE, EM/F1, UAS/LAS, and retrieval metrics each answer different questions; they cannot be replaced by a single aggregate score.

If the spectral norm of these factors is consistently less than 1, gradients vanish; if consistently greater than 1, gradients explode. LSTMs/GRUs mitigate this with gating and approximately linear memory paths. Transformer self-attention shortens the path between any two positions to a single attention operation, but at the cost of a quadratic attention matrix and memory/bandwidth pressure.

11.3 Dependency Parsing: Structured Prediction is Not an Attention Heatmap

Dependency parsing predicts a head and a dependency label for each token, outputting a directed tree. Its distinction from an attention heatmap is important: attention is a continuous weight, useful for diagnostics; a dependency tree is a discrete structure that must satisfy constraints like a single root, acyclicity, and one head per word.

The Arc-standard transition system constructs trees using stack, buffer, and arcs:

ActionPreconditionEffect
SHIFTbuffer is non-emptyPush the first token of the buffer onto the stack
LEFT_ARC(label)stack has at least two elementsMake the second-top element a dependent of the top element, then remove the second-top element
RIGHT_ARC(label)stack has at least two elementsMake the top element a dependent of the second-top element, then remove the top element
ROOT(label)Only one root candidate remainsAttach the root candidate to ROOT

Worked Example: Arc-standard parsing of I saw her

The target dependency tree is saw -> I (nsubj), saw -> her (obj), and finally ROOT -> saw (root). Using the arc-standard transition system, it can be constructed as follows:

stepstackbufferactionnew arc
0[][I, saw, her]SHIFT-
1[I][saw, her]SHIFT-
2[I, saw][her]LEFT_ARC(nsubj)saw -> I
3[saw][her]SHIFT-
4[saw, her][]RIGHT_ARC(obj)saw -> her
5[saw][]ROOT(root)ROOT -> saw

This example illustrates that parsing is not about independently classifying each token: the legality of an action depends on the current stack/buffer state, and an incorrect action will destroy the final tree structure. A transition trace should return the head, label, arcs, and stack/buffer state for each token so that tree construction can be audited step by step.

11.5 BERT / Encoder-only: MLM, Classification, and Extractive QA

A BERT-style encoder can bidirectionally see the left and right context within the input; it is not an autoregressive generator. One of its pre-training objectives is Masked Language Modeling: cross-entropy is only computed at masked positions, while unmasked tokens use an ignore index. An MLM diagnostic should return the masked-token loss, accuracy, and positions contributing to the loss, helping distinguish between "which tokens in the input are visible" and "which positions contribute to the training objective."

If the no-answer score for [CLS] is the highest, the model should return an empty answer. This differs from a decoder-only LLM generating "I don't know": BERT QA's answer must be bound to an input token span.

BIO sequence labeling

NER, slot filling, and token classification often label each token as B-TYPE, I-TYPE, or O. B-ORG I-ORG represents a continuous organization entity, and O represents outside an entity. A BIO decoder converts token-level tags into span-level output; for example, ["B-LOC", "I-LOC"] corresponds to a LOC span. If a sequence starts with I-LOC, many evaluation scripts will tolerate it as the start of a new entity; a strict system could also choose to report an error directly, but this must be clearly stated in the data protocol.

Entity recognition typically uses span-level F1, rather than just token accuracy. A predicted entity is considered a true positive only if its type, start, and end all exactly match the gold span; a boundary off by one token or a wrong type counts as a false positive, while the corresponding gold span remains a false negative. Span-level evaluation should convert this definition into computable TP/FP/FN, precision, recall, and F1.

Viterbi structured decoding

Per-token argmax ignores constraints between labels: for example, B-ORG followed by I-ORG is usually more reasonable than jumping back to O, and O directly followed by I-PER should often be penalized. Linear-chain CRFs or similar structured decoding methods formulate the path score as the sum of emission scores and transition scores:

Formula 11.7 Linear-chain path score

Viterbi dynamic programming saves the "best path score ending with a certain tag" and a backpointer at each step t, with a complexity of O(TK^2). Implementing this decoding process by hand makes clear how structured prediction transforms the output of a local classifier into a globally consistent label sequence.

Training a CRF also requires normalizing over all possible paths, not just the optimal one. The Forward algorithm replaces the max in Viterbi with logsumexp to compute the log-partition:

Formula 11.8 CRF log-partition

Log-space forward DP computes this normalization term, while CRF negative log likelihood computes the gold path score and returns log_partition - gold_score. It has the same state shape as Viterbi, but different semantics: Viterbi is used to predict the optimal path, while CRF NLL is used during training to compare the gold path against all competing paths.

11.6 Evaluation: BLEU, ROUGE, EM/F1, and Structural Metrics

Evaluation metrics must match the task structure. BLEU is commonly used for translation, ROUGE for summarization, Exact Match and token-level F1 for extractive QA, and UAS/LAS for dependency parsing. None of them are a complete measure of "language quality."

MetricCore CalculationApplicable TasksMain Limitation

ROUGE-LPrecision/Recall/F1 of the longest common subsequenceSummarization, Generation ComparisonBiased towards lexical overlap, does not guarantee factual consistency
EM/F1Exact match or token overlap after answer normalizationExtractive QA, Short AnswersRequires extra handling for multiple answers, aliases, and reasoning-based answers
UAS/LASWhether the head is correct, and whether the label is also correctDependency ParsingOnly evaluates syntactic structure, not equivalent to downstream task success

Worked Example: BLEU Clipping and ROUGE-L

BLEU does not simply count how many words in the candidate sentence appear in the reference sentence. Let the candidate be the the the the and the reference be the cat is here. Looking only at unigrams, the appears 4 times in the candidate, but at most 1 time in the reference, so the clipped match can only count as 1:

BLEU-1 clipped precision

This step prevents the model from inflating precision by repeating high-frequency words. If the candidate is cat sat and the reference is cat sat on mat, the unigram precision is 1, but the candidate length (2) is less than the reference length (4), so a brevity penalty is needed:

Brevity penalty

Therefore, a short sentence cannot get a perfect score just because every word matches. ROUGE-L focuses on the longest common subsequence. If the candidate is a b c d and the reference is a x b c, the longest common subsequence is a b c with length 3; precision = 3/4, recall = 3/4, F1 = 0.75. The sentence_bleu and rouge_l_f1 functions in the Ch11 homework implement these definitions as runnable functions.

Worked Example: Why QA EM/F1 Requires Answer Normalization First

Extractive QA often has multiple gold answers. During evaluation, normalization is performed first: lowercasing, removing punctuation, removing articles, and collapsing whitespace. If the prediction is The quick brown fox and the gold answers are quick brown fox and slow turtle, after normalization the prediction matches the first gold exactly, so Exact Match = 1 and token F1 is also 1.

If the prediction is quick fox and the gold is quick brown fox, then the token overlap is {quick, fox}, precision = 2/2 = 1, recall = 2/3, so:

QA token F1

This example explains why EM and F1 are reported together: EM strictly requires an exact string match, while F1 allows partial overlap. For open-ended, reasoning-based, or aliased answers, relying on a single gold string can underestimate correct outputs; therefore, QA datasets typically provide multiple references and take the highest EM/F1.

Therefore, an evaluation report cannot just say "BLEU is high" or "ROUGE is high." It must specify the task structure, candidate outputs, reference answer set, and metric limitations; otherwise, a single score can easily be misinterpreted as a complete judgment of language quality.

11.6A Modern LLM Evaluation Protocols: From a Single Score to Reproducible Conclusions

BLEU, ROUGE, EM/F1, and UAS/LAS are suitable for well-defined traditional tasks; modern LLMs also require evaluation of open-ended responses, long contexts, tool calls, RAG, refusal behavior, safety, and multi-turn dialogue. The most common mistake is to interpret a single benchmark score, a single LLM-as-judge win rate, or a set of human preferences as "the model is better." A course-level evaluation should first define the task, samples, inference settings, and judgment criteria before discussing conclusions.

Minimum Components of an Evaluation Protocol

LLM-as-judge Requires Calibration

LLM-as-judge can reduce the cost of evaluating open-ended outputs, but it suffers from position bias, length bias, style bias, and model homophily bias. A more robust setup is pairwise blind comparison: hide model names, randomly swap the A/B order, and let the judge decide which response better meets the rubric, recording ties.

Industrial evaluation also requires a reliability audit before using judge scores. OpenAI's eval guidelines suggest aligning the model grader with human labels before using it for optimization; works like MT-Bench / Chatbot Arena also caution that pairwise judges are affected by position, verbosity, and style biases. Therefore, this course does not permit reporting a single win rate. At least four questions must be answered: whether the same candidate is chosen after swapping the A/B order, whether the judge has an abnormal preference for the first position, whether it has an abnormal preference for longer responses, and whether the agreement rate with a small set of human gold labels is sufficient.

audit = judge_reliability_audit( [ {"comparison_id": "q1", "winner": "A", "response_a_id": "strong", "response_b_id": "weak", "response_a_tokens": 80, "response_b_tokens": 120, "gold_winner": "strong"}, {"comparison_id": "q1", "winner": "B", "response_a_id": "weak", "response_b_id": "strong", "response_a_tokens": 120, "response_b_tokens": 80, "gold_winner": "strong"}, ], thresholds={"min_swapped_consistency": 0.8, "max_position_bias": 0.1}, ) assert audit["overall_pass"]

This audit is not meant to dismiss LLM-as-judge, but to prevent using an unstable evaluator as the basis for deployment. If position bias or swapped consistency fails, first randomize/swap candidate order, modify the judge prompt, or change the judge model; if gold agreement fails, supplement with human annotation and error analysis, rather than continuing to use that judge for automatic hyperparameter tuning.

Pairwise win rate with ties

If the result of Model A vs. Model B is W=42, L=35, T=23, the win rate is (42+11.5)/100=53.5%. This number alone cannot prove that A is better in production; one must also check whether the sample size is sufficient, whether the question distribution represents the target scenario, what the failure categories are, and whether latency and cost have worsened.

Safety evaluation must distinguish between refusal and over-refusal

Safety is not about "the more refusals, the better." Samples should be categorized at least into harmful requests, benign sensitive requests, and ordinary tasks, with separate statistics for each:

  • Attack success rate: The proportion of harmful requests where the model provides unsafe assistance; lower is better.
  • Refusal rate: The proportion of requests the model refuses; needs to be viewed separately for harmful/benign categories.
  • Over-refusal rate: The proportion of benign requests that should have been helped but were incorrectly refused.
  • Task utility: The accuracy, helpfulness, or completion quality on ordinary tasks, to prevent safety optimization from sacrificing usability.

For example, Model X has a 95% refusal rate on the harmful set, but also a 60% refusal rate on the benign medical-summary set. This is not simply "safer"; it is severe over-refusal. The alignment evaluation in Ch09 and the deployment SLO in Ch10 should be interpreted together with this section: safety, quality, latency, and cost must all meet their targets.

Write modern evaluations as runnable diagnostics

A modern evaluation report should be reducible to runnable diagnostics. Pairwise judgment summaries calculate raw win rate, tie-adjusted win rate, and win rate grouped by task, showing that the same model might win on QA while tying or losing on math. Judge reliability audits check sample size, position bias, length bias, swap consistency, and human label agreement. Safety metrics split samples into harmful, benign sensitive, and ordinary groups, then report attack success, harmful refusal, over-refusal, and task utility. Benchmark summaries organize task, sample size, metrics, model, prompt, temperature, failure types, and non-extrapolable scope into a structured conclusion.

These diagnostics share one purpose: preventing a single average score from being mistaken for a complete engineering conclusion. Training engineers use them to judge whether alignment has harmed capability; inference engineers use them to judge whether quality, safety, latency, and cost meet deployment requirements together.

Worked Example: How to Write a Qualified Conclusion from the Same Result

Less rigorous writing: Model A is 8% better than Model B.

More rigorous writing: On n=300 Chinese customer service QA samples, with a fixed system prompt, temperature=0, max output=512, the pairwise win rate of Model A relative to Model B is 58%. The main improvements come from format adherence and citation completeness; the factual error rate dropped from 11% to 8%, but the P95 TTFT increased from 720ms to 980ms. Furthermore, this result does not cover English, code, multi-turn tool calls, or high-concurrency load. Therefore, it supports "under this Chinese customer service RAG workload, A is worth entering a canary experiment," but does not support "A is a generally stronger model."

🔑
Course Standard: An evaluation answer is not a list of scores, but an engineering conclusion constrained by conditions. You must specify the task, samples, inference settings, metrics, failure types, statistical uncertainty, and non-extrapolable scope.

11.6B Benchmark Validity: Contamination, Confidence Intervals, and Ablation

The difficulty of modern LLM evaluation is not just "which benchmark to choose," but judging whether the score can support an engineering decision. A model might win because its training data has seen the questions, its prompt is more suitable, its output is longer, it uses more sampling iterations, or the judge prefers its style. Course-level reports must disentangle these factors; otherwise, benchmarks become leaderboard screenshots rather than scientific comparisons.

1. Data contamination and split leakage are not the same problem

Split leakage usually occurs within one's own data pipeline: the same document, user session, question bank, or near-paraphrase enters train/dev/test simultaneously. It can be mitigated through document-level splitting, URL/question ID deduplication, n-gram overlap checks, embedding nearest-neighbor search, and manual spot checks. Benchmark contamination is broader: public test questions, answer explanations, discussion posts, or code repositories may have already entered pre-training or SFT data. It cannot always be completely eliminated, so the report must specify the data source, publication date, deduplication method, and handling of high-similarity samples.

Public benchmarks also have a lifecycle. A benchmark that was useful when first released can become saturated, repeatedly prompt-tuned, or contaminated after months of leaderboard optimization. OpenAI's 2026 discussion of SWE-bench Verified is a useful example: even a human-validated public benchmark can stop measuring the frontier once models, agents, and training data have adapted to it. In course reports, old public benchmarks should often be treated as regression tests, while private held-out sets and live/canary evals carry more evidence for deployment decisions.

Evaluation lifecycle: from public benchmarks to live evidence

Modern evaluation should be treated as an operational lifecycle rather than a final leaderboard snapshot. Public benchmarks are useful for rough comparability and regression checks, but frontier model and agent systems change quickly: tasks saturate, prompts become optimized, public solutions leak into data, and product workflows drift. A serious evaluation program therefore uses multiple evidence layers with different ownership, refresh cadence, and decision authority.

The engineering conclusion should cite which evidence layer supports each claim. Model A wins on a public benchmark supports only a broad comparison; Model A reduces citation errors on a private customer-service slice without increasing P95 latency or unsafe completions during a canary supports a deployment decision. Mixing these levels is a common source of overclaiming.

2. Small sample improvements must include uncertainty

If Model A answers 61 out of 100 questions correctly, and Model B answers 56, one cannot simply write "A improves by 5%." The difference of 5 samples could be due to question sampling fluctuation. The simplest approximation is to treat accuracy as a binomial proportion and estimate the confidence interval using the standard error:

Accuracy standard error

When n=100, p=0.61, the standard error is approximately 0.049, and the 95% interval is roughly 61% ± 9.6%. This shows that 61% and 56% can hardly support a strong conclusion with this sample size. If the same question set evaluates both models, prioritize paired comparisons: which questions A got right and B got wrong, and vice versa, because question difficulty is shared. For open-ended generation or RAG tasks, bootstrap resampling of samples is commonly used to estimate intervals for win rate, accuracy, or error rate.

3. Ablation experiments answer one question at a time

Ablation is not "running a few more configurations." It must make the effect of a single design variable interpretable. If you change the base model, prompt, retriever, reranker, temperature, and context length all at once, even if the score improves, you cannot know which change contributed. A rigorous LLM system ablation is usually organized in the following order:

QuestionKeep ConstantChange OnlyReport Simultaneously
Is RAG useful?Model, prompt, decoding parameters, sample setWhether to inject retrieval contextaccuracy, citation precision, input tokens, P95 latency
Is reranker useful?Retriever candidates, LLM, context budgetCandidate ranking strategyRecall@k, nDCG, final answer accuracy, rerank latency
Does DPO improve alignment?SFT base, eval set, prompt, temperatureDPO checkpointhelpfulness, safety refusal, length distribution, KL or behavioral drift
Is longer reasoning worth it?Model, questions, scoring rulesmax tokens, sampling count, or reasoning budgetpass@k, average output tokens, cost, TTFT/TPOT

Worked Example: Rewriting a 3% improvement into a judgeable conclusion

Insufficient writing: Adding a reranker improved the benchmark from 72% to 75%, the effect is better.

More rigorous writing: On the same set of n=500 Chinese regulatory QA samples, with a fixed generation model, prompt, temperature=0, max output=384, and context budget=1800 tokens, only the top-8 dense retrieval was changed to dense retrieval + cross-encoder rerank. End-to-end accuracy went from 72.0% to 75.0%, with a bootstrap 95% CI for the difference of approximately [0.4%, 5.8%]; retrieval Recall@8 remained unchanged, but citation precision improved from 68% to 79%; P95 latency increased by 140ms, average input tokens unchanged. This conclusion supports "the reranker improved context ranking for this regulatory QA workload, worth a canary test," but does not support "all RAG tasks should include a reranker."

🔑
Engineering Judgment: A credible benchmark conclusion must withstand at least four questions: Is the data clean? Is the comparison fair? Is the difference stable? Can the ablation explain the source of improvement? Missing one link, the score can only serve as a clue.

11.6C Safety & Social Impact: Risk Classification, Trigger Conditions, and Engineering Mitigations

An NLP/LLM engineering course cannot treat safety as an appendix before deployment. Model risk arises from the combination of data, training objectives, inference interfaces, application scenarios, and user groups; the same model has vastly different error costs in casual conversation, medical summaries, financial advice, educational tutoring, and code execution. Engineers must be able to break down "the model might be harmful" into risk categories that are discussable, measurable, and mitigable.

1. Risk is not a single dimension

2. Safety evaluation must be written for the scenario, not as a slogan

"Model safety" cannot be separated from the application scenario. The core risks for medical Q&A are misleading advice, privacy, and expression of uncertainty; for a code agent, they are execution permissions, supply chain, and data exfiltration; for educational tutoring, they are incorrect feedback, over-reliance on writing, and learning dependency. A safety report should first clearly describe the target scenario, users, model permissions, and error costs, then select samples and metrics.

Enterprise RAGDocuments within permissions, documents outside permissions, outdated documents, conflicting documents, missing citationsOnly looking at final answer accuracy, ignoring permissions and citations

Worked Example: Risk Decomposition for RAG Medical Q&A

Suppose a team wants to launch a patient-facing medical education RAG system. An insufficient approach is to only test whether "the answer looks professional." A more reasonable engineering decomposition is:

  1. Retrieval Risk: Whether relevant guidelines are recalled, whether outdated documents enter the context, and whether citations point to the correct paragraphs.
  2. Generation Risk: Whether the model extrapolates retrieved content into diagnostic conclusions, fabricates medication dosages, or gives definitive advice when information is lacking.
  3. Privacy Risk: Whether user input and logs contain names, phone numbers, medical record numbers, or other identifiable information.
  4. Refusal Boundary: General educational questions should be answered, while requests regarding emergencies, self-harm, or prescription adjustments should be redirected to professional help.
  5. Usability: Safety policies cannot refuse all medical questions; otherwise, the system becomes unusable for real user tasks.

Therefore, this system should at least report retrieval recall, citation accuracy, factual error rate, harmful advice rate, general medical education completion rate, emergency referral rate, PII log handling method, and limitation descriptions in the user interface. The goal is not to create more tables; it is to describe model behavior, data sources, user risks, and deployment boundaries in the same engineering language.

3. Safety, Alignment, Evaluation, and Serving Are the Same Pipeline

Chapter 9 discusses how to use SFT/DPO/GRPO to change model behavior, Chapter 10 discusses how to turn a model into a serviceable system, and this section discusses the risk boundaries of these behaviors in real-world scenarios. A model can improve in harmlessness while over-refusing on benign sensitive questions; it can also perform well on an offline safety set but fail in an Agent scenario due to excessive tool permissions. A credible result table should include safety slices alongside quality, latency, and cost, rather than just writing "safety has been considered" in the conclusion.

🔑
Engineering Takeaway: Social impact is not an abstract moral discussion, nor is it a single safety score. LLM engineers must be able to map risks to data, objective functions, interface permissions, user scenarios, and evaluation slices, and explain what capabilities, latency, or cost each mitigation action will sacrifice.

11.6D Agent and Workflow Evaluation: Final Answer, Trajectory, State, and Cost

Agent evaluation is stricter than single-turn answer evaluation because the model can search, edit files, call tools, write to databases, and interact with users over multiple turns. A correct final sentence is not enough: the trajectory may have used a forbidden tool, modified the wrong state, leaked private data, or spent an unacceptable amount of time and money. Recent benchmarks make this boundary clearer. SWE-bench Verified validates software-engineering tasks through hidden tests and human filtering. tau2-bench evaluates customer-service agents with domain policy, tool use, user simulation, and final state. BrowseComp focuses on hard-to-find but short and verifiable browsing answers, explicitly trading open-ended realism for grading reliability.

What Different Agent Benchmarks Actually Measure

tau / tau2 style benchmarksConversational tool use under domain policy and simulated user interactionFinal database or environment state, policy adherence, tool-call correctness, repeated-trial reliabilityUser simulator quality, domain coverage, policy completeness, and tool environment realism limit generalization

Agent Evaluation Must Preserve the Trace

For an agent task, store the full trace, not only the final answer. At minimum, log: model version, prompt/template, user turns, tool schemas, tool arguments, tool return values, retrieved documents, file patches, environment state before/after, latency, token usage, errors, and final evaluator decision. This is the only way to distinguish "the answer is wrong because retrieval missed evidence" from "the answer is right but the agent violated policy" or "the patch works but broke an unrelated test." OpenAI's evaluation guidance frames evals as task-specific, logged, calibrated, and continuous rather than one-time leaderboard checks; this is especially important for agents because tools and environments drift over time.

A senior engineer should therefore avoid writing "the agent scores 62%" without context. A usable conclusion is closer to: "On 500 human-validated software issues with hidden tests, using model X, scaffold Y, max steps Z, and a fixed environment image, the agent resolved 62% of tasks; unresolved tasks clustered around cross-file reasoning and environment setup; average cost was A, P95 wall time was B, and PASS_TO_PASS regressions were C. This supports trying the agent on low-risk issue triage with review, but not autonomous merge rights."

🔑
Engineering Judgment: Agent evaluation is a workflow audit. It must prove final task success, legal trajectory, correct state change, repeatability, bounded cost, and acceptable safety risk. A benchmark can reveal one slice of capability, but production readiness requires a traceable eval harness tied to the actual tools and policies.

11.7 Task Selection: When Not to Use a Decoder-only LLM

When choosing a model family, first consider the output space and error cost:

  • Fixed-label Classification: Encoder-only fine-tuning or small model classifiers are usually cheaper and more stable.
  • RAG Reranking: Cross-encoders can look at the query and document simultaneously to rank candidates; decoder-only LLM reranking is costly and format-unstable.
  • Extractive Answers: If the answer must come from the original document, span QA is more controllable than free generation.
  • Structured Output: Tasks like dependency parsing, NER, and slot filling can use structured heads or constrained decoding, and should not default to open-ended generation.
  • Open-ended Generation and Reasoning: When the output requires new text, code, explanations, or multi-step reasoning, decoder-only LLMs are more natural.
⚠️
Common Mistake: Describing BERT and GPT as "just different masks." Their attention visibility, training objectives, fine-tuning heads, output spaces, and serving costs are all different.

11.7A Engineering Selection Case Study: Decomposing Tasks into Training Objectives, Output Spaces, and Serving Costs

The question LLM engineers often face is not "which model is the largest," but "should this task be modeled as classification, extraction, ranking, structured prediction, or open-ended generation?" The same business goal can have multiple modeling approaches, and the training data, inference path, evaluation metrics, and deployment costs will all change accordingly.

TaskPreferred Model FamilyTraining SignalInference FormPrimary Risk
Document RelevanceBi-encoder / Cross-encoder RerankerQuery-document positive/negative samples, clicks, or human relevanceReranking after vector recallRecall misses the answer, reranker is too slow
Answer Must Come from SourceEncoder-only Extractive QAStart/end span labels or no-answer labelsSelect a valid span from the inputCannot generate summaries, span boundary errors
Entity, Slot, Field ExtractionBIO Tagger / CRF / Constrained DecodingToken-level BIO or span labelsOutput structured spansHigh token accuracy but wrong entity boundaries
Customer Service Answers, Explanations, CodeDecoder-only LLM / RAG + LLMSFT, preference data, retrieval context, and task feedbackAutoregressive generationHallucination, format drift, latency and cost
Translation, SummarizationEncoder-decoder or Decoder-only Instruction ModelSource-target paired dataConditional generation, can use beam or samplingLength bias, factual omission, metric-human preference mismatch

Case 1: Customer Service Knowledge Base Q&A

If the business requires "the answer must cite the original knowledge base text," directly letting a decoder-only LLM answer freely carries high risk. A more stable approach is usually a three-stage pipeline: bi-encoder for candidate document recall, cross-encoder reranker for fine ranking, and extractive QA or constrained generation for the answer span and citation. Only when the question requires cross-paragraph synthesis, rewriting, or explanation should a decoder-only LLM generate natural language answers based on the retrieval context.

The training objectives for such a system are also layered: the retriever uses contrastive learning or click data, the reranker uses relevance ranking labels, the QA head uses span labels, and the generator uses SFT/preference optimization. Attributing all errors to "the LLM is not strong enough" will obscure the real bottlenecks: it could be a recall miss, a reranking error, chunking that destroys the answer, a prompt that doesn't constrain citation, or the generation phase not faithfully using the context.

Case 2: Contract Field Extraction

Extracting fields like Party A, Party B, amount, and date from contracts has a limited output space. If open-ended generation is used, the model might rewrite the amount format, complete non-existent fields, or mix up multiple dates. A more suitable baseline is encoder-only token classification or span classification, followed by rule-based validation for amounts, dates, and entity types.

In training engineering, focus on span-level F1, not token accuracy. If a date entity June 8, 2026 is missing one day, token accuracy might still be high, but the field extraction is wrong. In inference engineering, focus on batch throughput and long document chunking: if each contract is dozens of pages, the cost of processing the full text with a cross-encoder might be unacceptable, requiring chunking to recall candidate paragraphs first, followed by fine-grained extraction.

Case 3: Open-ended Reasoning Answers

Math problems, code fixes, and complex planning are often not covered by fixed labels or spans, requiring a decoder-only LLM to generate intermediate reasoning and the final answer. At this point, classic NLP metrics are no longer sufficient; BLEU/ROUGE might penalize reasonable rewrites or reward wrong but lexically similar answers. More suitable metrics are task success rate, unit test/pass@k, self-consistency, human or LLM-as-judge hierarchical evaluation, and the latency/cost metrics from Chapter 10.

🔑
Engineering Judgment: Model family selection should be deduced from output constraints. Prioritize classification for fixed labels, extraction for answers from the source text, span/CRF/constrained decoding for structured fields, and decoder-only generation for open-ended explanations and reasoning. This ensures that training objectives, evaluation metrics, and inference costs are aligned.

11.8 Experimental Conclusions: From Metrics to Decidable Judgments

This section organizes code outputs, written derivations, reading materials, and experimental metrics into decidable conclusions. The core is not "running a demo successfully," but answering a clear technical question and allowing the reader to judge under what conditions the conclusion holds.

1. Formulate the Question as a Testable Proposition

A good technical question should be falsifiable by data and experiments. For example:

  • Training Engineering: Under a fixed token budget, does seq_len=128 reduce validation PPL compared to seq_len=64? What is the cost in tokens/s and step time?
  • Alignment Engineering: Does DPO improve preference win rate while maintaining base task accuracy and refusal boundaries? How do KL, output length, and failure types change?
  • Inference Engineering: Does increasing RAG top-k from 3 to 8 improve pass rate? What is its impact on P95 TTFT, prompt tokens, context packing misses, and cost?
  • Serving Engineering: When concurrency increases from 1 to 8, do P95/P99 latencies still meet the SLO? Is the bottleneck queuing, prefill, decode, retrieval, or post-processing?

These questions all have baselines, change items, metrics, and conclusion boundaries. In contrast, "building a chatbot" or "training a small model" is not a well-posed technical proposition because it cannot explain what change led to what effect.

2. The Conclusion Structure Should Cover Modeling, System, and Evaluation

PartTraining/Alignment Conclusion Should AnswerInference/Serving Conclusion Should Answer
ProblemWhat phenomenon the training or alignment change aims to improveWhat workload and SLO the service aims to satisfy
BaselineDefault data, model, lr, batch tokens, seedDefault API, no-RAG, or default concurrency
MethodData processing, optimizer, scheduler, checkpoint strategyRAG/tool/JSON, batching, capacity, degradation strategy
MetricsTrain/val loss, PPL, grad_norm, tokens/s, resume parityPass rate, P50/P95/P99, TTFT, TPOT, error rate, cost
AblationChange only lr, seq_len, batch, dropout, or data source at a timeChange only top-k, concurrency, context, JSON/retry, or capacity assumption at a time
Failure analysisLoss spike, NaN, train/val divergence, throughput drop, resume discontinuityRetrieval miss, format/tool error, safety refusal, tail latency, 429/5xx
DecisionWhether to continue training, roll back, scale up, change data, or adjust objectivesWhether to deploy, canary, degrade, rate-limit, or continue optimization

3. Negative Results Can Also Be Strong Conclusions

Experiments do not require every method to "improve" something. If an ablation does not bring significant improvement, as long as the experimental design is clear, it is still a valuable conclusion. For example: seq_len=128 only reduces validation PPL by 0.03, but tokens/s drops by 35%, and the fluctuation across three seeds is about 0.05. Then a reasonable conclusion is "the current setup does not justify adopting longer sequences." This is more rigorous than only picking the best number.

4. Bind Conclusions to Reproduction Commands and Limitations

Conclusions must specify the reproduction path: data version, commit, run command, seed, hardware, model configuration, checkpoint or service configuration, evaluation set, and stress test parameters. They must also specify where extrapolation is not valid: a tiny corpus cannot represent trillion-token training, a local MockEngine cannot represent real GPU serving, a small knowledge base RAG cannot represent an enterprise knowledge base, and a single safety set cannot represent all user risks.

The reader should be able to answer three things: First, is the question clear? Second, is the experiment sufficient to support the conclusion? Third, if the system is scaled to a larger model, larger data, or a real service, which assumptions will break first?

🔑
Engineering Takeaway: Training directions must prove the process is reproducible, recoverable, and interpretable; inference directions must prove that service quality, latency, cost, and risk are decidable under the target workload. Experiments without baselines, ablations, failure analysis, and conclusion boundaries cannot support engineering decisions.

11.10 Summary

  • RNN/LSTM explained sequential recurrence, BPTT, and the long-range dependency problem.
  • Dependency parsing is discrete structure prediction, not equivalent to an attention heatmap.
  • Seq2Seq attention models the source sequence and target prefix separately, forming the basic encoder-decoder structure.
  • BERT/encoder-only models are suitable for classification, reranking, embedding, token classification, and extractive QA.
  • BLEU, ROUGE, EM/F1, span F1, UAS/LAS each measure different task structures; modern LLM evaluation must also describe data, inference settings, judge bias, safety refusals, statistical uncertainty, and conclusion boundaries.
