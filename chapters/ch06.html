6.1 GPT-2 架构全景——从 token ID 到 logits 的完整数据流

经过前 5 章的积累，我们终于可以将所有组件组装成一个完整的 GPT 模型。GPT-2 small (124M) 的架构可以用三个主要阶段概括：

  1. Embedding 阶段：Token Embedding + Position Embedding → 将离散 token ID 映射为连续向量
  2. 计算阶段：12 层 Transformer Block 堆叠 → 逐层变换表示
  3. 输出阶段：Final Norm + LM Head → 将隐藏状态映射为词表概率分布
┌──────────────────────────────────────────────────────────┐ │ GPT-2 (完整模型) │ │ Input: [t1, t2, ..., tn] (词表大小 50257) │ │ │ │ │ ▼ │ │ ┌─────────────┐ ┌──────────────────┐ │ │ │ Token Emb. │ │ Position Emb. │ │ │ │ (50257×768) │ │ (1024×768) │ │ │ └──────┬──────┘ └────────┬─────────┘ │ │ └────────── (+) ─────┘ │ │ ▼ │ │ ┌──────────────────┐ │ │ │ Dropout │ │ │ └──────────────────┘ │ │ │ │ │ ▼ (×12) │ │ ┌────────────────────┐ │ │ │ GPT2Block 1 │ d_model=768, n_heads=12 │ │ ├────────────────────┤ │ │ │ ... │ │ │ ├────────────────────┤ │ │ │ GPT2Block12 │ │ │ └─────────┬──────────┘ │ │ │ │ │ ▼ │ │ ┌──────────────────┐ │ │ │ LayerNorm │ (最终输出归一化) │ │ └──────────────────┘ │ │ │ │ │ ▼ │ │ ┌──────────────────┐ │ │ │ LM Head │ Linear(768, 50257) │ │ └──────────────────┘ │ │ │ │ │ ▼ │ │ logits: (batch, seq, vocab=50257) │ └──────────────────────────────────────────────────────────┘

GPT-2 与后续模型的关键区别

  • 绝对位置编码：GPT-2 使用可学习的 Position Embedding（1024 个位置 × 768 维），而非 RoPE。RoPE 是 Llama、DeepSeek 等后续模型使用的
  • LayerNorm with bias：GPT-2 在每个 Block 中使用带 bias 的 LayerNorm（不是 RMSNorm）
  • GELU 激活：GPT-2 的 FFN 使用 GELU，不是 SwiGLU
  • Weight Tying：GPT-2 共享 Embedding 和 LM Head 的权重

在后续章节中，我们将看到这些设计选择如何被后续模型逐一改进。

🖥️ 练习 2：GPTModel 组装

任务：实现 GPTModel——完整的 GPT-2 模型。

要求：

class GPTModel(nn.Module): """完整 GPT-2 模型 — Embedding → Blocks × 12 → Norm → LM Head""" def __init__(self, config: GPTConfig): super().__init__() self.config = config # Token Embedding self.token_embedding = nn.Embedding(config.vocab_size, config.d_model) # Position Embedding (绝对位置编码) self.pos_embedding = nn.Embedding(config.max_seq_len, config.d_model) self.dropout = nn.Dropout(config.dropout) # GPT-2 Block 堆叠：LayerNorm + causal self-attention + GELU MLP self.blocks = nn.ModuleList([GPT2Block(config) for _ in range(config.n_layers)]) # 最终归一化 self.norm = nn.LayerNorm(config.d_model, bias=config.bias) # LM Head self.lm_head = nn.Linear(config.d_model, config.vocab_size, bias=False) self.apply(self._init_weights) # Weight Tying if config.tie_weights: self.lm_head.weight = self.token_embedding.weight def _init_weights(self, module): if isinstance(module, nn.Linear): nn.init.normal_(module.weight, mean=0.0, std=0.02) if module.bias is not None: nn.init.zeros_(module.bias) elif isinstance(module, nn.Embedding): nn.init.normal_(module.weight, mean=0.0, std=0.02) def forward(self, input_ids, mask=None): batch_size, seq_len = input_ids.shape device = input_ids.device # 1. Token Embedding x = self.token_embedding(input_ids) # (b, s, d_model) # 2. Position Embedding pos_ids = torch.arange(seq_len, device=device).unsqueeze(0) x = x + self.pos_embedding(pos_ids) # (b, s, d_model) # 3. Dropout x = self.dropout(x) # 4. Transformer Blocks for block in self.blocks: x = block(x, mask=mask) # 5. Final Norm x = self.norm(x) # 6. LM Head logits = self.lm_head(x) # (b, s, vocab_size) return logits # 快速测试 config = GPTConfig() model = GPTModel(config) x = torch.randint(0, config.vocab_size, (2, 64)) logits = model(x) assert logits.shape == (2, 64, config.vocab_size) loss = logits.sum() loss.backward() print(f"模型参数: {sum(p.numel() for p in model.parameters()):,}") print(f"输出 shape: {logits.shape}") print("前向 + 反向传播通过 ✓")

6.3A 从 logits 到语言模型目标：为什么输出是 [batch, seq, vocab]

完整 GPTModel 的前向传播到 logits 还没有结束。语言模型训练真正优化的是：在每个位置 ，用当前位置的隐藏状态预测下一个 token 。因此 logits[:, t, :] 不是预测当前位置 token，而是预测下一位置 token 的分类分布。

自回归分解

训练时通常把输入和标签错开一位：

# tokens: [BOS, The, cat, sat, EOS] inputs = tokens[:, :-1] # [BOS, The, cat, sat] labels = tokens[:, 1:] # [The, cat, sat, EOS] logits = model(inputs) # (batch, seq-1, vocab) loss = F.cross_entropy(logits.reshape(-1, vocab), labels.reshape(-1))

Ch06 作业中的 causal_lm_loss_from_logits(logits, input_ids) 让学生把这一步写成函数：logits[:, :-1, :] 对齐 input_ids[:, 1:]，并用 ignore_index 跳过 padding 或不参与训练的位置。这个函数不应该使用 logits[:, 1:, :]，也不应该让 logits[:, t, :] 预测 input_ids[:, t]。

Teacher forcing 与并行训练

训练阶段并不是让模型一步步生成再把生成结果喂回去，而是使用真实前缀并行预测所有下一 token。这叫 teacher forcing。它让长度为 的序列在一次 forward 中产生 个分类任务，因此训练可以充分利用 GPU 并行性。推理阶段则不同：模型只能使用自己已经生成的 token，逐步扩展上下文。

Softmax 只在 loss 中需要

LM Head 输出的是未归一化分数 logits。训练时直接把 logits 传给 F.cross_entropy，因为它内部会稳定地计算 log-softmax 和负对数似然：

Next-token cross entropy

不要在模型 forward 里先做 softmax 再传给 cross entropy；那会损失数值稳定性，也会让梯度更差。

Causal leakage 的最小反例

如果 causal mask 错了，模型可能在位置 看到 ，训练 loss 会异常容易下降，但推理时这种信息不可用。一个简单检查是：固定前缀，只改变未来 token，比较前缀位置 logits 是否变化。

a = torch.tensor([[10, 20, 30, 40]]) b = torch.tensor([[10, 20, 99, 88]]) # 前两个 token 相同，未来 token 不同 logits_a = model(a) logits_b = model(b) assert torch.allclose(logits_a[:, :2], logits_b[:, :2], atol=1e-5)

这个测试不是质量评测，而是因果结构检查：前缀位置的输出不能依赖未来 token。它把第 3 章的 causal mask、第 6 章的 logits shape 和第 7 章的 next-token loss 串成同一个训练目标。

6.3B Worked Example：手算一个 batch 的 label shift 与 loss 分母

假设一个 batch 中有两条已经 padding 到长度 5 的样本，PAD 对应的标签用 -100 表示，交给 F.cross_entropy(..., ignore_index=-100) 跳过：

sample 1 input_ids: [BOS, I, like, NLP, EOS] sample 2 input_ids: [BOS, GPT, EOS, PAD, PAD]

模型 forward 仍然会为每个输入位置都输出一个 vocab 分类分布，形状是 [2, 5, |V|]。训练 next-token loss 时，只取前 4 个位置的 logits，并让它们预测右移一位后的 token：

logits used: positions 0 1 2 3 sample 1 pred: BOS -> I, I -> like, like -> NLP, NLP -> EOS sample 2 pred: BOS -> GPT, GPT -> EOS, EOS -> PAD, PAD -> PAD labels before ignore: sample 1 labels: [I, like, NLP, EOS] sample 2 labels: [GPT, EOS, PAD, PAD] labels after padding mask: sample 1 labels: [I, like, NLP, EOS] sample 2 labels: [GPT, EOS, -100, -100]

因此这个 batch 的平均 loss 只除以 6 个有效预测位置，而不是 2 * 4 = 8 个位置，也不是原始序列的 2 * 5 = 10 个位置。最后一个输入位置的 logits 没有对应的“下一个 token”标签，padding 位置也不代表真实语言建模目标。

带 ignore_index 的 batch mean

这也是为什么实现时应先构造 shift_logits = logits[:, :-1, :] 和 shift_labels = input_ids[:, 1:]，再把 shift_labels 中的 padding 改成 ignore_index。如果把 logits[:, 1:, :] 拿去预测 input_ids[:, 1:]，模型就在用位置 t 的隐藏状态预测同一个位置的 token，训练目标会退化成“复述输入 token”，不再是自回归语言建模。

6.4 深度理论：Weight Tying——哪些模型用，哪些不用，为什么？

Weight Tying（权重共享）让嵌入层（Token Embedding）和输出层（LM Head）共享同一个权重矩阵。更严谨地说，输入端用第 行向量表示 token ；输出端把隐藏状态 与每个 token 行向量做点积，得到 vocab logits。它不是数学意义上的逆映射，而是同一组 token 向量同时承担“读入表示”和“输出分类原型”两种角色。

Weight Tying 共享方式

参数量影响

对于 GPT-2 small：词表 50257 × d_model 768 = 38.6M 参数——占总模型 124M 的 31%。共享后这 38.6M 参数只计算一次。

模型Weight Tying？词表大小节省参数量占总参数比例
GPT-1是~40K~30M~25%
GPT-2是50,25738.6M31%
GPT-3 (175B)否50,257~38.6M< 0.1%
Llama 2/3否32K / 128K~0.1B-1.7B< 1%
DeepSeek-V3否129K~0.9B< 0.2%

为什么有些大模型不用？

随着非 embedding 参数占比上升，Weight Tying 的参数节省比例会下降：在百亿到千亿参数模型里， 往往不再主导总参数量。Untied LM Head 的好处是输入表示和输出分类器可以分别学习，适合需要更大输出端容量、特殊词表设计或多模态输出接口的架构。但这不是固定规则：是否 tying 取决于词表大小、隐藏维度、总参数规模、训练稳定性和模型家族的工程取舍。

反向传播的影响

共享时，嵌入层的梯度来自两个方向：一是 embedding lookup 路径，二是 LM Head 分类路径。被输入序列用到的 token 行会从 lookup 路径拿到梯度；所有出现在 softmax 分类器里的 token 行都会通过输出分布得到梯度。这个耦合有时像正则化，有时会限制输入端和输出端分别适配任务，不能简单归结为“小模型一定好、大模型一定坏”。

tied LM Head 梯度

其中 是 cross entropy 对 logits 的梯度：有效 token 行为 (softmax - one_hot) / N_valid，被 ignore_index 跳过的行全为 0。Ch06 作业中的 tied_lm_head_gradients 要求学生手写这三步，并和 autograd 对齐。

🔑
结论：Weight Tying 首先是一个架构约束：它节省 个 LM Head 参数，并让输入 token 空间和输出 token 分类器共享几何结构。参数节省是否重要，要放到具体模型的 vocab、hidden size 和总参数账本里计算。

6.6 训练前的完整性检查——前向传播测试

在训练之前，你需要确认模型的前向传播能正确运行。这里的关键检查项是：

实现要点

  • Shape 正确：从 (batch, seq_len) 的 token ID 到 (batch, seq_len, vocab_size) 的 logits，整个链条没有被破坏
  • 数值稳定：随机初始化下 logits 在合理范围内（均值 ~0，标准差 ~1-10）。如果出现 NaN 或 Inf，说明初始化有问题
  • 梯度流通：从 logits 的标量损失到所有参数的梯度都能正确计算（每个参数的 .grad 不为 None）
  • 参数共享验证：如果 tie_weights=True，确认 lm_head.weight 和 token_embedding.weight 是同一个张量对象

一个常见的训练失败原因是初始化标准差过大。GPT-2 使用 0.02，但如果误用 0.1，深层网络的 logits 可能会出现 NaN。这是因为残差流的方差在 12 层中逐层累加：

残差流的方差累加

当 过大时， 层的累加会超出 float16 的表达范围。这也是为什么 _init_weights 中的 std=0.02 不是随意选择的——它来自 GPT-2 论文中对残差方差的分析。

💡
实用建议：写一个 test_forward_pass() 函数作为 CI 的一部分。每次修改模型架构后运行它，可以在几分钟内而不是几小时后发现问题。

6.6A 工程视角：一个完整 GPT 实现必须同时满足什么

把 GPT 组件接起来之后，工程上还要回答三个问题：训练目标是否和数据对齐，参数和显存账本是否解释得清，推理时是否遵守自回归约束。下面把本章的关键实现点串成一张工程检查表，它对应的是模型设计能力，不是单纯的单元测试。

1. 张量契约：从 token id 到 next-token loss

完整 decoder-only LM 的 forward contract 是：

input_ids: [B, T] token + pos emb: [B, T, d_model] blocks output: [B, T, d_model] logits: [B, T, vocab_size]

这里的 logits[:, t, :] 预测的是 input_ids[:, t+1]，不是当前位置 token。因此训练 loss 的核心对齐是：

shift_logits = logits[:, :-1, :] shift_labels = input_ids[:, 1:] loss = F.cross_entropy( shift_logits.reshape(-1, vocab_size), shift_labels.reshape(-1), ignore_index=-100, )

最后一个输入位置没有 next-token label；padding 或不参与训练的位置必须用 ignore_index 排除。这个对齐错误通常不会导致 shape 报错，却会把语言建模目标变成“复述当前位置 token”，是初学实现里最隐蔽的错误之一。

2. 参数账本：为什么 GPT-2 small 是 124,439,808

GPT-2 small 的参数量不是一个需要背诵的数字，而是架构账本的结果。默认配置为 V=50257, T_max=1024, d=768, L=12, d_ff=3072。主要参数项为：

token embedding: V * d position embedding: T_max * d per block attention: d * 3d + 3d bias + d * d + d bias per block MLP: d * 4d + 4d bias + 4d * d + d bias per block LayerNorm: 4d final LayerNorm: 2d LM head: 0 if tied, V * d if untied

在 weight tying 下，LM head 和 token embedding 是同一个参数对象，只计一次。训练工程师需要能把总参数拆成 embedding、position embedding、blocks、final norm 和 LM head，因为优化器状态、checkpoint 大小、张量并行切分和显存预算都依赖这个分解。

3. Weight tying：共享参数不等于只更新目标 token

共享权重时，输出层 logits 写成 。交叉熵梯度会先作用到所有 vocab logits，再回传到 hidden states 和 embedding 矩阵：

tied LM Head 梯度路径

因此非目标词表行也会通过 softmax 概率得到输出端梯度。被输入序列索引到的 token 行还会从 embedding lookup 路径得到梯度。理解这点之后，weight tying 就不只是“省参数”，而是输入 token 空间和输出分类器共享几何结构的建模约束。

4. Causal mask：推理约束必须在训练中成立

训练时使用 teacher forcing，可以一次性看到整段序列；但模型内部的每个位置仍然只能关注过去和当前位置。dense attention 的因果约束是：

allowed[i, j] = j <= i scores[~allowed] = -inf weights = softmax(scores)

若 mask 方向写反或广播维度错误，训练 loss 可能异常顺利下降，因为模型偷看了未来 token；但推理时未来 token 不存在，质量会崩。一个小型 sanity check 是固定前缀、替换未来 token，过去位置 logits 不应变化。

5. 初始化和位置编码：GPT-2 公式不能和现代 Llama 公式混用

GPT-2 风格实现使用 learned absolute position embedding、LayerNorm、GELU MLP、带 bias 的线性层和常见的 N(0, 0.02) 初始化。Llama/DeepSeek 风格模型常用 RoPE、RMSNorm、SwiGLU 和 bias-free projection。两类模型都属于 decoder-only Transformer，但参数量公式、权重加载方式和训练稳定性细节不同。

如果输入长度超过 max_seq_len，learned position embedding 没有对应行，模型必须报错或显式扩展位置表。这里和 RoPE 的长上下文外推问题不同，不能用同一套解释。

6. MoE：总容量和每 token 计算必须分开说

MoE 把 FFN 替换成多个 experts，router 为每个 token 选择少数专家。对 bias-free SwiGLU expert：

expert_params = 3 * d_model * expert_hidden total_expert_params = (n_routed_experts + shared_experts) * expert_params activated_expert_params_per_token = (top_k + shared_experts) * expert_params capacity_to_compute_ratio = total_expert_params / activated_expert_params_per_token

这说明“模型总参数量”和“每 token 激活计算量”不是同一个指标。推理工程中，MoE 还会引入 dispatch/combine、跨设备通信、expert load skew 和 tail latency。因此 top-k 越小不必然越快，专家越多也不必然更好。

🔑
本节结论：完整 GPT 组装的核心能力是把 forward shape、next-token objective、参数量、梯度路径、因果约束和稀疏激活资源账本放在同一个系统里理解。只有这样，后续第 7 章的训练循环和第 10 章的推理服务才有可靠的模型基础。

6.6B Checkpoint 兼容性：state_dict 不是一袋张量

完成 GPTModel.forward() 之后，下一步不是只把 model.state_dict() 保存下来，而是让这个 checkpoint 能被另一个进程、另一个框架或推理服务准确恢复。对训练和推理工程师来说，checkpoint 是架构契约：它同时绑定配置、参数命名、张量 shape、tokenizer、数值精度和前向传播语义。

1. 一个可交付 checkpoint 至少包含什么

最小可用包通常不只是一份权重文件。以 decoder-only LM 为例，工程上需要同时保存：

config.json # vocab_size, n_layers, n_heads, d_model, max_seq_len, norm/rope/mlp 类型 model.safetensors # 或 pytorch_model.bin；参数名、shape 和 dtype tokenizer.json # 或 tokenizer.model/merges.txt/vocab.json；词表和 special tokens generation_config.json # 可选；默认 max_new_tokens、temperature、top_p、eos_token_id 等 training_args.json # 可选；继续训练时需要 optimizer/scheduler/数据切分信息

如果只保存权重、不保存 tokenizer，同一段文本可能被切成不同 token id，logits 对比就没有意义。如果只保存权重、不保存配置，加载端可能用错层数、hidden size、RoPE/absolute position embedding 或 LayerNorm/RMSNorm，错误有时会在 shape 阶段暴露，有时会以质量下降的方式延迟暴露。

3. GPT-2 权重转置为什么容易写错

PyTorch nn.Linear(in_features, out_features) 的参数 shape 是 [out_features, in_features]，前向传播等价于 x @ W.T + b。GPT-2 的 HuggingFace 实现早期沿用了 OpenAI 的 Conv1D 命名，内部权重常以 [in_features, out_features] 形式存放。因此把 HF GPT-2 权重加载到自研 nn.Linear 时，很多投影矩阵需要转置。

def copy_linear_from_hf(linear, hf_weight, hf_bias=None, transpose=True): src = hf_weight.t() if transpose else hf_weight assert linear.weight.shape == src.shape linear.weight.data.copy_(src) if hf_bias is not None: assert linear.bias.shape == hf_bias.shape linear.bias.data.copy_(hf_bias) # 例：HF GPT-2 c_attn 是 fused QKV projection # 自研实现若也是 nn.Linear(d_model, 3 * d_model)，通常需要 transpose=True

转置判断不能只看名字，要看两端 shape 和前向公式。若你的自研实现把 Q、K、V 拆成三个线性层，则还要先把 HF 的 fused QKV 权重沿输出维切成三段，再分别复制到 q_proj/k_proj/v_proj。

4. logit parity 的定位顺序

加载预训练权重后，最实用的等价性检查是同一批 input_ids 下比较 logits。定位顺序建议从最不会被训练随机性影响的项开始：

  1. 确认两边使用同一个 tokenizer，且 input_ids 完全相同。
  2. 确认 config 完全一致：层数、hidden size、head 数、上下文长度、LayerNorm eps、是否使用 bias、位置编码类型。
  3. 确认两边都处于 eval()，dropout 已关闭。
  4. 确认所有加载参数的 missing_keys 和 unexpected_keys 都能解释；不能把关键投影层的缺失当成无害 warning。
  5. 逐层比较 hidden states：embedding 后、每个 block 后、final norm 后、lm head 后，找出第一个产生大误差的位置。
  6. 若 embedding 后已不一致，优先检查 token embedding、position embedding 和 position ids；若 attention 后开始不一致，优先检查 QKV 顺序、mask 方向、head reshape 和投影转置。

在 fp32、同 tokenizer、同架构、无 dropout 的条件下，GPT-2 small 的 logits 最大绝对误差通常应接近浮点舍入误差量级。若误差达到 1e-2 甚至更大，通常不是“预训练模型本来就有差异”，而是加载映射或前向实现仍有语义不一致。

🔑
本节结论：模型实现的成熟度体现在能否跨越“随机初始化能跑”和“预训练权重能等价加载”之间的差距。前者验证 shape 和基本梯度，后者验证命名、布局、位置编码、归一化、mask、tokenizer 与数值模式共同构成的完整模型契约。

6.8 深度理论：MoE 架构——把容量和每 token 计算分开

Mixture of Experts (MoE) 是 GPT-2 架构之外最重要的 LLM 扩展。它将每层的 FFN 替换为多个"专家"网络，每个 token 只激活其中少数专家，实现了 稀疏激活——总参数量巨大但每次前向的计算量只取决于激活参数。

DeepSeekMoE 的核心设计

MoE 前向计算

以常见的 DeepSeekMoE 风格设计为例，课程只关注可迁移的结构要点：

  • 多个路由专家：router 为每个 token 选择 top-K 个专家，而不是激活全部专家。
  • 共享专家：所有 token 都经过少量 shared experts，用来保留通用变换路径。
  • 细粒度专家：单个 expert 通常比 dense FFN 更小，但 expert 数量更多。
  • 核心问题：总参数量决定容量上限；每 token 激活专家数决定单次前向计算量；负载均衡决定吞吐是否稳定。
┌─ DeepSeekMoE FFN 层 ────────────────────────────┐ │ │ │ 输入 x ──→ Router ──→ top-8 专家选择 │ │ (Linear + Softmax + TopK) │ │ │ │ ┌──────────────────────────────────┐ │ │ │ Expert 1 │ Expert 2 │ ... │ E256 │ │ │ │ (SwiGLU) │ (SwiGLU) │ │ │ │ │ └───┬──────┴───┬──────┘ └──┬────┘ │ │ │ │ │ │ │ └───(+)────┘ │ │ │ │ 共享 Expert ──┘ │ │ │ (always on) │ │ ▼ │ │ 输出 y = 路由加权和 + 共享专家输出 │ └────────────────────────────────────────────────────┘

为什么只激活少量专家仍然有效？

MoE 的核心假设是条件计算：不同 token 不必经过完全相同的 FFN 参数。router 根据隐藏状态选择专家，使数学符号、代码片段、实体名、格式 token 等不同输入模式走向不同的参数子集。这个假设是否成立取决于训练数据、router 表达能力、专家容量和负载均衡；如果 router 坍缩到少数专家，MoE 会退化成昂贵且不稳定的稠密模型近似。

🔑
核心 trade-off：MoE 用更多总参数扩大容量，用 top-K 稀疏激活控制每 token 计算量。但它额外引入 router 训练、expert load balancing、跨设备通信和 batch 内 token dispatch 问题；这些系统成本必须和容量收益一起分析。

作业中的 moe_parameter_budget 会把这个 trade-off 写成可计算的参数预算。若每个 expert 是 bias-free SwiGLU FFN，单个 expert 参数量为 3 * d_model * d_ff，分别对应 gate/up/down 三个线性矩阵。MoE 的 total expert parameters 需要乘以全部路由专家和 shared experts；每 token activated expert parameters 只乘以 top_k 个路由专家和 shared experts。router 本身还包含 d_model * n_experts 个门控参数。这样可以明确区分“模型容量来自全部专家”和“单次 token 计算只经过少量专家”。

Worked Example：手算 MoE 参数预算与负载偏置

设一个 toy MoE 层使用 d_model=16、d_ff=32、8 个 routed experts、top_k=2，并有 1 个 shared expert。每个 bias-free SwiGLU expert 有 3 * 16 * 32 = 1536 个参数：gate、up、down 三个矩阵。

再看负载均衡。若 4 个专家在一个 batch 中收到 token 计数 [80, 10, 5, 5]，理想负载是 0.25，实际负载是 [0.80, 0.10, 0.05, 0.05]。动态偏置更新 bias += lr * (target_load - load_ratio)，取 lr=0.04 时，专家 0 的增量是 0.04 * (0.25 - 0.80) = -0.022，专家 2 的增量是 0.04 * (0.25 - 0.05) = 0.008。因此过载专家下一步更难被选中，欠载专家更容易进入 top-K；这正是 AuxLossFreeBalancer.update 的方向性要求。

Router 概率、top-k 归一化与梯度路径

MoE router 首先把每个 token 的隐藏状态 投影成专家 logits：

Router logits and probabilities

top-k 路由只保留概率最大的 个专家。因为被丢弃专家的概率质量不再参与输出，保留下来的权重必须重新归一化：

Top-k mixture weights

这一步解释了作业为什么检查 top_weights.sum(dim=-1) == 1。没有重新归一化时，不同 token 的 expert 输出会被不同的残余概率质量缩放，模型的残差流尺度会随 router 分布变化，训练更不稳定。

Worked Example：MoE load-balancing loss 同时看概率和分配

设 4 个 token 对 3 个 experts 的 router probabilities 分别为 [0.7,0.2,0.1]、[0.6,0.3,0.1]、[0.1,0.8,0.1]、[0.2,0.2,0.6]，top-1 选择的 expert ids 是 [0,0,1,2]。实际分配比例 load 是 [0.5,0.25,0.25]；平均 router probability 是 [0.4,0.375,0.225]。Switch-style auxiliary loss 可写成 n_experts * sum_i load_i * prob_i，所以该例为 3 * (0.5*0.4 + 0.25*0.375 + 0.25*0.225) = 1.05。

这个 loss 的作用不是让每个 token 平均激活所有专家，而是惩罚 batch 级别的 router collapse。只看 top-k counts 会忽略 router softmax 是否已经偏向少数专家；只看平均概率又会忽略实际 top-k 分配是否已经拥塞。Ch06 作业中的 moe_load_balance_loss 因此同时返回 load_fraction、mean_router_prob 和 loss。

Capacity、token dispatch 与负载不均衡

参数预算只回答“一个 token 理论上激活多少专家”，还没有回答“一个 batch 能否顺利执行”。实际 MoE 层需要把 token 分发到专家上。若每个 expert 的容量为 个 token，expert 收到 个 token，则：

Expert overflow

即使没有显式丢 token，极端不均衡也会让最忙的 expert 决定整层延迟，其他设备或专家空等。因此 MoE 的质量和吞吐不仅取决于 top-k，还取决于 load distribution、capacity factor、专家并行通信和 dispatch/combine kernel 的实现。

6.9 编程练习 5：实现 DeepSeekMoE Router——Top-K 门控 + Aux-Loss-Free 动态偏置

🖥️ 练习 5：DeepSeekMoE Router—Top-K Gating + Dynamic Bias

任务：实现 DeepSeekMoE 的参数预算、路由器（Router）和负载均衡器（Aux-Loss-Free Balancer）。

参数预算要求：

  • moe_parameter_budget：使用 bias-free SwiGLU expert 的 3 * d_model * expert_hidden 参数量
  • 返回 router 参数、全部 expert 参数、每 token 激活 expert 参数、激活比例和 capacity-to-compute ratio
  • 检查 top_k 必须在 [1, n_routed_experts] 内，维度和专家数必须为正数

路由部分要求：

  • MoERouter：一个 nn.Linear(d_model, n_experts) 计算路由得分，然后 softmax + topk 选择 top-K 专家
  • 返回 (top_weights, top_indices)——重新归一化后的门控权重和被选中的专家索引
  • 支持 n_experts=256, top_k=8（DeepSeek 配置）

负载均衡器要求：

  • AuxLossFreeBalancer：维护每个专家的动态偏置 bias: (n_experts,)
  • update(token_counts)：根据每个专家的 token 计数调整偏置——过载专家降低偏置，欠载专家增加偏置
  • apply_bias(gate_logits)：在 softmax 之前将偏置加到 logits 上
  • 验证：均衡后专家的负载比例在平均值附近波动
# 预期用法
router = MoERouter(d_model=768, n_experts=256, top_k=8)

# 完全均衡时: max ≈ min ≈ 1/256 ≈ 0.0039
class MoERouter(nn.Module): """DeepSeekMoE 路由器 — Top-K 门控""" def __init__(self, d_model, n_experts=256, top_k=8): super().__init__() self.n_experts = n_experts self.top_k = top_k self.gate = nn.Linear(d_model, n_experts, bias=False) def forward(self, x, bias=None): """ x: (batch, seq_len, d_model) bias: (n_experts,) 或 None — 动态偏置 返回: (weights, indices) 各 (b, s, top_k) """ logits = self.gate(x) # (b, s, n_experts) if bias is not None: logits = logits + bias.unsqueeze(0).unsqueeze(0) weights = F.softmax(logits, dim=-1) # (b, s, n_experts) # Top-K 选择 top_weights, top_indices = torch.topk( weights, self.top_k, dim=-1 ) # 各 (b, s, top_k) # 重新归一化 top-k 权重 top_weights = top_weights / top_weights.sum(dim=-1, keepdim=True) return top_weights, top_indices class AuxLossFreeBalancer: """Aux-Loss-Free 负载均衡 — DeepSeek-V3 使用动态偏置替代辅助损失来平衡专家负载。 过载专家降低偏置（降低被选中的概率），欠载专家增加偏置。 """ def __init__(self, n_experts, bias_update_rate=0.01, bias_clip=0.1): self.n_experts = n_experts self.bias_update_rate = bias_update_rate self.bias_clip = bias_clip self.bias = torch.zeros(n_experts) def update(self, token_counts): """根据每个专家的 token 计数更新偏置""" total = token_counts.sum().float() if total == 0: return load_ratio = token_counts / total # 实际负载比例 target_load = 1.0 / self.n_experts # 理想负载 delta = self.bias_update_rate * (target_load - load_ratio) self.bias += delta self.bias.clamp_(-self.bias_clip, self.bias_clip) def apply_bias(self, gate_logits): """在路由 logits 上应用动态偏置""" return gate_logits + self.bias.to(gate_logits.device) def get_load_stats(self, token_counts): """返回负载均衡统计信息""" total = token_counts.sum().float() load_ratio = token_counts / total if total > 0 else token_counts ideal = 1.0 / self.n_experts return { 'max_load': load_ratio.max().item(), 'min_load': load_ratio.min().item(), 'std_load': load_ratio.std().item(), 'ideal_load': ideal, } # 集成测试 router = MoERouter(768, n_experts=256, top_k=8) balancer = AuxLossFreeBalancer(256) x = torch.randn(4, 64, 768) # 模拟训练步 for step in range(10): weights, indices = router(x, balancer.bias) # 统计 token_counts = torch.zeros(256) token_counts.scatter_add_(0, indices.flatten(), torch.ones_like(indices.flatten())) balancer.update(token_counts) if step in [0, 4, 9]: stats = balancer.get_load_stats(token_counts) print(f"Step {step}: max={stats['max_load']:.6f}, " f"min={stats['min_load']:.6f}, std={stats['std_load']:.6f}")

💡 动态偏置 vs 辅助损失：辅助损失通过梯度反向传播微调路由器参数来平衡负载，计算量小但引入了额外的超参数 α。动态偏置不经过反向传播——它是前向过程中对路由 logits 的直接调整。DeepSeek-V3 报告显示，在其训练设定下动态偏置达到的均衡效果与辅助损失相当，并减少了辅助损失权重调参；不同模型和数据分布仍需要实测。
