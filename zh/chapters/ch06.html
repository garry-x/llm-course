6.1 GPT-2 架构总览：从 token id 到 logits

前 5 章把零件都做出来了，现在把它们组成 GPT-2 small (124M)。完整数据流可以分三段：

  1. Embedding 阶段：Token Embedding + Position Embedding，把离散 token id 变成连续向量。
  2. 计算阶段：堆叠 12 层 Transformer Block，逐层变换表示。
  3. 输出阶段：Final LayerNorm + LM Head，把 hidden state 映射到 vocabulary logits。

图 6-1：GPT-2 small (124M) 的参数拆解。Embedding 在小模型里占比很高。

GPT-2 和现代 decoder-only 模型的差异

  • 位置编码：GPT-2 用可学习 absolute position embedding，长度表是 1024 x 768；Llama/DeepSeek 常用 RoPE。
  • Norm：GPT-2 用带 bias 的 LayerNorm；很多现代模型用 RMSNorm。
  • FFN：GPT-2 用 GELU MLP；很多现代模型用 SwiGLU。
  • Weight tying：GPT-2 共享 token embedding 和 LM head 权重。

所以如果目标是复现 GPT-2 124M 参数量，就不能直接拿 Ch05 的 RMSNorm/SwiGLU/Llama 风格 Block 替换。

6.3 编程练习 2：组装完整 GPTModel

现在用 GPTConfig 和 GPT-2 style 组件组装完整模型。结构是：Embedding -> Blocks x 12 -> Final Norm -> LM Head。

练习 2：GPTModel Assembly

  • token_embedding = nn.Embedding(vocab_size, d_model)。
  • pos_embedding = nn.Embedding(max_seq_len, d_model)。
  • blocks = nn.ModuleList([...])，包含 n_layers 个 GPT2Block。
  • norm = nn.LayerNorm(d_model)。
  • lm_head = nn.Linear(d_model, vocab_size, bias=False)。
  • 如果 tie_weights=True，令 lm_head.weight = token_embedding.weight。
  • Linear 和 Embedding 用 N(0, 0.02) 初始化，bias 初始化为 0。
class GPTModel(nn.Module): def __init__(self, config: GPTConfig): super().__init__() self.config = config self.token_embedding = nn.Embedding(config.vocab_size, config.d_model) self.pos_embedding = nn.Embedding(config.max_seq_len, config.d_model) self.dropout = nn.Dropout(config.dropout) self.blocks = nn.ModuleList([GPT2Block(config) for _ in range(config.n_layers)]) self.norm = nn.LayerNorm(config.d_model, bias=config.bias) self.lm_head = nn.Linear(config.d_model, config.vocab_size, bias=False) self.apply(self._init_weights) if config.tie_weights: self.lm_head.weight = self.token_embedding.weight def _init_weights(self, module): if isinstance(module, nn.Linear): nn.init.normal_(module.weight, mean=0.0, std=0.02) if module.bias is not None: nn.init.zeros_(module.bias) elif isinstance(module, nn.Embedding): nn.init.normal_(module.weight, mean=0.0, std=0.02) def forward(self, input_ids, mask=None): B, T = input_ids.shape assert T <= self.config.max_seq_len pos_ids = torch.arange(T, device=input_ids.device).unsqueeze(0) x = self.token_embedding(input_ids) + self.pos_embedding(pos_ids) x = self.dropout(x) for block in self.blocks: x = block(x, mask=mask) x = self.norm(x) return self.lm_head(x) # [B,T,V]

最小验收：随机输入 [2,64] 后，输出必须是 [2,64,50257]，并且一个标量 loss 能反向传播到所有参数。

6.3A 从 logits 到语言模型目标：为什么输出是 [batch, seq, vocab]

GPTModel 输出 logits 还不是训练目标。自回归语言模型在位置 t 用当前 hidden state 预测下一个 token x_{t+1}。也就是说，logits[:, t, :] 预测的是下一位，不是当前位置 token。

# tokens: [BOS, The, cat, sat, EOS] inputs = tokens[:, :-1] # [BOS, The, cat, sat] labels = tokens[:, 1:] # [The, cat, sat, EOS] logits = model(inputs) loss = F.cross_entropy(logits.reshape(-1, vocab), labels.reshape(-1))

如果直接让 logits[:, t, :] 预测 input_ids[:, t]，训练目标会退化成“复读当前 token”，不是 causal LM。

Teacher forcing 和并行训练

训练时不会让模型一步步生成再喂回去，而是用真实前缀并行预测所有 next tokens，这叫 teacher forcing。推理时不同，只能使用已经生成出来的 token。

softmax 只应该放在 loss 里

F.cross_entropy 内部已经稳定地做了 log-softmax 和 NLL。模型 forward 里不要先 softmax，否则数值稳定性和梯度都会变差。

未来信息泄漏检查

固定前缀，只改未来 token，前缀位置的 logits 不应该变化：

a = torch.tensor([[10, 20, 30, 40]]) b = torch.tensor([[10, 20, 99, 88]]) assert torch.allclose(model(a)[:, :2], model(b)[:, :2], atol=1e-5)

6.3B 手算 label shift 和 loss 分母

假设一个 batch 有两个样本，padding 后长度都是 5。PAD 对应的 label 设为 -100，传给 F.cross_entropy(..., ignore_index=-100) 跳过。

sample 1 input_ids: [BOS, I, like, NLP, EOS] sample 2 input_ids: [BOS, GPT, EOS, PAD, PAD] logits used: positions 0..3 sample 1 labels: [I, like, NLP, EOS] sample 2 labels: [GPT, EOS, -100, -100]

这个 batch 的平均 loss 分母是 6 个有效预测位置，不是 2*4=8，也不是原始长度 2*5=10。最后一个输入位置没有“下一个 token”标签，padding 位置也不是有效语言建模目标。

6.4 Weight Tying：哪些模型用，为什么用

Weight tying 让输入侧 token embedding 和输出侧 LM Head 共享同一个矩阵。输入时，第 i 行是 token i 的向量；输出时，hidden state 和每个 token row 做点积得到 vocabulary logits。

W_lm_head = W_token_embedding # [vocab_size, d_model]

参数量影响

GPT-2 small 的词表是 50257，d_model=768，embedding 参数量是 50257 * 768 = 38,597,376，约占 124M 的 31%。共享权重后，这 38.6M 只算一次。

模型是否常用 tying原因
GPT-2 small是embedding 占比高，节省明显
GPT-3 / Llama / DeepSeek 大模型常见为否embedding 占总参数比例变小，不共享能让输入表示和输出分类器分开学习

反向传播影响

共享权重时，embedding 矩阵有两条梯度来源：输入 lookup 路径，以及输出 softmax 分类路径。它不仅是“省参数”，也是一个几何约束：输入 token 空间和输出 token classifier 共享同一组向量。

6.4A 例子：tied LM Head 和 next-token loss 怎么接上

设词表大小 |V|=4，hidden 维度 d=2，embedding 矩阵 E: [4,2]。Transformer 输出 hidden states H: [3,2]，则 tied LM Head 的 logits 是：

Z = H @ E.T # [3,4]

输入序列 [0,1,2] 只有两个 next-token 监督项：Z[0] 预测 1，Z[1] 预测 2。Z[2] 在训练样本里没有下一位标签，但推理时可以用来预测下一个 token。

6.5 编程练习 3：逐层核对 124M 参数量

参数量不是背出来的，是架构 ledger 算出来的。GPT-2 small 默认配置是 V=50257, T_max=1024, d=768, L=12, d_ff=3072。

练习 3：Parameter Counter

  • 遍历 model.named_parameters()。
  • 按 Token Embedding、Position Embedding、每层 Attention、每层 MLP、Final Norm 分组打印。
  • 验证总量为 124,439,808。
  • 确认 embedding 约占 38.6M / 124.4M = 31%。
def count_parameters(model): total = 0 for name, param in model.named_parameters(): n = param.numel() total += n print(f"{name:<40} {n:>12,}") print(f"Total: {total:,}") assert total == 124_439_808 return total

如果总数不对，优先检查：LM head 是否真的 tied、GPT-2 Block 是否用了 LayerNorm/GELU/bias、position embedding 是否是 1024 x 768。

6.6 预训练前完整性检查：Forward Pass Test

训练前先确认 forward 链路本身可靠：

  • shape：[B,T] 到 [B,T,V] 没断。
  • 数值：随机初始化下 logits 没有 NaN/Inf，均值和标准差在合理范围。
  • 梯度：从标量 loss 能反向传播到所有参数。
  • 共享权重：lm_head.weight is token_embedding.weight 为 True。

GPT-2 使用 std=0.02 初始化不是随便选的。初始化太大时，残差流方差会在 12 层里不断累积，低精度训练更容易溢出。建议把 test_forward_pass() 放进最小 CI，每次改模型结构后先跑它。

6.6A 工程视角：完整 GPT 实现要同时满足什么

1. Tensor contract

input_ids: [B,T] token + pos emb: [B,T,d_model] blocks output: [B,T,d_model] logits: [B,T,vocab_size]

训练 loss 的对齐是：

shift_logits = logits[:, :-1, :] shift_labels = input_ids[:, 1:] loss = F.cross_entropy( shift_logits.reshape(-1, vocab_size), shift_labels.reshape(-1), ignore_index=-100, )

2. Parameter ledger

token embedding: V * d position embedding: T_max * d per block attention: d * 3d + 3d bias + d * d + d bias per block MLP: d * 4d + 4d bias + 4d * d + d bias per block LayerNorm: 4d final LayerNorm: 2d LM head: 0 if tied, V * d if untied

3. Causal mask

训练时 teacher forcing 让整段序列同时进入模型，但每个位置仍然只能 attend 到过去和自己。mask 方向错时，loss 可能下降得很漂亮，但推理时质量会崩。

4. GPT-2 公式不要和 Llama 公式混用

GPT-2 是 learned absolute position + LayerNorm + GELU + bias + N(0,0.02)。Llama/DeepSeek 常见是 RoPE + RMSNorm + SwiGLU + bias-free。两者都是 decoder-only，但参数量公式、权重加载和稳定性细节不同。

6.6B Checkpoint 兼容性：state_dict 不是一袋 tensor

完成 GPTModel.forward() 后，下一步不是简单保存 model.state_dict()。checkpoint 是一个架构契约，必须把配置、参数名、tensor shape、tokenizer、数值精度和 forward 语义绑在一起。

可交付包至少包含什么

config.json model.safetensors tokenizer.json / vocab.json / merges.txt generation_config.json training_args.json # 可选，用于继续训练或审计

6.6C 现代 Decoder 架构 ledger：怎样读模型报告

GPT-2 是清晰的教学基线。现代工业模型报告通常还会混合 dense/MoE、RoPE/GQA/MLA、total vs active params、多模态 token、thinking budget、量化和导出约束。不要只看“参数更多”。

报告里的说法工程含义比较前要问什么
total parameterscheckpoint 和权重驻留成本expert 是常驻、分片还是 offload？
active parameters每个 token 实际计算路径是否包含 shared experts、attention、embedding、output head？
context lengthposition 机制和 KV budget 的上限是训练得到、扩展得到，还是只外推？
thinking budget推理时允许额外 reasoning tokens质量、延迟、成本和停止条件怎么一起报？

成熟的 release ledger 至少要给出：config、tokenizer、参数账、训练目标、serving 语义、支持的 decoding modes、context 策略、量化格式、评测边界和回滚方式。

6.7 编程练习 4：加载 HuggingFace GPT-2 权重并验证一致性

最严格的验证方式，是把官方 GPT-2 small 权重加载到自己的实现里，用同一批 input_ids 比较 logits。

练习 4：HuggingFace Weight Loading and Verification

  • 用 transformers.GPT2LMHeadModel.from_pretrained("gpt2") 加载官方权重。
  • 建立命名映射：例如 transformer.h.0.attn.c_attn.weight 到 blocks.0.attn.c_attn.weight。
  • HF GPT-2 的 Conv1D 权重常需要转置后拷贝到 nn.Linear。
  • 你的实现必须使用 GPT-2 的 LayerNorm + GELU + bias，而不是 RMSNorm/SwiGLU。
  • 同一输入下 logits 最大差异应小于 1e-4。
def copy_linear_from_hf(linear, hf_weight, hf_bias=None, transpose=True): src = hf_weight.t() if transpose else hf_weight assert linear.weight.shape == src.shape linear.weight.data.copy_(src) if hf_bias is not None: assert linear.bias.shape == hf_bias.shape linear.bias.data.copy_(hf_bias)

转置不能只靠名字判断，要看两边的 shape 和 forward 公式。误差如果从 attention 后才变大，优先检查 QKV 顺序、mask 方向、head reshape 和 projection transpose。

6.8 MoE 架构：把容量和每 token 计算解耦

Mixture of Experts (MoE) 把每层 FFN 换成多个 expert，并且每个 token 只激活其中少数几个。这叫 sparse activation：模型可以有很大的总参数量，但单次 forward 的计算主要由被激活的专家决定。

DeepSeekMoE 风格的核心结构

  • Routed Experts：router 为每个 token 选择 top-k experts。
  • Shared Experts：所有 token 都经过少量 shared experts，保留通用变换路径。
  • Fine-grained Experts：单个 expert 往往比 dense FFN 小，但 expert 数更多。
  • 核心账本：总参数决定容量上限，激活 expert 数决定每 token 计算，load balancing 决定吞吐稳定性。

例子：MoE 参数预算和动态 bias

设 d_model=16、d_ff=32、8 个 routed experts、top_k=2、1 个 shared expert。每个 bias-free SwiGLU expert 参数量是 3 * 16 * 32 = 1536。

项目计算结果
总 expert 参数(8+1)*153613,824
router 参数16*8128
每 token 激活 expert 参数(2+1)*15364,608
容量/计算比13,824/4,6083.0

如果 4 个 experts 的 token counts 是 [80,10,5,5]，理想负载是 0.25。动态 bias 更新 bias += lr * (target_load - load_ratio) 会降低过载 expert 的 bias，提高欠载 expert 的 bias。

Top-k 权重要重新归一化

router softmax 后只保留 top-k experts，被丢掉的概率质量不再参与输出，所以保留的权重必须重新归一化：

alpha_i = p_i / sum_{j in TopK(p)} p_j y = sum_i alpha_i * E_i(x)

否则不同 token 的 expert 输出会因为残余概率质量不同而改变 residual stream 尺度。

容量、dispatch 和负载倾斜

MoE 还要把 token dispatch 到 experts。若某个 expert 收到太多 token，就会成为整层延迟瓶颈；如果实现选择丢弃超容量 token，还会直接影响质量。所以 MoE 的“每 token 激活参数少”不自动等于训练或推理更快。

6.9 编程练习 5：实现 DeepSeekMoE Router

练习 5：Top-K Gating + Aux-Loss-Free Dynamic Bias

  • moe_parameter_budget：用 3*d_model*expert_hidden 计算 bias-free SwiGLU expert 参数量。
  • MoERouter：用 nn.Linear(d_model, n_experts) 得到 logits，再 softmax + topk。
  • 返回 (top_weights, top_indices)，shape 是 [B,T,top_k]。
  • top-k 权重要重新归一化，保证最后一维和为 1。
  • AuxLossFreeBalancer：维护每个 expert 的动态 bias，过载降 bias，欠载升 bias。
class MoERouter(nn.Module): def __init__(self, d_model, n_experts=256, top_k=8): super().__init__() self.n_experts = n_experts self.top_k = top_k self.gate = nn.Linear(d_model, n_experts, bias=False) def forward(self, x, bias=None): logits = self.gate(x) if bias is not None: logits = logits + bias.view(1, 1, -1) probs = F.softmax(logits, dim=-1) top_weights, top_indices = torch.topk(probs, self.top_k, dim=-1) top_weights = top_weights / top_weights.sum(dim=-1, keepdim=True) return top_weights, top_indices class AuxLossFreeBalancer: def __init__(self, n_experts, bias_update_rate=0.01, bias_clip=0.1): self.n_experts = n_experts self.bias_update_rate = bias_update_rate self.bias_clip = bias_clip self.bias = torch.zeros(n_experts) def update(self, token_counts): total = token_counts.sum().float() if total == 0: return load_ratio = token_counts / total target = 1.0 / self.n_experts self.bias += self.bias_update_rate * (target - load_ratio) self.bias.clamp_(-self.bias_clip, self.bias_clip)

Dynamic bias 不走反向传播，是在 forward 里直接调整 routing logits。它减少了 auxiliary loss 权重的调参负担，但不同模型和数据上仍要实测负载、质量和吞吐。

6.10 本章小结

  • 你把 token embedding、position embedding、12 层 GPT-2 Block、final norm 和 LM head 组成了完整 GPTModel。
  • 完整 GPT 的 forward contract 是 [B,T] -> [B,T,D] -> [B,T,V]。
  • logits[:, t, :] 预测下一位 token，训练时必须做 label shift，并用 ignore_index 跳过 padding。
  • GPT-2 的 weight tying 在小模型里节省大量参数；在大模型里是否共享要看词表、hidden size、总参数和模型族设计。
  • GPT-2 small 的 124,439,808 参数来自明确的 architecture ledger，不是一个孤立数字。
  • HF 权重加载要处理命名映射、tensor 转置、fused QKV、shared weights、eval mode 和 tokenizer 一致性。
  • MoE 用更多总参数换容量，用 top-k sparse activation 控制每 token 计算，但带来 router、负载均衡、dispatch 和通信成本。
  • DeepSeekMoE 风格的 dynamic bias 通过调节 router logits 平衡 expert 负载，不通过辅助损失反向传播。
