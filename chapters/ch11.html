11.1 为什么 LLM 课程仍要学习 Classic NLP

CS224N 风格的神经 NLP 主线从词向量、RNN、dependency parsing、seq2seq attention、Transformer、BERT 一路发展到大语言模型。跳过中间模型会带来一个常见误解：以为所有 NLP 问题都应被改写成 prompt-based generation。实际上，许多生产系统把不同模型族组合起来：BERT 或 DeBERTa 做 reranking，bi-encoder 做 retrieval embedding，encoder-only head 做抽取式 QA，decoder-only LLM 做答案生成。

本章的重点不是记住旧模型名字，而是掌握三类稳定概念：

  • 归纳偏置：RNN 按时间递归，parser 输出树，seq2seq 分离源序列和目标序列，BERT 双向读上下文。
  • 目标函数：causal LM 预测下一个 token，MLM 只在 mask 位置计算 loss，seq2seq 最大化条件生成概率，parser 最大化结构动作或 head/label 概率。
  • 评测指标：BLEU、ROUGE、EM/F1、UAS/LAS 和 retrieval metrics 各自回答不同问题，不能用一个总分替代。

11.3 Dependency Parsing：结构化预测不是 attention heatmap

Dependency parsing 为每个 token 预测一个 head 和一个 dependency label，输出是一棵有向树。它和 attention heatmap 的区别很重要：attention 是连续权重，可用于诊断；dependency tree 是离散结构，要满足单根、无环、每个词一个 head 等约束。

Arc-standard transition system 用 stack、buffer 和 arcs 构造树：

动作前置条件效果
SHIFTbuffer 非空把 buffer 首 token 推入 stack
LEFT_ARC(label)stack 至少两个元素次栈顶依附于栈顶，然后移除次栈顶
RIGHT_ARC(label)stack 至少两个元素栈顶依附于次栈顶，然后移除栈顶
ROOT(label)只剩一个根候选把根候选连到 ROOT

Worked Example：Arc-standard 解析 I saw her

目标依存树是 saw -> I (nsubj)、saw -> her (obj)，最后 ROOT -> saw (root)。用 arc-standard transition system 可以按下表构造：

stepstackbufferactionnew arc
0[][I, saw, her]SHIFT-
1[I][saw, her]SHIFT-
2[I, saw][her]LEFT_ARC(nsubj)saw -> I
3[saw][her]SHIFT-
4[saw, her][]RIGHT_ARC(obj)saw -> her
5[saw][]ROOT(root)ROOT -> saw

这个例子说明 parsing 不是给每个 token 独立分类：动作是否合法取决于当前 stack/buffer 状态，错误 action 会破坏最终树结构。Ch11 作业中的 run_arc_standard_transitions 正是让学生执行这张 transition table，并返回每个 token 的 head、label、arcs 和 trace。

11.5 BERT / Encoder-only：MLM、分类与抽取式 QA

BERT-style encoder 可以双向看见输入中的左右上下文，不是自回归生成器。它的预训练目标之一是 Masked Language Modeling：只在被 mask 的位置计算 cross entropy，未 mask 的 token 使用 ignore index。课程作业中的 masked_lm_loss_from_logits 会显式返回 masked-token loss、accuracy 和参与 loss 的位置，帮助区分“输入中有哪些 token 可见”和“哪些位置贡献训练目标”。

维度Encoder-only / BERTDecoder-only / GPT
可见性双向 self-attentioncausal mask，只看左侧前缀
预训练目标MLM：预测 mask 位置next-token prediction
典型用途分类、reranking、embedding、token classification、span QA开放式生成、代码、对话、reasoning
输出约束常有固定 label 或输入 span生成新 token 序列

抽取式 QA 常把输入组织为 [CLS] question [SEP] passage [SEP]，模型分别预测每个位置作为答案起点和终点的 logits。合法 span 的分数为：

若 [CLS] 的 no-answer 分数最高，模型应返回空答案。这和 decoder-only LLM 生成“我不知道”不同：BERT QA 的答案必须绑定到输入 token span。

BIO sequence labeling

NER、slot filling 和 token classification 常把每个 token 标成 B-TYPE、I-TYPE 或 O。B-ORG I-ORG 表示一个连续组织实体，O 表示实体外部。课程作业中的 bio_tags_to_spans(tokens, tags) 把 token-level 标签解码成 span-level 输出，例如 ["B-LOC", "I-LOC"] 对应一个 LOC span。若一个序列以 I-LOC 开头，很多评测脚本会把它容错为新实体开始；严格系统也可以选择直接报错，但必须在数据协议中写清楚。

实体识别通常用 span-level F1，而不是只看 token accuracy。一个预测实体只有在 type、start 和 end 都和 gold 完全一致时才算 true positive；边界少一个 token 或类型错了都算 false positive，同时对应 gold span 仍是 false negative。Ch11 作业中的 span_f1(gold_spans, pred_spans) 把这个定义落到可计算的 TP/FP/FN、precision、recall 和 F1。

Viterbi structured decoding

逐 token argmax 会忽略标签之间的约束：例如 B-ORG 后面接 I-ORG 通常比跳回 O 更合理，O 后面直接接 I-PER 往往应被惩罚。Linear-chain CRF 或类似结构化解码把路径分数写成 emission score 加 transition score：

公式 11.7 Linear-chain path score

Viterbi 动态规划在第 t 步保存“以某个 tag 结尾的最佳路径分数”和 backpointer，复杂度为 O(TK^2)。Ch11 作业中的 viterbi_decode 让学生手写这个解码过程，理解结构化预测如何把局部分类器输出变成全局一致的标签序列。

训练 CRF 时还需要归一化所有可能路径，而不是只看最优路径。Forward algorithm 把 Viterbi 里的 max 换成 logsumexp，计算 log-partition：

公式 11.8 CRF log-partition

抽取式 QA：从 start/end logits 选择合法 span

BERT-style QA head 不生成自由文本，而是在输入 passage 中选择一个连续 span。设输入是 [CLS] who founded OpenAI [SEP] Sam Altman founded OpenAI [SEP]，候选答案必须来自 passage token 位置。start/end logits 最高的组合可能如下：

合法 span 要满足 start <= end 且长度不超过 max_answer_len。若 max_answer_len >= 2，Sam Altman 的分数为 4.0 + 3.5 = 7.5，高于 no-answer 的 [CLS] 分数 1.0 + 1.0 = 2.0，所以返回该 span。若 [CLS] 分数最高，系统返回 no-answer；这与 decoder-only LLM 生成一段解释不同，是一个受输入 span 约束的预测问题。

🔑
统一视角：MLM、token classification、CRF、extractive QA 都把 encoder 输出接到不同 prediction head 上。它们的共同优势是输出空间受控；共同边界是不能自然生成输入中不存在的新文本。

11.6 评测：BLEU、ROUGE、EM/F1 与结构指标

评测指标必须匹配任务结构。翻译常用 BLEU，摘要常用 ROUGE，抽取式 QA 常用 Exact Match 和 token-level F1，dependency parsing 用 UAS/LAS。它们都不是“语言质量”的完整度量。

指标核心计算适用任务主要局限
BLEUclipped n-gram precision + brevity penalty机器翻译对同义改写和语义等价不敏感
ROUGE-L最长公共子序列的 precision/recall/F1摘要、生成比较偏向词面重合，不能保证事实一致
EM/F1答案规范化后 exact match 或 token overlap抽取式 QA、短答多答案、别名和推理型答案需要额外处理
UAS/LAShead 是否正确，label 是否也正确依存句法分析只评估句法结构，不等于下游任务成功

Worked Example：BLEU clipping 与 ROUGE-L

BLEU 不是简单数候选句里有多少词出现在参考句中。设 candidate 为 the the the the，reference 为 the cat is here。只看 unigram 时，candidate 中 the 出现 4 次，但 reference 中最多只出现 1 次，所以 clipped match 只能算 1 次：

BLEU-1 clipped precision

这一步防止模型靠重复高频词刷高 precision。若 candidate 是 cat sat，reference 是 cat sat on mat，unigram precision 为 1，但候选句长度 2 小于参考句长度 4，需要 brevity penalty：

Brevity penalty

因此短句不能因为每个词都命中就得到满分。ROUGE-L 关注最长公共子序列。若 candidate 为 a b c d，reference 为 a x b c，最长公共子序列是 a b c，长度为 3；precision = 3/4，recall = 3/4，F1 = 0.75。Ch11 作业中的 sentence_bleu 与 rouge_l_f1 正是把这些定义写成可运行函数。

Worked Example：QA 的 EM/F1 为什么要先规范化答案

抽取式 QA 常有多个 gold answer，评测时先做规范化：小写、去标点、去冠词，再压缩空格。若预测为 The quick brown fox，gold answers 为 quick brown fox 和 slow turtle，规范化后预测与第一个 gold 完全一致，因此 Exact Match = 1，token F1 也是 1。

若预测为 quick fox，gold 为 quick brown fox，则 token overlap 是 {quick, fox}，precision = 2/2 = 1，recall = 2/3，所以：

QA token F1

这个例子解释了为什么 EM 和 F1 要同时报告：EM 严格要求字符串完全匹配，F1 允许部分重叠。对于开放式、推理式或有别名的答案，仅靠一个 gold string 会低估正确输出；因此 QA 数据集通常提供多个 reference，并取最高 EM/F1。

因此评测报告不能只写“BLEU 高”或“ROUGE 高”。必须说明任务结构、候选输出、参考答案集合和指标局限，否则一个分数很容易被误读成完整的语言质量判断。

11.6A 现代 LLM 评测协议：从单一分数到可复现结论

BLEU、ROUGE、EM/F1 和 UAS/LAS 适合定义清楚的传统任务；现代 LLM 还要评估开放式回答、长上下文、工具调用、RAG、拒答、安全和多轮对话。这里最常见的错误，是把一个 benchmark 分数、一次 LLM-as-judge 胜率或一组人工偏好直接解释成“模型更好”。课程级评测应先定义任务、样本、推理设置和判定标准，再谈结论。

评测协议的最小组成

推理设置模型版本、prompt 模板、system prompt、temperature、top-p、max tokens、工具/RAG 配置两个模型使用不同 prompt 或采样参数，比较不公平

LLM-as-judge 不是免费人工评审

LLM-as-judge 可以降低开放式输出评测成本，但它有位置偏置、长度偏置、风格偏置和模型同源偏置。一个更稳妥的设置是 pairwise blind comparison：隐藏模型名，随机交换 A/B 顺序，让 judge 只根据 rubric 判断哪个回答更符合要求，并记录 tie。

Pairwise win rate with ties

若模型 A 对模型 B 的结果是 W=42、L=35、T=23，win rate 是 (42+11.5)/100=53.5%。这个数字不能单独证明 A 在生产中更好；还要看样本量是否足够、题目分布是否代表目标场景、失败类别是什么、延迟和成本是否变差。

安全评测要区分拒答和过度拒答

安全不是“拒答越多越好”。应至少把样本分成 harmful requests、benign sensitive requests 和普通任务，并分别统计：

  • Attack success rate：有害请求中模型给出不安全帮助的比例，越低越好。
  • Refusal rate：模型拒答的比例，需要按 harmful/benign 分开看。
  • Over-refusal rate：本应帮助的 benign 请求被错误拒答的比例。
  • Task utility：普通任务上的正确率、helpfulness 或 completion quality，防止安全优化牺牲可用性。

例如模型 X 在 harmful set 上拒答率 95%，但在 benign medical-summary set 上也拒答 60%，这不是单纯“更安全”，而是出现了严重 over-refusal。Ch09 的对齐评估和 Ch10 的上线 SLO 应与这里合并解读：安全、质量、延迟和成本必须同时过关。

把现代评测写成可运行函数

Ch11 作业把这一节落到三个函数上。summarize_pairwise_judgments 接收 blind pairwise 记录，计算 raw win rate、tie-adjusted win rate 和按任务分组的胜率，帮助学生看到同一个模型可能在 QA 胜出、在 math 持平或落后。safety_evaluation_metrics 要求把样本分成 harmful、benign sensitive 和 ordinary 三组，分别计算 attack success、harmful refusal、over-refusal 和 task utility。benchmark_result_summary 则把 task、sample size、metrics、model、prompt、temperature、失败类型和不可外推范围组织成结构化结论。

这三个函数的共同目的，是防止把“一个平均分”误读成完整工程结论。训练工程师需要用它判断对齐是否损害能力；推理工程师需要用它判断上线后质量、安全、延迟和成本是否同时满足要求。

Worked Example：同一个结果如何写成合格结论

不严谨写法：模型 A 比模型 B 好 8%。

更严谨写法：在 n=300 个中文客服问答样本上，固定 system prompt、temperature=0、max output=512，模型 A 相对模型 B 的 pairwise win rate 为 58%，主要提升来自格式遵循和引用完整性；事实错误率从 11% 降到 8%，但 P95 TTFT 从 720ms 升到 980ms，并且该结果没有覆盖英文、代码、多轮工具调用或高并发负载。因此它支持“在这个中文客服 RAG workload 下，A 值得进入灰度实验”，不支持“A 是通用更强模型”。

🔑
课程标准：评测答案不是分数列表，而是一个受条件约束的工程结论。你必须说明任务、样本、推理设置、指标、失败类型、统计不确定性和不可外推范围。

11.6B Benchmark Validity：污染、置信区间与消融

现代 LLM 评测的难点不只是“选哪个 benchmark”，而是判断这次分数能否支持工程决策。一个模型可能因为训练数据见过题目、prompt 更适配、输出更长、采样次数更多或 judge 偏好它的风格而获胜。课程级报告需要把这些因素拆开，否则 benchmark 会变成排行榜截图，而不是科学比较。

1. 数据污染和切分泄漏不是同一个问题

切分泄漏通常发生在自己的数据管线里：同一文档、同一用户会话、同一题库或近似改写同时进入 train/dev/test。它可以通过文档级切分、URL/题目 ID 去重、n-gram overlap、embedding 近邻检索和人工抽查降低风险。Benchmark contamination更广：公开测试题、答案解析、讨论帖或代码仓库可能已经进入预训练或 SFT 数据。它不一定能被完全排除，因此报告中要说明数据来源、公开时间、去重方法和高相似样本处理方式。

现象可能原因应做的检查结论边界
测试集分数远高于新题集benchmark 题目被训练见过，或 prompt 针对旧题优化按发布日期切分，查 exact/near duplicate，构造 held-out 新题旧 benchmark 只能作为回归测试，不能证明泛化
dev set 每次调参都提升，test 不升dev 被反复优化，逐渐变成训练信号冻结 final test，保留 shadow eval，只在里程碑运行dev 分数只代表开发进度，不代表最终能力
RAG 问答引用正确但答案错误检索命中不等于生成忠实，或 gold answer 过窄分开统计 retrieval recall、citation precision 和 answer accuracy不能用端到端 accuracy 直接定位瓶颈
LLM judge 偏好新模型但人工不认可长度、语气、格式或同源模型偏置随机 A/B 顺序，控制输出长度，抽样人工复核judge 胜率只是辅助信号

2. 小样本提升要给不确定性

如果模型 A 在 100 道题上答对 61 道，模型 B 答对 56 道，不能直接写“A 提升 5%”。这 5 个样本的差异可能来自题目抽样波动。最简单的近似是把 accuracy 当成二项比例，用标准误差估算置信区间：

Accuracy standard error

当 n=100、p=0.61 时，标准误差约 0.049，95% 区间大约是 61% ± 9.6%。这说明 61% 和 56% 在这个样本量下很难支撑强结论。若同一批题目同时评测两个模型，还应优先看 paired comparison：哪些题 A 对 B 错，哪些题 B 对 A 错，因为题目难度是共享的。开放式生成或 RAG 任务常用 bootstrap 对样本重采样，估计 win rate、accuracy 或 error rate 的区间。

3. 消融实验一次只回答一个问题

消融不是“多跑几个配置”。它要让一个设计变量的作用可解释。若同时把 base model、prompt、retriever、reranker、temperature 和 context length 都换掉，即使分数提升，也无法知道是谁贡献了提升。一个严谨的 LLM 系统消融通常按下面顺序组织：

问题保留不变只改变要同时报告
RAG 是否有用模型、prompt、解码参数、样本集是否注入检索上下文accuracy、citation precision、输入 token、P95 latency
reranker 是否有用retriever 候选、LLM、context budget候选排序策略Recall@k、nDCG、最终答案正确率、rerank 延迟
DPO 是否改善对齐SFT base、eval set、prompt、temperatureDPO checkpointhelpfulness、安全拒答、长度分布、KL 或行为漂移
更长推理是否值得模型、题目、评分规则max tokens、采样次数或 reasoning budgetpass@k、平均输出 token、成本、TTFT/TPOT

Worked Example：把 3% 提升改写成可判断结论

不充分写法：加入 reranker 后 benchmark 从 72% 提升到 75%，效果更好。

更严谨写法：在同一组 n=500 中文法规 QA 样本上，固定生成模型、prompt、temperature=0、max output=384 和 context budget=1800 tokens，只把 top-8 dense retrieval 改为 dense retrieval + cross-encoder rerank。端到端 accuracy 从 72.0% 到 75.0%，bootstrap 95% CI 的差值约为 [0.4%, 5.8%]；retrieval Recall@8 不变，但 citation precision 从 68% 到 79%；P95 latency 增加 140ms，平均输入 token 不变。这个结论支持“reranker 改善了该法规 QA workload 的上下文排序，值得灰度”，不支持“所有 RAG 任务都应加入 reranker”。

🔑
工程判断：可信 benchmark 结论至少要同时经受四个问题：数据是否干净，比较是否公平，差异是否稳定，消融是否能解释提升来源。少一个环节，分数都只能作为线索。

11.6C Safety & Social Impact：风险分类、触发条件与工程缓解

高校级 NLP/LLM 课程不能把 safety 当成上线前的附录。模型风险来自数据、训练目标、推理接口、应用场景和用户群体的组合；同一个模型在闲聊、医疗摘要、金融建议、教育辅导和代码执行中的错误成本完全不同。工程师需要能把“模型可能有害”拆成可讨论、可测量、可缓解的风险类别。

1. 风险不是一个维度

风险类别典型触发条件工程上先看什么常见缓解

2. 安全评估要按场景写，而不是按口号写

“模型安全”不能脱离应用场景。医疗问答的核心风险是误导性建议、隐私和不确定性表达；代码 Agent 的核心风险是执行权限、供应链和数据外传；教育辅导的核心风险是错误反馈、过度代写和学习依赖。课程项目中应先写清目标场景、用户、模型权限和错误成本，再选择样本和指标。

场景必须覆盖的样本切片不充分的评估方式
医疗科普问答普通科普、紧急症状、用药、隐私信息、应建议就医的问题只看回答是否流畅或是否拒答
企业 RAG权限内文档、权限外文档、过期文档、冲突文档、引用缺失只看最终答案正确率，不看权限和引用
代码 Agent只读任务、写文件任务、联网任务、危险命令、依赖安装只看 pass rate，不看工具调用边界
多模态文档理解小字、表格、签名、人脸、身份证件、图表数值用普通 VQA 分数代表文档安全性

Worked Example：RAG 医疗问答的风险分解

假设团队要上线一个面向患者的医疗科普 RAG。一个不充分的方案是只测“回答是否看起来专业”。更合理的工程分解是：

  1. 检索风险：相关指南是否被召回，过期文档是否进入 context，引用是否指向正确段落。
  2. 生成风险：模型是否把检索内容外推成诊断结论，是否编造药量，是否在缺少信息时给确定建议。
  3. 隐私风险：用户输入和日志是否包含姓名、电话、病历号或可识别信息。
  4. 拒答边界：普通科普应回答，急症、自我伤害、处方调整等请求应转向专业帮助。
  5. 可用性：安全策略不能把所有医学问题都拒掉，否则系统在真实用户任务中不可用。

因此该系统至少应报告 retrieval recall、引用准确率、事实错误率、有害建议率、普通医学科普完成率、急症转介率、PII 日志处理方式和用户界面中的限制说明。这里的重点不是制造更多表格，而是让模型行为、数据来源、用户风险和部署边界能被同一套工程语言描述。

3. Safety 与 alignment、evaluation、serving 是同一条链路

第 9 章讨论的是如何用 SFT/DPO/GRPO 改变模型行为，第 10 章讨论的是如何把模型变成可服务系统，本节讨论的是这些行为在真实场景中的风险边界。一个模型可以在 harmlessness 上提升，同时在良性敏感问题上过度拒答；也可以在离线 safety set 上表现好，却因为工具权限过大而在 Agent 场景中失败。课程项目应把安全切片放进与质量、延迟、成本并列的结果表，而不是在结论里只写一句“已考虑安全”。

🔑
核心判断：社会影响不是抽象道德讨论，也不是单一安全分数。LLM 工程师要能把风险映射到数据、目标函数、接口权限、用户场景和评测切片，并说明每个缓解动作会牺牲什么能力、延迟或成本。

11.7 任务选型：什么时候不用 Decoder-only LLM

选择模型族时先看输出空间和错误成本：

  • 固定标签分类：encoder-only fine-tuning 或小模型分类器通常更便宜、更稳定。
  • RAG reranking：cross-encoder 可以同时看 query 和 document，给候选排序；decoder-only LLM rerank 成本高且格式不稳定。
  • 抽取式答案：若答案必须来自文档原文，span QA 比自由生成更可控。
  • 结构输出：dependency parser、NER、slot filling 等任务可用结构化 head 或 constrained decoding，不应默认开放式生成。
  • 开放式生成与推理：当输出需要新文本、代码、解释或多步 reasoning，decoder-only LLM 更自然。
⚠️
常见错误：把 BERT 和 GPT 说成“只是 mask 不同”。它们的 attention 可见性、训练目标、fine-tuning head、输出空间和服务成本都不同。

11.7A 工程选型案例：把任务拆成训练目标、输出空间和服务成本

LLM 工程师经常面对的问题不是“哪个模型最大”，而是“这个任务应该被建模成分类、抽取、排序、结构化预测还是开放式生成”。同一个业务目标可以有多种建模方式，训练数据、推理路径、评测指标和上线成本都会随之变化。

案例 1：客服知识库问答

如果业务要求“答案必须引用知识库原文”，直接让 decoder-only LLM 自由回答风险较高。更稳的方案通常是三段式：bi-encoder 召回候选文档，cross-encoder reranker 精排，extractive QA 或受约束生成给出答案 span 与引用。只有当问题需要跨段综合、改写或解释时，再让 decoder-only LLM 基于检索上下文生成自然语言回答。

这类系统的训练目标也分层：retriever 用对比学习或点击数据，reranker 用相关性排序标签，QA head 用 span label，生成器用 SFT/偏好优化。把所有错误都归因于“LLM 不够强”会掩盖真正瓶颈：可能是召回没命中、重排错序、chunk 切分破坏答案、prompt 没约束引用，或生成阶段没有忠实使用上下文。

案例 2：合同字段抽取

合同中抽取甲方、乙方、金额、日期等字段，输出空间是有限结构。若用开放式生成，模型可能改写金额格式、补全不存在字段或把多个日期混在一起。更合适的 baseline 是 encoder-only token classification 或 span classification，再用规则校验金额、日期和实体类型。

训练工程上要关注 span-level F1，而不是 token accuracy。一个日期实体 2026 年 6 月 8 日 少预测一天，token accuracy 可能仍然很高，但字段抽取是错的。推理工程上要关注批量吞吐和长文档切片：如果每份合同几十页，cross-encoder 全文处理成本可能不可接受，需要先分块召回候选段落，再做精抽取。

案例 3：开放式推理回答

数学题、代码修复和复杂规划通常不能被固定标签或 span 覆盖，需要 decoder-only LLM 生成中间推理和最终答案。此时 Classic NLP 指标不再充分，BLEU/ROUGE 可能惩罚合理改写，也可能奖励错误但词面相似的答案。更合适的是任务成功率、unit test/pass@k、self-consistency、人工或 LLM-as-judge 分层评测，以及第 10 章的延迟/成本指标。

🔑
工程判断：模型族选择要从输出约束倒推。固定标签优先分类，输入原文答案优先抽取，结构化字段优先 span/CRF/约束解码，开放式解释和推理才优先 decoder-only 生成。这样训练目标、评测指标和推理成本才能彼此一致。

11.8 Capstone 项目报告：从实验结果到可判定结论

CS224N 这样的高校课程会把 final project 看成一次小型研究或工程论文训练：学生不仅要实现系统，还要说明问题、方法、实验、失败和边界。本课程的训练工程与推理工程 capstone 也应达到同样标准。最终项目报告的核心不是“我跑通了脚本”，而是回答一个明确问题，并让读者能判断这个结论在什么条件下成立。

1. 把项目问题写成可检验命题

好的项目问题应该能被数据和实验反驳。例如：

  • 训练工程：在固定 token budget 下，seq_len=128 是否比 seq_len=64 降低 validation PPL？代价是多少 tokens/s 和 step time？
  • 对齐工程：DPO 是否提高偏好胜率，同时保持基础任务正确率和拒答边界？KL、输出长度和失败类型如何变化？
  • 推理工程：RAG top-k 从 3 增到 8 是否提高 pass rate？它对 P95 TTFT、prompt tokens、context packing miss 和成本有什么影响？
  • 服务工程：concurrency 从 1 到 8 时，P95/P99 是否仍满足 SLO？瓶颈是排队、prefill、decode、检索还是后处理？

这些问题都有 baseline、改动项、指标和结论边界。相比之下，“做一个聊天机器人”或“训练一个小模型”不是合格的课程项目问题，因为它无法说明什么改变带来了什么效果。

2. 报告结构应覆盖建模、系统和评测

3. 负结果也可以是强结论

高校课程项目不要求每个方法都“提升”。如果一个 ablation 没有带来显著改善，只要实验设计清楚，它仍然是有价值的结论。例如：seq_len=128 的 validation PPL 只降低 0.03，但 tokens/s 下降 35%，且三次 seed 波动约 0.05，那么合理结论是“当前设置不足以支持更长序列值得采用”。这比只挑一个最好数字更严谨。

4. 把结论绑定到复现命令和限制条件

报告最后必须说明复现路径：数据版本、commit、运行命令、seed、硬件、模型配置、checkpoint 或服务 URL、评测集和压测参数。也必须说明不能外推的地方：tiny corpus 不能代表万亿 token 训练，本地 MockEngine 不能代表真实 GPU serving，小知识库 RAG 不能代表企业知识库，单一安全集合不能代表所有用户风险。

最终报告的读者应该能回答三件事：第一，项目提出的问题是否清楚；第二，实验是否足以支持结论；第三，如果把系统推向更大模型、更大数据或真实服务，哪些假设会最先失效。

🔑
核心判断：Capstone 是课程能力的收束点。训练项目要证明训练过程可复现、可恢复、可解释；推理项目要证明服务质量、延迟、成本和风险在目标 workload 下可判断。没有 baseline、ablation、失败分析和结论边界的项目，只是 demo，不是高校课程级项目。

11.10 小结

  • RNN/LSTM 解释了顺序递归、BPTT 和长距离依赖问题。
  • Dependency parsing 是离散结构预测，不等于 attention heatmap。
  • Seq2Seq attention 把源序列和目标前缀分开建模，是 encoder-decoder 的基本形式。
  • BERT/encoder-only 适合分类、reranking、embedding、token classification 和抽取式 QA。
  • BLEU、ROUGE、EM/F1、span F1、UAS/LAS 各自衡量不同任务结构；现代 LLM 评测还要说明数据、推理设置、judge 偏置、安全拒答、统计不确定性和结论边界。
