8.1 自回归生成的数学原理

在第七章训练之后，我们得到了一个能够预测下一个 token 的模型。然而，模型本身只是一个“评分器”：给定前缀，它输出一个概率分布。如何将这个概率分布转化为实际的文本序列，是文本生成的核心问题。

自回归分解

生成的数学基础仍然是自回归分解——序列的联合概率被分解为条件概率的乘积：

公式 8.1 自回归生成

在每个步骤中，模型基于已有的前缀 ，输出一个词表上的概率分布 。然后，通过从该分布中采样或取最大值得到 。这个过程逐个 token 进行，直到生成结束 token 或达到最大长度。

两个阶段：预填充和解码

生成过程分为两个计算特征完全不同的阶段：

  • 预填充：用户的输入 prompt 一次性送入模型。这是一个计算密集型阶段——大量矩阵乘法并行执行。瓶颈是 GPU 算力（FLOPS）。首 token 时间（TTFT）由该阶段主导。
  • 解码： token 逐个生成。每步只输入一个 token，利用 KV Cache 进行增量计算。这通常是带宽密集型阶段——在小批量或低并发下，GPU 计算利用率可能显著低于预填充阶段。后续吞吐量（TPS/TPOT）由该阶段主导。
🔑
重要概念： TTFT（Time to First Token）主要受预填充、排队、网络和路由影响；用户的体验是“等待第一个字符出现的时间”。TPS/TPOT 主要受解码、批处理、KV cache 和带宽影响；用户的体验是“打字速度”。不同模型、上下文长度、并发级别和服务提供商之间数值差异很大。可复现的生成基准测试应在固定的模型、prompt 集合、上下文长度、并发级别和服务配置下报告这些指标。

8.2 为什么贪心解码会失败——重复陷阱与曝光偏差

贪心解码每一步都选择概率最高的 token：。这是最简单的策略，但有两个根本缺陷。

缺陷 1：重复循环

一旦模型进入一个高概率的局部模式，后续的 argmax 步骤可能会持续选择相似或相同的 token。这在训练数据中的常见短语、标点或功能词附近尤其明显。原因通常是多种因素的组合：过于尖锐的条件分布、上下文中已存在的重复模式、没有随机探索以及弱停止约束。这些因素可能会使贪心解码陷入重复循环；不应将其简化为单个 Key/Query 自强化故事。

缺陷 2：曝光偏差

训练时，模型每一步看到的是真实的真实 token。然而推理时，模型看到的是自己生成的 token。这两种情境下的分布不同——训练时的条件分布是 ，而推理时是 。某一步的误差会在后续步中被放大；这就是误差累积。

温度参数的熵控制视角

一个常见的补救方法是引入随机性：从概率分布中采样，而不是总是取最大值。温度参数 控制这种随机性的程度：

公式 8.2 温度采样
  • ：退化为 argmax（完全确定性）
  • ：原始分布
  • ：退化为均匀分布（完全随机）

温度的"热力学类比"：在玻尔兹曼分布中，温度控制着系统的"探索"程度——高温度使系统探索更多状态（更丰富的词汇），低温度使系统利用已知的高概率模式（输出更确定）。

8.3 编程练习一：实现贪心解码

尽管贪心解码有缺陷，但它是理解所有其他采样策略的起点。实现它，然后观察其问题。

练习一：贪心解码

任务：实现 generate_greedy(model, input_ids, max_new_tokens) 函数。

要求：

  • 输入 input_ids: (1, seq_len)，逐个生成最多 max_new_tokens 个新 token
  • 每一步取最后一个位置的 logits 的 argmax（logits[:, -1, :]）
  • 检测 EOS token，遇到时提前停止
  • 返回完整的 token 序列
# Expected usage
prompt = tokenizer.encode("Deep learning is", return_tensors='pt')
output = generate_greedy(model, prompt, max_new_tokens=50)
print(tokenizer.decode(output[0]))
# Possible output: "Deep learning is deep learning is deep learning is..."
def generate_greedy(model, input_ids, max_new_tokens=100): """Greedy decoding: select the token with the highest probability at each step""" model.eval() device = next(model.parameters()).device input_ids = input_ids.to(device) for _ in range(max_new_tokens): with torch.no_grad(): logits = model(input_ids) # Take the logits at the last position next_logits = logits[:, -1, :] # (1, vocab_size) next_token = next_logits.argmax(dim=-1, keepdim=True) # (1, 1) input_ids = torch.cat([input_ids, next_token], dim=-1) if next_token.item() == model.tokenizer.eos_token_id: break return input_ids

💡 观察：即使在小型模型上，贪心解码也极易陷入重复。注意 -1 中的 logits[:, -1, :] —— 我们取“最后一个位置”的 logits，即模型对 下一个 token 的预测。

8.4 编程练习 2：实现 Temperature Sampling

Temperature sampling 通过对 logits 施加除法 来控制分布的"尖锐度"。核心变化是：argmax → softmax + multinomial。

练习 2：Temperature Sampling

任务：实现 generate_temperature(model, input_ids, max_new_tokens, temperature)。

要求：

  • 在应用 softmax 之前，将 logits 除以 temperature
  • 使用 torch.multinomial 从分布中采样
  • 处理 的特殊情况：退化为 argmax（避免除以零）
  • 在相同 prompt 下多次运行应产生不同结果（由于随机采样）
# Expected usage
prompt = tokenizer.encode("The future of artificial intelligence", return_tensors='pt')

# Low temperature: high determinism
output1 = generate_temperature(model, prompt, 100, temperature=0.2)
# High temperature: high diversity
output2 = generate_temperature(model, prompt, 100, temperature=1.5)

# Different runs should be different
output3 = generate_temperature(model, prompt, 100, temperature=0.8)
output4 = generate_temperature(model, prompt, 100, temperature=0.8)
assert output3.tolist() != output4.tolist()
def generate_temperature(model, input_ids, max_new_tokens=100, temperature=1.0): """Temperature sampling: use temperature to control the sharpness of the distribution""" model.eval() device = next(model.parameters()).device input_ids = input_ids.to(device) for _ in range(max_new_tokens): with torch.no_grad(): logits = model(input_ids) next_logits = logits[:, -1, :] # (1, vocab_size) # T=0 degrades to greedy if temperature == 0: next_token = next_logits.argmax(dim=-1, keepdim=True) else: scaled = next_logits / temperature probs = torch.softmax(scaled, dim=-1) next_token = torch.multinomial(probs, num_samples=1) input_ids = torch.cat([input_ids, next_token], dim=-1) if next_token.item() == model.tokenizer.eos_token_id: break return input_ids

💡 处理 T=0：当 T→0 时， 可能导致数值溢出。实际上，我们从不除以真正的 0——在 T=0 时直接使用 argmax，仅在 T>0 时进行采样。

8.5 从 Temperature 到 Top-K / Top-P：截断低概率尾部

Temperature 采样有帮助，但词汇量大的情况下，softmax 尾部仍然有一长串概率极低的 token。即便使用 ，模型仍可能采样到低质量或离题的 token。Top-K 和 Top-P 通过截断低概率尾部来降低这种风险。

Top-K 的逻辑

只保留概率最高的 K 个 token，重新归一化，然后从这个集合中采样。K 是一个固定值（GPT-2 默认 K=40）。

Top-K 的固有问题

K 是固定的。在某些上下文中，可能只有 3 个合理的选择（K=40 引入了太多噪声）；而在另一些上下文中，可能有 100 个合理的延续（K=40 截断了太多可能性）。例如：

  • "美国的首都是____" — 分布极其尖锐；"Washington" 的概率可能超过 90%。使用 K=40 时，你仍从 40 个 token 中采样，有很小概率选到 "New York"（明显错误）
  • "为她写一首关于____的诗" — 分布非常平坦；K=40 可能对创造性限制过强

Top-P（Nucleus）的自适应方案

Top-P（Holtzman 等人，2020）不固定 K，而是选择累积概率达到 P 的最小 token 集合：

  1. 将 token 按概率从高到低排序
  2. 从最高概率开始累加，直到累积概率 （典型值 0.9 或 0.95）
  3. 只从这个“核心集”中采样

当分布尖锐时，核心集很小（"Washington" 例子可能只有 1-3 个 token）；当分布平坦时，核心集会自动扩大。它不需要手动固定 K，是开放文本生成中常用的采样策略之一；实际部署时仍需结合 temperature、重复惩罚、停止条件和任务约束来调整参数。

示例：同一分布下 Top-K 与 Top-P 的候选集对比

假设在某生成步骤中只有 6 个候选 token，logits 为 [4, 3, 2, 1, 0, -1]。使用 temperature T=1 后，softmax 概率近似为：

logit43210-1
probability0.6340.2330.0860.0320.0120.004
cumulative0.6340.8670.9530.9850.9961.000

top-k=3 总是保留 A,B,C，然后重新归一化得到近似 [0.718, 0.264, 0.097]。top-p=0.9 从最高概率开始累加，直到累积概率超过 0.9；这里 A+B=0.867 不够，加上 C 后达到 0.953，所以候选集也是 A,B,C。如果改为 top-p=0.8，则候选集只需要 A,B。

关键区别不在于这个单独例子，而在于分布形状变化时的行为。如果分布非常尖锐，例如 [0.92, 0.04, 0.02, 0.01, 0.01]，top-p=0.9 只保留概率最高的 token；如果分布平坦，例如 [0.22, 0.20, 0.18, 0.16, 0.14, 0.10]，top-p=0.9 需要保留前 5 个 token。固定的 top-k 不能根据上下文不确定性自动调整候选集大小。

8.5A 采样分布记录：从原始logits到最终token

实际的生成系统不会直接对原始logits做softmax。一个生成步骤通常先应用多种变换：重复惩罚、有效token约束、temperature、top-k/top-p截断，然后才重新归一化和采样。顺序写错会改变最终分布，也会使logprob、entropy和调参结论失去意义。

1. 推荐的处理顺序

一个可复现的生成策略应该使用明确声明的处理顺序。实际系统可能有更多处理器，但每一步都应该显式写入生成策略：

raw logits -> repetition penalty / frequency penalty -> token constraints or grammar mask -> temperature scaling -> top-k or top-p filtering -> softmax over remaining tokens -> sample or argmax

重复惩罚应在截断前应用；否则，已出现的token可能先进入top-k/top-p候选集，惩罚就不能及时生效。约束mask应在归一化前应用，将非法token的logits设为-inf；后续的softmax只在有效候选中分配概率。Temperature改变分布的尖锐度，而top-k/top-p改变候选集；它们不是同一类操作。

2. 实例：同一步骤logits的处理轨迹

假设当前词表只有5个token，原始logits为：

token: A B C D E raw logits: 4.0 3.0 2.0 1.0 -1.0

A 在历史中出现过，使用重复惩罚=2.0；当前语法只允许 A/B/C；temperature=1.0；top-p=0.8。处理轨迹可记为：

阶段ABCDE描述
原始logits4.03.02.01.0-1.0模型原始输出
重复惩罚2.03.02.01.0-1.0A的logit为正，除以惩罚
约束mask2.03.02.0-inf-infD/E当前非法
top-p=0.82.03.02.0-inf-infB的概率最高，加上A或C达到阈值

最终概率不是原始logits的softmax，而是经过处理的logits在保留候选上的softmax。如果top-p在概率相等时保留 B, A，则候选概率约为 B=0.731, A=0.269；如果实现保留 B, C，概率同构但候选token不同。因此，稳定的平局处理、token id顺序以及报告候选集都很重要。

3. logprob、entropy和候选集必须绑定到同一分布

生成API常常返回 logprobs、top token列表或entropy。必须说明这些量来自哪个分布：

  • 原始模型logprob：无约束、未截断，适合比较模型自身对token的偏好。
  • 处理后logprob：经过惩罚、mask、temperature和截断后的概率，适合解释本次实际采样。
  • entropy：如果在截断后计算，它只代表候选集内的不确定性，不能直接与原始全词表entropy比较。
  • 选中token：token id、文本片段、处理后概率以及是否受约束影响都应报告。

如果一个系统声称“模型对这个答案很有信心”，但引用的是top-p截断和JSON语法mask之后的概率，这个说法是不严谨的。约束分布中的高概率仅表示“它在有效集内可能性最大”，并不等同于模型在开放词表上信心很高。

4. 调参时关注什么

调参temperature、top-p和重复惩罚时，不要仅凭少数几个生成文本做判断。至少同时记录以下内容：

kept_token_count / vocab_size selected_token_probability entropy_after_filtering repetition rate or distinct-n stop reason: eos / stop string / max_new_tokens / constraint failure generated tokens and latency

这些量有助于区分几种问题类型：枯燥的输出可能因为温度过低或候选集太小；跑题的输出可能因为温度过高或top-p过宽；重复可能需要重复惩罚、停止条件或训练数据处理；格式错误更适合用约束解码而不是继续降低温度。

🔑
本节要点： 生成并不是“softmax后采样”那么简单。严谨的生成实验必须报告从原始logits到处理分布的变换链；否则实验无法复现，也无法判断质量变化来自模型、采样策略还是约束系统。

8.6 编程练习3-4：实现Top-K和Top-P采样

练习3：Top-K采样

任务：实现 generate_topk(model, input_ids, max_new_tokens, k, temperature)。

要求：

  • 使用 torch.topk 取概率最高的K个logits
  • 对Top-K子集进行softmax归一化
  • 采样后，使用 gather 将索引映射回原始词汇表ID
# Expected usage
prompt = tokenizer.encode("Tell me a story", return_tensors='pt')
output = generate_topk(model, prompt, 100, k=40, temperature=0.8)
def generate_topk(model, input_ids, max_new_tokens=100, k=40, temperature=1.0): """Top-K采样：仅保留概率最高的K个token""" model.eval() device = next(model.parameters()).device input_ids = input_ids.to(device) for _ in range(max_new_tokens): with torch.no_grad(): logits = model(input_ids) next_logits = logits[:, -1, :] / temperature # 取top-k top_logits, top_indices = torch.topk(next_logits, k, dim=-1) # 在top-k子集上采样 top_probs = torch.softmax(top_logits, dim=-1) sampled_idx = torch.multinomial(top_probs, num_samples=1) # 映射回原始词汇表ID next_token = top_indices.gather(-1, sampled_idx) input_ids = torch.cat([input_ids, next_token], dim=-1) if next_token.item() == model.tokenizer.eos_token_id: break return input_ids

练习4：Top-P（Nucleus）采样

任务：实现 generate_topp(model, input_ids, max_new_tokens, p, temperature)。

要求：

  • 将概率分布按降序排序，计算累积概率
  • 关键：首先使用 cumsum >= p 标记超过nucleus的位置，然后将掩码右移一位——确保第一个达到或超过p的token被保留，并且nucleus尽可能小
  • 掩码后，重新归一化剩余概率（确保它们之和为1）
  • 从归一化后的分布中采样
# Expected usage
prompt = tokenizer.encode("Tell me a story", return_tensors='pt')
output = generate_topp(model, prompt, 100, p=0.9, temperature=0.8)
def generate_topp(model, input_ids, max_new_tokens=100, p=0.9, temperature=1.0): """Top-P（Nucleus）采样：选择累积概率达到p的最小token集""" model.eval() device = next(model.parameters()).device input_ids = input_ids.to(device) for _ in range(max_new_tokens): with torch.no_grad(): logits = model(input_ids) next_logits = logits[:, -1, :] / temperature probs = torch.softmax(next_logits, dim=-1) # 按降序排序 sorted_probs, sorted_indices = torch.sort(probs, descending=True, dim=-1) # 计算累积概率，掩码尾部 cumsum = torch.cumsum(sorted_probs, dim=-1) mask = cumsum >= p mask[..., 1:] = mask[..., :-1].clone() mask[..., 0] = False sorted_probs = sorted_probs.masked_fill(mask, 0.0) # 重新归一化 renormalized = sorted_probs / sorted_probs.sum(dim=-1, keepdim=True) # 采样 sampled_idx = torch.multinomial(renormalized, num_samples=1) next_token = sorted_indices.gather(-1, sampled_idx) input_ids = torch.cat([input_ids, next_token], dim=-1) if next_token.item() == model.tokenizer.eos_token_id: break return input_ids

💡 为什么要右移掩码？ cumsum >= p 标记第一个达到阈值的token；将掩码右移一位后，该token被保留，后续的尾部token被去掉。这样就得到了累积概率达到p的最小nucleus。

8.7A 如何评估生成策略：多样性不等于质量，低重复率不等于正确性

在比较解码策略时，最常见的错误是只看一两个例子，或者只报告 distinct-n。生成质量是一个多目标问题：同一策略可能在提高多样性的同时降低事实性、格式稳定性或可复现性。严格的实验需要固定 任务类型、采样参数、随机种子、输出长度和评估指标。

为什么要进行参数扫描

Temperature、top-k 和 top-p 不是独立的开关。较高的 temperature 通常增加熵；较低的 top-p 截断尾部；它们的组合可能相互抵消或放大。合理的实验应在同一提示集上比较多组参数，例如：

temperature in {0.2, 0.7, 1.0} top_p in {0.8, 0.9, 0.95} max_tokens fixed seeds at least 3 metrics distinct-2, repetition rate, task score, latency

如果结论仅来自一个提示、一个种子和一个样本输出，通常不足以确定策略优劣。对于开放式生成，还应保存失败案例：重复循环、提前停止、跑题、事实错误、格式损坏和过于保守都是不同的失败模式。

重复惩罚的作用

重复惩罚不是新的概率分布也不是训练目标；它是在 softmax 和采样之前应用于 logits 的一个处理器。常见规则是：对于前缀中已出现的 token，如果 logit 为正，则除以惩罚值；如果 logit 为负，则乘以惩罚值，从而相对降低这些 token 的概率。正确的实现应在贪心、top-k 或 top-p 选择之前应用此变换，然后验证它确实改变了候选排名。

工作示例：从 Logits 处理器到最终采样分布

给定 logits [4.0, 3.0, -1.0]，生成的 token ids 为 [0,2]，重复惩罚为 2.0。首先，处理重复 token：token 0 的正 logit 从 4.0 变为 2.0，token 2 的负 logit 从 -1.0 变为 -2.0，得到 [2.0,3.0,-2.0]。如果随后使用 top_p=0.8，经过 softmax 后，token 1 的概率约为 0.721，尚未达到 0.8；加上 token 0 后累积概率约为 0.986，因此核心词为 tokens {1,0}，token 2 被截断。

在最终采样之前，对保留集重新归一化：token 0 和 token 1 的概率约为 [0.269, 0.731]，熵约为 0.582，概率最高的 token 是 1。一个有用的分布记录表会记录处理后的 logits、保留的候选集、归一化概率、熵和当前最高概率 token；这将“采样参数改变输出”的陈述变成可检查的概率迹。

策略选择的实用判断

  • 代码、数学、抽取：偏好低温度或 beam / pass@k；多样性不是主要目标，正确性和可复现性更重要。
  • 创意写作、头脑风暴：可以提高 temperature/top-p，但必须监控重复、话题漂移和安全边界。
  • 结构化输出：采样策略不能替代 schema 约束；优先使用结构化解码或输出验证。
  • 线上服务：需要同时考虑质量和 TPOT/吞吐量；高 top-p 可能增加输出长度和后处理失败率。
🔑
如何撰写实验结论：不要写“top-p 最好”。相反，写“在特定的提示集、长度限制、temperature、top-p、随机种子和指标下，top-p=0.9 相比贪心降低了重复率，但增加了输出方差；人工检查显示一些样本更加发散。”

8.7B 束搜索：搜索多个高概率候选，而非仅取单步最佳

贪心解码只保留每一步概率最高的 token，容易在早期陷入局部最优。束搜索保留 num_beams 条候选路径。每一步，将每条路径扩展出几个下一个 token，然后按累积对数概率排序，保留前几个。

长度偏置

直接累积对数概率会天然偏向更短的序列，因为每多生成一个 token 就增加一个非正的 log 概率。在机器翻译、摘要和结构化候选生成中，通常使用长度归一化或长度惩罚来缓解这一问题：

长度归一化束评分

length_normalized_score 和 beam_search 背后的概念对应这一过程。一个紧凑的 batch-size-1 实现足以展示束表、原始得分和归一化得分之间的关系，然后再引入生产级批处理和早停策略。

示例：两步束搜索与长度归一化

设 num_beams=2。第一步，从提示词扩展出两个候选：A 的对数概率为 -0.2，EOS 的对数概率为 -0.4。如果允许早停，EOS 是一个长度为 1 的完整短答案；A 需要继续生成。

步候选新 logp累积 logp长度alpha=1 归一化得分
1EOS-0.4-0.41-0.400
1A-0.2-0.21-0.200
2A B-0.3-0.52-0.250
2A C-1.0-1.22-0.600

如果只考虑累积 logp，短答案 EOS 的 -0.4 高于 A B 的 -0.5，因此束可能会偏向过短的输出。使用 s(y)=logp/|y| 后，A B 的得分为 -0.25，优于 EOS 的 -0.4。这就是长度惩罚的作用：它不是无条件奖励长句，而是减少“每增加一个 token 就多加一个负数”导致的短句偏置。

何时使用束搜索

  • 适用场景：目标明确的任务，如翻译、抽取、代码候选、短数学答案、受模式约束的输出。
  • 不可盲目使用：在开放式聊天和创意写作中，束搜索可能产生高概率但平庸、重复或过短的输出。
  • 必须报告：束大小、长度惩罚、最大长度、是否使用早停，以及候选排序指标。
🔑
边界说明：束搜索是一种搜索策略，不是训练目标，也不能保证事实正确性。它能更系统地探索高概率区域，但如果模型本身的概率倾向于错误答案，束搜索只会更可靠地找到高概率错误。

8.7C 推理生成：用更多测试时计算换取更高准确率

对于数学、代码、规划以及多跳问答，生成策略并不仅仅是"选择哪个 token"。模型可能先生成中间推理步骤，然后再选出最终答案。近期的推理方法可以统一在同一个问题下：训练完成后，你是否愿意用更多的测试时计算（test-time compute）来换取更高的任务成功率？

常见方法

方法做法收益来源成本/风险
思维链（Chain-of-Thought）让模型在给出答案前先写出中间步骤将隐式计算展开为 token 序列步骤看似合理，但可能事实上错误
自一致性（Self-consistency）采样多条推理路径，对最终答案进行投票降低单条路径的偶然错误成本随采样数量近似线性增长；投票仅适用于可聚合的答案
Best-of-N生成 N 个候选，用奖励模型或规则筛选将生成转化为搜索与重排重排器偏差可能选出风格好但事实错误的答案
验证器/评判者（Verifier / Critic）训练或调用独立模型判断解答是否正确将"求解"与"评判"分离验证器覆盖不足可能放过类似错误
过程监督（Process supervision）评估每个推理步骤而非仅最终答案更容易定位中间错误标注成本高；步长粒度难以定义

为何与普通采样不同

普通的 top-p 或 temperature 主要控制输出分布的熵；而推理生成关心的是搜索空间。同一个问题可以生成多条推理路径，每条路径都有自己的中间状态和最终答案。对于有标准答案的任务，评估重点不是 distinct-n，而是 pass@k、多数投票准确率、验证器准确率、平均输出 token 数，以及每个正确答案的成本。推理评估应使用 1 - C(n-c,k)/C(n,k) 来估计从 k 个样本中抽取 n 个时至少有一个正确答案的概率；还应定义最终答案提取器，将多个候选输出聚合为投票分布，并同时报告总生成 token 数和平均 token 数。

实例讲解：pass@k 不是 c/n

假设对于同一个编程问题，我们采样了 n=10 个候选，其中有 c=2 个通过了测试。如果看单次采样成功率，它是 c/n = 0.2。但 pass@3 问的是：如果从这 10 个中随机抽 3 个提交，只要至少一个是正确的就算成功。失败事件是抽到的 3 个全部错误：

pass@3 对于 n=10, c=2

因此，对于同一组候选，pass@1 是 0.2，pass@3 约为 0.533。这说明 pass@k 衡量的是"在给定 k 次尝试预算下至少命中一次的概率"，而不是单次采样的平均准确率。在报告推理实验时，必须明确说明 n、k、采样参数以及平均生成 token 数；否则，准确率的提升可能仅仅是因为花费了更多的测试时计算。

还有第二个边界：最终答案的 pass@k 并不能证明推理路径是正确的。一个候选可能通过了单元测试、匹配了数值答案、或被多数投票选中，但其思维链却是无效的。因此，最近关于 RLVR 评估的工作认为，当任务要求可信推理时，既要检查最终答案也要检查推理轨迹。在课程报告中，请分别报告答案正确率、轨迹有效性以及每正确且有效答案的成本；不要仅凭更高的 pass@k 就说模型学会了更好的推理。

如何报告计算预算

如果某种方法将准确率从 60% 提升到 72%，但每个问题的生成次数从 1 次增加到 32 次，你不能只报告准确率。你至少应该报告：

  • 采样数量： N、temperature、top-p、最大推理 token 数。
  • 准确率： 单采样准确率、pass@k、多数投票准确率或重排后的 top-1 准确率。
  • 成本： 平均生成 token 数、TTFT/TPOT、每问题延迟、每问题 token 成本。
  • 失败类型： 算术错误、遗漏条件、无效代码、错误投票、验证器误判。
🔑
工程要点： 推理能力并不是"输出更长的解释"。严谨的实验必须证明更多的中间 token 或更多候选确实能提升任务成功率，并且必须说明消耗了多少测试时计算。

8.7D 思考模式、思考预算与运行时路由

最近的推理模型已将“投入多少思考”变为一个推理时的控制变量，而不是模型的固定属性。Qwen3技术报告描述了一个统一的框架，同时包含思考模式（thinking mode）和非思考模式（non-thinking mode），外加一个思考预算（thinking budget）机制，该机制根据任务复杂度分配更多或更少的生成推理token。对工程师而言，这应视为一个运行时策略问题：模型、提示模板、最大推理token数、候选数量、验证器、延迟目标和回退路径共同定义了用户可见的行为。

8.7E 推理控制器：Effort、隐藏 Token、不完整输出与工具延续

当推理成为请求级别的控制时，生成系统需要一个控制器，而不是固定的解码方案。OpenAI 的 推理模型文档 将其具体化：reasoning.effort 在速度与 token 使用量之间权衡更深度的推理，隐藏推理 token 占用上下文并按输出 token 计费，如果在产生可见输出之前达到 token 上限，则响应可能不完整。Qwen3 的 thinking-budget 设计从开源模型的角度指向相同的工程模式：模型可能支持多种推理模式，但产品必须决定何时每种模式值得其成本。

1. Effort 等级是一个评估旋钮，而非质量保证

只有在代表性评估显示出可衡量的改进，足以证明延迟和成本合理时，增加 effort 才有用。低 effort 更适合短工具计划、提取和客服路由；中等 effort 通常是平衡的起点；高或超高 effort 应保留给困难推理、长期研究、调试、安全审查或异步智能体工作流。如果提示词的成功标准不明确、停止规则薄弱或工具访问不受限制，更多推理会导致过度思考、不必要的搜索和输出退化。

2. 隐藏推理 token 依然是容量和成本

推理 token 可能在最终答案中不可见，但它们仍然消耗输出预算和上下文空间。这造成两个工程陷阱。首先，max_output_tokens 必须同时覆盖隐藏推理和可见答案 token；否则模型可能将整个预算用于思考并返回不完整的响应。其次，长提示词、检索到的上下文、工具痕迹和隐藏推理竞争同一个上下文窗口，因此长上下文 RAG 请求可能需要比短数学问题更低的推理预算。

3. 生产级推理控制器需要回退机制

一个健壮的控制器应该定义当运行不完整、验证器拒绝答案、工具调用超时或 token 预算耗尽时发生什么。典型的回退包括要求澄清、返回带有不确定性的部分答案、用更高 effort 或更小上下文重试一次、路由到专用模型、或推迟到异步工作流。重点是回退规则必须像解码参数一样被评估：衡量恢复成功率、增加的延迟、以及回退是否掩盖了系统性模型失败。

8.8 DeepSeek MTP – 多 Token 预测的理论

多 Token 预测（Multi-Token Prediction, MTP）是 DeepSeek-V3 中使用的一种训练和推理技术。传统语言模型每次只预测一个未来 token，而 MTP 允许模型同时预测多个未来 token。

训练阶段的 MTP：多输出头架构

每个 MTP 头是一个小型 MLP + LayerNorm，参数量远少于主 Transformer。它们共享底层特征表示，仅在输出层有独立参数。

双重好处

  • 训练收益：更密集的监督信号。每个 token 位置不仅要预测下一个 token，还要预测之后的 token。这有助于模型学习更长距离的依赖关系。DeepSeek-V3 报告提到了 MTP 带来的训练效率提升；具体幅度取决于模型、数据和训练设置，原文并未将其推广为一个通用比率。
  • 推理收益：推理时，MTP 头可以用作推测解码的草稿模型。由于 MTP 头参数量远少于完整 Transformer，它可以快速生成候选 token，然后由主模型在单次前向传播中验证。
💡
DeepSeek-V3 报告选择了 （预测 1 个额外 token），因为在他们的实验中，更大的 显示出边际收益递减。这里的要点是理解“多个预测头既可以服务于训练，也可以用于草稿生成”，而不是记住某个固定的收益比率。

8.9 编程练习 6：实现一个简化的投机解码

投机解码的核心思想非常巧妙：使用一个小模型（草稿模型）快速生成候选 token，然后使用一个大模型（目标模型）并行验证这些候选。当满足拒绝采样条件时，可以保留目标模型的采样分布；实际加速取决于草稿模型的速度、接受率、批处理以及系统瓶颈，论文和系统报告中通常显示 1.x 到数倍的提升。

算法流程

  1. 草稿阶段：使用轻量级草稿模型（例如 100M）自回归生成 个候选 token（通常 或 5）
  2. 验证阶段：将候选 token + 前缀输入目标模型（例如 7B），一次前向传播得到所有位置的 logits
  3. 拒绝采样：检查每个位置上目标/草稿模型的概率分布——如果差异过大，则拒绝并从修正后的分布重新采样
  4. 接受：接受的 token 和重新采样的 token 一起添加到输出序列中

任务：实现 speculative_decoding(target_model, draft_model, input_ids, gamma=4, max_new_tokens=100, eos_token_id=None)。

要求：

  • 草稿模型使用 Top-P 采样生成 gamma 个候选 token
  • 目标模型验证：一次前向传播计算所有候选位置的目标概率和草稿概率
  • 拒绝采样：以概率 min(1, p_target / p_draft) 接受每个候选
  • 拒绝时，从 max(0, p_target - p_draft) 重新采样
  • 如果所有候选都被接受，则额外从目标模型采样一个 token

def speculative_decoding(target_model, draft_model, input_ids, gamma=4, max_new_tokens=100, eos_token_id=None): """Speculative Decoding: Draft → Verify → Rejection Sampling → Distribution-Preserving Acceleration""" device = next(target_model.parameters()).device prefix = input_ids.to(device) n_orig = prefix.size(1) proposed = 0 accepted_count = 0 while prefix.size(1) - n_orig < max_new_tokens: draft_ids = prefix.clone() with torch.no_grad(): for _ in range(gamma): logits = draft_model(draft_ids) probs = torch.softmax(logits[:, -1, :], dim=-1) next_token = torch.multinomial(probs, 1) draft_ids = torch.cat([draft_ids, next_token], dim=-1) target_logits = target_model(draft_ids) draft_logits = draft_model(draft_ids) draft_tokens = draft_ids[:, -gamma:] proposed += draft_tokens.size(1) target_probs = torch.softmax(target_logits[:, -(gamma + 1):-1, :], dim=-1) draft_probs = torch.softmax(draft_logits[:, -(gamma + 1):-1, :], dim=-1) accepted = [] n_accepted = 0 for t in range(gamma): token = draft_tokens[0, t].item() p = target_probs[0, t, token].item() q = max(draft_probs[0, t, token].item(), 1e-12) if p >= q or torch.rand(1).item() < p / q: accepted.append(token) n_accepted += 1 accepted_count += 1 else: residual = (target_probs[0, t] - draft_probs[0, t]).clamp(min=0) residual = residual / residual.sum() accepted.append(torch.multinomial(residual, 1).item()) break if n_accepted == gamma: extra_probs = torch.softmax(target_logits[:, -1, :], dim=-1) accepted.append(torch.multinomial(extra_probs, 1).item()) accepted = accepted[:max_new_tokens - (prefix.size(1) - n_orig)] prefix = torch.cat([prefix, torch.tensor([accepted], device=device)], dim=-1) if eos_token_id is not None and accepted[-1] == eos_token_id: break stats = { "proposed": proposed, "accepted": accepted_count, "acceptance_rate": accepted_count / proposed if proposed else 0.0, } return prefix, stats

💡 分布保持分析：当草稿/目标概率、采样温度和拒绝采样实现均满足论文假设时，最终输出分布可以与直接从目标模型采样保持一致。加速不能仅凭 判断：如果每轮生成 个草稿 token，目标模型验证一次，草稿模型每步成本是目标模型每步成本的 倍，那么每轮时间约为 个目标模型步长单位；真正的分子是该轮实际写入输出的 token 数，取决于接受率。

8.10 上下文窗口与注意力汇聚

上下文窗口 (Context Window) 是模型单次生成时能够“看到”的最大 token 数量。窗口越长，帮助越大，但前提是模型仍能从中检索、优先处理并使用相关信息。

  • 长距离依赖： 小说中的伏笔可能跨越数百页后揭露；函数定义在文件开头，但调用在文件末尾。更长的上下文意味着更好的跨段落推理能力。
  • 少样本学习 (Few-shot Learning)： 在提示词中放置更多示例（上下文示例）能提升模型性能。上下文越长，容纳的示例就越多。
  • 多轮对话： 维持完整的对话历史。一个 128K 窗口大约可以容纳 10 万字符的对话记录。

注意力汇聚 (Attention Sink)

近期研究发现了一个有趣的现象——在长序列中，模型倾向于将大量注意力权重分配给初始 token（通常是 <BOS> 或句子起始 token），即使这些 token 与当前生成无关。这被称为 注意力汇聚 (Attention Sink)（Xiao et al., 2024）。

原因：训练时序列长度有限（例如 4K），导致模型形成一种“残余注意力偏差”——初始 token 吸收了过多的注意力分数。当推理时序列长度增加，这种偏差被放大，模型将大量注意力浪费在无关的早期 token 上。

8.10A 生成策略部署决策：将质量、格式、延迟和成本放在同一张表上

课程中介绍的贪婪解码、top-p、束搜索、自一致性、推测解码和受约束解码并非一组互斥的按钮。实际服务通常将它们组合成一个生成策略：首先根据任务确定输出空间，然后选择采样/搜索方法，最后使用格式约束、停止条件和成本预算来保护系统。

1. 首先确定输出空间

如果输出必须是 JSON、SQL、函数调用或固定枚举，则应优先考虑受约束解码或结构化输出，而不是依靠提示词提醒模型"请只输出 JSON"。如果输出是开放式的解释，那么调整 temperature、top-p 和重复惩罚是主要关注点。如果任务有可验证的答案，例如代码或数学，采样多样性本身不是目标；必须通过 pass@k、单元测试或验证器将多样本生成转换为正确率。

2. 然后确定风险边界

每个生成策略都应明确说明其失败边界。例如，在客户服务 RAG 中，低温度可以降低随机性，但不能保证引文忠实；束搜索可以搜索高概率候选，但不能修复遗漏的检索；自一致性可以减少单次推理错误的机会，但对于系统性误解可能导致多数投票错误；结构化解码可以保证语法有效性，但不能保证语义字段正确性。

3. 成本必须按每个完整请求计算

一份部署报告不能只写模型的 tokens/s。一个请求的端到端成本包括提示 token、生成的 token、推理样本、RAG 检索/重排序、工具调用、结构化解码开销、重试和排队。一个简化的核算可以描述为：

request_cost ~= prompt_tokens + generated_tokens * samples + reranker_calls * reranker_cost + tool_calls * tool_cost + retries * retry_cost

如果 samples 从 1 增加到 16，pass@k 可能会提高，但 P95 延迟和 token 成本也会几乎线性增加，除非系统执行并行采样并且具有足够的批量/硬件余量。推测解码可以减少目标模型验证的次数，但它也引入了草稿模型计算和调度复杂性。

3A. 推理/测试时计算需要预算决策

继 OpenAI o1、DeepSeek-R1 和 Kimi k1.5 之后，许多系统使用"多思考一会儿"作为质量改进方法：增加推理 token，采样多个解题路径，使用验证器或单元测试进行 Best-of-N 选择，或将长链蒸馏为短输出。工程报告不能只报告准确率的提升；还必须报告每问题样本数、平均输出 token 数、P95 延迟、验证器调用次数、每正确答案的成本以及边际收益。

测试时计算预算报告将这种判断形式化为部署决策。它比较贪婪解码、自一致性、Best-of-N、验证器重排序或推理模型层级，并输出质量、token 预算、延迟、成本和边际效率信号，以及推荐的部署策略。

8.12 结构化解码——JSON 模式与约束生成

在 API 和 Agent 场景中，我们常常需要 LLM 输出特定格式的结构化数据（JSON、YAML），而不是自由文本。仅让 LLM“尝试输出 JSON”是不够的——它偶尔会输出不完整的 JSON 或多余的文字。

约束解码原理

约束解码根据目标语法在每一步采样时动态修改 logits：将不符合语法的 token 的概率设为 -inf，从而保证生成结果遵循指定格式。

约束解码

实现：基于 JSON Schema 的 Token 过滤

以 JSON 为例：将 JSON Schema 编译成一个确定有限自动机（DFA）。在每个解码步骤，将词汇表中的每个 token 输入 DFA 判断是否合法——只有合法的 token 才能被采样。例如，当前缀是 {"name": 时，下一个 token 必须是 "。

练习 7：实现 Token 约束掩码

任务：实现 apply_token_constraints(logits, allowed_token_ids)，将每个批次行中非法 token 的 logits 设置为 -inf。

要求：

  • 输入 logits 形状为 [batch, vocab]
  • allowed_token_ids 可以是一个所有样本共享的 token 集合，也可以是每个样本单独的集合
  • 每行必须至少保留一个合法 token；如果非法 id 或空集则应报错
  • 应用约束后，执行 greedy、top-k 或 top-p；概率仅重新归一化到合法 token 集合上
# Example: In the second row, token 2 originally has the highest logit, but the grammar currently only allows 0 or 1
logits = torch.tensor([[5.0, 4.0, 3.0],
                       [1.0, 2.0, 9.0]])
allowed = [[1, 2], [0, 1]]
masked = apply_token_constraints(logits, allowed)
next_token = sample_next_token(masked, strategy="greedy")
# next_token == [[1], [1]]

8.12A 结构化输出的可靠性：模式合法性不等于语义正确性

结构化输出应被理解为一个契约堆栈，而非单一功能。JSON模式、严格的JSON Schema、正则表达式/文法约束、工具调用模式、后验证以及重试修复各自保护不同的边界。OpenAI的Structured Outputs文档强调了模式遵循；vLLM和SGLang通过JSON schema、正则表达式、选项、文法或EBNF提供了类似的受限输出选项。这些机制使可解析性更加可靠，但它们并不能证明所选字段值是真实、完整、安全或允许执行的。

3. 同时报告模式质量和语义质量

结构化输出实验不应止步于“JSON合法性为99%”。一份有用的报告应包括模式通过率、重试次数、延迟开销、语义字段准确率、无支持引文率、工具授权失败率以及执行后失败率。如果受限解码提高了可解析性，但增加了延迟或隐藏了系统性的提取错误，那么在发布结论中必须提及这一边界。

对于资深推理工程师而言，正确的部署声明是狭隘的：structured decoding guarantees the output shape under the supported schema; application validation decides whether the content is true and safe to execute. 混淆这两个层级是智能体事件的常见原因：一个合法的工具调用JSON对象仍然可能删除错误的资源、引用不支持该主张的文档或提交不安全的查询。

8.12B 约束解码引擎：CFG、FSM、Schema 和服务成本

结构化输出不仅仅是一种提示技术。在生产推理中，它通常作为服务时的约束引擎实现：请求提供 schema、正则表达式、选项集合、语法或工具签名；后端将该契约编译成约束状态；每个解码步在采样前屏蔽非法 token。OpenAI Structured Outputs、vLLM Structured Outputs、SGLang Structured Outputs、Outlines 类库和 guidance 类库都遵循这个大模式，尽管它们在支持的 schema 特性、语法后端、批处理行为和延迟开销上有所不同。

本节补充了之前的可靠性讨论。约束解码能让表面格式有效，但不能让内容真实。推理工程师仍然需要在解码器返回可解析对象之后，运行验证器、证据检查、权限检查和失败指标。
