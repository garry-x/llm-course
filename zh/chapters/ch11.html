11.1 为什么大语言模型课程仍需要学习经典的自然语言处理

本课程覆盖的经典 NLP 主题包括词向量、RNN、依存句法分析、seq2seq + 注意力机制、Transformer 和 BERT，最后延伸到大型语言模型。如果跳过这些中间模型，容易产生一个常见误解：所有 NLP 问题都应该改成基于 prompt 的生成方式。实际上，很多生产系统会组合不同的模型族：用 BERT 或 DeBERTa 做重排序，用双编码器做检索嵌入，用仅编码器模型做抽取式问答，用仅解码器大语言模型做答案生成。

本章的重点不是记住旧的模型名称，而是掌握三个核心概念：

  • 归纳偏置 (Inductive Bias)：RNN 按时间递归，句法分析器输出树结构，seq2seq 把源序列和目标序列分开，BERT 双向读取上下文。
  • 目标函数 (Objective Functions)：因果语言模型预测下一个 token，MLM 仅在掩码位置计算损失，seq2seq 最大化条件生成概率，句法分析器最大化结构动作或头/标签概率。
  • 评估指标 (Evaluation Metrics)：BLEU、ROUGE、EM/F1、UAS/LAS 以及检索指标各自回答不同的问题，不能用单一的汇总分数代替它们。

11.3 依存句法分析：结构化预测并非注意力热力图

依存句法分析为每个 token 预测一个中心词和一个依存标签，输出一棵有向树。它与注意力热力图的重要区别在于：注意力是连续权重，可用于诊断；而依存树是离散结构，必须满足单一根节点、无环、每个词只有一个中心词等约束。

Arc-standard 转移系统使用 SHIFT、LEFT_ARC 和 RIGHT_ARC 动作构建依存树：

动作前置条件效果
SHIFT缓冲区非空将缓冲区第一个 token 压入栈顶
LEFT_ARC(label)栈中至少有两个元素将栈中第二个元素设为栈顶元素的依存，然后移除第二个元素
RIGHT_ARC(label)栈中至少有两个元素将栈顶元素设为栈中第二个元素的依存，然后移除栈顶元素
ROOT(label)只剩一个根候选将该根候选连接到 ROOT

示例：I saw her 的 Arc-standard 分析

目标依存树为 saw -> I (nsubj)、saw -> her (obj)，最后是 ROOT -> saw (root)。使用 Arc-standard 转移系统，按以下步骤构建：

步骤栈缓冲区动作新弧
0[][I, saw, her]SHIFT-
1[I][saw, her]SHIFT-
2[I, saw][her]LEFT_ARC(nsubj)saw -> I
3[saw][her]SHIFT-
4[saw, her][]RIGHT_ARC(obj)saw -> her
5[saw][]ROOT(root)ROOT -> saw

这个例子说明句法分析不是独立地对每个 token 进行分类：一个动作的合法性取决于当前栈/缓冲区的状态，错误的动作会破坏最终的树结构。因此，转移轨迹需要记录每个 token 的中心词、标签、弧以及栈/缓冲区状态，以便检查树的构建过程。

11.5 BERT / 仅编码器：MLM、分类和抽取式问答

BERT风格的编码器可以同时看到输入左右两侧的上下文；它不是自回归生成器。预训练目标之一是掩码语言建模：交叉熵只在掩码位置计算，未掩码的token用忽略索引处理。MLM诊断需要返回掩码token的损失、准确率以及导致损失的位置，这有助于区分“输入中哪些token是可见的”和“哪些位置对训练目标有贡献”。

BIO序列标注

NER、槽位填充和token分类通常把每个token标注为 B-TYPE、I-TYPE 或 O。B-ORG I-ORG 表示一个连续的组织实体，O 表示实体外部。BIO解码器将token级标签转换为跨度级输出；例如，["B-LOC", "I-LOC"] 对应一个LOC跨度。如果一个序列以 I-LOC 开头，许多评估脚本会将其当作新实体的开始；严格的系统也可以直接报告错误，但这必须在数据协议中明确说明。

实体识别通常使用 跨度级F1，而不仅仅是token准确率。只有当预测实体的 type、start 和 end 全部与黄金跨度完全匹配时，才视为真正例；边界差一个token或类型错误计为假正例，而对应的黄金跨度仍为假负例。跨度级评估需要将这一定义转换为可计算的TP/FP/FN、精确率、召回率和F1。

11.6 评估：BLEU、ROUGE、EM/F1 与结构指标

评估指标必须匹配任务结构。BLEU 常用于翻译，ROUGE 用于摘要，Exact Match 和 token 级 F1 用于抽取式问答，UAS/LAS 用于依存句法分析。这些指标没有一个是“语言质量”的完整度量。

指标核心计算适用任务主要局限
BLEU裁剪的 n-gram 精确度 + 简短惩罚机器翻译对改写和语义等价不敏感
ROUGE-L最长公共子序列的精确度/召回率/F1摘要、生成比较偏向词汇重叠，不能保证事实一致性
EM/F1答案标准化后的精确匹配或 token 重叠抽取式问答、简短回答需要额外处理多个答案、别名和推理型答案
UAS/LAShead 是否正确，以及标签是否也正确依存句法分析只评估句法结构，不等同于下游任务成功

11.6A 现代 LLM 评估协议：从单一分数到可复现结论

BLEU、ROUGE、EM/F1 和 UAS/LAS 适用于定义明确的传统任务；现代 LLM 还需要评估开放式回答、长上下文、工具调用、RAG、拒答行为、安全性和多轮对话。最常见的错误是将单一基准分数、单一 LLM-as-judge 胜率或一组人类偏好解读为“该模型更好”。课程评估应该先定义任务、样本、推理设置和评判标准，再讨论结论。

LLM-as-judge 需要校准

LLM-as-judge 可以降低评估开放式输出的成本，但它存在位置偏差、长度偏差、风格偏差和模型同质偏差。更稳健的做法是成对盲比：隐藏模型名称，随机交换 A/B 顺序，让判断者决定哪个响应更符合评分标准，并记录平局。

工业级评估在使用判断者分数之前需要做可靠性审计。OpenAI 的评估指南建议在使用模型评分器进行优化之前，先使其与人类标签对齐；MT-Bench / Chatbot Arena 等工作也警告说，成对判断者会受到位置、冗长和风格偏差的影响。因此，本课程不允许报告单一胜率。至少必须回答四个问题：交换 A/B 顺序后是否选择相同的候选者、判断者是否对第一个位置有异常偏好、是否对较长的响应有异常偏好、以及与一小批人工黄金标签的一致率是否足够。

带平局的成对胜率

如果模型 A 与模型 B 的结果是 W=42、L=35、T=23，则胜率为 (42+11.5)/100=53.5%。仅凭此数字不能证明 A 在生产环境中更好；还必须检查样本量是否充足、问题分布是否代表目标场景、失败类别是什么、以及延迟和成本是否恶化。

安全评估必须区分拒答和过度拒答

安全性并非“拒绝越多越好”。样本至少应分为有害请求、良性敏感请求和普通任务三类，并分别统计：

  • 攻击成功率：模型对有害请求提供不安全帮助的比例；越低越好。
  • 拒答率：模型拒绝请求的比例；需要对有害/良性类别分别查看。
  • 过度拒答率：本应提供帮助但被错误拒绝的良性请求比例。
  • 任务效用：在普通任务上的准确率、有用性或完成质量，以防止安全优化牺牲可用性。

例如，模型 X 在有害集上的拒答率为 95%，但在良性医疗摘要集上的拒答率也为 60%。这不能简单说是“更安全”，而是严重的过度拒答。第 09 章的对齐评估和第 10 章的部署 SLO 应与本节一起解读：安全性、质量、延迟和成本都必须达到各自的目标。

将现代评估编写为可运行的诊断程序

一份现代评估报告应当可以简化为可运行的诊断程序。成对判断摘要计算原始胜率、平局调整胜率和按任务分组的胜率，显示同一个模型可能在问答任务上获胜，而在数学任务上持平或失败。判断者可靠性审计检查样本量、位置偏差、长度偏差、交换一致性和人类标签一致率。安全指标将样本分为有害、良性敏感和普通三类，然后报告攻击成功率、有害拒答率、过度拒答率和任务效用。基准摘要将任务、样本量、指标、模型、提示、温度、失败类型和不可外推范围组织成结构化结论。

这些诊断程序共享一个目的：防止单一平均分被误认为是完整的工程结论。训练工程师用它们来判断对齐是否损害了能力；推理工程师用它们来判断质量、安全性、延迟和成本是否共同满足部署要求。

11.6B 基准测试有效性：污染、置信区间与消融实验

评估现代大语言模型时，难点不只是“该选哪个基准测试”，更在于判断分数能否支撑工程决策。一个模型胜出，可能是因为训练数据已经见过考题、它的 prompt 更合适、它的输出更长、它使用了更多采样轮次，或者评判者更喜欢它的风格。课程报告必须厘清这些因素；否则，基准测试就变成了排行榜截图，而非科学比较。

1. 数据污染与数据划分泄露不是同一个问题

数据划分泄露通常发生在自身的数据流水线内：同一份文档、用户会话、题库或近似改写文本同时进入了训练/开发/测试集。可以通过文档级划分、URL/题目ID去重、n-gram重叠检查、embedding最近邻搜索以及人工抽查来缓解。基准测试污染则更为宽泛：公开的测试题、答案解析、讨论帖或代码仓库可能已经混入了预训练或SFT数据。它并不总能被完全消除，因此报告中必须说明数据来源、发布日期、去重方法以及对高相似度样本的处理方式。

公开基准测试也有其生命周期。一个在刚发布时很有用的基准测试，经过几个月的排行榜优化后，可能会饱和、被反复 prompt 调优或者被污染。OpenAI 在2026年关于 SWE-bench Verified 的讨论是一个不错的例子：即使是一个经过人工验证的公开基准测试，一旦模型、智能体和训练数据都适应了它，它就可能不再衡量前沿水平。在课程报告中，旧的公开基准测试通常应作为回归测试，而私有保留集和在线/canary评估则为部署决策提供更有力的证据。

评估生命周期：从公开基准测试到在线证据

现代评估应被视为一个运营生命周期，而非最终的排行榜快照。公开基准测试适用于粗略的比较和回归检查，但前沿模型和智能体系统变化很快：任务会饱和、prompt 会变得优化、公开解决方案会泄露到数据中、产品工作流会发生偏移。因此，一个严肃的评估程序应使用多个证据层，这些层具有不同的所有权、刷新频率和决策权限。

示例分析：RAG 医疗问答的风险分解

假设一个团队想推出一款面向患者的医学教育 RAG 系统。一个不充分的做法是只测试“答案看起来是否专业”。更合理的工程分解方式是：

  1. 检索风险：是否召回了相关指南，过时文档是否被引入上下文，引用是否指向了正确的段落。
  2. 生成风险：模型是否将检索内容外推到诊断结论，捏造了药物剂量，或在信息不足时给出了确定性的建议。
  3. 隐私风险：用户输入和日志是否包含姓名、电话号码、病历号或其他可识别信息。
  4. 拒绝边界：一般教育性问题应予以回答，而涉及紧急情况、自残或处方调整的请求应转介给专业人士。
  5. 可用性：安全策略不能拒绝所有医疗问题，否则系统将无法服务于真实用户任务。

因此，该系统至少应报告检索召回率、引用准确率、事实错误率、有害建议率、一般医学教育完成率、紧急转介率、个人身份信息日志处理方式以及用户界面中的局限性描述。目标不是创建更多的表格，而是用同一种工程语言来描述模型行为、数据来源、用户风险和部署边界。

3. 安全、对齐、评估与提供服务是同一流水线

第 9 章讨论了如何使用 SFT/DPO/GRPO 改变模型行为，第 10 章讨论了如何将模型转变为可服务的系统，而本节讨论的是这些行为在真实场景下的风险边界。一个模型可能在无害性上有所提升，同时对良性的敏感问题过度拒绝；它也可能在离线安全集上表现良好，但由于工具权限过大而在 Agent 场景中失效。一份可信的结果表应在质量、延迟和成本旁边包含安全切片，而不是仅仅在结论中写上“已考虑安全性”。

🔑
工程要点：社会影响不是抽象的道德讨论，也不是单一的安全分数。LLM 工程师必须能够将风险映射到数据、目标函数、接口权限、用户场景和评估切片上，并解释每项缓解措施会牺牲哪些能力、延迟或成本。

11.6D Agent 与工作流评估：最终答案、轨迹、状态与成本

Agent 评估比单轮答案评估要求更高，因为模型可以搜索、编辑文件、调用工具、写入数据库，并在多轮交互中与用户沟通。只有最终答案正确还不够：轨迹可能使用了禁止的工具、改了不该改的状态、泄露了隐私数据，或者花了不可接受的时间和金钱。最近的基准测试让这些要求更明确了。SWE-bench Verified 通过隐藏测试和人工过滤来验证软件工程任务。tau2-bench 结合领域策略、工具使用、用户模拟和最终状态来评估客服 Agent。BrowseComp 专注于难以找到但简短且可验证的浏览答案，明确用开放的逼真性换取评分可靠性。

11.7 任务选择：何时不应使用 Decoder-only 大语言模型

选择模型系列时，首先考虑输出空间和错误代价：

  • 固定标签分类：仅编码器微调或小型模型分类器通常成本更低、更稳定。
  • RAG 重排序：Cross-encoder 可以同时查看 query 和文档，对候选进行排序；decoder-only 大语言模型做重排序成本高，输出格式也不稳定。
  • 抽取式答案：如果答案必须来自原始文档，Span QA 比自由生成更可控。
  • 结构化输出：依存分析、NER、槽位填充等任务可以使用结构化 head 或约束解码，不应默认采用开放生成。
  • 开放生成与推理：当输出需要新文本、代码、解释或多步推理时，decoder-only 大语言模型更自然。
⚠️
常见错误：把 BERT 和 GPT 描述为“只是不同的 mask”。它们的注意力可见性、训练目标、微调 head、输出空间和服务成本都不同。

11.7A 工程选择案例研究：将任务分解为训练目标、输出空间和服务成本

大语言模型工程师常问的不是“哪个模型最大”，而是“这个任务应该建模为分类、抽取、排序、结构化预测，还是开放生成？”同一个业务目标可以有多种建模方法，训练数据、推理路径、评估指标和部署成本都会随之改变。

案例 1：客服知识库问答

如果业务要求“答案必须引用原始知识库文本”，让仅解码器 LLM 自由回答风险很高。更稳妥的做法是使用三阶段流水线：双编码器用于候选文档召回、交叉编码器重排序器用于精细排序、抽取式问答或受限生成用于答案跨度及引用。只有当问题需要跨段落综合、改写或解释时，才让仅解码器 LLM 基于检索上下文生成自然语言答案。

这类系统的训练目标也是分层的：检索器使用对比学习或点击数据，重排序器使用相关性排序标签，问答头使用跨度标签，生成器使用 SFT/偏好优化。把所有错误都归到“LLM 不够强”上，可能会掩盖真正的瓶颈：可能是召回遗漏、重排序错误、分块破坏了答案、提示未限制引用，或是生成阶段没有忠实地使用上下文。

案例 2：合同字段抽取

从合同中抽取甲方、乙方、金额和日期等字段的输出空间是有限的。如果使用开放生成，模型可能会改写金额格式、补全不存在的字段或混淆多个日期。更合适的基线是用仅编码器的 token 分类或跨度分类，然后对金额、日期和实体类型进行基于规则的验证。

在训练工程中，应该关注跨度级别的 F1，而不是 token 准确率。如果一个日期实体 June 8, 2026 少了一天，token 准确率可能仍然很高，但字段抽取却是错误的。在推理工程中，应关注批量吞吐量和长文档分块：如果每份合同有几十页，使用交叉编码器处理全文的成本可能无法接受，需要先分块召回候选段落，再进行细粒度抽取。

案例 3：开放式推理答案

数学问题、代码修复和复杂规划通常无法用固定标签或跨度覆盖，需要仅解码器 LLM 生成中间推理过程和最终答案。此时，经典的 NLP 指标往往不够用；BLEU/ROUGE 可能会惩罚合理的改写，或奖励错误但词汇相似的答案。更合适的指标是任务成功率、单元测试/pass@k、自一致性、人工或 LLM-as-judge 分层评估，以及第 10 章中的延迟/成本指标。

🔑
工程判断：选择模型家族时，应该从输出约束出发。固定标签优先用分类，答案来自原文优先用抽取，结构化字段优先用跨度/CRF/受限解码，开放式解释和推理优先用仅解码器生成。这样训练目标、评估指标和推理成本才能相互对齐。

11.8 实验结论：从指标到可判定的判断

本节把代码输出、手动推导、阅读材料和实验指标整理成可判定的结论。目的是回答一个清晰的技术问题，并让读者能判断结论在什么条件下成立。

1. 将问题表述为可检验的命题

好的技术问题应该能被数据和实验证伪。例如：

  • 训练工程：在固定token预算下，seq_len=128相比seq_len=64是否降低了验证PPL？代价（tokens/s和step时间）是多少？
  • 对齐工程：DPO是否在维持基础任务准确率和拒绝边界的同时，提高了偏好胜率？KL、输出长度和失败类型如何变化？
  • 推理工程：将RAG top-k从3增加到8是否提高了通过率？对P95 TTFT、prompt tokens、context packing miss率和成本有何影响？
  • 服务工程：当并发从1增加到8时，P95/P99延迟是否仍然满足SLO？瓶颈是排队、prefill、decode、检索还是后处理？

这些问题都有基线、变更项、指标和结论边界。相比之下，“构建一个聊天机器人”或“训练一个小模型”不是定义良好的技术命题，因为它无法解释什么变化导致了什么效果。

3. 负面结果也是有力的结论

实验不要求每个方法都“提升”指标。如果某个消融没有带来显著改进，只要实验设计清晰，仍然是一个有价值的结论。例如：seq_len=128仅将验证PPL降低了0.03，但tokens/s下降了35%，且三个seed上的波动约为0.05。那么合理的结论是“当前设置不足以证明采用更长序列是合理的”。这比只挑选最优数字更加严谨。

4. 将结论与复现命令和局限性绑定

结论要指定复现路径：数据版本、commit、运行命令、seed、硬件、模型配置、检查点或服务配置、评估集、压力测试参数。还要指出外推无效的方面：一个小型语料库不能代表万亿token训练，本地MockEngine不能代表真实GPU服务，小型知识库RAG不能代表企业知识库，单一安全测试集不能代表所有用户风险。

读者应该能回答三个问题：第一，问题是否清晰？第二，实验是否足以支撑结论？第三，如果系统扩展到更大的模型、更大的数据或真实服务，哪些假设会首先失效？

🔑
工程要点：训练方向要证明过程是可复现、可恢复且可解释的；推理方向要证明在目标工作负载下服务质量、延迟、成本和风险是可判定的。没有基线、消融、失败分析和结论边界的实验，不能支持工程决策。

11.10 总结

  • RNN/LSTM 讲解了序列循环、BPTT 和长距离依赖问题。
  • 依存句法分析是一种离散结构预测任务，不同于注意力热力图。
  • Seq2Seq 注意力分别对源序列和目标前缀建模，构成了基本的编码器-解码器结构。
  • BERT/仅编码器模型适用于分类、重排序、嵌入、词元分类和抽取式问答。
  • BLEU、ROUGE、EM/F1、span F1、UAS/LAS 分别衡量不同的任务结构。现代 LLM 评估还需描述数据、推理设置、评判者偏差、安全拒绝、统计不确定性和结论边界。
