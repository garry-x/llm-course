4.1 从单头到多头——为什么一个头不够

第3章实现了单头自注意力：一个维的 Q/K/V 空间负责所有匹配模式。但语言中的关系是多方面的——单个 token 需要同时从多个“视角”匹配其他 token。

具体场景：为什么一个头不够？

考虑句子 "The cat sat on the mat because it was comfortable."

当模型处理单词 "it" 时，它需要回答多个问题：

  • 句法问题："it" 是单数代词；它的先行词在句法上应该是单数名词——这需要一种特定的匹配规则
  • 语义问题："it" 应该指代“最合理”的事物——猫坐在垫子上，所以可能是 "the cat" 或 "the mat"——这需要另一种匹配规则
  • 位置问题："it" 前面的名词是 "mat"，再往前是 "cat"——模型需要区分位置5和位置1的 token

单头注意力只能使用一个 维的空间来编码所有这些匹配规则。这迫使句法、语义和位置的证据进入同一个相似度函数，这通常过于局限。

多头注意力的解决方案：并行子空间

多头注意力（Multi-Head Attention, MHA）的核心思想是：将输入的 维空间拆分成 个维度为 的子空间，每个头独立地在其子空间中计算注意力，最后所有头的输出通过输出投影 拼接并融合：

设 d_model=8，使用 n_heads=2，则每个头的维度为 d_k=4。如果单头注意力将 Q/K/V 投影到 8 维，它需要 3 * 8 * 8 = 192 个 QKV 参数；两个头各需 3 * 8 * 4 = 96 个参数，总计 2 * 96 = 192。

方案Q/K/V 投影空间QKV 参数数量注意力分数形状
单头1 个 8 维空间3 * 8 * 8 = 192[T, T]
2 头 MHA2 个 4 维空间2 * 3 * 8 * 4 = 192[2, T, T]

两者的参数数量相同，但 MHA 在不同投影空间中产生两张注意力图，然后将两个 4 维的头输出拼接回 8 维，并通过 W_O 进行混合。它的表示优势来源于“多个可学习的相似度函数并行工作”，而不是简单地增加参数。

换句话说，多头注意力并非“变大”，而是将现有的参数预算重新组织成多个更小的子空间。每个子空间学习不同的注意力模式，最后的 负责将这些分散的信息整合在一起。

这是一种固定预算的分解：模型不是添加更大的注意力模块，而是将表示空间拆分成几个更小的子空间，这些子空间可以专门处理不同的匹配模式。

🔑
为什么参数相同但效果更好？每个头有独立的 Q/K/V 投影，可以学习不同的相似度函数；输出投影 再将子空间结果重新组合。MHA 的优势来自表示分解和可学习的混合，而不是“更多头、更多参数”。

4.2 编程练习 1：实现 MultiHeadAttention

🖥️ 练习 1：MultiHeadAttention 模块

任务：实现一个完整的 MultiHeadAttention 模块。输入 x: (batch, seq_len, d_model)，输出 (output, attn_weights)。

要求：

  • 使用单个 nn.Linear(d_model, 3 * d_model) 进行融合的 QKV 投影，然后通过 reshape 和 unbind 将 Q、K、V 分离到每个 head。
  • 使用 transpose 将 head 维度移到第 1 维（batch 之后），形状为 (b, h, s, d_k)。
  • 实现缩放点积注意力（可选因果掩码支持）。
  • 使用 W_o 输出投影将拼接后的多头结果映射回 。
  • 返回 (output, attn_weights)，其中 attn_weights 的形状为 (b, n_heads, seq_len, seq_len)。
# Expected usage
mha = MultiHeadAttention(d_model=768, n_heads=12, dropout=0.1)
x = torch.randn(2, 16, 768)
mask = torch.tril(torch.ones(16, 16)).unsqueeze(0).unsqueeze(0)
out, weights = mha(x, mask)
assert out.shape == (2, 16, 768)

# Check attention weight normalization

import torch import torch.nn as nn import torch.nn.functional as F class MultiHeadAttention(nn.Module): """Multi-Head Attention — Fused QKV Projection + Efficient Head Separation""" def __init__(self, d_model, n_heads, dropout=0.1): super().__init__() assert d_model % n_heads == 0 self.d_model = d_model self.n_heads = n_heads self.d_k = d_model // n_heads self.W_qkv = nn.Linear(d_model, 3 * d_model, bias=False) self.W_o = nn.Linear(d_model, d_model, bias=False) self.dropout = nn.Dropout(dropout) def forward(self, x, mask=None): # x: (batch, seq_len, d_model) batch, seq_len, _ = x.shape # 1. Fused projection + separate Q/K/V qkv = self.W_qkv(x) # (b, s, 3*d) qkv = qkv.reshape(batch, seq_len, 3, self.n_heads, self.d_k) q, k, v = qkv.unbind(dim=2) # each (b, s, h, d_k) # 2. Move head dimension forward: (b, h, s, d_k) q = q.transpose(1, 2) k = k.transpose(1, 2) v = v.transpose(1, 2) # 3. Scaled dot-product attention scores = torch.matmul(q, k.transpose(-2, -1)) / (self.d_k ** 0.5) if mask is not None: scores = scores.masked_fill(mask == 0, float('-inf')) attn_weights = F.softmax(scores, dim=-1) attn_weights = self.dropout(attn_weights) out = torch.matmul(attn_weights, v) # (b, h, s, d_k) # 4. Concatenate heads out = out.transpose(1, 2).contiguous() out = out.reshape(batch, seq_len, self.d_model) # 5. Output projection out = self.W_o(out) return out, attn_weights

💡 为什么融合 QKV 投影？ GPU 在处理大矩阵乘法时效率更高——一次 (d, 3d) 矩阵乘法比三个 (d, d) 矩阵乘法之和更快，因为 GPU 的并行计算单元能更好地利用更大的矩阵块。这是工程实践中常见的“先融合再分离”模式。

4.3 深入探讨：为什么多个注意力头可以学习不同的表示子空间？

初始化驱动的分化

在实践中，多个注意力头能够学习不同模式的关键原因在于随机初始化。每个头的 Q/K/V 投影矩阵 初始时具有不同的随机值。在训练过程中，不同的初始化导致每个头沿不同的梯度方向优化——就像从不同的起点攀登不同的山峰。

数学视角：跨头的梯度差异

假设两个头的初始投影矩阵分别为 和 ，且 。它们的输出 和 也会不同。通过损失函数的梯度，——参数更新方向不同，导致它们逐渐分化。

如果所有头学到的完全相同会怎样？

理论上这是可能的；但实际上，长时间保持完全相同通常不太可能，因为：

  • 随机初始化打破了对称性
  • 不同头的训练数据流（梯度）天然不同——因为每个头看到的注意力分布不同
  • 即使两个头在训练早期相似，SGD 的随机性也会放大细微差异

事实上，Michel 等人（2019）的研究表明，训练后相当比例的头是冗余的（移除它们不影响性能）。这意味着模型实际需要的"有效头数"可能远小于设计头数——这也是 GQA 和 MLA 压缩 KV 头数的理论基础之一。

头之间的竞争与合作

有趣的是，注意力头之间存在竞争关系。如果一个头已经捕获了某种句法模式，其他头再去捕获相同模式就没有"边际收益"了。这促使每个头去发现不同的模式——类似于负相关学习（Negative Correlation Learning），这是一种经典的集成学习策略。

🔑
机制：多头注意力将子空间分解与差异化初始化相结合。总参数量不变，但将一个大空间拆分成多个较小的子空间，让每个子空间专注于不同的匹配模式。这好比在固定参数预算下，使用几个狭窄的专家而不是一个宽泛的匹配器。

4.3A 多头注意力并非简单集成：表征分解、冗余性与压缩空间

"多头学到不同的模式"这一直觉没错，但还不够严谨。MHA 并不是把多个完整模型的投票做平均，而是在固定的 宽度内，将注意力的匹配与聚合分解到多个更窄的子空间中，再通过输出投影 重新组合。

MHA 的分解形式

表达能力来自"不同的相似度函数"

每个 head 有自己的 ，因此它是在不同的投影空间中计算相似度。某个 head 可能让括号、缩进或分隔符变得相似，另一个 head 可能让主语和谓语变得相似，还有一个专门负责当前位置附近的局部模式。关键在于：这些 head 的输出并不是最终答案，而是供下一层继续处理的中间表示。

为什么会出现 head 冗余

训练后，并非每个 head 都同等重要。原因可分为三类：

  • 任务需求低于设计容量。如果当前层只需要少数几种主要路由模式，额外的 head 就可能学到相似的功能。
  • 输出投影可以重新加权。 能削弱某些 head 的贡献；某个 head 的注意力可能看上去很活跃，但这并不代表它对 logits 有强烈影响。
  • 相邻层能补偿。在移除或压缩一些 head 之后，后续层有时仍然可以恢复足够的信息，因此性能下降可能不会立即显现。

这如何解释 GQA 和 MLA

GQA 的基本假设是：并非每个 query head 都必须拥有独立的 K/V 表示。多个 query head 可以共享一组 K/V，通过各自的 Q 投影仍然可以提出不同的查询。MLA 则更进一步，将 K/V 信息压缩到低维潜空间中，使用时再重建出所需的表示。两者都利用了同一个事实：模型设计中名义上的 head 数量，并不等于推理时必须保留的独立 K/V 信息量。

现象理论解释工程后果
多个 head 学到相似的模式任务路由需求低于设计 head 数量，或输出投影削弱了部分 head可尝试 head 剪枝、GQA 或低秩压缩
GQA 质量接近 MHAQ 的多样性仍在，K/V 的多样性由组共享KV Cache 显著减少，适合高并发推理
MQA 可能导致质量下降所有 Q 共享一组 K/V，检索空间过于狭窄内存效率最高，但在长文本和复杂任务中可能更敏感
MLA 需要解压缩和额外的投影低维潜空间存储主要信息，而非完整的 K/V 张量缓存更小，但实现复杂度和算子融合要求更高
⚠️
不要把 head 数量直接当作能力指标。更多 head 提供了更精细的子空间分解，但也可能增加冗余；更少的 KV head 提升了推理效率，但可能丢失检索多样性。严格比较时必须同时考察 loss、下游任务、长上下文行为、吞吐量和 KV Cache。

4.5 MHA vs MQA vs GQA——KV Cache的演进

推理瓶颈：KV Cache

在自回归生成中，每步只生成一个token ，但需要与所有历史token计算注意力。如果每步都重新计算所有历史token的K和V，计算量将是 。因此，我们将历史token的K和V向量缓存起来——这就是KV Cache。

MQA：最激进的压缩策略

MQA（Shazeer, 2019）让所有Q头共享同一组K和V。KV Cache缩减为MHA的 。代价是KV的多样性完全丧失——所有注意力头只能从同一个K、V空间中检索信息。

GQA：质量与缓存成本的折中

GQA（Ainslie等人, 2023）将Q头划分为 个组，每组共享一个K/V头。Llama 2 70B使用了8个KV头（），将KV Cache降低了8倍，同时保留了一定的K/V多样性。

Ch04作业中的 gqa_head_mapping(n_heads, n_kv_heads) 函数将这种分组关系表达为一个显式列表。如果 n_heads=8, n_kv_heads=2, 那么每4个Q头共享一个KV头，得到映射 [0,0,0,0,1,1,1,1]；如果 n_kv_heads=1, 就是MQA；如果 n_kv_heads=n_heads, 则退化为标准MHA。在实现中，repeat_kv_heads 只是在注意力计算前将 [B,H_{kv},T,D] view扩展到 [B,H_q,T,D] 的语义；KV Cache仍然只需存储 H_{kv} 份K/V。

GQA的实际收益取决于任务、模型规模、n_heads/n_kv_heads 比例以及训练方法。Ainslie等人的实验表明，在适当的分组下，可以在保持接近MHA质量的同时，显著降低解码阶段的KV cache和带宽压力；但如果 n_kv_heads 过小，可能会丢失K/V多样性。训练后转换也可以通过取均值或重排MHA检查点的K/V头来获得GQA初始化，然后继续微调。

💡
学习重点： MQA/GQA并没有改变自注意力的公式，而是改变了“缓存多少组独立的K/V”。质量问题的根源在于共享K/V后表征能力下降，而系统收益来自解码阶段缓存和内存带宽的降低。

4.5A 示例：GQA 头映射与 KV 缓存预算

4.5B 训练 Attention 和推理 KV Cache 不是相同的布局问题

当学习者首次实现 MHA/GQA 时，他们通常只关心 [B, H, T, D] 张量能否计算出正确输出。然而，训练工程和推理工程使用同一组权重解决两个不同的问题：训练阶段需要高吞吐量处理整个序列，而推理阶段需要在每一步将新 token 追加到历史缓存。理解这一区别是解释为什么 GQA/MLA 对 decode 阶段特别重要的关键。

1. 训练阶段：在全序列上并行计算

在训练或 prefill 阶段，一次处理一个长度为 T 的完整序列。标准实现将 Q/K/V 投影为：

q: [B, H_q, T, D] k: [B, H_kv, T, D] v: [B, H_kv, T, D]

此时，所有 Q/K/V 值均来自当前 batch 的隐藏状态。在 MHA 中，H_q = H_kv；在 GQA 中，H_q > H_kv，并且 K/V 按组扩展以匹配 query head 数量，然后计算 attention；在 MLA 中，先存储低维潜在表示，然后在计算分数时通过解压缩矩阵或矩阵吸收恢复等效的 K/V 效果。训练期间的主要压力来自大矩阵乘法、attention 分数的 T x T 计算以及激活内存。

2. Decode 阶段：每步只有一个新 token

在自回归解码的第 t 步，只输入最新的 token，query 长度通常为 1：

new_q: [B, H_q, 1, D] new_k/new_v:[B, H_kv, 1, D] cache_k/v: [B, H_kv, t, D] attn score: [B, H_q, 1, t]

此时，attention 分数不是一个大 T x T 矩阵，而是每个请求从历史缓存中进行读取。decode 的瓶颈通常是 HBM 带宽和缓存读写，而不是单步 FLOPs。GQA/MLA 的直接好处是减少 cache_k/v 的体积和读取带宽；它们不会神奇地减少所有模型权重，也不会自动提高 prefill 的整体吞吐量。

3. 缓存布局决定系统能否高效调度

在实际的推理服务中，一个 batch 内的请求通常长度不同。如果每个请求将其 KV cache 存储为连续张量，短请求会浪费空间，长请求会导致碎片化，释放和重用内存变得困难。推理引擎通常将缓存拆分为块/页，并维护从“逻辑 token 位置 → 物理缓存块”的映射。第 10 章的 PagedAttention 将详细介绍调度；目前请记住：GQA/MLA 决定每个 token 需要存储多少 KV 内容，而 PagedAttention 决定这些内容如何放入内存并在不同请求之间重用。

4.5C 将 MHA 检查点转换为 GQA：压缩并非仅仅修改配置

一个常见的生产场景是拥有一个 MHA 检查点，其推理瓶颈是 KV Cache。将其从 H_q=H_kv 转换为使用 H_q > H_kv 的 GQA 不能仅仅通过修改配置中的 n_kv_heads 来完成。K/V 投影权重、优化器状态、缓存布局以及质量评估都需要重新对齐。

1. 如何折叠权重

假设原始模型有 H_q=8 个 K/V 头，你想将其转换为 H_kv=2。每 4 个原始 K/V 头将变成 1 个共享 KV 头。最常见的初始化方法是将原始 K/V 投影按组平均：

old K heads: [K0, K1, K2, K3, K4, K5, K6, K7] new K heads: [mean(K0..K3), mean(K4..K7)] old V heads: [V0, V1, V2, V3, V4, V5, V6, V7] new V heads: [mean(V0..V3), mean(V4..V7)]

Q 投影和输出投影 W_o 通常保持原始头数不变，因为 GQA 的目标是减少 K/V 表示，而不是减少查询头的多样性。转换后，前向传播首先获取形状为 [B,H_kv,T,D] 的 K/V，然后通过 [B,H_q,T,D] 将其对齐到 repeat_kv_heads 以进行注意力计算。

2. 为什么需要继续训练（Uptraining）

分组平均只是一个初始化。最初，每个查询头可以访问独立的 K/V 空间；转换后，多个查询头共享相同的 K/V，因此信息路由能力降低。通常需要一段短期的持续训练，让 Q、K、V 以及后续层适应共享 K/V 的约束。这个阶段通常称为 uptraining 或 conversion fine-tuning。

检查项观察内容异常的含义
转换前后的 Logit 一致性相同输入的 Logits 发生变化，但不应完全失控头重塑、平均维度或 RoPE 位置维度错误
短上下文验证损失转换后损失增加，uptraining 后部分恢复若无法恢复，可能是 H_kv 过小或数据不匹配
长上下文任务检查检索、引用、代码和多轮对话K/V 多样性降低可能首先损害复杂的长距离依赖
解码 KV 字节数减少了 H_q/H_kv若未减少，可能重复的 K/V 被写入了缓存
TPOT / tokens/s在解码阶段改善更明显；预填充阶段改善幅度可能不同瓶颈可能在于权重带宽、调度、采样或网络

3. 优化器和检查点不能直接复用

转换后，K/V 投影的形状发生变化，原始优化器状态中对应的 AdamW m/v 将不再具有相同的形状。安全的做法是将转换后的模型视为新的检查点：保留可映射的权重，为形状发生变化的部分重新初始化优化器状态，并在配置中明确指定 n_heads、n_kv_heads、头分组、RoPE 设置和缓存布局。否则，在恢复训练或加载用于推理时，state_dict 的键可能匹配但语义不一致。

4. 报告结论必须同时包含质量和系统收益

一份严谨的 MHA→GQA 转换报告至少应包含：

conversion: MHA H_q=32,H_kv=32 -> GQA H_q=32,H_kv=8 init: group-mean K/V projection, Q/O unchanged training: uptraining tokens, lr schedule, data mixture, validation loss quality: short-context PPL, long-context retrieval, code/math/chat slices serving: KV cache GB/request, P50/P95 TPOT, throughput at target concurrency boundary: tasks where quality regressed or cache was not the bottleneck

这也是本章与第 10 章的联系点：GQA 的架构压缩只有在推理工作负载中进行度量时，才构成一个完整的工程闭环。如果质量下降超过了服务收益，或者瓶颈实际上在于预填充、网络、RAG 检索或采样后处理，那么减少 KV 头可能不是正确的优化方向。

def repeat_kv_heads(kv, n_rep): """kv: [B, H_kv, T, D] -> [B, H_kv * n_rep, T, D]""" if n_rep == 1: return kv return kv.repeat_interleave(n_rep, dim=1) class GroupedQueryAttention(nn.Module): """Grouped Query Attention — KV heads < Q heads""" def __init__(self, d_model, n_heads, n_kv_heads, dropout=0.1): super().__init__() assert d_model % n_heads == 0 assert n_heads % n_kv_heads == 0 self.d_model = d_model self.n_heads = n_heads self.n_kv_heads = n_kv_heads self.d_k = d_model // n_heads self.n_rep = n_heads // n_kv_heads self.W_q = nn.Linear(d_model, n_heads * self.d_k, bias=False) self.W_k = nn.Linear(d_model, n_kv_heads * self.d_k, bias=False) self.W_v = nn.Linear(d_model, n_kv_heads * self.d_k, bias=False) self.W_o = nn.Linear(d_model, d_model, bias=False) self.dropout = nn.Dropout(dropout) def forward(self, x, mask=None): batch, seq_len, _ = x.shape q = self.W_q(x).reshape(batch, seq_len, self.n_heads, self.d_k) k = self.W_k(x).reshape(batch, seq_len, self.n_kv_heads, self.d_k) v = self.W_v(x).reshape(batch, seq_len, self.n_kv_heads, self.d_k) q = q.transpose(1, 2) # (b, n_heads, s, d_k) k = k.transpose(1, 2) # (b, n_kv_heads, s, d_k) v = v.transpose(1, 2) k = repeat_kv_heads(k, self.n_rep) # (b, n_heads, s, d_k) v = repeat_kv_heads(v, self.n_rep) scores = torch.matmul(q, k.transpose(-2, -1)) / (self.d_k ** 0.5) if mask is not None: scores = scores.masked_fill(mask == 0, float('-inf')) attn_weights = F.softmax(scores, dim=-1) attn_weights = self.dropout(attn_weights) out = torch.matmul(attn_weights, v) out = out.transpose(1, 2).contiguous() out = out.reshape(batch, seq_len, self.d_model) out = self.W_o(out) return out, attn_weights

4.7 DeepSeek MLA——将 KV 压缩到隐空间

GQA 的局限性

GQA 将 KV head 的数量从 减少到了 ，但每个 KV head 仍然存储的是一个 维向量。我们能否对单个 KV 向量本身进行压缩？这便是 DeepSeek MLA 的核心创新。

低秩分解的核心思想

MLA（Multi-head Latent Attention）将 KV 存储视为一个低秩近似问题：

MLA 低秩压缩

其中 是压缩矩阵， 是解压矩阵。关键点在于：。

矩阵吸收——减少显式解压的开销

MLA 真正的巧妙之处在于推理过程：解压矩阵 可以被吸收到 Q 投影 中，因为：

矩阵吸收

这意味着在推理时：

  • 在计算每一层的 Q 时，先对 Q 应用 变换；在工程实现中，这可以与 Q 投影融合，从而减少显式解压带来的额外开销，但仍然会产生投影、RoPE 分支和注意力计算的计算成本
  • K 可以直接与隐向量 做点积，无需解压
  • KV Cache 只需要存储隐向量

Ch04 作业中的 mla_absorbed_attention_scores(q, latent_kv, up_k_weight) 函数要求学习者验证这种等价性。输入 q 的形状是 [B,H,T_q,D]，隐层缓存是 [B,T_k,L]，K 解压矩阵权重是 [H*D,L]。参考测试首先显式计算 K = W_{UK} c 得到 qK^T，然后将 W_{UK} 按 head 重排成 [H,D,L]，并计算 (q W_{UK}) c^T；两者必须逐元素相等。这一步表明，MLA 的缓存节省来源于线性代数的结合律，而非让 K/V 信息“凭空消失”。

DeepSeek-V2/V3 中的配置

参数值描述

与 MHA 相比：MLA 只缓存一个共享的隐向量 （使用时从中分别重建 K 和 V），再加上一个解耦的 RoPE 键 ，总计大约 维/token；相比之下，MHA 是 ——约 28 倍压缩（DeepSeek 报告减少约 93%）。注意：缓存的是单个隐向量，而不是 K 和 V 各一个。

示例演练：RoPE 如何将相对位置编码到点积中

RoPE 将每两个维度视为一个 2D 平面，并根据 token 位置旋转 Q 和 K。假设在某个频率下，位置前进 1 对应的旋转是 30°，旋转前 q = [1, 0]，k = [1, 0]。如果 query 在位置 5，key 在位置 2，它们分别被旋转了 150° 和 60°，那么点积等于它们之间夹角 90° 的余弦：

2D RoPE 点积

如果整个文本段向右平移 10 个位置，query 变成位置 15，key 变成位置 12。两者的绝对旋转角度都变了，但相对差仍然是 3 个位置，点积仍然是 cos(90°)=0。这是 RoPE 的核心特性：注意力得分能够感知相对距离，而不仅仅是记住绝对位置索引。

这个性质来源于旋转矩阵的正交性和可加性：R_m^T R_n = R_{n-m}。正是因为旋转依赖于位置 m, n，所以不能随意与 MLA 的固定解压矩阵互换。内容向量可以进行矩阵吸收，但位置旋转部分通常需要保留并单独计算。

解耦 RoPE 的必要性

MLA 面临一个关键问题：RoPE 旋转本身是一个线性正交变换，但它依赖于 token 位置；这种位置相关的旋转通常不能与固定的解压矩阵互换，也无法统一吸收到同一个 Q 投影中。如果将 RoPE 应用于解压后的 K，那么内容部分的矩阵吸收就不再直接有效。

DeepSeek 的解决方案是解耦 RoPE：将 K 分成两部分——内容部分（通过低秩压缩/解压）和 RoPE 部分（单独投影，不压缩）。在推理时，只有内容部分参与矩阵吸收，而 RoPE 部分则单独计算其点积贡献。

🔑
为什么 MLA 重要： 在标准 MHA 中，KV Cache 占用了 GPU 总内存的 30-40%（长上下文时甚至更高）。MLA 将 KV Cache 减少了约 93%，使得 DeepSeek-V2/V3 的推理成本仅为同等规模 Llama 3 的约 1/7。这是 DeepSeek 经济高效模型家族的核心技术支柱之一。

mla_cache_per_token = 512                       # single shared latent vector c^KV (not 2×)
print(f"Compression ratio: {mha_cache_per_token / mla_cache_per_token:.0f}x")  # 16x
class MLA(nn.Module): """Simplified Multi-head Latent Attention — Low-Rank KV Compression""" def __init__(self, d_model, n_heads, d_latent, dropout=0.1): super().__init__() assert d_model % n_heads == 0 self.d_model = d_model self.n_heads = n_heads self.d_k = d_model // n_heads self.d_latent = d_latent # Q Projection self.W_q = nn.Linear(d_model, n_heads * self.d_k, bias=False) # KV Low-Rank Compression (shared compression matrix) self.W_dkv = nn.Linear(d_model, d_latent, bias=False) # KV Decompression self.W_uk = nn.Linear(d_latent, n_heads * self.d_k, bias=False) self.W_uv = nn.Linear(d_latent, n_heads * self.d_k, bias=False) self.W_o = nn.Linear(n_heads * self.d_k, d_model, bias=False) self.dropout = nn.Dropout(dropout) def forward(self, x, mask=None): batch, seq_len, _ = x.shape n_heads, d_k, d_latent = self.n_heads, self.d_k, self.d_latent # Q Projection q = self.W_q(x).reshape(batch, seq_len, n_heads, d_k) q = q.transpose(1, 2) # (b, n_heads, s, d_k) # KV: Compress -> Decompress latent = self.W_dkv(x) # (b, s, d_latent) k = self.W_uk(latent).reshape(batch, seq_len, n_heads, d_k) v = self.W_uv(latent).reshape(batch, seq_len, n_heads, d_k) k = k.transpose(1, 2) v = v.transpose(1, 2) # Standard Attention scores = torch.matmul(q, k.transpose(-2, -1)) / (d_k ** 0.5) if mask is not None: scores = scores.masked_fill(mask == 0, float('-inf')) attn_weights = F.softmax(scores, dim=-1) attn_weights = self.dropout(attn_weights) out = torch.matmul(attn_weights, v) out = out.transpose(1, 2).contiguous() out = out.reshape(batch, seq_len, self.d_model) out = self.W_o(out) return out, attn_weights, latent # latent is used to simulate KV Cache

💡 在实际的 MLA 实现中，解压缩矩阵 在推理时被吸收到 Q 投影中，因此每一步都不需要实际的“解压缩”操作。为了教学清晰，我们在这里显式地执行完整的压缩->解压缩流程。

4.9 编程练习 5：对比 MHA、GQA 和 MLA 的 KV Cache 大小

现在让我们定量比较三种注意力机制的 KV Cache 大小。这对于理解它们在长上下文场景中的实际影响至关重要。

def compute_kv_cache_size(d_model, n_heads, n_kv_heads, d_latent, seq_len): """Calculate KV Cache size for MHA / GQA / MLA""" d_k = d_model // n_heads bytes_per_float = 4 # float32 # MHA: cache K of n_heads * d_k + V of n_heads * d_k per token mha_per_token = 2 * n_heads * d_k mha_total = mha_per_token * seq_len * bytes_per_float # GQA: cache K of n_kv_heads * d_k + V of n_kv_heads * d_k per token gqa_per_token = 2 * n_kv_heads * d_k gqa_total = gqa_per_token * seq_len * bytes_per_float # MLA: cache only one shared latent vector c^KV (d_latent) per token, reconstruct K and V during decompression mla_per_token = d_latent mla_total = mla_per_token * seq_len * bytes_per_float def fmt_gb(bytes_): return f"{bytes_ / (1024**3):.1f} GB" print(f"{'Type':<8} {'Per Token (floats)':<20} {'Per Token (KB)':<16} {'128K Total Size':<12}") print("-" * 60) print(f"{'MHA':<8} {mha_per_token:<20,} {mha_per_token*4/1024:<16.1f} {fmt_gb(mha_total)}") print(f"{'GQA':<8} {gqa_per_token:<20,} {gqa_per_token*4/1024:<16.1f} {fmt_gb(gqa_total)}") print(f"{'MLA':<8} {mla_per_token:<20,} {mla_per_token*4/1024:<16.1f} {fmt_gb(mla_total)}") print() print(f"GQA vs MHA: {mha_per_token/gqa_per_token:.1f}x savings") print(f"MLA vs MHA: {mha_per_token/mla_per_token:.1f}x savings") return { 'mha_per_token': mha_per_token, 'gqa_per_token': gqa_per_token, 'mla_per_token': mla_per_token, } # DeepSeek-V2/V3 configuration compute_kv_cache_size( d_model=7168, n_heads=64, n_kv_heads=8, d_latent=512, seq_len=131072 )

💡 预算单位：每个 token 的元素数量仅适用于比较架构；实际部署的内存还需乘以 batch_size * n_layers * seq_len * dtype_bytes。Ch04 作业中的 compare_kv_cache_budget 函数需要将这些维度整合到一张表中，以避免将每 token 的比值误认为是整机内存。

4.10 本章小结

从单头注意力出发，你沿着注意力机制的演进路径实现了三代多头注意力变体：

  • ✅ 多头注意力 (MHA) — 并行 个子空间，参数量与单头相同但表示能力更强
  • ✅ 头冗余与压缩原理 — 理解为什么名义上的头数并不等于必须缓存的独立 K/V 信息量
  • ✅ 参数量等价性验证 — 从代码层面证明 MHA 与单头注意力的参数量完全相同
  • ✅ 分组查询注意力 (GQA) — 减少 KV 头数，广泛应用于 Llama 2/3
  • ✅ 多头潜在注意力 (MLA) — 将 KV 低秩压缩到潜在空间，只缓存共享的潜在向量，实现约 28x 的 KV Cache 压缩
  • ✅ KV Cache 量化对比 — 128K 上下文下：MHA 7GB vs GQA 0.9GB vs MLA 0.25GB

这种从 MHA 到 GQA 再到 MLA 的演进路径，本质上是在模型质量、推理效率与内存约束之间寻求更好的平衡。在下一章中，我们将把这些注意力组件与 FFN、RMSNorm 等模块结合起来，构建一个完整的 Transformer Block。
