6.1 GPT-2 架构总览 — 从 Token ID 到 Logits 的完整数据流

在积累前 5 章内容之后，我们终于可以将所有组件组装成一个完整的 GPT 模型。GPT-2 small（124M）的架构可以概括为三个主要阶段：

  1. 嵌入阶段（Embedding Stage）：Token Embedding + Position Embedding → 将离散的 token ID 映射为连续向量
  2. 计算阶段（Computation Stage）：12 层 Transformer Block 堆叠 → 逐层变换表示
  3. 输出阶段（Output Stage）：最终 Norm + LM Head → 将隐藏状态映射到词表概率分布

GPT-2 与后续仅解码器（Decoder-Only）模型的关键区别

  • 绝对位置编码（Absolute Position Encoding）：GPT-2 使用可学习的位置嵌入（1024 个位置 × 768 维），而非 RoPE。RoPE 被后续模型如 Llama 和 DeepSeek 采用。
  • 带偏置的 LayerNorm：GPT-2 在每个 Block 中使用带偏置的 LayerNorm（而非 RMSNorm）
  • GELU 激活函数：GPT-2 的 FFN 使用 GELU，而非 SwiGLU
  • 权重共享（Weight Tying）：GPT-2 在 Embedding 和 LM Head 之间共享权重

在后续章节中，我们将看到现代仅解码器模型如何改变这些设计选择，以实现更长的上下文、更好的稳定性和更低的推理成本。

6.3A 从 Logits 到语言建模目标：为什么输出是 [batch, seq, vocab]

完整 GPTModel 的前向传播到 logits 并不是终点。语言模型训练的真正优化目标是：在每个位置 ，利用当前位置的隐藏状态来预测下一个 token 。因此，logits[:, t, :] 预测的不是当前位置的 token，而是下一个位置 token 的类别分布。

自回归分解

在训练时，输入和标签通常错开一个位置：

# tokens: [BOS, The, cat, sat, EOS] inputs = tokens[:, :-1] # [BOS, The, cat, sat] labels = tokens[:, 1:] # [The, cat, sat, EOS] logits = model(inputs) # (batch, seq-1, vocab) loss = F.cross_entropy(logits.reshape(-1, vocab), labels.reshape(-1))

第 6 章作业中的 causal_lm_loss_from_logits(logits, input_ids) 函数要求学习者实现这一步：将 logits[:, :-1, :] 与 input_ids[:, 1:] 对齐，并使用 ignore_index 跳过 padding 或不参与训练的位置。该函数不应使用 logits[:, 1:, :]，也不应让 logits[:, t, :] 预测 input_ids[:, t]。

Teacher Forcing 与并行训练

训练阶段并不涉及让模型逐步生成并把结果反馈回去，而是使用真实的前缀并行预测所有下一个 token。这种方法称为 teacher forcing。它可以让一个长度为 的序列在一次前向传播中产生 个分类任务，从而在训练时充分利用 GPU 的并行能力。推理阶段则不同：模型只能使用自己已经生成的 token，逐步扩展上下文。

Softmax 只在 Loss 中需要

LM Head 输出的是未归一化的分数，logits。训练时，logits 直接传入 F.cross_entropy，因为它内部会稳定地计算 log-softmax 和负对数似然：

下一个 token 的交叉熵

不要在模型的前向传播中先对 logits 应用 softmax 再传给交叉熵，否则会牺牲数值稳定性，导致梯度变差。

因果泄漏的简单示例

如果因果掩码不正确，模型在位置 可能看到 ，导致训练损失异常容易下降，但推理时这些信息不可用。一个简单的检查方法是：固定前缀，只改变未来的 token，比较前缀位置的 logits 是否变化。

a = torch.tensor([[10, 20, 30, 40]]) b = torch.tensor([[10, 20, 99, 88]]) # 前两个 token 相同，后面的 token 不同 logits_a = model(a) logits_b = model(b) assert torch.allclose(logits_a[:, :2], logits_b[:, :2], atol=1e-5)

这个测试不是质量评估，而是因果结构检查：前缀位置的输出不能依赖于未来的 token。它把第 3 章的因果掩码、第 6 章的 logits 形状以及第 7 章的下一个 token 损失整合到了同一个训练目标中。

6.3B 动手示例：手动计算一个批次的标签偏移与损失分母

假设一个批次包含两个样本，均已填充到长度5。对应于 PAD 的标签由 -100 表示，该参数被传入 F.cross_entropy(..., ignore_index=-100) 以跳过这些位置：

sample 1 input_ids: [BOS, I, like, NLP, EOS] sample 2 input_ids: [BOS, GPT, EOS, PAD, PAD]

模型前向传播仍会为每个输入位置输出一个词汇表分类分布，形状为 [2, 5, |V|]。在计算下一个 token 的损失时，只使用前 4 个位置的 logits，它们负责预测右移一个位置后的 token：

用到的 logits： 位置 0 1 2 3 样本 1 预测： BOS -> I, I -> like, like -> NLP, NLP -> EOS 样本 2 预测： BOS -> GPT, GPT -> EOS, EOS -> PAD, PAD -> PAD 忽略前的标签： 样本 1 标签： [I, like, NLP, EOS] 样本 2 标签： [GPT, EOS, PAD, PAD] 经过填充掩码后的标签： 样本 1 标签： [I, like, NLP, EOS] 样本 2 标签： [GPT, EOS, -100, -100]

因此，这个批次的平均损失除以 6 个有效的预测位置，而不是 2 * 4 = 8 个位置，也不是原始序列长度 2 * 5 = 10 个位置。最后一个输入位置的 logits 没有对应的“下一个 token”标签，而填充位置也不代表真正的语言建模目标。

使用 ignore_index 的批次平均

这就是实现过程中需要先构造 shift_logits = logits[:, :-1, :] 和 shift_labels = input_ids[:, 1:]，然后将 shift_labels 中的填充位置改为 ignore_index 的原因。如果使用 logits[:, 1:, :] 来预测 input_ids[:, 1:]，模型就会用位置 t 的隐状态去预测同一位置的 token，导致训练目标退化为“重复输入 token”，不再是自回归语言建模。

6.4 理论：权重绑定（Weight Tying）——哪些模型使用，哪些不用，为什么？

权重绑定（Weight Tying）让嵌入层（Token Embedding）和输出层（LM Head）共享同一个权重矩阵。更准确地说，输入侧使用第 个行向量表示 token ；输出侧计算隐藏状态 与每个 token 行向量的点积，得到词表 logits。这在数学上不是逆映射，而是同一套 token 向量同时承担“输入表示”和“输出分类原型”的双重角色。

权重绑定共享方式

参数量影响

对于 GPT-2 small：词表大小 50257 × d_model 768 = 38.6M 参数——占模型总参数 124M 的 31%。使用绑定时，这 38.6M 参数只计一次。

为什么一些大型模型不用它

随着非嵌入层参数占比增大，权重绑定节省的参数比例下降：在数百亿甚至千亿参数的模型中， 往往不再是总参数量的主导因素。不绑定 LM Head 的好处是输入表示和输出分类器可以独立学习，适用于需要更大输出容量、特殊词表设计或多模态输出接口的架构。但这并非固定规则：是否使用绑定取决于词表大小、隐藏维度、总参数量级、训练稳定性以及模型家族工程上的权衡。

对反向传播的影响

当绑定时，嵌入层的梯度来自两条路径：一条来自嵌入查找路径，另一条来自 LM Head 分类路径。输入序列用到的 token 行会收到来自查找路径的梯度；softmax 分类器中出现的所有 token 行都会通过输出分布收到梯度。这种耦合有时起到正则化作用，有时又限制了输入侧和输出侧独立适应任务的能力。不能简单断言“小模型一定受益，大模型一定受损”。

绑定的 LM Head 梯度

其中 是交叉熵关于 logits 的梯度：对于有效的 token 行，它是 (softmax - one_hot) / N_valid；被 ignore_index 跳过的行全为零。Ch06 作业中的 tied_lm_head_gradients 函数要求学习者手动计算这三步，并与 autograd 对齐。

6.6A 工程视角：一个完整的GPT实现必须同时满足哪些条件

连接GPT组件之后，工程上还需要回答三个问题：训练目标是否与数据对齐，参数和内存账本是否可解释，以及自回归约束在推理阶段是否被遵循。下面，我们把本章的关键实现点串成一份工程检查清单，它对应的是模型设计能力，而不仅仅是单元测试。

1. Tensor Contract：从 token id 到 next-token loss

一个完整 decoder-only LM 的前向合约是：

input_ids: [B, T] token + pos emb: [B, T, d_model] blocks output: [B, T, d_model] logits: [B, T, vocab_size]

这里，logits[:, t, :] 预测的是 input_ids[:, t+1]，而不是当前位置的 token。因此，训练损失的核心对齐方式是：

shift_logits = logits[:, :-1, :] shift_labels = input_ids[:, 1:] loss = F.cross_entropy( shift_logits.reshape(-1, vocab_size), shift_labels.reshape(-1), ignore_index=-100, )

最后一个输入位置没有下一个 token 标签；填充位或不参与训练的位置必须通过 ignore_index 排除。这种对齐错误通常不会引起形状错误，但它会把语言建模目标变成"重复当前位置的 token"，是初学者实现中最隐蔽的错误之一。

2. Parameter Ledger：为什么 GPT-2 small 是 124,439,808

GPT-2 small 的参数数量不是一个需要记住的数字，而是架构账本的结果。默认配置是 V=50257, T_max=1024, d=768, L=12, d_ff=3072。主要的参数项有：

token embedding: V * d position embedding: T_max * d per block attention: d * 3d + 3d bias + d * d + d bias per block MLP: d * 4d + 4d bias + 4d * d + d bias per block LayerNorm: 4d final LayerNorm: 2d LM head: 0 if tied, V * d if untied

在 weight tying 下，LM head 和 token embedding 是同一个参数对象，只计数一次。训练工程师需要能将总参数分解为 embedding、position embedding、blocks、final norm 和 LM head，因为优化器状态、检查点大小、tensor parallelism 分区以及内存预算都依赖于这种分解。

3. Weight Tying：共享参数并不意味着只更新目标 token

当权重共享时，输出层的 logits 写作 。交叉熵梯度首先作用于所有 vocab logits，然后反向传播到隐藏状态和 embedding 矩阵：

Tied LM Head 梯度路径

因此，不对应目标 token 的行也会通过 softmax 概率接收到输出梯度。被输入序列索引到的 token 行也会从 embedding lookup 路径接收梯度。理解这一点之后，weight tying 就不仅仅是"节省参数"，而是一种建模约束，它强制输入 token 空间和输出分类器共享一个几何结构。

4. Causal mask：推理约束必须在训练期间成立

训练时，teacher forcing 允许模型一次看到整个序列；但模型内部的每个位置仍然只能关注过去和当前的位置。稠密注意力的因果约束是：

allowed[i, j] = j <= i scores[~allowed] = -inf weights = softmax(scores)

如果 mask 方向反了或 broadcast 维度不正确，训练损失可能会异常平滑地下降，因为模型在偷看未来的 token；然而在推理时，未来的 token 并不存在，质量会崩溃。一个小型的 sanity check 是固定前缀，替换未来 token，并验证过去位置的 logits 是否不变。

5. Initialization and Positional Encoding：GPT-2 公式不能与现代 Llama 公式混用

GPT-2 风格的实现使用学习到的绝对位置 embedding、LayerNorm、GELU MLP、带 bias 的线性层以及常见的 N(0, 0.02) 初始化。Llama/DeepSeek 风格的模型通常使用 RoPE、RMSNorm、SwiGLU 和无 bias 的投影层。两种模型类型都是 decoder-only Transformer，但它们在参数数量公式、权重加载方式和训练稳定性细节上有所不同。

如果输入长度超过 max_seq_len，学习到的位置 embedding 没有对应的行，模型要么报错，要么显式扩展位置表。这与 RoPE 的长上下文外推问题不同，不能用相同的推理来解释。

6. MoE：总容量和每 token 计算量必须分开讨论

MoE 将 FFN 替换为多个专家，路由器为每个 token 选择少数几个专家。对于一个无 bias 的 SwiGLU 专家：

expert_params = 3 * d_model * expert_hidden total_expert_params = (n_routed_experts + shared_experts) * expert_params activated_expert_params_per_token = (top_k + shared_experts) * expert_params capacity_to_compute_ratio = total_expert_params / activated_expert_params_per_token

这表明"模型总参数"和"每 token 激活的计算量"不是同一个指标。在推理工程中，MoE 还引入了 dispatch/combine、跨设备通信、专家负载偏斜和尾部延迟。因此，较小的 top-k 不一定更快，更多的专家也不一定更好。

🔑
本节要点： 完整 GPT 组装的核心能力是在一个系统中同时理解 forward shape、next-token 目标、参数数量、梯度路径、因果约束以及稀疏激活的资源预算。只有这样，第 7 章的训练循环和第 10 章的推理服务才能拥有可靠的模型基础。

3. 为什么 GPT-2 权重转置容易出错

PyTorch 的 nn.Linear(in_features, out_features) 参数形状为 [out_features, in_features]，其前向传播等价于 x @ W.T + b。早期 HuggingFace 对 GPT-2 的实现遵循了 OpenAI 的 Conv1D 命名惯例，其中内部权重通常以 [in_features, out_features] 格式存储。因此，当将 HF GPT-2 权重加载到自定义的 nn.Linear 时，许多投影矩阵需要进行转置。

def copy_linear_from_hf(linear, hf_weight, hf_bias=None, transpose=True): src = hf_weight.t() if transpose else hf_weight assert linear.weight.shape == src.shape linear.weight.data.copy_(src) if hf_bias is not None: assert linear.bias.shape == hf_bias.shape linear.bias.data.copy_(hf_bias) # 示例：HF GPT-2 的 c_attn 是一个融合的 QKV 投影 # 如果自定义实现也使用 nn.Linear(d_model, 3 * d_model)，通常需要 transpose=True

是否进行转置不能仅通过名称判断；你必须检查双方的形状和前向计算公式。如果你的自定义实现将 Q、K、V 拆成了三个独立的线性层，你必须先将 HF 融合的 QKV 权重沿着输出维度切分成三部分，然后分别复制到 q_proj/k_proj/v_proj。

4. Logit 一致性的诊断顺序

加载预训练权重后，最实用的等价性检查是比较同一批 input_ids 的 logit。推荐的诊断顺序从受训练随机性影响最小的项开始：

  1. 确认双方使用相同的 tokenizer，并且 input_ids 完全相同。
  2. 确认 config 完全一致：层数、隐藏层大小、注意力头数、上下文长度、LayerNorm eps、是否使用 bias、位置编码类型。
  3. 确认双方均处于 eval() 模式且 dropout 已禁用。
  4. 确认所有已加载参数的 missing_keys 和 unexpected_keys 都能够解释；不要将缺失的关键投影层视为无害警告。
  5. 逐层比较隐藏状态：嵌入层之后、每个 block 之后、最终 norm 之后、lm head 之后，找到产生较大误差的第一个位置。
  6. 如果误差始于嵌入层之后，优先检查 token embedding、position embedding 和 position ids；如果误差始于注意力层之后，优先检查 QKV 顺序、掩码方向、注意力头重塑以及投影转置。

在 fp32、相同 tokenizer、相同架构且无 dropout 的情况下，GPT-2 small 的最大绝对 logit 误差通常应接近浮点舍入误差的量级。如果误差达到 1e-2 或更大，通常并非“预训练模型本身存在差异”，而是加载映射或前向实现仍存在语义不一致。

6.8 理论：MoE 架构——分离容量与每 token 计算量

混合专家模型（MoE） 是 LLM 在 GPT-2 架构之上最重要的扩展之一。它将每一层中的 FFN 替换为多个“专家”网络，并且每个 token 只激活其中少数几个。这就产生了稀疏激活：模型可以拥有极大的总参数量，而前向传播的成本主要取决于被激活的参数。

DeepSeekMoE 的核心设计

MoE 前向计算

以常见的 DeepSeekMoE 风格设计为例，本课程只关注可迁移的结构要点：

  • 多个路由专家： 路由器为每个 token 选择 top-K 个专家，而不是激活所有专家。
  • 共享专家： 所有 token 都经过少数共享专家，以保留一条通用的变换路径。
  • 细粒度专家： 单个专家通常比密集 FFN 更小，但专家总数更多。
  • 核心问题： 总参数量决定容量上限；每 token 激活的专家数决定单次前向传播的计算成本；负载均衡决定吞吐稳定性。

为什么只激活少数专家也能工作

MoE 的核心假设是条件计算：不同的 token 不需要经过完全相同的 FFN 参数。路由器根据隐藏状态选择专家，将不同的输入模式——如数学符号、代码片段、实体名称和格式 token——引导到不同的参数子集。这个假设是否成立取决于训练数据、路由器的表示能力、专家容量以及负载均衡；如果路由器坍缩到少数几个专家上，MoE 就会退化为一种昂贵又不稳定的密集模型近似。

🔑
核心权衡： MoE 使用更多总参数来增加容量，并用 top-K 稀疏激活来控制每 token 计算量。但它也引入了额外问题：路由器训练、专家负载均衡、跨设备通信以及批次内 token 分发；这些系统开销必须与容量收益一起分析。

同样的权衡可以用可计算的参数量预算来形式化。如果每个专家是一个无偏置的 SwiGLU FFN，那么单个专家的参数量是 3 * d_model * d_ff，对应三个线性矩阵：gate、up 和 down。专家总参数包括所有路由专家和共享专家；每 token 激活的专家参数只包含 top_k 个路由专家和共享专家。路由器本身还包含 d_model * n_experts 个门控参数。这就把“模型容量来自所有专家”和“单 token 计算只经过少数专家”分离开来。

示例练习：MoE 负载均衡损失同时考虑概率和赋值

假设 4 个 token 在 3 个专家上的路由器概率分别为 [0.7,0.2,0.1]、[0.6,0.3,0.1]、[0.1,0.8,0.1] 和 [0.2,0.2,0.6]，top-1 选中的专家 id 为 [0,0,1,2]。实际赋值比例 load 是 [0.5,0.25,0.25]；平均路由器概率是 [0.4,0.375,0.225]。Switch 风格的辅助损失可写作 n_experts * sum_i load_i * prob_i，因此对这个例子它是 3 * (0.5*0.4 + 0.25*0.375 + 0.25*0.225) = 1.05。

这个损失的目的不是让每个 token 平等激活所有专家，而是惩罚批次级别的路由器坍缩。只看 top-k 计数会忽略路由器 softmax 是否已经偏向少数专家；只看平均概率会忽略实际的 top-k 赋值是否已经拥塞。因此，Ch06 作业中的 moe_load_balance_loss 同时返回 load_fraction、mean_router_prob 和损失。

load_ratio) self.bias += delta self.bias.clamp_(-self.bias_clip, self.bias_clip) def apply_bias(self, gate_logits): """将动态偏置应用到路由 logits""" return gate_logits + self.bias.to(gate_logits.device) def get_load_stats(self, token_counts): """返回负载均衡统计信息""" total = token_counts.sum().float() load_ratio = token_counts / total if total > 0 else token_counts ideal = 1.0 / self.n_experts return { 'max_load': load_ratio.max().item(), 'min_load': load_ratio.min().item(), 'std_load': load_ratio.std().item(), 'ideal_load': ideal, } # 集成测试 router = MoERouter(768, n_experts=256, top_k=8) balancer = AuxLossFreeBalancer(256) x = torch.randn(4, 64, 768) # 模拟训练步骤 for step in range(10): weights, indices = router(x, balancer.bias) # 统计信息 token_counts = torch.zeros(256) token_counts.scatter_add_(0, indices.flatten(), torch.ones_like(indices.flatten())) balancer.update(token_counts) if step in [0, 4, 9]: stats = balancer.get_load_stats(token_counts) print(f"Step {step}: max={stats['max_load']:.6f}, " f"min={stats['min_load']:.6f}, std={stats['std_load']:.6f}")

💡 动态偏置 vs. 辅助损失：辅助损失通过梯度反向传播微调路由器参数以平衡负载，计算成本低但引入了一个额外的超参数 α。动态偏置不经过反向传播——它是在前向传播过程中对路由 logits 的直接调整。DeepSeek-V3 报告在其训练设置下，动态偏置达到的平衡效果与辅助损失相当，并减少了调优辅助损失权重的需要；但针对不同模型和数据分布仍需要实际测量。
