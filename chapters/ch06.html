6.1 GPT-2 架构全景

经过前 5 章的积累，我们终于可以组装一个 完整的 GPT 模型了。GPT-2 的架构可以用三个主要阶段来概括：

  1. Token Embedding + Position Embedding — 将 token ID 映射为稠密向量并加入位置信息
  2. 12x Transformer Block (Decoder) — 每层包含 Masked Self-Attention + FFN
  3. LM Head — 将最终 hidden state 映射回词表概率分布

完整的数据流图如下：

┌──────────────────────────────────────────────────────────┐ │ GPT-2 (完整模型) │ │ │ │ Token IDs: [t1, t2, ..., tn] (词表大小 50257) │ │ │ │ │ ▼ │ │ ┌─────────────┐ ┌──────────────────┐ │ │ │ Token Emb. │ │ Position Emb. │ │ │ │ (50257×768) │ │ (1024×768) │ │ │ └──────┬──────┘ └────────┬─────────┘ │ │ └────────── (+) ─────┘ │ │ ▼ │ │ ┌──────────────────┐ │ │ │ Dropout │ │ │ └──────────────────┘ │ │ │ │ │ ▼ (×12 层) │ │ ┌────────────────────┐ │ │ │ TransformerBlock 1 │ d_model=768, n_heads=12 │ │ ├────────────────────┤ │ │ │ TransformerBlock 2 │ │ │ ├────────────────────┤ │ │ │ ... │ │ │ ├────────────────────┤ │ │ │ TransformerBlock12 │ │ │ └─────────┬──────────┘ │ │ │ │ │ ▼ │ │ ┌──────────────────┐ │ │ │ RMSNorm │ (最后输出归一化) │ │ └──────────────────┘ │ │ │ │ │ ▼ │ │ ┌──────────────────┐ │ │ │ LM Head │ (nn.Linear(768, 50257)) │ │ └──────────────────┘ │ │ │ │ │ ▼ │ │ logits: (batch, seq, vocab=50257) │ └──────────────────────────────────────────────────────────┘

输入是 token ID 序列（[batch, seq_len]），输出是词表上的概率分布（[batch, seq_len, vocab_size]）。注意这里 GPT-2 使用 绝对位置编码（可学习的 Position Embedding），而不是 RoPE（RoPE 是 Llama 等后续模型使用的）。

6.2 动手：GPTConfig

首先定义一个配置类来保存模型的超参数。GPT-2 small (124M) 的配置如下：

from dataclasses import dataclass from typing import Optional @dataclass class GPTConfig: """GPT-2 模型配置 — 124M 参数的默认值""" # 词表与序列 vocab_size: int = 50257 # GPT-2 BPE 词表大小 max_seq_len: int = 1024 # 最大序列长度 # 模型维度 d_model: int = 768 # 隐藏层维度 n_heads: int = 12 # 注意力头数 n_layers: int = 12 # Transformer Block 数 d_ff: Optional[int] = None # FFN 中间维度（默认 4*d_model） dropout: float = 0.1 # Dropout 概率 # 注意力配置 attn_dropout: float = 0.1 # Attention dropout bias: bool = True # 线性层是否使用 bias（GPT-2 用，Llama 不用） # 位置编码 use_rope: bool = False # GPT-2 使用绝对位置编码，不启用 RoPE # 权重共享 tie_weights: bool = True # 是否共享嵌入和 LM Head 权重

每个参数的含义：

  • vocab_size=50257 — GPT-2 的 BPE 词表（256 基础字节 + 50000 合并 + 1 EOS）
  • max_seq_len=1024 — GPT-2 支持的最大序列长度
  • d_model=768 — 所有层的隐藏维度，也是每个 token 的表示维度
  • n_heads=12 — 多头注意力头数，768/12 = 64 维每头
  • n_layers=12 — 12 层 Transformer Block 堆叠
  • d_ff=3072 — FFN 中间层维度（4 * 768 = 3072）。如果使用 SwiGLU，调整为 2048

6.3 动手：组装 GPTModel

现在组装完整的 GPTModel。它整合了嵌入层、位置编码、Transformer Block 堆叠、最终归一化和 LM Head：

import torch import torch.nn as nn class GPTModel(nn.Module): """完整 GPT-2 模型 架构: Embedding → 12×TransformerBlock → RMSNorm → LM Head """ def __init__(self, config: GPTConfig): super().__init__() self.config = config # Token 嵌入 (词表大小 × d_model) self.token_embedding = nn.Embedding( config.vocab_size, config.d_model ) # 位置嵌入 (最大序列长度 × d_model) — 绝对位置编码 self.pos_embedding = nn.Embedding( config.max_seq_len, config.d_model ) # Dropout self.dropout = nn.Dropout(config.dropout) # Transformer Block 堆叠 self.blocks = nn.ModuleList([ TransformerBlock( d_model=config.d_model, n_heads=config.n_heads, d_ff=config.d_ff or config.d_model * 4, dropout=config.dropout, ) for _ in range(config.n_layers) ]) # 最终 RMSNorm self.norm = RMSNorm(config.d_model) # LM Head — 将隐藏状态映射回词表概率 self.lm_head = nn.Linear( config.d_model, config.vocab_size, bias=False ) # 权重共享（可选） if config.tie_weights: self.lm_head.weight = self.token_embedding.weight # 初始化权重 self._init_weights() def _init_weights(self): """根据 GPT-2 论文的初始化方案""" for module in self.modules(): if isinstance(module, nn.Linear): nn.init.normal_(module.weight, mean=0.0, std=0.02) if module.bias is not None: nn.init.zeros_(module.bias) elif isinstance(module, nn.Embedding): nn.init.normal_(module.weight, mean=0.0, std=0.02) def forward(self, input_ids, mask=None): """前向传播 Args: input_ids: (batch, seq_len) token ID 序列 mask: 可选的自定义注意力掩码 Returns: logits: (batch, seq_len, vocab_size) """ batch_size, seq_len = input_ids.shape device = input_ids.device # 1. Token Embedding x = self.token_embedding(input_ids) # (b, s, d_model) # 2. Position Embedding pos_ids = torch.arange(seq_len, device=device).unsqueeze(0) x = x + self.pos_embedding(pos_ids) # (b, s, d_model) # 3. Dropout x = self.dropout(x) # 4. Transformer Blocks for block in self.blocks: x = block(x, mask=mask) # 5. 最终归一化 x = self.norm(x) # 6. LM Head — 投影到词表 logits = self.lm_head(x) # (b, s, vocab_size) return logits

关键设计决策：

  • 循环而非顺序模块：使用 nn.ModuleList + for 循环，而不是 nn.Sequential。这便于调试每层的输出、添加层间连接（如 mHC）、以及可视化梯度流
  • 位置编码在嵌入后立即添加：GPT-2 的位置编码与 token 嵌入相加（而非拼接），这是经典做法
  • 最后归一化：在所有 Block 之后再加一次 RMSNorm，确保输出进入 LM Head 前尺度稳定

6.4 Weight Tying

Weight Tying（权重共享）是指让嵌入层（Embedding）和输出映射层（LM Head）共享同一个权重矩阵。这意味着 token_embedding.weight 和 lm_head.weight 是同一个张量。

这个想法最早由 Press & Wolf (2017) 在 Using the Output Embedding to Improve Language Models 中提出。它基于一个直观的观察：嵌入层决定"哪些 token 在语义上相似"，LM Head 决定"下一个 token 应该是什么"——这本质上是同一个任务的"逆"操作。

# Weight Tying 的实现 model = GPTModel(config) # 共享前: 两个不同的参数 print(model.token_embedding.weight.shape) # (50257, 768) print(model.lm_head.weight.shape) # (50257, 768) print(id(model.token_embedding.weight) == id(model.lm_head.weight)) # → True (当 tie_weights=True 时) # 修改嵌入会同时影响 LM Head with torch.no_grad(): old_val = model.token_embedding.weight[0, 0].clone() model.token_embedding.weight[0, 0] = 100.0 assert model.lm_head.weight[0, 0] == 100.0 # 同一张量
模型使用 Weight Tying？参数量影响
GPT-1是-38.6M (50257 × 768)
GPT-2是-38.6M
GPT-3 (175B)否+ ~1.7B
Llama 1/2/3否+ ~1.7B (基于 128K 词表)
DeepSeek-V2/V3否+ ~1.7B (词表 129K+ 更大)

为什么不共享的模型更多？随着模型规模增大，词表投影层的参数量占比越来越小（在 175B 模型中 <1%），共享带来的节省微不足道。而且不共享可以给模型更大的 表达能力——嵌入层和 LM Head 可以学习不同的语义映射，尤其对于大词表模型（如 Llama 3 的 128K 词表）。

💡
反向传播的影响：Weight Tying 使得嵌入层同时接收来自词嵌入梯度和 LM Head 梯度的"双重训练信号"。这在小模型上是一种有效的正则化，但在大模型上，反而可能限制嵌入层的灵活性。

6.5 动手：124M 参数精算

让我们仔细计算 GPT-2 small (124M) 的每一个参数。精确的参数推导对理解模型规模至关重要：

参数分解（GPT-2 small 124M）

逐项详细解释：

  • Token Embedding: 50257 × 768。每个 token 映射为一个 768 维向量。这是整个模型中最大的单层（38.6M，占总参数的 31%）
  • Position Embedding: 1024 × 768。最多支持 1024 个位置，每个位置一个 768 维向量
  • Per Block Attention: Q K V O 四个 768×768 矩阵 + 3 个 bias（Q、K、V 各一个，O 没有 bias 因为 GPT-2 的实现）。4 × 768² + 3 × 768 = 2,361,600
  • Per Block FFN: 768×3072 + 3072×768 = 两个线性层，加上 bias。两层加起来 2 × 768 × 3072 + 768 + 3072 = 4,722,432
  • LM Head: 当启用 Weight Tying 时参数为 0，因为与 Token Embedding 共享权重

6.6 动手：随机权重前向传播测试

在训练之前，先验证模型的前向传播能正确运行——检查输出的形状、数值范围，以及梯度是否能反向传播：

def test_forward_pass(model, config): """测试 GPT 模型的前向传播""" batch_size, seq_len = 2, 64 device = next(model.parameters()).device # 随机 token ID（在词表范围内） input_ids = torch.randint(0, config.vocab_size, (batch_size, seq_len)) # 前向传播 logits = model(input_ids) # --- shape 验证 --- expected_shape = (batch_size, seq_len, config.vocab_size) assert logits.shape == expected_shape, \ f"期望 {expected_shape}，实际 {logits.shape}" print(f"输出 shape 正确: {logits.shape}") # --- 数值范围验证 --- print(f"logits 范围: [{logits.min().item():.2f}, {logits.max().item():.2f}]") print(f"logits 均值: {logits.mean().item():.4f}") print(f"logits 标准差: {logits.std().item():.4f}") # --- 梯度流通验证 --- loss = logits.sum() loss.backward() grad_norm = 0.0 for p in model.parameters(): if p.grad is not None: grad_norm += p.grad.norm().item() ** 2 grad_norm = grad_norm ** 0.5 print(f"总梯度范数: {grad_norm:.4f}") assert grad_norm > 0, "梯度为零，反向传播失败！" print("梯度流通正常 ✓") # --- 内存占用 --- mem_mb = torch.cuda.memory_allocated() / 1024**2 if torch.cuda.is_available() else 0 if mem_mb: print(f"GPU 内存占用: {mem_mb:.1f} MB") print("前向传播测试全部通过 ✓") # 运行测试 config = GPTConfig() model = GPTModel(config) test_forward_pass(model, config)

这个测试看起来简单，但它验证了几个关键假设：

  • 维度正确：从 token ID 到 logits 的完整链条没有被破坏
  • 可微：梯度能正确地从输出流回到所有参数
  • 数值稳定：随机初始化下输出在合理的数值范围内（不会 exploding 到 NaN）

如果模型初始化时 logits 出现 NaN 或梯度爆炸，通常是由于 _init_weights 中的标准差过大（GPT-2 用 0.02，如果设为 0.1 则可能出现问题）。

6.7 HuggingFace 权重加载

为了验证我们的实现与官方 GPT-2 一致，我们可以加载 HuggingFace 的预训练权重并进行比较：

from transformers import GPT2Model, GPT2Config as HFGPT2Config def load_hf_weights_to_model(hf_model, our_model): """将 HuggingFace GPT-2 权重加载到我们的实现中""" state_dict = {} # 映射 HuggingFace 的命名到我们的命名 mapping = { 'wte.weight': 'token_embedding.weight', 'wpe.weight': 'pos_embedding.weight', 'ln_f.weight': 'norm.gamma', 'ln_f.bias': 'norm.beta', # GPT-2 用 LayerNorm (gamma/beta) } for hf_name, param in hf_model.state_dict().items(): # 处理 Block 内部参数 if hf_name.startswith('h.'): # h.0.attn.c_attn.weight → blocks.0.attention.qkv.weight parts = hf_name.split('.') layer_idx = int(parts[1]) # ... 详细的映射逻辑取决于具体实现 else: our_name = mapping.get(hf_name) if our_name: state_dict[our_name] = param our_model.load_state_dict(state_dict, strict=False) # 验证输出一致性 input_ids = torch.randint(0, 50257, (1, 64)) with torch.no_grad(): hf_out = hf_model(input_ids).last_hidden_state our_out = our_model(input_ids) diff = (hf_out - our_out).abs().max().item() print(f"最大差异: {diff:.6f}") assert diff < 1e-4, f"差异过大: {diff}" print("与 HuggingFace 输出一致 ✓")
⚠️
注意事项：HuggingFace 的 GPT-2 使用 LayerNorm（含 bias），而我们的默认实现使用 RMSNorm（无 bias）。在使用权重加载时需要对应调整。另外，GPT-2 的 c_attn 是 QKV 合并的（一个 Linear 输出 3×768=2304 维），这和我们分开的 Q/K/V 需要做 reshape 映射。

6.8 DeepSeekMoE 架构

MoE (Mixture of Experts) 是 GPT-2 架构之外最重要的 LLM 扩展。DeepSeekMoE（DeepSeek-V2, 2024）将每层的 FFN 替换为多个"专家"网络，每个 token 只激活其中少数专家——实现了 稀疏激活。

公式 6.1 MoE 层

其中 是第 i 个专家（通常是一个 SwiGLU FFN）， 是门控（gating）函数，每个 token 只激活 top-k 个专家。

DeepSeekMoE 的核心设计：

  • N=256, K=8: 总共 256 个专家，每个 token 只激活 8 个（top-8 路由）
  • Fine-grained experts: 每个专家的维度较小（d_ff = 1536），通过增加专家数而非扩大每个专家的规模来提升容量
  • Shared experts: 额外 1 个共享专家，所有 token 必定激活，用于捕获"通用知识"
  • 671B 总参数，仅激活 37B: 这就是"671B 参数只激活 37B"的由来——总参数量巨大但计算量只取决于激活参数
┌─ DeepSeekMoE Block ────────────────────────────────┐ │ │ │ 输入 x ────→ Router ──→ top-8 专家选择 │ │ (batch) │ (linear + softmax) │ │ │ │ │ ▼ │ │ ┌──────────────────────────────────────┐ │ │ │ Expert 1 │ Expert 2 │ ... │ Expert N│ │ │ │ (FFN) │ (FFN) │ │ (FFN) │ │ │ └───┬───────┴───┬──────┘ └───┬──────┘ │ │ │ │ │ │ │ └────(+)────┘ 共享 Expert ──┘ │ │ │ (always active) │ │ ▼ │ │ 输出 y = 激活专家输出的加权和 │ └──────────────────────────────────────────────────────┘
class DeepSeekMoE(nn.Module): """DeepSeekMoE 层 — 稀疏激活 Mixture of Experts N=256 个专家, K=8 top-k 路由, 1 个共享专家 """ def __init__(self, d_model, n_experts=256, top_k=8, d_ff=1536): super().__init__() self.n_experts = n_experts self.top_k = top_k # 门控路由器 self.gate = nn.Linear(d_model, n_experts, bias=False) # 256 个专家 FFN self.experts = nn.ModuleList([ SwiGLU(d_model, d_ff) for _ in range(n_experts) ]) # 共享专家（所有 token 都经过） self.shared_expert = SwiGLU(d_model, d_ff) def forward(self, x): batch, seq, dim = x.shape # 1. 路由：计算每个 token 到每个专家的得分 gate_logits = self.gate(x) # (b, s, N) gate_scores = F.softmax(gate_logits, dim=-1) # 2. Top-k 选择 top_k_scores, top_k_indices = torch.topk( gate_scores, self.top_k, dim=-1 ) # (b, s, K), (b, s, K) # 3. 稀疏激活（实际实现中需要更高效的 dispatch） final = torch.zeros_like(x) for b in range(batch): for s in range(seq): for k in range(self.top_k): expert_idx = top_k_indices[b, s, k] weight = top_k_scores[b, s, k] expert_out = self.experts[expert_idx](x[b:b+1, s:s+1]) final[b, s] += weight * expert_out.squeeze() # 4. 添加共享专家 final = final + self.shared_expert(x) return final
🔑
实际实现注意：上述代码仅为教学目的简化版。实际 DeepSeekMoE 使用高效的 dispatch/combine 机制——通过专家并行（Expert Parallelism）在多个 GPU 间分布专家，使用 All-to-All 通信路由 token。计算过程不逐个循环，而是使用诸如 torch.nn.functional.scatter 或 Triton 自定义 kernel 进行批处理。

6.9 Aux-Loss-Free 负载均衡

MoE 的一个核心挑战是 负载均衡——如果路由器倾向于选择少数专家，大多数专家得不到训练，模型容量会退化。

传统方法（GShard, Switch Transformer）使用 辅助损失 (auxiliary loss) 来约束专家负载均衡：

公式 6.2 传统负载均衡辅助损失

其中 是分配给专家 i 的 token 比例， 是路由器分配给专家 i 的平均概率。这个辅助损失 以梯度方式 强制路由器平衡分配。

DeepSeek-V3 提出了 Aux-Loss-Free 负载均衡——放弃辅助损失，改用 动态偏置 (dynamic bias)：

class AuxLossFreeBalancer: """Aux-Loss-Free 负载均衡 — DeepSeek-V3 使用动态偏置替代辅助损失。 当专家过载时降低其偏置，欠载时增加偏置。 """ def __init__(self, n_experts, bias_update_rate=0.01): self.n_experts = n_experts self.bias = torch.zeros(n_experts) self.bias_update_rate = bias_update_rate self.target_load = 1.0 / n_experts def update(self, token_counts): """根据每个专家的 token 计数更新偏置""" # token_counts: (n_experts,) — 各专家分配的 token 数 total = token_counts.sum() load_ratio = token_counts / total # 各专家的负载比例 # 调整偏置: 过载 → 降低偏置, 欠载 → 增加偏置 delta = self.bias_update_rate * (self.target_load - load_ratio) self.bias += delta # 可选: 截断偏置范围 self.bias.clamp_(-0.1, 0.1) def apply_bias(self, gate_logits): """在路由之前应用偏置""" return gate_logits + self.bias.to(gate_logits.device)

Aux-Loss-Free 的核心优势：

  • 无 token 丢弃：传统方法中过载的专家会丢弃 token（Switch Transformer 的容量因子）。Aux-Loss-Free 通过动态偏置让路由自然平衡，不需要丢弃任何 token
  • 零额外梯度：偏置更新是启发式的（不通过反向传播），不干扰主损失函数的梯度
  • 训练更稳定：辅助损失引入了一个额外的超参数 α，需要精细调优。Aux-Loss-Free 彻底消除了这个负担
💡
实际效果：DeepSeek-V3 报告显示，Aux-Loss-Free 方法在 2.788T token 的训练中保持了接近理想的负载均衡（最大专家负载 < 平均负载的 1.05 倍），同时没有引入辅助损失带来的模型质量损失。
