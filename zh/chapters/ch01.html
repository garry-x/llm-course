1.1 Tokenizer 的位置：从文本接口到工程成本

Tokenizer 是 LLM 的第一层接口。它把原始字符串转成 token ID，同时决定序列长度、embedding 矩阵大小、上下文窗口利用率和服务计费口径。本章把 tokenization 当作训练和推理共同遵守的模型合约，而不是一个随手做掉的预处理步骤。

  • 训练侧：改 tokenization 规则，会改 token 分布、batch token 数、embedding 参数量和数据质量诊断。
  • 推理侧：同一段文本的 token 数，会影响上下文容量、KV Cache 显存、延迟和调用成本。
  • 发布侧：checkpoint、chat template、special token 和 tokenizer 文件必须一起版本化，否则模型接口会错位。

下面的实现只保留 BPE 最小、可解释的核心：统计相邻 pair 的频率，按优先级合并，保存 merge table，并在 encoding/decoding 时使用同一套规则。理解这条线之后，再比较 WordPiece、Unigram LM 和 byte-level tokenizer，就会变成工程取舍问题，而不是名词记忆。

1.3 什么是 token：一个信息论视角

LLM 不直接处理字符或字符串，而是处理 token。token 是模型看到的最小文本单元，每个 token 对应一个整数 ID。模型真正拿到的是整数序列，不是自然语言文本。

从信息论看 tokenization

假设一段文本包含的信息量是 H bit。如果用 n 个 token 表示它，每个 token 平均承载 H/n bit。Tokenizer 的目标可以粗略理解为：在 vocabulary size 为 V 的条件下，让一段文本需要的 token 数尽量少。这就是经典的压缩取舍。

  • 更大的 vocabulary：每个 token 信息密度更高，序列更短，但 embedding 矩阵 R^{V x d} 参数更多。
  • 更小的 vocabulary：参数更少，但序列更长，训练计算和推理延迟都会上升。

为什么不直接用单词：Zipf 定律

自然语言大致符合 Zipf 定律：第 k 高频词的频率和 1/k 成正比。这意味着大量词只出现很少次数。如果直接用“词”作为 vocabulary，长尾词会造成严重稀疏，未登录词（OOV）也无法处理。

Subword Tokenization 的思路是在字符和单词之间找一个中间粒度：常见词可以保留成完整 token，罕见词拆成更小的 subword，实在没有见过的词还能退回到字符或 byte 层面。

BPE 的 byte-level 设计

GPT 系列的 BPE tokenizer 通常工作在 byte 层面，基础 vocabulary 是 256 个 byte 值，而不是字符。英文、中文、阿拉伯语、emoji，底层都会先变成 UTF-8 byte 序列，再在 byte 序列上做合并。好处是：任何字符串都能被 tokenize，不会 OOV。

🔑
关键数据：GPT-4 的 cl100k_base vocabulary 大约 100K token，Llama 3 大约 128K token。中文、日文等 CJK 文本通常比英文更“费 token”：一个中文字符常常需要约 1.5-2 个 token，而英文 token 平均能覆盖约 3.3 个字符。

1.4 把 BPE 看成数据驱动压缩：从频率归纳符号

BPE（Byte Pair Encoding）最早由 Gage 在 1994 年作为通用数据压缩方法提出，后来由 Sennrich 等人在 2016 年引入神经机器翻译。它不是 NLP 专属技巧，而是把通用压缩策略改造成文本建模接口。

核心观察：高频模式就是冗余

在任何 byte 序列里，总会有一些相邻 byte pair 反复出现。比如英文里 "t" + "h" 很常见。BPE 的策略是把高频相邻 pair 合成一个新符号，让序列变短。每次合并都相当于发现一个重复模式，并把它抽象成更高层的 token。

算法流程

输入：文本 T，目标 vocabulary size V 初始化：Vocab = 所有 byte 值 0..255 for i in 1..(V - 256): 统计当前 token 序列中所有相邻 pair 的出现次数 选择出现次数最高的 pair (a, b) 新增 token c = a + b 把当前序列中所有不重叠的 (a, b) 替换成 c 返回：vocabulary 和 merge rules

每次合并都会让序列长度减少，减少量大致等于这个 pair 的出现次数。做完 V - 256 次合并后，vocabulary 从 256 扩到 V，同时训练语料的 token 序列明显变短。

为什么合并最高频 pair 是合理的

从压缩角度看，当前步骤里合并最高频 pair，可以最大化这一步的序列长度下降。这是贪心启发式，不保证全局最优；效果还取决于语料、pre-tokenization 和 vocab budget。

从信息论角度看，经常相邻出现的符号对通常值得词汇化，但原始频率不等于互信息。高频 pair 可能只是两个高频单符号碰到了一起。所以更准确地说，BPE 优化的是局部压缩收益，不是 PMI 或语义关联的最优估计。

一个具体例子

Corpus: "low lower lowest" 初始序列: l, o, w, _, l, o, w, e, r, _, l, o, w, e, s, t Round 1: "l"+"o" 出现 3 次 -> 合并成 token 256 ("lo") 序列: lo, w, _, lo, w, e, r, _, lo, w, e, s, t Round 2: "lo"+"w" 出现 3 次 -> 合并成 token 257 ("low") 序列: low, _, low, e, r, _, low, e, s, t Round 3: "e"+"r" 出现 2 次 -> 合并成 token 258 ("er") 序列: low, _, low, er, _, low, e, s, t

"e"+"r" 到第 3 轮才合并，是因为前两轮 "l"+"o" 和 "lo"+"w" 的频率更高。合并顺序完全由频率和 tie-breaking 规则决定，这就是 BPE 的“学习信号”。

每轮合并后必须重新统计 pair

实现 BPE 最常见的错误，是第一轮统计完 pair 后一直复用旧计数。BPE 统计的对象不是原始字符串，而是当前 token 序列。一次合并会改变相邻关系，下一轮必须重新运行 _get_stats。

阶段当前序列主要 pair 计数下一步选择
初始l o w _ l o w e r _ l o w e s t(l,o)=3, (o,w)=3, (w,e)=2按 tie-breaking 选择 (l,o)
合并 (l,o)->lo 后lo w _ lo w e r _ lo w e s t(lo,w)=3, (w,e)=2选择 (lo,w)
合并 (lo,w)->low 后low _ low e r _ low e s t(_,low)=2, (low,e)=2从新的最高频 pair 中选择

第一轮后旧的 (o,w)=3 已经不存在，因为 o 被并进了 lo。如果继续按旧计数合并，代码要么找不到这个 pair，要么得到和当前 tokenization 不一致的 vocabulary。

1.5 编程练习 1：统计相邻 token pair 频率

BPE 训练的第一步，是拿到当前 token 序列并统计所有相邻 pair 的出现次数。这个统计结果决定每一轮要合并哪个 pair。

练习 1：实现 _get_stats

任务：实现 _get_stats(ids)。输入是一串整数 token ID，输出是所有相邻 pair 的频率字典。

  • 输入 ids 是 list[int]。
  • 输出是 dict[tuple[int, int], int]。
  • 对序列 [1,2,3,4]，统计 (1,2)、(2,3)、(3,4)。
assert _get_stats([1, 2, 3, 1, 2]) == {(1, 2): 2, (2, 3): 1, (3, 1): 1} assert _get_stats([]) == {} assert _get_stats([42]) == {}
from collections import Counter def _get_stats(ids): """Count all adjacent token pairs.""" return dict(Counter(zip(ids, ids[1:])))

1.6 细看：为什么频率选择是一种压缩启发式

压缩视角

如果一个 pair 出现了 f 次，把它合成一个新 token，理论上可以让当前序列少 f 个位置。选择最高频 pair，就是在当前这一步拿到最大的局部压缩收益。

和 Huffman Coding 的区别

Huffman Coding 给已有符号分配不同长度的编码；BPE 则是在序列里归纳新的符号。Huffman 关心码长，BPE 关心把哪些相邻片段变成 token。两者都利用频率，但解决的问题不同。

频率不等于互信息

一个 pair 高频，并不说明它一定有强语义关系。BPE 的目标是让序列变短，不是发现语言学意义上的词。工程上这反而是优点：它简单、稳定、可复现，也容易和模型文件一起发布。

1.7 编程练习 2：实现 pair 替换

选出要合并的 pair 后，需要在当前序列里做不重叠替换。这一步非常关键：如果允许重叠，BPE 的训练和编码就会不一致。

练习 2：实现 _merge

任务：实现 _merge(ids, pair, idx)，把序列中的目标 pair 替换成新 token ID。

assert _merge([1, 2, 1, 2, 3], (1, 2), 99) == [99, 99, 3] assert _merge([1, 1, 1], (1, 1), 99) == [99, 1]
def _merge(ids, pair, idx): out = [] i = 0 while i < len(ids): if i < len(ids) - 1 and (ids[i], ids[i + 1]) == pair: out.append(idx) i += 2 else: out.append(ids[i]) i += 1 return out

1.8 BPE 训练循环：从 byte 到 vocabulary

训练 BPE 时，先把文本转成 UTF-8 bytes，再把每个 byte 当作初始 token ID。之后重复三步：统计当前序列的 pair，选择最高频 pair，合并成新 token。

def train(text, vocab_size): ids = list(text.encode("utf-8")) merges = {} next_id = 256 for _ in range(vocab_size - 256): stats = _get_stats(ids) if not stats: break pair = max(stats, key=stats.get) ids = _merge(ids, pair, next_id) merges[pair] = next_id next_id += 1 return merges

复杂度上，最简单实现每一轮都要重新扫描当前序列。如果目标 vocab 很大、语料很大，就需要增量统计和更复杂的数据结构。但教学实现保留全量重算，更容易看清循环不变量。

1.9 编程练习 3：实现 BPE 训练函数

这一练习把 _get_stats 和 _merge 串起来。你需要返回 merge table，并保证每一轮都基于当前 token 序列重新统计。

练习 3：BPE Training Function

  • 输入文本先用 UTF-8 编成 bytes。
  • 新 token ID 从 256 开始。
  • 每轮合并最高频 pair；没有 pair 时提前停止。
  • 保存 pair -> new_id，供 encode 使用。

1.10 编码和解码：从规则到完整 tokenizer

Decoding：简单且完全可逆

解码时，只需要把每个 token ID 展开回它对应的 byte 序列，再把 bytes 按 UTF-8 解回字符串。只要 vocabulary 保存完整，decode 不需要搜索，也不需要模型。

Encoding：按 merge rank 做确定性贪心合并

编码不是重新统计输入文本里的频率，而是使用训练阶段学到的 merge rules。每次选择当前序列中 rank 最靠前的可合并 pair，直到不能再合并。这样同一个 tokenizer 对同一段文本会得到确定结果。

重点：训练时按频率学规则；编码时按规则的优先级应用规则。不要把这两件事混在一起。

1.11 编程练习 4-5：实现完整 encode/decode

练习 4：实现 decode

把 token ID 序列展开成 bytes，再用 UTF-8 解码。遇到非法 byte 序列时，可以用 errors="replace" 保证函数不崩。

def decode(ids, vocab): tokens = b"".join(vocab[i] for i in ids) return tokens.decode("utf-8", errors="replace")

练习 5：实现 encode

把文本转成 byte ID，然后反复查当前序列里哪个 pair 在 merge table 里 rank 最靠前，合并它。

def encode(text, merges): ids = list(text.encode("utf-8")) while len(ids) >= 2: stats = _get_stats(ids) pair = min(stats, key=lambda p: merges.get(p, float("inf"))) if pair not in merges: break ids = _merge(ids, pair, merges[pair]) return ids

1.12A 怎么评估一个 tokenizer

核心维度

  • 可逆性：decode(encode(x)) == x 是否成立。
  • token 成本：平均 token 数、P95 token 数、不同语言/领域的 token per character。
  • 参数成本：vocab size 对 embedding 矩阵大小的影响。
  • 公平性：中文、代码、emoji、少数语言是否被明显惩罚。
  • 接口稳定性：special token、chat role token、tool-call token 是否有清晰版本合约。

一个小实验

准备英文、中文、代码、emoji 混合文本，分别用不同 vocab size 训练 tokenizer，记录 round-trip 是否成功、平均 token 数、P95 token 数和 embedding 参数量。不要只看一个全局平均值，必须按语言和领域拆开看。

1.12B Tokenizer 是模型接口合约

1. Special token 必须语义稳定

BOS、EOS、PAD、chat role token、tool-call token 都不是普通字符串。训练、推理、评估和发布必须对它们有同一套解释。

2. Chat template 决定 SFT 的监督边界

同样一段对话，template 不同，assistant mask 和 loss 位置就会不同。SFT 训练时必须明确哪些 token 参与 loss，哪些只是上下文。

3. 改 vocabulary 等于改模型 shape

vocab size 变化会改变 embedding 和 LM head 的形状。如果 checkpoint 和 tokenizer 不匹配，模型即使能加载，也会在语义上错位。

4. Tokenizer 版本必须跟模型一起发布

模型权重、tokenizer 文件、special token、chat template 和服务参数应该作为一个 release package 管理。

1.12C Chat Template 与多模态接口：一条消息，一份 token 合约

真实系统里，一条消息不只是纯文本。它可能包含 role、tool call、image placeholder、audio segment、引用块或结构化输出约束。Tokenizer 和 template 必须一起定义这些内容怎样进入 token 序列。

  • 训练和推理要一致：训练时看到的 role token、分隔符、工具调用格式，推理时也要一致。
  • 多模态占位要稳定：图像、音频或视频特征如何映射到 token/向量序列，要和模型配置一起发布。
  • 工具调用要可验证：工具名、参数、返回值边界最好有 schema 或明确 delimiter。

1.12D 多语言 tokenizer 训练：从语料比例到 token 成本

1. 语料比例会改变 merge rules

如果训练语料主要是英文，高频 merge 会优先服务英文。中文、代码或少数语言可能被拆得更碎，推理成本更高。

2. 用 fertility 衡量不同语言的 token 负担

可以用 tokens_per_character 或 tokens_per_word 衡量 token 负担。报告 tokenizer 质量时，不要只报全局平均，要按语言、领域和长度分组。

3. Tokenizer 决策会传到训练和推理

token 数变多，会影响训练 token budget、上下文长度、KV Cache 显存、延迟和计费。Tokenizer 是一个底层工程决策，不是前处理细节。

1.13 本章小结：你实现了一个完整 BPE Tokenizer

  • 文本先变成 UTF-8 bytes，再进入 byte-level BPE。
  • BPE 训练反复统计当前序列的相邻 pair，并合并最高频 pair。
  • 每轮合并后必须重新统计 pair，不能复用旧计数。
  • 编码使用训练好的 merge rules，按 rank 贪心应用；解码只需要查表展开 bytes。
  • Tokenizer 会影响模型 shape、训练预算、KV Cache 成本、chat template 和发布合约。
