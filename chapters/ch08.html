8.1 自回归生成原理

我们已经构建了一个完整的 GPT 模型——6 层 Transformer、12 头注意力、768 维隐藏层、约 1.24 亿参数。但模型本身只是一个"评分器"：给定一个 token 序列，它预测下一个 token 的概率分布。如何将概率转化为实际的文本输出，这就是文本生成（Text Generation）要解决的问题。

大语言模型的生成过程是自回归（Autoregressive）的：

每一步，模型看到已经生成的 token ，输出下一个 token 的概率分布 ，然后从该分布中采样或取最大值得到 。这个过程重复，直到生成结束 token 或达到最大长度。

生成过程分为两个阶段：

  • Prefilling（预填充）：将用户输入的 prompt 一次性送入模型，计算所有层的 K/V 缓存。这是计算密集阶段——大量矩阵乘法并行执行。首 token 延迟 = Prefilling 时间。
  • Decoding（解码）：逐 token 生成，每次只输入一个 token，利用 KV Cache 增量计算。这是带宽密集阶段——每次只生成一个 token，GPU 利用率低（~3-5%）。后续吞吐 = 每秒生成的 token 数。
🔑
重要概念：首 token 延迟（Time to First Token, TTFT）由 Prefilling 主导，用户感知为"等待第一个字出现的时间"；后续吞吐（Tokens Per Second, TPS）由 Decoding 主导，用户感知为"打字速度"。两者是独立的优化目标。ChatGPT 的 TTFT 约 200-500ms，TPS 约 30-60 tok/s。

本章的核心是实现一个生成器（Generator）——一个将模型输出 logits 转化为文本 token 序列的模块。我们将实现 4 种采样策略，然后深入探讨生成质量评估、上下文窗口、推测解码（Speculative Decoding）和 DeepSeek 的 Multi-Token Prediction（MTP）。

8.2 动手：Greedy Decoding

贪心解码（Greedy Decoding）是最简单的策略——每一步选择概率最高的 token：

实现非常直接——对 logits 取 argmax：

def generate_greedy(model, input_ids, max_new_tokens=100): """贪心解码：每一步选概率最高的 token。""" model.eval() device = next(model.parameters()).device input_ids = input_ids.to(device) seq_len = input_ids.size(1) for step in range(max_new_tokens): with torch.no_grad(): logits = model(input_ids) # (1, seq_len, vocab_size) next_logits = logits[:, -1, :] # 最后一个位置 next_token = next_logits.argmax(dim=-1, keepdim=True) # (1, 1) input_ids = torch.cat([input_ids, next_token], dim=-1) seq_len += 1 if next_token.item() == model.tokenizer.eos_token_id: break return input_ids

严重问题：贪心解码有一个极其隐蔽的陷阱——死循环（Repetition Loop）。模型一旦生成了一个高频 token（如"的"或"the"），后续 argmax 很可能继续选择同一个 token，导致无穷无尽的重复输出。

# 示例输入 prompt = "深度学习是人工智能的一个重要分支，它" input_ids = tokenizer.encode(prompt, return_tensors='pt') output = generate_greedy(model, input_ids, max_new_tokens=50) print(tokenizer.decode(output[0])) # 典型输出（可能有重复）： # "深度学习是人工智能的一个重要分支，它的发展非常迅速，它的发展非常迅速，它的发展非常迅速..."
⚠️
argmax 的盲区：贪心解码永远选择"当前最优"，但语言中"次优"的选择往往更自然。例如"我吃了一个____"中"苹果"的概率略低于"水果"，但前者更合理。更关键的是，argmax 没有回溯机制——一步选错，后续全部受影响。这就是曝光偏差（Exposure Bias）：训练时模型看到的是真实文本，推理时看到的是自己的输出。

解决方案是引入随机性——从概率分布中采样，而不是取最大值。接下来我们会看到三种越来越精细的采样策略。

8.3 动手：Temperature Sampling

温度采样（Temperature Sampling）在 argmax 和完全均匀采样之间提供了一个连续控制旋钮。核心是温度参数 ：

其中 是 logits， 是温度：

  • ：softmax 退化为 argmax（确定性的，等价于贪心）
  • ：原始概率分布，不做任何调整
  • ：分布更均匀，更多随机性（创意更多但可能跑题）
  • ：分布更尖锐，聚焦于高概率 token（更保守）
def generate_temperature(model, input_ids, max_new_tokens=100, temperature=1.0): """温度采样：用 temperature 控制分布的尖锐程度。""" model.eval() device = next(model.parameters()).device input_ids = input_ids.to(device) for step in range(max_new_tokens): with torch.no_grad(): logits = model(input_ids) next_logits = logits[:, -1, :] # (1, vocab_size) # 应用温度 scaled_logits = next_logits / temperature # 从 softmax 分布中采样 probs = torch.softmax(scaled_logits, dim=-1) next_token = torch.multinomial(probs, num_samples=1) # (1, 1) input_ids = torch.cat([input_ids, next_token], dim=-1) if next_token.item() == model.tokenizer.eos_token_id: break return input_ids

的数值问题：当 时，除法 会导致数值溢出。实际实现中从来不会真的除以 0： 时直接用 argmax， 时才做采样。另一种常见做法是设定 作为下限。

温度参数的应用实际上是 Boltzmann 分布 的变体——在统计力学中，温度控制系统的"探索"程度。高温使系统探索更多状态（更多样的词汇），低温使系统利用已知高概率模式（更确定的输出）。这也是"温度"这个名称的来源。

💡
实际用法：ChatGPT 默认温度是 0.7 或 0.8——在确定性和创造性之间取得平衡。代码生成任务常用 （确定性更重要），创意写作任务常用 。但即使 ，低概率 token 仍有可能被采样到——这就是 Top-K 和 Top-P 要解决的问题。

8.4 动手：Top-K 采样

Top-K 采样的思路非常直观：只从概率最高的 K 个 token 中采样，忽略"长尾"的低概率 token。这可以防止模型从概率极低的、不合理的 token 中采样。

GPT-2 的默认参数是 。Hugging Face Transformers 也用 作为默认值。

def generate_topk(model, input_ids, max_new_tokens=100, temperature=1.0, k=40): """Top-K 采样：只保留概率最高的 K 个 token。""" model.eval() device = next(model.parameters()).device input_ids = input_ids.to(device) for step in range(max_new_tokens): with torch.no_grad(): logits = model(input_ids) next_logits = logits[:, -1, :] / temperature # 找到 top-k 的 logits 和 indices top_logits, top_indices = torch.topk(next_logits, k, dim=-1) # 用 top-k logits 计算 softmax 并采样 top_probs = torch.softmax(top_logits, dim=-1) # (1, k) sampled_idx = torch.multinomial(top_probs, num_samples=1) # (1, 1) # 将 sampled_idx 映射回词表 id next_token = top_indices.gather(-1, sampled_idx) # (1, 1) input_ids = torch.cat([input_ids, next_token], dim=-1) if next_token.item() == model.tokenizer.eos_token_id: break return input_ids

Top-K 的固有问题：K 是一个固定值。在某些上下文中可能只有 3 个合理的选择（K=40 引入了太多噪声），而在另一些上下文中可能有 100 个合理的延续（K=40 截断了太多可能性）。换句话说，K 无法自适应。

例如：模型预测"美国的首都是____"，此时概率分布极度尖锐——"华盛顿"的概率可能超过 90%。K=40 意味着我们仍然从 40 个候选中随机采样，这极大概率仍然选到"华盛顿"（没问题），但小概率选到"纽约"或"洛杉矶"（明显错误）。而在创意写作中，分布往往更平缓，K=40 可能过于限制创造力。

8.5 动手：Top-P (Nucleus) 采样

Top-P（Nucleus）采样由 Holtzman et al. (2020) 提出，是目前最主流的采样策略。它解决了 Top-K 的"K 固定"问题：不是固定选择 K 个 token，而是选择累积概率达到 的最小 token 集合。

具体来说：

  1. 将 token 按概率从高到低排序
  2. 从最高概率开始累加，直到累积概率达到 （通常 0.9 或 0.95）
  3. 只从这个"核心集"（nucleus）中采样

当分布尖锐时，核心集很小（如上面"华盛顿"的例子可能只有 1-3 个 token 就达到 P=0.9）；当分布平缓时，核心集自动扩大。这完全由数据驱动，无需手动调整 K。

def generate_topp(model, input_ids, max_new_tokens=100, temperature=1.0, p=0.9): """Top-P (Nucleus) 采样：选择累积概率达到 p 的最小 token 集。""" model.eval() device = next(model.parameters()).device input_ids = input_ids.to(device) for step in range(max_new_tokens): with torch.no_grad(): logits = model(input_ids) next_logits = logits[:, -1, :] / temperature probs = torch.softmax(next_logits, dim=-1) # (1, vocab_size) # 按概率降序排序 sorted_probs, sorted_indices = torch.sort(probs, descending=True, dim=-1) # 计算累积概率 cumsum = torch.cumsum(sorted_probs, dim=-1) # 找到累积概率 > p 的位置，将其后的概率置 0 mask = cumsum - sorted_probs > p # 关键：减去自身，使"第一个 >p"也被包含 sorted_probs[mask] = 0.0 # 归一化（被 mask 的 token 概率为 0） renormalized = sorted_probs / sorted_probs.sum(dim=-1, keepdim=True) # 从归一化后的分布中采样 sampled_idx = torch.multinomial(renormalized, num_samples=1) # (1, 1) next_token = sorted_indices.gather(-1, sampled_idx) # (1, 1) input_ids = torch.cat([input_ids, next_token], dim=-1) if next_token.item() == model.tokenizer.eos_token_id: break return input_ids
🔑
P 的选择： 是最常用值，兼顾多样性和质量。 更激进（核心集更大）， 更保守。Google 的 PaLM 使用 ，而 Anthropic 的 Claude 使用 。实际应用中，温度 + Top-P 通常联合使用——先用温度调整分布形状，再用 Top-P 截断尾部。

8.6 动手：采样策略对比实验

现在我们把所有策略放在一起对比。用同一个 prompt 和同一个模型，观察不同策略的输出差异：

class Generator: """统一的生成器，支持多种采样策略。""" def __init__(self, model, tokenizer): self.model = model self.tokenizer = tokenizer self.device = next(model.parameters()).device def generate(self, prompt: str, strategy: str = "top-p", max_new_tokens: int = 100, temperature: float = 0.8, k: int = 40, p: float = 0.9) -> str: """ 统一的生成接口。 strategy: "greedy" | "temperature" | "top-k" | "top-p" """ input_ids = self.tokenizer.encode( prompt, return_tensors='pt' ).to(self.device) for step in range(max_new_tokens): with torch.no_grad(): logits = self.model(input_ids) next_logits = logits[:, -1, :] # —— 策略选择 —— if strategy == "greedy": next_token = next_logits.argmax(dim=-1, keepdim=True) elif strategy == "temperature": scaled = next_logits / temperature probs = torch.softmax(scaled, dim=-1) next_token = torch.multinomial(probs, 1) elif strategy == "top-k": scaled = next_logits / temperature top_v, top_i = torch.topk(scaled, k, dim=-1) probs = torch.softmax(top_v, dim=-1) idx = torch.multinomial(probs, 1) next_token = top_i.gather(-1, idx) elif strategy == "top-p": scaled = next_logits / temperature probs = torch.softmax(scaled, dim=-1) sorted_p, sorted_i = torch.sort(probs, descending=True, dim=-1) cumsum = torch.cumsum(sorted_p, dim=-1) mask = cumsum - sorted_p > p sorted_p[mask] = 0.0 sorted_p /= sorted_p.sum(dim=-1, keepdim=True) idx = torch.multinomial(sorted_p, 1) next_token = sorted_i.gather(-1, idx) input_ids = torch.cat([input_ids, next_token], dim=-1) if next_token.item() == self.tokenizer.eos_token_id: break return self.tokenizer.decode(input_ids[0]) # 对比实验 gen = Generator(model, tokenizer) prompt = "人工智能的未来发展将" print("=== Greedy ===") print(gen.generate(prompt, "greedy")[:200]) print("\n=== Temperature (T=0.8) ===") print(gen.generate(prompt, "temperature", temperature=0.8)[:200]) print("\n=== Top-K (K=40, T=0.8) ===") print(gen.generate(prompt, "top-k", temperature=0.8, k=40)[:200]) print("\n=== Top-P (P=0.9, T=0.8) ===") print(gen.generate(prompt, "top-p", temperature=0.8, p=0.9)[:200])

典型观察：

  • Greedy：最短、最确定性、最可能陷入重复循环。输出通常是"安全但无聊"的——统计上最可能的延续往往也是最平淡的。
  • Temperature：更多样化，但 T 过高时可能跑题甚至输出乱码。每次运行结果不同。
  • Top-K：在多样性和质量间取得平衡，但偶尔出现不合理 token（因为 K 固定无法适应）。
  • Top-P：质量最高的策略——合理且多样，极少出现荒谬输出。这是当前工业界最常用的策略。

8.7 生成质量评估

如何衡量生成文本的质量？主要从以下三个维度出发：

1. Perplexity（困惑度）：衡量模型对文本的"惊讶程度"。给定真实 token 序列 ：

PPL 越低越好——模型对生成文本的预测越自信。但 PPL 与人类判断并不完全一致：重复文本的 PPL 往往很低（重复模式容易预测），但人类不会认为它是高质量输出。

def compute_perplexity(model, tokenizer, text: str) -> float: """计算文本的困惑度。""" enc = tokenizer.encode(text, return_tensors='pt').to(device) with torch.no_grad(): logits = model(enc) # (1, seq_len, vocab_size) # 计算每个位置的交叉熵损失 shift_logits = logits[:, :-1, :].contiguous() shift_labels = enc[:, 1:].contiguous() loss = F.cross_entropy( shift_logits.view(-1, shift_logits.size(-1)), shift_labels.view(-1), reduction='mean' ) return torch.exp(loss).item() text1 = "人工智能是计算机科学的一个重要分支。" text2 = "人工智能人工人工人工人工人工人工" print(f"PPL(合理文本): {compute_perplexity(model, tokenizer, text1):.2f}") print(f"PPL(重复文本): {compute_perplexity(model, tokenizer, text2):.2f}")

2. 多样性（Diversity）：使用 distinct-n 指标——生成文本中不同的 n-gram 比例：

def distinct_ngrams(text: str, n: int = 2) -> float: """计算 distinct-n 分数：不同的 n-gram 占所有 n-gram 的比例。""" tokens = text.split() if len(tokens) < n: return 0.0 ngrams = set() for i in range(len(tokens) - n + 1): ngrams.add(' '.join(tokens[i:i+n])) return len(ngrams) / (len(tokens) - n + 1) def diversity_report(text: str) -> dict: return { 'distinct-1': distinct_ngrams(text, 1), 'distinct-2': distinct_ngrams(text, 2), 'distinct-3': distinct_ngrams(text, 3), }

3. 重复率（Repetition Rate）：检测生成的文本中有多少内容是重复的：

def repetition_rate(text: str, window: int = 3) -> float: """检测文本中重复 n-gram 的比例。""" tokens = text.split() if len(tokens) < window * 2: return 0.0 seen = set() repeat_count = 0 for i in range(len(tokens) - window + 1): gram = ' '.join(tokens[i:i+window]) if gram in seen: repeat_count += 1 else: seen.add(gram) return repeat_count / (len(tokens) - window + 1)
💡
评估的"圣三一"：Perplexity 衡量模型置信度，distinct-n 衡量词汇多样性，重复率衡量局部连贯性。它们共同构成了生成质量的量化评估体系。但最终判断还是需要人类评估（Human Evaluation）——LLM-as-a-Judge（用更强模型评估弱模型的输出）是目前工业界常用的自动评估替代方案。

8.8 上下文窗口

上下文窗口（Context Window）是模型在一次生成中能"看到"的最大 token 数量。为什么越长越好？因为：

  • 长程依赖：小说的伏笔需要数百页后揭晓；代码中一个函数可能在文件开头定义、末尾调用。更长的上下文意味着模型能更好地处理这些跨段落关系。
  • 少样本学习：在 prompt 中放入更多示例（in-context examples），模型表现更好。上下文越长，可容纳的示例越多。
  • 多轮对话：ChatGPT 等应用需要在上下文窗口中保持完整的对话历史。128K 窗口约可容纳 10 万字的对话历史（约一本 300 页的书）。

但为什么有上限？主要有两个瓶颈：

计算复杂度：标准 Self-Attention 的计算量是 ，其中 是序列长度。当 从 4K 增加到 128K，计算量增加 倍——从几 GFLOPs 到几 TFLOPs。这也是 FlashAttention 和 MLA（Multi-head Latent Attention）等优化存在的根本原因。

Attention Sink（注意力沉没）：近期的研究发现了一个有趣的现象——在长序列中，模型会将大量的注意力权重分配给初始 token（通常是 <BOS> 或句子开头的 token），即使这些 token 与当前生成的内容无关。这种现象被称为 Attention Sink（Xiao et al., 2024）。

⚠️
Attention Sink 的成因：如果在训练时序列长度有限，模型会产生"残差注意力偏移"——初始 token 吸收了多余的注意力分数。当推理时序列变长，这种偏移被放大，导致模型将大量注意力浪费在无关的早期 token 上。解决方案是添加一个专门的"窗口基线"（如 StreamingLLM）或在训练中对早期 token 施加 attention 惩罚。

目前主流模型的上下文窗口：GPT-4 128K、Claude 3 200K、Gemini 1.5 Pro 1M、DeepSeek-V2 128K、DeepSeek-V3 128K。1M 上下文意味着模型可以一次处理整本《三体》三部曲（约 90 万字）。

8.9 DeepSeek MTP — 多 Token 预测

多 Token 预测（Multi-Token Prediction, MTP）是 DeepSeek-V3 的一项重要技术创新。传统语言模型每次只预测一个未来的 token，而 MTP 让模型一次预测多个未来 token（DeepSeek-V3 使用 ，即预测 2 个 token，而 DeepSeek-V4 扩展到更大的跨度）。

训练时的 MTP：模型有 个独立的输出头（MTP Head），每个头预测不同偏移的未来 token：

其中 是第 个 MTP Head。每个头有自己的参数，但所有头共享底层 Transformer 表示。

class MTPHead(nn.Module): """MTP 预测头：预测向后偏移 d 个位置的 token。""" def __init__(self, d_model: int, vocab_size: int, d: int): super().__init__() self.d = d # 每个 MTP Head 是一个小 MLP + LayerNorm self.norm = nn.LayerNorm(d_model) self.linear = nn.Linear(d_model, vocab_size, bias=False) def forward(self, hidden_states: torch.Tensor) -> torch.Tensor: """ hidden_states: (batch, seq_len, d_model) returns: logits (batch, seq_len, vocab_size) """ return self.linear(self.norm(hidden_states)) class MultiTokenPredictionModel(nn.Module): """带 MTP 的 GPT 模型。""" def __init__(self, base_model, num_mtp_heads: int = 2): super().__init__() self.base = base_model # 底层 Transformer self.main_head = nn.Linear(base_model.config.d_model, base_model.config.vocab_size) self.mtp_heads = nn.ModuleList([ MTPHead(base_model.config.d_model, base_model.config.vocab_size, d=d+1) for d in range(num_mtp_heads) ]) def forward(self, input_ids: torch.Tensor): """返回主头 + 所有 MTP 头的 logits。""" hidden = self.base(input_ids) # (batch, seq_len, d_model) main_logits = self.main_head(hidden) mtp_logits = [head(hidden) for head in self.mtp_heads] return main_logits, mtp_logits

双重增益：

  • 训练增益：MTP 为模型提供了更密集的监督信号。每个 token 位置不仅要预测下一个 token，还要预测下下个 token。这迫使模型学习更长距离的依赖关系。DeepSeek-V3 的实验表明，MTP 使训练效率提升约 15-20%。
  • 推理增益：MTP Head 在推理时可以用作推测解码的草稿模型（下一节详细讲解）。由于 MTP Head 的参数远小于完整 Transformer，它们可以快速生成候选 token，然后由主模型一次性验证。

DeepSeek-V3 选择 （额外预测 1 个 token），因为实验发现更大的 带来的边际收益递减。但 DeepSeek-V4 将 MTP 进一步扩展，与推测解码深度集成。

8.10 动手：推测解码

推测解码（Speculative Decoding）由 Leviathan et al. (2023) 和 Chen et al. (2023) 独立提出。它的核心思想非常巧妙：用一个小模型（草稿模型）快速生成候选 tokens，然后由大模型（目标模型）并行验证这些候选，在不改变输出分布的前提下实现 ~1.8x-3x 的加速。

推测解码的步骤如下：

  1. 草稿：用轻量草稿模型（如 100M 参数）自回归生成 个候选 token（通常 或 5）
  2. 验证：将候选 tokens + 前缀一并送入目标模型（如 7B 参数），做一次完整的 forward 计算，获得所有位置的 logits
  3. 拒绝采样：逐位置检查目标模型和草稿模型的概率分布——如果两者分歧过大，拒绝该候选 token，从修正分布中重新采样
  4. 接受：被接受的 token 和重新采样的 token 一起加入输出序列
def speculative_decoding( target_model: nn.Module, # 大模型（精确但慢） draft_model: nn.Module, # 小模型（快速但可能不准） tokenizer, # 共享分词器 input_ids: torch.Tensor, # 前缀 gamma: int = 4, # 草稿步数 max_new_tokens: int = 100, ) -> torch.Tensor: """ 推测解码：小模型草稿 → 大模型验证 → 拒绝采样修正。 """ device = next(target_model.parameters()).device prefix = input_ids.to(device) n_orig = prefix.size(1) while prefix.size(1) - n_orig < max_new_tokens: # —— 步骤 1: 草稿阶段 —— draft_ids = prefix.clone() with torch.no_grad(): for _ in range(gamma): logits = draft_model(draft_ids) next_logit = logits[:, -1, :] # 草稿模型用目标策略（如 top-p）采样 next_token = sample_top_p(next_logit, p=0.9) draft_ids = torch.cat([draft_ids, next_token], dim=-1) draft_tokens = draft_ids[:, -gamma:] # (1, gamma) # —— 步骤 2: 验证阶段 —— with torch.no_grad(): target_logits = target_model(draft_ids) draft_logits = draft_model(draft_ids) # 获取目标模型和草稿模型在第 0..gamma-1 步的概率 target_probs = torch.softmax( target_logits[:, -(gamma+1):-1, :], dim=-1 ) # (1, gamma, vocab_size) draft_probs = torch.softmax( draft_logits[:, -(gamma+1):-1, :], dim=-1 ) # (1, gamma, vocab_size) # —— 步骤 3: 拒绝采样 —— accepted = [] n_accepted = 0 rng = torch.quasirandom.SobolEngine(1, scramble=True) for t in range(gamma): token = draft_tokens[0, t].item() q = draft_probs[0, t, token].item() # 草稿模型对 token 的概率 p = target_probs[0, t, token].item() # 目标模型对 token 的概率 # 接受准则: 以 min(1, p/q) 的概率接受 if p >= q: accepted.append(token) n_accepted += 1 else: r = rng.draw(1).item() if r < p / q: accepted.append(token) n_accepted += 1 else: # 从修正分布 (p - q)_+ 中重新采样 residual = (target_probs[0, t] - draft_probs[0, t]).clamp(min=0) residual /= residual.sum() # 归一化 corrected = torch.multinomial(residual, 1).item() accepted.append(corrected) # 接受后停止（剩余候选被丢弃） break if n_accepted == gamma: # 所有候选被接受，额外从目标模型采样一个 token last_logit = target_logits[:, -1, :] extra_token = sample_top_p(last_logit, p=0.9) accepted.append(extra_token.item()) # —— 步骤 4: 添加接受的 token —— prefix = torch.cat([ prefix, torch.tensor([accepted], device=device) ], dim=-1) return prefix

为什么这能保证输出分布不变？拒绝采样算法的精妙之处在于：虽然我们用了草稿模型的近似分布，但通过精确的接受/拒绝准则和修正采样，最终的输出分布在数学上等价于直接用目标模型自回归生成。这意味着推测解码是一种无损加速方法。

8.11 本章总结

你已经实现了：

  • ✅ 自回归生成的基本原理——Prefilling + Decoding 两阶段
  • ✅ Greedy Decoding——简单但容易陷入重复循环
  • ✅ Temperature Sampling——用温度控制分布的尖锐程度
  • ✅ Top-K 采样——截断到 K 个最高概率 token
  • ✅ Top-P (Nucleus) 采样——累积概率阈值，动态调整候选集大小
  • ✅ 统一的 Generator 类——四种策略一键切换
  • ✅ 生成质量评估体系——PPL / distinct-n / 重复率
  • ✅ 上下文窗口的概念和限制——计算复杂度与 Attention Sink
  • ✅ DeepSeek MTP——一次预测多个未来 token，训练/推理双重增益
  • ✅ 推测解码——小模型草稿 + 大模型验证，无损加速 ~1.8x

文本生成是 LLM 与用户交互的"最后一步"。采样策略的选择直接决定了用户对模型输出的感知——太确定则无聊且重复，太随机则跑题且荒谬。在工业界，Top-P（P=0.9）+ Temperature（T=0.7-0.8）被绝大多数模型采纳为默认配置。
